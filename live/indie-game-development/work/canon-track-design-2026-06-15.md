# Дизайн творческого трека «Канон» + план реализации (структура)

**Что это:** записанная проработка (продукт в `work/`, НЕ состояние). Фиксирует, как творческий/канон-трек направления `indie-game-development` вплетается в Direction OS, и служит планом «форм-шагов». Ничего из `live/**`-состояния (TREE/NOW/CHARTER/knowledge) этот документ не меняет.
**Дата:** 2026-06-15. **Направление:** indie-game-development. **Статус:** дизайн УТВЕРЖДЁН владельцем; структура ещё НЕ сформирована (этот документ — первый реальный артефакт).

---

## 0. Решение в одну строку
Канон (мир/лор/сюжет/механики/визуал) ведётся в УЖЕ существующей ветке дерева **`g-d3a8`** как обычные converge/`canon-forge`-сессии — **без правки ядра OS**, параллельно инженерной ставке **`g-9c41`**, **не становясь вторым активным bet** и **не отдельным направлением**. Контент канона живёт в **отдельном git-репозитории**. Движок остаётся приоритетом.

Почему так (а не иначе): враждебный аудит (7 агентов) отверг правку гейта G1 «на месте» (модели «ставка-на-линию» / «трек-тег») — они ломают G1/G2/G5/G8 и не влезают в бюджет ядра (~1494/1500 слов). «Канон как своё направление» проходит механически, но переворачивает собственное правило владельца «канон ≠ второй bet» — оставлено только как запасной вариант, если позже понадобится постоянный выделенный bet-слот. Ключевой сдвиг, снявший затык: **«активная ставка» ≠ «параллельный прогресс»** — канон полноценно идёт параллельно как converge-сессии, не занимая bet-слот.

## 1. Как это вплетено в workflow (два уровня)

**Уровень 1 — дерево OS (крупно).** Канон = ветка **`g-d3a8`**, сосед ветки «движок» (`g-9c41`). Дерево держит только ЦЕЛИ. Движок и канон — соседние ветки → не мешают.

**Уровень 2 — внутри репо канона (детально).** Внутри ветки живёт **граф вопросов** (DAG): каждая заметка-вопрос несёт `depends_on`. Этот граф лежит в репо канона, НЕ в `NOW.md` движка — поэтому ставка движка каноном вообще не затрагивается.

**Почему не мешает разработке:** разные репозитории (движок → GasCoopGame; канон → отдельный репо), разные чаты, разные плеи. Встречаются только в общем дереве, которое лишь *записывает прогресс*. `NOW.md` (ставка движка) канон не трогает.

---

## 2. Свой процесс внутри ноды = локальный плей `canon-forge`
Канон нельзя гнать плоским workflow — нужен свой рецепт (включая образы). Механизм OS для этого — **локальный плей** (`live/indie-game-development/plays/canon-forge.md`, вызов `play: local/canon-forge`; разрешено KERNEL §6 + EXTENDING.md, ≤600 слов, ядро не меняется).

**Рецепт `canon-forge` — работа над ОДНИМ вопросом:**
1. Берём вопрос из `canon-status` (тот, что можно сейчас).
2. **Рамка** — что решаем + что уже заморожено у «родителей» (не противоречить).
3. **3 варианта** — AI предлагает разные направления → владелец выбирает/правит (со-творчество; лор НЕ выдумывается агентом).
4. **Черновик текста.**
5. **Гейты** ремесла (см. §4) — не прошёл → назад к 4.
6. **Образы — ЕСЛИ у вопроса есть смысл их иметь (шаг необязательный):** image-бриф → подставляем стиль-якорь из «дизайн-карты» → генерим СЕТ в ChatGPT (GPT Image) → владелец выбирает → кладём в репо канона рядом с текстом.
7. **Заморозка** → официальная карточка (текст + выбранные образы); граф подрастает новыми под-вопросами.

## 3. Процедура состояния `canon-status` (render-only)
По запросу печатает текущее состояние графа канона (ничего не меняет):
- ✅ что заморожено; ▶️ что можно взять сейчас (все зависимости готовы); ⏳ что заблокировано (и чем); 🔀 что можно вести параллельно (несвязанные вопросы); 📦 какой кластер весь зелёный → можно собирать главу-артбук.

Это и есть «один чат показывает следующий шаг». Реализация — отдельная локальная процедура или расширение `digest`.

## 4. Гейты ремесла (проверенные, реальные авторы — встроены как `knowledge/`-чек-листы)
- **Нарратив — «Ворота обещания»:** Сандерсон (обещание→прогресс→расплата), Чехов (ружьё), Хичкок (саспенс>сюрприз), Кинг (террор>хоррор>отвращение — держит тон ТРИЛЛЕРА, не сплаттера), Трубай (want vs need).
- **Интрига — «Ворота дна айсберга»:** Абрамс (mystery-box) + провал LOST, Хемингуэй (айсберг), Нокс/Ван Дайн (честная игра), Стернберг (любопытство/саспенс/сюрприз). *Реframe (на подпись владельцу): сверхъестественное/вещество РАЗРЕШЕНО, но должно быть УСТАНОВЛЕНО до развязки (Чехов на лоре).*
- **Мир — «Ворота несущего канона»:** Харрисон (не строить лишнего), Сандерсон 1/2/3 (правило-до-развязки · ограничения>силы · расширяй-прежде-чем-добавлять), Толкин (вторичная вера). «Спина прежде ветки»: мир прежде газа.
- **Механика — «Ворота кооп-решения»:** Мейер (интересные решения), Шелл (линзы), Sirlin (глубина≠сложность), MDA (эмерджентность), кооп-взаимозависимость.
- **Арт — «Ворота стиль-библии»:** Гёрни (палитра/нотан), Эгглстон (color-script), Блок (контраст), силуэт-тест, язык камеры.

Полные чек-листы лягут в `knowledge/` с реальным читателем.

## 5. Хранилище и визуал (один источник, два вида)
- **Единственная правда** — атомарные `.md`+YAML в репо канона (один файл = одна мысль; поля: `id, type, area, status, depends_on, images[], gate{}`). Образы — файлы под `/assets`, имена в нижнем регистре через дефис, относительные ссылки.
- **Вид для владельца** — красивые MOC-индексы (текст + образы инлайн).
- **Вид для AI** — сгенерированный топосортированный бандл (ChatGPT/Claude не противоречат друг другу).
- **«Дизайн-карта» (style bible)** — записанный документ с ОБЩИМИ правилами визуала (палитра, свет, настроение, материалы, силуэт). На конкретный вопрос чат ВЫВОДИТ из неё конкретный промпт. Образы консистентны за счёт стиль-якоря + опорных кадров (reference plates) + одной чат-сессии (НЕ через seed).
- **Выход** — вложенный граф: завершённый кластер собирается в **главу-артбук** (текст+образы, авто-кросс-ссылки). Артбук = ПРОЕКЦИЯ, не хранилище. Подача: авто-сайт для основного объёма (Obsidian reading-view / Quartz) + **AFFiNE** (локально, бесплатно, Edgeless-холст) для «геройских» разворотов вручную. Anytype отклонён (облачный тариф урезан 2026-06-11; не верстальщик).

## 6. ПОРЯДОК: сеттинг раньше стиля (важная поправка владельца 2026-06-15)
Визуальный стиль ВЫВОДИТСЯ из мира, а не наоборот. Поэтому «Визуальный стиль» — **не корень графа**, а зависит от первого слоя сеттинга.
**Стартовый порядок узлов:** `Creative North Star` (опыт + жанр-обещание + тон) → база мира (происхождение/город/кто мы) → `Визуальный стиль` (дизайн-карта, выводится из базы) → дальше каждый узел получает консистентные образы.
Ранние «пощупать глазами» картинки допустимы как ЧЕРНОВЫЕ mood-кадры (скретч, не канон); залоченный стиль — после базы мира.
Зерно, уже бывшее в `g-d3a8`: фарм-корпорация → спуск в подземный город → неизвестное внизу.

## 7. ФОРМ-ШАГИ (план реализации — каждый = своя задача/сессия)
1. **Записать дизайн** — ЭТОТ документ (сделано).
2. **Завести репо канона** + скелет: `/questions`, `/canon`, `/maps` (MOC + StyleBible/дизайн-карта), `/assets`, `/board.canvas`, `/build` (AI-бандл), шаблон карточки, `.gitignore`. (Действие владельца + скелет от меня.)
3. **Написать плей `canon-forge`** (≤600 слов) + процедуру **`canon-status`**.
4. **Разложить `g-d3a8` на под-узлы** (North Star → база мира → стиль → персонажи/фракции/механики) — правка `TREE.md`, через сессию map/shape с явным одобрением владельца (G9). **НЕ делать без отдельного «да».**
5. **Мелочи:** чек-листы гейтов N/M/мир/механика/арт → `knowledge/` (с реальным читателем); убрать устаревший «≤10% часов» гейт из `map_order` (висит на отменённой вехе «первый клип») — owner-approved правка дерева.

**После форм-шагов:** каждая канон-тема = свой свежий чат, одна задача, берётся из `canon-status`, ведётся по `canon-forge`, пишется обратно. Разные темы — в разных чатах, параллельно.

## 8. Жёсткие ограничения (свежий чат обязан соблюсти)
- **Ядро OS и существующие гейты НЕ менять.** Эта затея = ноль правок ядра. `live/**`-состояние меняется только через RESULT/одобрение.
- Движок `g-9c41` остаётся ЕДИНСТВЕННОЙ активной ставкой; канон вторым активным bet НЕ становится.
- **Со-творчество:** не выдумывать лор/механики/ассеты владельца — AI даёт варианты, решает владелец. Generic-вывод в одиночку отвергается.
- Маркетинг-политика: AI-картинки — для ВНУТРЕННЕГО понимания/концептов можно; в ПУБЛИЧНЫЙ маркетинг AI-генерация картинок/видео/голоса нельзя.
- Один чат = одна задача. Этот чат был ДИЗАЙНОМ; работа над каноном идёт в отдельных чатах после форм-шагов.
