# Дизайн маркетингового трека «marketing-forge» (ПРОЦЕСС)

## 0. Решение в одну строку
Маркетинг ведётся как обычные `marketing-forge`-сессии — без правки ядра OS, параллельно ставке `g-9c41`, не становясь вторым активным bet (G1). Аналог канон-трека: своя мини-инструкция (`marketing-forge`) + экран «что дальше» (`marketing-status`) + свой граф вопросов в `work/marketing/`. Движок — приоритет; маркетинг заполняет параллельную ёмкость неравномерно.

## 1. Как вплетено (как в каноне)
- Дерево OS: маркетинг = ветка `g-2f8c`, сосед движка `g-9c41` и канона `g-d3a8`. Дерево держит только ЦЕЛИ (G2).
- Внутри трека: граф маркетинговых вопросов (DAG) живёт в `work/marketing/`, НЕ в `NOW.md`. Markdown-first, без спец-тулинга.

## 2. Рецепт `marketing-forge` (одна карточка за раз)
Полный плей: `plays/marketing-forge.md`. Суть: **Рамка → Ресёрч + вводные владельца → Проверка кросс-трек → Выбор (владелец) → Драфт → Гейт голоса (владелец) → Заморозка (владелец)**. Карточки двух видов: DECISION (решение) и ARTIFACT (что сделать / где / с каким сообщением).

**Ключевая разница с каноном:** там гейт — крафт-чек-листы; здесь гейт — **РЕСЁРЧ** (у владельца нет маркетинговой экспертизы, и он не обязан её выдавать — каждое решение опирается на ресёрч + его вводные, а не на его знание соцсетей). Любые цифры/цели ВЫВОДЯТСЯ внутри процесса с обоснованием — не задаются заранее владельцем и не выдумываются AI.

«Один чат = одна карточка»: зашёл, когда есть время → взял карточку → обсудил → закрыл. Неравномерно. Экран `marketing-status` показывает, что можно брать сейчас.

## 3. Кросс-трек (как просил владелец)
Карточке нужно из мира/лора → запрос в **канон** (`cross_track`; владелец переносит в канон-сессию; маркетинг канон-репо не пишет). Позже нужен показываемый момент или приоритет → **рекомендация в разработку** (только то, что ставка уже гарантирует; без добавления объёма, G1/G2; владелец переносит). Это руль, которым нода со временем может направлять разработку — но всегда через решение владельца.

## 4. Делегирование (опциональный клапан, не ставка)
По умолчанию владелец делает всё сам по чётким инструкциям карточки. Делегируемая логистика (расписание, кросс-пост, админ рассылки/Discord, сбор метрик) может уйти жене/брату, ЕСЛИ владелец перегружен. Голос и решения — за владельцем. На помощников не ставим.

## 5. Жёсткие ограничения
- Ядро OS и гейты не менять; `live/**` — только через RESULT/одобрение.
- `g-9c41` — единственная активная ставка; маркетинг вторым bet не становится (G1); `NOW.md` не трогает.
- Со-творчество: голос / хук / лор не выдумываем — ресёрч даёт варианты, решает владелец; generic в одиночку отвергается.
- Публичный маркетинг: AI-генерация картинок / видео / голоса запрещена (устав).
- Один чат = одна карточка.

## 6. Шейп-валидность (G6) — на уровне ПРОЦЕССА
**Cut list (реальные отсечения):**
- ОТСЕЧЕНО: запечённый go-to-market план / стратегия — выводится ВНУТРИ процесса, не здесь.
- ОТСЕЧЕНО: пред-заданные «этапы» — путь выводится внутри (внешние даты — это ограничения, не структура процесса).
- ОТСЕЧЕНО: цифры / цели, написанные владельцем заранее или выдуманные AI — выводятся по карточке с обоснованием.
- ОТСЕЧЕНО: каденс по неделям — ритм по «есть что сказать», неравномерный.

**Линзы устава:** commercial/traction + audience-workflow — СЛУЖИТ (процесс = сама машина аудитории; пред-мортемы #6/#7). Core depth / co-op-first / technical / scope — не нужно как работа процесса (владеют другие ноды; объём держится цельностью процесса).

**Самое рискованное допущение:** процесс выдаёт не-generic, в голосе владельца, и владелец РЕАЛЬНО может его вести без экспертизы. ТЕСТ = первый прогон (карточка `q-foundation`) с образцом через гейт голоса и вердиктом владельца.

## 7. Стартовый граф (минимальный — процесс растит сам)
- `q-foundation` ▶️ — первая карточка: внутри процесса (ресёрч + вводные владельца) определяем нашу ситуацию, цели и путь; порождает остальные карточки. (Тут же — первый тест процесса.)
- `q-studio-identity` ▶️ — имя студии + один ник (независимая).
- `q-positioning-hook` ⏳ — хук / позиционирование; BLOCKED → запрос в канон (пример кросс-трека).
Дальнейшие карточки (каналы, артефакты, захват, деньги, каденс) РОЖДАЕТ `q-foundation`, когда процесс выведет путь — не заводим заранее.
