Контроль-слой — как этим пользоваться

В любом новом чате работают фразы: «что можем делать» — покажет этот борд свежим; «делай рефакторинг» — запустит ночную линию по бэклогу; «дневное ревью» — сверка направления с записью находок; «давай пройдёмся по развилкам» — решения из «Ждёт тебя» по одному, с вариантами и рекомендациями; «обнови борд» — принудительная перегенерация панели.

Словарь борда

лег — одна порция инженерной работы, проходящая полный контроль качества (план → стройка → независимая проверка); примерно день. · dev / main — рабочая ветка кода / основная «чистовая» ветка; в main работа попадает только через твой гейт. · мерж-гейт — твоё «да» на вливание работы в основную ветку. · PLAN / BUILD — две отдельные сессии одной работы: сначала план (ты одобряешь и он замораживается), потом стройка строго по нему. · G5 / опровержение — независимая свежая проверка, которая ПЫТАЕТСЯ доказать, что сделано неправильно; не смогла — работа принята. · контрольная сумма (чексум) — число-«отпечаток» состояния симуляции; совпадает у всех игроков — значит игра синхронна. · ставка / аппетит / стоп-линия — текущая большая цель / её бюджет в легах / дата или лимит, после которого работа не продлевается молча. · спайк — быстрый бросовый эксперимент ради ответа на один вопрос: код выбрасывается, знание остаётся. · канон / рамка / якорь — закон игры / принятая основа дизайна (ещё не закон) / материал, который нравится и хранится, но не закон.

Оформление параллельной работы в OS

борд = рендер поверх NOW + git · четыре артефакта · ничего в ядре OS не меняем

1 · Карта площадок (knowledge)

  • Таблица: линия → checkout/ветка → нужен ли Unity Editor → машина
  • Правила Unity: один проект = один редактор; линия = свой worktree
  • Очередь мержей + пред-/пост-полётные чек-листы лега

2 · LAUNCH-блок в каждом CALL

  • lane / venue / машина / editor или headless
  • base @SHA + обязательный §Re-sync
  • conflict_surface (какие файлы трогает) + depends_on + слот очереди мержей

3 · Конвенция open_calls в NOW

  • Одна строка на CALL: линия, площадка, база, зависимости, слот
  • Писарь ведёт как обычно — борд из этого рендерится

4 · Пьеса «борд» (локальная, render-only)

  • Новый чат: «что можем делать» → В РАБОТЕ / МОЖНО ЗАПУСТИТЬ (с инструкцией запуска) / ЖДЁТ ТЕБЯ / ОЧЕРЕДЬ МЕРЖЕЙ
  • Ничего не пишет — только читает NOW + git обоих репо
Площадки: ПК = редактор №1 (main — owner-eye и мержи), №2 (dev_2 — линия B рендер), №3 по надобности (новый worktree lab — линия C; цена: первый импорт Library, лимит — RAM). Линия A (ядро) редактора почти не требует — engine-free ядро гоняется headless. MacBook: сессии планирования/писаря/G5 (Unity не нужен) + headless-ядро после установки dotnet+pwsh + кандидат в «машину №2» для интернет-спайка лока (или облачная Windows-VM — чище для первого замера). Один и тот же project-folder в два редактора открыть нельзя — поэтому линия = свой worktree, конфликтов не будет.

Ночной рефакторинг: процесс запуска

твоя команда: «делай рефакторинг» · мерж — утренним батчем

Вечером (ты, 1 минута)

  • Новый чат: «делай рефакторинг» — всё
  • Опционально: «только ядро» / «только доки» / «N штук»

Ночью (само)

  • Пред-полёт: читает борд/NOW — какие зоны СВОБОДНЫ от летящих легов; занятые зоны не трогает
  • Берёт верхние элементы из work/refactor/backlog.md, чьи зоны свободны
  • Каждый элемент — свой worktree/ветка, параллельно
  • Доказательство на каждом: голдены байт-в-байт + чексум + 1525 тестов + mutation не хуже + оба скана
  • Следом на каждый — G5-опровержение свежей сессией («поведение изменилось? найди») — тоже ночью
  • Что не доказалось — ОТКАТ и пометка в бэклоге, в отчёт честно

Утром (ты, ~5 минут)

  • Отчёт в борде: «доказано N, отклонено M, ждут мержа»
  • Мерж в main — твой батч (ворота не ослабляем): смотришь отчёт → «мержи» → очередь мержей по одному с полным прогоном

Границы ночного

  • Ночью — только headless-доказуемые зоны: ядро, тесты, tools, доки
  • Адаптер/рендер-рефакторы — дневные (нужен твой 5-мин глаз)
  • Строго ноль изменения поведения; никаких «попутных улучшений»
  • В main ночью не мержится НИЧЕГО
Для установки: (1) бэклог work/refactor/backlog.md — наполняется стартовым read-only сканом; (2) локальная пьеса plays/refactor-night.md с этим процессом; (3) строка-указатель в NOW, чтобы фраза «делай рефакторинг» из любого нового чата находила пьесу. Позже можно повесить на расписание — но старт всегда работает и вручную.
Слой контроля для автономных треков (расширение борда): каждый трек = parallel_track в NOW со своим гроссбухом в work/<трек>/; возвращает RESULT с evidence и встроенной адверсарной проверкой (свежая сессия пытается опровергнуть выводы ДО того, как они попадут к тебе); пишет только в свою рабочую область — в состояние только через писаря; твоя валидация — батч-слот, тем же ритмом, что вердикты. Рефакторинг дополнительно: строго нулевое изменение поведения (доказуемо голденами/чексумом — в детерминированном ядре это проверка сильнее человеческого ревью), никогда не трогает зону, где летит фичевый лег, и всегда уступает мерж-слот фичам.

Стенд «Газовая лаборатория»

владельческий верстак · спроектирован по реальному коду 10.07
Что это: одна постоянная сцена-верстак (эволюция существующей песочницы, не новая рига): ты в игре, не в инспекторе — свободная камера, палитра инструментов (спавн газа по клику, вентили/джеты/извержения/двери, пауза/шаг/скорость, заготовленные сценарии), приборы (проба клетки по наведению, масса по типам, стоимость тика, лампа детерминизма AGREE/DIVERGE) и возможность вселиться в капсулу и ходить. Это не одноразовая проверялка: капсула-контроллер, подключение дозы и большой уровень — это одновременно и работа дороги Полигона М1; стенд потом же станет площадкой грейбокса и источником захвата материалов для витрины.
P0

Верстак v0 — управлять всем, что уже есть

~1 лег (неделя с приёмкой) · ноль изменений ядра

Что появится

  • Игровая палитра: зеркалит весь существующий API песочницы (пауза/шаг/скорость, burst/puff, вентиль/джет/извержение, двери/клапаны, FLOOD, выбор уровня)

  • Клик по клетке → проба: концентрация, тип, потоки, телеграф
  • HUD: тик, ms/тик со спарклайном, активные грани, масса по типам
  • Лампа детерминизма: прогон дважды → AGREE/DIVERGE (код уже написан, просто не положен в сцену)
  • Панель-слот «БЛИЖНИЙ/ДАЛЬНИЙ УРОВЕНЬ: НЕ ПОСТРОЕНО» — честность с первого дня

Чего в v0 не будет (честно)

  • Оверлея спящих/бодрых клеток — перечисление фронта внутри ядра закрыто, нужен воротный лег (P1)
  • Больших уровней — сначала чинить рендер (P1): сейчас он каждый кадр считает полный чексум (28 мс на ангаре) и рисует куб-объект на клетку
  • Сохранения своих сессий (сериализатор — на полке)
Итог P0: нажал Play — и всё, что построено за месяц, у тебя в руках, из игры, без инспектора.
P1–P2

Увидеть оптимизации + ходить и дышать

P1: 1 воротный лег в ядро + 1 адаптер-лег · P2: 1–2 лега (= работа дороги М1)

P1 · оптимизации на масштабе

  • Читальный шов в ядро: счётчик и перечисление активных клеток + счётчики работы шага (read-only, вне чексума, полные ворота)
  • Чинка рендера: чексум по счётчику ревизий, отсечка кубов на больших сценах
  • Оверлей «бодрых» клеток поверх облака — впервые ВИДНО, как спит/просыпается фронт
  • Сцены: HANGAR-196K (та же геометрия, что в замерах — числа сверяемы), MINE-SHAFT (вертикаль, плавучесть)

P2 · тело игрока + ходьба + доза (= дорога)

  • P2a0 сначала проверяет PuppetMaster root authority и сетевую пригодность; production-контроллер выбирается только после A/B/C verdict
  • Победивший контроллер/physics-handoff даёт WASD и будущую оболочку игрока
  • Подключение дозы раз-в-тик к живому полю — это и есть пре-работа слайса урона
  • HUD дозы; манекены-пробы; телеграф-оверлей + лог взрывов

P3–P4 · после (по потребности)

  • P3: источники на честном сетевом пути (после мержа dev-лега), настоящий выбор типов, покраска реакций A/B
  • P4: BIG-OPEN-1M (128×128×64 = 1,05 млн клеток) — «поставил источник, ушёл, смотришь»; данж растёт к октябрьскому грейбоксу; пресеты камер для GIF
  • Полка (не в плане): рекордер сценариев, 2М-стресс, синк двух инстансов по сети
Стоп-линия после P2: P3/P4 идут, только если сессии не нужны дороге М1. Стенд не конкурирует с дорогой — P0+P2 и есть её куски, просто с приборами.
Честные потолки (исправлено раскопками 10.07): охрана 2^21 клеток — это защита от переполнения, НЕ потолок уровня (в коде прямо: «NOT to cap level size»; она суммирует объёмы комнат, и 150 средних комнат ≈ 223 тыс. клеток = ~11% от неё). Реальные потолки другие: слабый компьютер держит ~27–28 тыс. одновременно-активных клеток @50 мс — тихий 150-комнатный уровень движок несёт уже сейчас (стоимость ∝ активному фронту, доказано замерами), полностью залитый — нет; это чинит отложенный S4-rollup, который по подписанному SPEC «При ~150 комнатах ОБЯЗАТЕЛЕН (не опционален)». Полный чексум — 28 мс за обход (дешёвый дайджест = часть S4-пакета). Рендер рисует максимум 128 м по оси (fail-closed) — чинится рендер-окном вокруг камеры: малый лег адаптера, ноль правок ядра. «Кубы делятся возле игрока» (детальная сфера) — спроектировано, не построено, отдельный слот. До мержа dev-лега кнопки впрыска носят бейдж INTERIM.
Словарик: грейбокс — уровень из серых болванок без арта, чтобы проверить геймплей; standalone-билд — игра как отдельный .exe, а не запуск из редактора Unity; вертикальный срез — маленький кусок игры, где есть всё сразу (управление, цель, смерть, звук), а не одна система целиком; капсула — обложка игры в Steam, главный конверсионный арт; вишлист — «в желаемое» в Steam, главная валюта до релиза; слот Next Fest — право участвовать, даётся игре один раз навсегда; прототип-лента — отдельная «песочница» в проекте для быстрого кода оболочки, не трогающая проверенное ядро симуляции; телеметрия — автоматический сбор данных, как играют люди (сколько минут, где умерли); min-spec — слабый компьютер, на котором игра обязана работать.
