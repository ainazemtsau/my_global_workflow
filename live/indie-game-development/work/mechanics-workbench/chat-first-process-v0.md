# Mechanics Workbench — Chat-First Process v0

Status: process draft, не canon.
Purpose: владелец работает через чат и один вопрос за раз; файлы являются backend-картой для агента.

## 1. Исправление процесса

Владелец не должен ходить по файлам и искать, какой вопрос обсуждать.

Правильный режим:

> Карты и ledger лежат в repo, но в работу они входят только через новый chat/CALL: один вопрос, почему он сейчас, что он блокирует, что не обсуждаем, какой результат нужен.

Файлы нужны не чтобы владелец их читал руками, а чтобы агент:

- не терял идеи;
- видел blocker graph;
- не тащил parked-темы в текущую сессию;
- мог после owner-approved результата обновить drafts/state строго через RESULT.

## 2. Как выглядит одна рабочая сессия

Каждая сессия обсуждает ровно один главный вопрос.

Owner-facing packet в начале сессии:

1. **Вопрос.** Одной нормальной фразой.
2. **Почему сейчас.** Что без этого нельзя обсуждать дальше.
3. **Что блокирует.** Какие следующие вопросы зависят от ответа.
4. **Что уже известно.** Коротко, без ссылки "иди прочитай файл".
5. **Какие варианты на столе.** 2-5 вариантов, каждый с "почему может быть хорош" и "bad, because".
6. **Что не обсуждаем.** Parked topics for this session.
7. **Какой результат нужен.** Например: accept / revise / reject / split / park.

После обсуждения:

- если owner говорит "можно записывать" — пишется RESULT/state_changes;
- если есть сомнения — checkpoint/captures only;
- если вопрос оказался неправильным — return to map and create corrected next question.

## 3. Что запрещено в chat-first режиме

- "Открой файл и сам найди раздел".
- Несколько главных вопросов в одной сессии.
- Скрытый выбор концепта внутри формулировки.
- Слова без определения как будто они уже решены.
- Обсуждать `ЛЁГКИЕ`, economy, roster, title, exact tools, если текущий вопрос про floor proof.
- Записывать canon или state без owner review/sign.

## 4. Что агент обязан делать

- Сам приносить нужный контекст в чат.
- Говорить, почему вопрос стоит именно сейчас.
- Называть downstream, который вопрос блокирует.
- Парковать все боковые идеи с dependency.
- Спорить с вариантом, а не продавать его.
- В конце давать короткое "что решили / что не решили / следующий вопрос".

## 5. Как используются карты

Карты остаются backend:

- `area-map-v0.md` — где вопрос находится в системе;
- `idea-ledger-v0.md` — что нельзя потерять;
- `core-proof-skeleton-v0.1.md` — current draft, не owner homework;
- `question-queue-v0.md` — какая очередь вопросов сейчас предлагается.

Владелец может читать файлы, если хочет, но процесс не должен этого требовать.
