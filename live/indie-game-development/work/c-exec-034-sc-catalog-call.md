## Why this slice

Sc-types laid a 3-layer param substrate (common parent → meta-type → type) but per-type differentiation is TODAY only two integer overrides (spread `SpreadKp`, buoyancy `RatioToAir`) + reagent-class coords; the reserved per-type extension structs (`GasInstanceReserved`, `GasLookReserved`) are EMPTY/unwired; a new common param means editing 4 load-bearing files; authoring is hand-built C# object lists (only the ReactionSandbox demo + test fixtures — no canonical roster module). Sc-reactions gave a data-driven reaction engine but its custom outcome/telegraph handlers are NOT wired into the production tick (a data-authored outcome folds into the hash yet has no runtime handler on the zero-arg Step path). So the substrate exists but there is no SOLID, CONVENIENT, EXTENSIBLE way to author several genuinely-different types with their own laws — the CENTRAL mechanic. This slice closes that.

## goal (outcome, not method)

An author (human or AI) can define **several genuinely-different gas types AS DATA** — each carrying its own laws via an **EXTENSIBLE per-type parameter set** (a new param, and a new type, are DATA only — ZERO engine code branches, ZERO enum/switch, and — for a new param — ZERO edits to Core/** engine source), inheriting common → meta → type, folded deterministically into the session handshake — and author **reactions between them** on the existing engine, such that the types **behave genuinely differently in the running sim** and any authored reaction's outcome **actually fires in the production tick**. The owner SEES, in the engine debug view, several data-defined types that behave distinctly and react. **ENGINE-ONLY — no visual/shader/look work; no mod/DLC/external-loader; no procedural generation; no env-conditioned-change mechanism; no new gas physics** (all design-aware, §boundaries/§design-aware).

**approach token = `data-defined-types-extensible-per-type-laws-generic`** — the ONE substrate this leg proves: per-type laws are EXTENSIBLE DATA applied by a GENERIC engine (add a param = a declared schema entry + data, no Core/** edit; add a type = data), folded into the ContentHash, ZERO type-identity control flow. ## the handshake hash (lockstep determinism — DO NOT skip; three tripwires)

Per-type LAWS are determinism-load-bearing: two peers whose types differ in a sim-affecting param must desync at the session-start handshake, never silently mid-run (Факт-1 / §2 «чексумм ЗЕЛЁНЫЙ, физика мусор»). Today `ComputeContentHash` folds ceiling + parent + type-count + per-type (TypeId, MetaTypeId, **resolved** EffectiveKp/EffectiveWeight) + reactions. Every new per-type sim-affecting param MUST fold into the SAME hash, obeying ALL three:

1. **Fold the RESOLVED value, UNIFORMLY, default included.** Fold the resolved (post-inherit) per-type value inside the canonical-TypeId loop — never the raw override + which-layer-it-was-set-on. Fold it for EVERY type including the default baseline, so `IsDefault` (structural `hash == default._contentHash`) stays SOUND. **Byte-identity is produced ONE LAYER UP** — `VoxelField.MeaningChecksum` skips the whole type/reaction fold when `!IsDefault`; it is NOT «the fold contributes nothing for the default». So: the handshake-SEED literal `Default.ContentHash()` MAY legitimately change when the fold surface grows (it is not a physics golden — any test pinning it is UPDATED, not treated as a regression); the byte-identity RED asserts the single-default-type **loopback/MeaningChecksum golden** is unchanged.
2. **Length/arity guard every variable segment.** FNV has no delimiting. Precede every variable-length segment with its count (as the code already does: `ordered.Length`, `_sortedRules.Length`, …). If a type may carry a VARIABLE population of params, a per-type param a type OMITS still folds its RESOLVED default at a FIXED position/arity (or a per-type present-count precedes its entries) — so `{A}` vs `{}` vs `{A=default}` cannot alias to the same running hash.
3. **RED controls (all three directions):**
   - **DIVERGENCE:** two rosters differing in ANY resolved sim-affecting per-type param value, OR the param schema (a different declared default / inherit-rule for a param a type OMITS — it changes that type's resolved value), MUST diverge the hash.
   - **CONVERGENCE (authoring order):** the SAME content authored in a DIFFERENT order (types/params permuted) → IDENTICAL hash.
   - **CONVERGENCE (inheritance path):** the SAME resolved per-type law reached via meta-inherit vs type-explicit-same-value → IDENTICAL hash (proves the fold uses the RESOLVED value, not raw override + layer — else two behaviourally-identical rosters spuriously desync green≠green).
   - **PARTITION-ALIASING:** two rosters that fold identically ONLY if a count/arity guard is missing MUST diverge.
   Unknown param-id / TypeId = LOUD build error, never a silent skip. **A param that is NOT sim-affecting (a future look param) MUST NOT fold — but NO such param is built this slice** (§design-aware d).
