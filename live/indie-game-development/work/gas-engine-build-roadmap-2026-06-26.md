## Зафиксированные решения (сходимость)

- **Архитектура** (triple-vetted, 3 состязательных воркфлоу): слойный координирующий grid (газ 50 / структура 25 / размещение 25-резерв), grid авторитетен, гибрид с кап-сферой визуала. Облака-как-авторитет отвергнуты.
- **Комнаты:** дефолт — ~однородные малые/средние. **Большие залы НЕ спец-кейсим сейчас, отложено-расширяемо** (owner): если упрутся по замеру — (а) авторить зал как сегменты-комнаты с дверьми-во-всю-стену (дальнее грубее даётся даром), либо (б) far-only `mid-2.5D` + sub-partition. Этажи друг над другом = просто стопка комнат (объём-нейтрально).
- **Точность B:** full-3D там, где живёт настоящий-B вопрос; большинство B реклассифицируется в дешёвый монотон-порог по грубому тоталу. State-driven-near-demote + lazy-resolve — отвергнуты.
- **Сеть:** lockstep (только инпуты по проводу) — оставляем; это и есть то, что делает большой детальный газ дешёвым по каналу. State-sync хуже.
- **Применено:** Факт 2 (слойный, ратифицирован) + сплит разрешений + контракт меток + правило двери + кап-сфера (4м@50см старт, ADR) + предикат монотон-B + settled_flag. `c-exec-013` → `c-exec-014`. `mid-2.5D` отложен (резерв).

## Исходная задача — решена

Тред начался с **S0 / ридера уровня Dungeon Architect** + баг с дверью (газ открыл всю стену) + идея «поразносить поверхности по типам». **Решено архитектурой:** ридер = слой СТРУКТУРЫ (читает геометрию DA + метки на модулях: геометрические валидируются против меша, семантические авторишь ты); дверь = настоящая апертура; «типы» = маркерная топология. Это строит `c-exec-014`.

### Шаг 4 / замер — важная оговорка про железо

Газосим — это **CPU**; RTX 4090 ему не помогает; топовый процессор владельца делает замеры **оптимистичными**. Бенчмарк показывает **клетки + мс**, а вывод делаем по **бюджету слабого железа** (потолок ~200к активных клеток на слабом ядре), с запасом — НЕ «у меня на риге плавно = шипим». **Первая build-задача = «проба зала»** (1 игрок в большом открытом объёме, full-3D, Burst/Jobs) — это #1 неизмеренное число всей архитектуры; оно решает, нужны ли вообще оптимизации больших залов. + RED-оракул монотон-B (full-sim vs порог, с не-монотонным контролем, который ОБЯЗАН провалиться).

## Ритм

- По одному маленькому шагу; каждый кончается играбельной сценой + зелёными автотестами, прежде чем следующий.
- Сборка идёт в `GasCoopGame_dev` (MCP + ветка dev); владелец гоняет играбельные сцены + бенчмарки; OS применяет RESULT-ы домой отдельным writer-ом.
- `mid-2.5D` / sub-partition / position-hints / blob-render / placement-механика — отложенные резервы, включаются по необходимости (замер), не сейчас.
