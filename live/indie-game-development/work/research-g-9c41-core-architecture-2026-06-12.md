# Архитектурный бриф ядра g-9c41 — ресёрч c-arch-001

Сессия s-arch-001 (research), 2026-06-12. Назначение: вход для возобновления шейпа в c-shape-003 (выбор подхода — твой). Статус: **информирует, не решает**. P6-модель сети не пересматривалась (host-authoritative сим, кастомный chunked-delta поток газа, ghosts только для игроков/объектов, GPU никогда не авторитетен) — бриф проектирует, ЧТО течёт через эту модель.

Теги достоверности по всему тексту:
- **[факт]** — проверено в коде / официальных доках / первоисточнике, источник с датой;
- **[сообщается]** — форум / вики / блог / новость, первоисточник не вскрыт;
- **[вывод]** — инженерная оценка из явно названных допущений (вся арифметика §5 — здесь).

Каждый импорт из архива — с явной причиной (§8, правило clean-start).

| Под-вопрос | Вердикт |

| **(a) Модель поля** | Лестница из трёх ступеней: **T1 room-graph** (всегда, на всём уровне, авторитет массы) → **T2-окна** (воксели 25/50 см только вокруг игроков и событий, реконсиляция к T1) → **render-LOD** (клиент, вне авторитета). ARCH-1 содержит подходы **C и A как конфигурации** (T2 выключен / T2 минимальными окнами) — архитектура не предрешает твой выбор. ## §1. Словарь (дополнение к словарю shape-файла)

| Термин | Что это |
|---|---|
| **Лестница грануляции** | Упорядоченный набор уровней детализации поля с правилами перехода (что когда активно, как сохраняется масса при переходах). |
| **T2-окно** | Ограниченная область (комната/группа комнат), где включена воксельная детализация. Вне окон газ живёт только в T1. |
| **Дормантность** | Сворачивание T2-окна в T1-скаляры, когда регион неактивен. Аналог у SS14 — excited groups; у старого проекта — Gas.Interest. |
| **Chunk (чанк)** | Блок 8×8×8 ячеек — единица «грязи» и отправки по сети. Фиксирован в ЯЧЕЙКАХ, не в метрах: при 25 см это куб 2 м, при 50 см — 4 м. Поэтому wire-формат не зависит от размера ячейки. |
| **Change-feed** | Поток «что изменилось в слое за тик» (грязные чанки/узлы) с номером ревизии. Это едят сеть, рендер и другие механики. |
| **TopologyDocument** | Единственный формат, в котором ядро принимает уровень от любого генератора (см. §4.1). |
| **Data plane / control plane** | Данные слоёв и их перевозка / порядок фаз тика и права записи. |
| **Quantization (квантование)** | Упаковка float в 8–16 бит для сети (с масштабом на чанк). |
| **RLE** | Сжатие повторов; на гладких полях даёт 4–10× [сообщается, Noita EW]. |
| **UTP** | Unity Transport — транспортный слой под NfE; рядом с NfE можно открыть собственный канал/драйвер [вывод из доков, §5.4]. |
| **Seal-валидация** | Проверка герметичности объёма на ингесте: flood-fill изнутри; достижимость «наружу» мимо порталов = ошибка уровня. |
| **Hotness / interest** | Приоритет региона для детализации и отправки (близость игроков, величина изменений). Прямой потомок старого Gas.Interest. |

## §2. (a) Модель газового поля — лестница грануляции

### 2.1 Что реально делают прецеденты

- **SS14** (код прочитан агентом 2026-06-12, github.com/space-wizards/space-station-14): тайл **1 м** (объём ячейки 2.5 м³); 9 газов, массив молей паддится до 12 float «for SIMD reasons» (~48 Б моли + температура; активный тайл ~250–300 Б с служебными полями) [факт]. Алгоритм — порт LINDA + опциональный monstermos (по умолчанию включён) [факт]. **Дормантность**: тайл просыпается при нетривиальном обмене с соседом; активные тайлы сливаются в excited groups; после 4 циклов покоя группа усредняется (SelfBreakdown), после 16 — полностью засыпает (Dismantle) [факт, AtmosphereSystem.LINDA.cs / ExcitedGroup.cs]. Атмос тикает **15 Гц** при игровых 30 Гц, бюджет **3 мс/тик** с time-slicing (недоделанное доделывается следующим тиком) [факт, CCVars]. Monstermos-выравнивание капится 200/2000 тайлов на событие — бюджетирование каскада [факт].
- **SS13-родословная**: ZAS = зоны-комнаты с выравниванием по связям (это T1 в чистом виде) [сообщается, Aurora wiki]; философия LINDA дословно: *«We don't simulate every tile, and most of the changes… simulating as little as we can get away with»* [факт, tgstation Atmospherics.md]; monstermos добавлен ради ЧИТАЕМОСТИ прорывов — мгновенная декомпрессия вместо медленной диффузии [факт, PR 2019-12-17].
- **Stationeers** (3D, MP): атмосфера **на ячейку 2 м** (8000 л) + комнаты как оверлей, душащий обмен (room detection сдаётся выше 1200 кубов — большее считается «миром»); трубы — отдельные графы сетей [факт/сообщается, вики]. Атмос на выделенном потоке, целевой тик **500 мс**, на больших базах деградирует до ≥1 с [сообщается]; в декабре 2025 чинили лаг-спайк «миров с >5000 атмосфер» [сообщается, патчноут v0.2.6085]; признан «longstanding severe desync» у клиентов с высоким пингом [факт, оф. анонс]. Транспортный инцидент верифицирован по Steam news API: стабильный пуш ~2026-01-08/09 → **реверт 2026-01-09** → **перевыпуск 2026-01-28** [факт].
- **ONI** (2D, сингл): ячейка ~1 м [вывод из вики-единиц], **один элемент на ячейку** (газы вытесняют, не смешиваются в ячейке) [факт, вики]; сим в нативном потоке (SimDLL), обмен сообщениями с C#; карты 256×384 ≈ 98k ячеек [факт/сообщается].
- **Space Engineers** (3D, MP): герметичность — flood-fill по ячейкам грида, кислород комнаты — **один скаляр** (`IMyAirVent.GetOxygenLevel()`); исторически выключалась по умолчанию из-за производительности; известные MP-глюки ре-прессуризации при подходе игрока [факт API / сообщается].
- **Stormworks**: воксели машин 0.25 м, но газы — смеси **на отсек** [сообщается].
- **Barotrauma** (2D co-op, шипнут коммерчески): вода/кислород — скаляры **на hull** (комнату), синк порогами (§5.4) [факт, код].

**Сводный факт по прецедентам: суб-метрового объёмного многогазового 3D нет ни в одной шипнутой игре. Самый мелкий 3D — 2 м (Stationeers), и тот душит обмен комнатами. Все остальные 3D-мультиплеерные схлопывают газ в скаляры по комнатам.** Твои 25 см — за пределом прецедентов (это и есть moat), и единственный масштабируемый путь к ним — иерархия + дормантность; обе техники прецедентами доказаны (excited groups, rooms-overlay, твой же Gas.Interest).

### 2.2 Лестница (ступени и переходы)

- **Ступень 0 — T1 room-graph.** Узел = объём (комната/коридор/шахта/вент-путь), рёбра = порталы. На узел: массы по видам газа + температура + ревизия. **Авторитет массы всегда здесь, на всём уровне, без исключений.** Прецедент работоспособности: в старом проекте T1 был авторитетом и работал в сингле [факт-импорт §8.3].
- **Ступень 1 — T2-окно.** Воксели 25/50 см внутри активных регионов. Инвариант: T2 хранит РАСПРЕДЕЛЕНИЕ массы узла, не собственную массу — сумма по ячейкам окна ≡ массе T1-узла (реконсиляция по образцу старого Gas.T2: «ambient + bounded primitives, reconciled to T1» [факт-импорт §8.3]; у нас вместо примитивов — воксели, инвариант тот же).
- **Ступень 2 — render-LOD.** Клиентская интерполяция/шум/волюметрика поверх принятых данных. Вне авторитета, вне этого ресёрча (узел g-7e15; render proof slice — критерий 6 узла).
- **Переходы.** Wake: игрок в interest-радиусе ∨ порталный поток > ε ∨ событие/источник → окно раскрывается, сидируется из T1-массы + компактного хинта распределения (центроид+разброс, ~8–16 Б на узел; приём — прямой потомок старого `GasT3RoomSeedingBuilder` [факт-импорт §8.3]). Sleep: все триггеры ниже порогов N тиков подряд → массы сворачиваются в T1, хинт сохраняется. Сохранение массы точное по построению: масса ПЕРЕДАЁТСЯ между представлениями, не пересоздаётся.
- **Роль иерархии T1/T2** (прямой вопрос CALL): T1 — не «дешёвый режим», а позвоночник: глобальная консистентность всего уровня, дешёвый whole-level сим, дешёвая сеть для дальних регионов, естественная единица дормантности и interest. T2 — линза локальной правды там, где смотрят и где происходит. Это та же пара, что уже работала в сингле у старого проекта — импортирована как evidence FOR feasibility [причина в §8.3].

### 2.3 Конфигурируемая ячейка (R2)

- Размер ячейки — **параметр региона** (`resolutionClass` в TopologyDocument: hero=25 см, default=50 см, допустим и coarse), плюс глобальный override в конфиге стенда для A/B-прогонов.
- Контракты не знают размера ячейки: все API в мировых координатах; чанк фиксирован в ячейках (8³) → wire-формат, память чанка и код обработки инвариантны к размеру.
- Цена 25 vs 50 в 3D: **×8 памяти, ×8–16 тика, ×~8 сети** на ту же геометрию облака (§5). Конфигурируемость — рычаг производительности и сети, не только качества.
- Acceptance-хук для шейпа: одна и та же сцена стенда прогоняется на 25 и 50 **без правок кода** — это и есть проверка «ячейка действительно конфиг».

### 2.4 Виды газов

Не вопрос этого ресёрча (двухуровневая декларативная модель мета-газ/конфиг уже в done_when узла), но поле обязано хранить K видов на узел/ячейку, K — из конфига. Планка прецедента: SS14 держит 9 видов (+SIMD-паддинг) [факт]; наша рабочая «3–5 на сцену» дешевле прецедента.

## §3. (b) Грид как ткань интеграции

### 3.1 Чем грид ВЛАДЕЕТ (узко, по A1)

1. **Пространство**: world→cell→region маппинг; chunked-хранилище слоёв (NativeArray, SoA).
2. **Топология**: volumes + portals + surfaces; единственный писатель через command→validate→commit→revision++ [факт-импорт A1, §8.1].
3. **Реестр слоёв**: схема, resolutionClass, владелец-писатель.
4. **Change-feeds**: грязные чанки/узлы per layer per tick + ревизии.
5. **Стабильные ID** всего перечисленного [факт-импорт A1 + guardrail, §8.1–8.2].

Чем НЕ владеет: семантикой механик (диффузия, реакции, урон), рендером, сетью, gameplay-правилами. A1 дословно запретил Grid-god-object («do_not_make_Grid_a_god_object_for_Gas_AI_interaction_multiplayer_Unity_presentation_or_gameplay_state») [факт, 2026-05-23].

### 3.2 Кандидат #1 — твоя гипотеза: слоёная решётка

Каждая механика — слой(и) на гриде; общение — через грид.

**За:** dirty-tracking один на всех — сеть, дормантность, interest и debug получают ОДИН механизм вместо четырёх; SoA-слои Burst-дружелюбны [факт-якорь: CA-бенчи ~1М ячеек ≈ 4 мс jobbed на NativeArray, §5.3]; новая механика = новый слой без правки чужого кода (важно для R4 «сим переживает концепт»); старый проект фактически так жил: Grid — авторитет топологии/дверей/якорей, Gas — потребитель через явный событийный мост `IGasGridEventHandler`, прямой доступ к хранилищу запрещён доками [факт-импорт §8.5].

**Слабость в чистом виде:** «синк через грид» без дисциплины вырождается в событийный суп с недетерминированным порядком — лечится поправками вердикта (3.5).

### 3.3 Альтернатива A — чистый ECS (DOTS-native)

Всё — entities/components/systems; поле — компоненты ячеек; общение — ECS-queries/события.

**За:** одна парадигма с NfE-миром; tooling/дебаг DOTS.
**Против [факт/сообщается]:** entity-per-cell — измеренный анти-паттерн (1М ячеек: NativeArray-джобы ≈ 4 мс против 900–1200 мс у ECB-подхода; чистый NativeArray 2.3M @ 13 мс) [сообщается, Unity discussions CA-тред]; поле в DynamicBuffer-ghost'ах упирается в 32-слотовую историю снапшотов на чанк на соединение (×память, ×рекомпрессия) [факт, NfE доки]; полевые данные всё равно уезжают в NativeArray — «чистый ECS» вырождается в гибрид сам собой.
**Вердикт:** ECS — для сущностей (игроки, объекты, девайсы, NfE ghosts). НЕ для поля.

### 3.4 Альтернатива B — сервисная сетка (точечные контракты, без общей ткани)

Каждая механика — сервис со своим хранилищем; общение — пары типизированных интерфейсов.

**За:** нулевая инфраструктура на старте; зависимости явные.
**Против:** контракты растут O(n²) по числу механик (газ↔двери, газ↔разрушение, огонь↔газ, огонь↔двери…); dirty-tracking/interest/дормантность/репликация дублируются в каждом сервисе — ровно то, что у слоёной решётки общее по построению; R4 хочет одно узкое переносимое ядро, не пучок пар; и A1 уже выбрал «Grid as topology/spatial authority with narrow consumers» [факт].
**Вердикт:** №3 как целое; её дисциплина (типизированные контракты, никакого общего стейта в обход) импортируется поправкой 2.

### 3.5 Вердикт по (b)

**Гипотеза #1 подтверждается как data plane, с тремя поправками:**

1. **Синк = ревизионные change-feeds в фиксированных фазах тика** (commands → topology commit → T1 → T2 → publish/network), не свободные события посреди тика. Прецеденты фазовой дисциплины — твой же старый `GridSpatialEventBus` (фазы, запрет публикации «назад», контроль глубины каскадов) [факт-импорт §8.4] и update-order из nucleus-спеки (topology → revision → gas reads → gas tick → snapshot/delta → observation) [факт-импорт §8.6].
2. **Запись:** у слоя ровно один писатель-система; межмеханическое влияние — командами/модификаторами владельцу (форма — modifier/command records из nucleus-спеки §5 [факт-импорт §8.6]); топология — только через commit.
3. **Грид узкий** (3.1); семантика — в механиках. Анти-god-object — действующее правило A1.

**Главный отрицательный факт: ни один продукт — ассет или OSS — не даёт сетевой авторитетный 3D-газ под Unity. Пять независимых углов поиска упёрлись либо в GPU-VFX, либо в «пишите сами»** [факт].

**Вердикт:** ядро и ткань — **строим** (ткань тонкая: chunked-словарь NativeArray-слоёв — порядок «дни, не недели» [вывод]); **покупаем края**: NfE (закреплён P6), DA/PGG (procgen, §4), рендер-ассеты только как визуал за швом (вердикт slice). SS14-атмос — легальный (MIT, с атрибуцией) донор алгоритмов и архитектурных решений: излучения excited groups, капы каскадов, формат клиентского оверлея; технически это референс, не библиотека (вшит в RobustToolbox) [факт].

## §4. (c) Контракт поедания procgen

### 4.1 TopologyDocument v0 (черновик контракта)

```
meta      { seed, generatorId+version, units: meters }
volumes[] { id, archetype (room|corridor|shaft|ventPath|techRoom),
            bounds (AABB | cell-set), verticalBand,
            resolutionClass (hero|default|coarse) }          ← R2 живёт здесь
portals[] { id, volumeA→volumeB (или exterior), type (door|opening|vent|shaftLink),
            geometry (позиция+габарит), initialState, flowLimit }
surfaces[]{ id, volumeA↔volumeB|exterior, area, breachable: bool }
            ← латентные порталы: scripted breach = активация surface
              → новый портал + revision++ (никакой системы разрушений)
anchors[] { id, volumeId, type (spawn|source|sink|observation), pos }
```

Прямое развитие `grid_topology_needs_from_gas` + mutation boundary из nucleus-спеки [факт-импорт §8.6]. Ядро не видит генератора; генератор не видит ядра (A1). Ингест-пайплайн: **adapter → compile (валидация: уникальность id, симметрия соседств, seal-тест) → присвоение стабильных runtime-id → publish revision 0** — по образцу старых `GasTopologyStartupValidator` и GridV2 `Source → ChamberGraphCompiler → RuntimeStore` [факт-импорт §8.3–8.4].

### 4.2 Маппинг DA → TopologyDocument [факт, API-доки DA, 2026-06-12]

- **Grid Builder**: `GridDungeonModel` → `Cells: List<Cell>` (Id, Bounds, CellType Room/Corridor, AdjacentCells, ConnectedRooms) + `Doors: CellDoor[]` → volumes+portals реконструируются из модели напрямую, **без парсинга сцены** [факт].
- **GridFlow**: `GridFlowDungeonModel.layoutGraph` (FlowLayoutGraph — высокоуровневый граф уровня) + `tilemap` [факт].
- **Snap / SnapGridFlow**: query-интерфейсы (комнаты, двери, bounds, дистанции от старта/цели; changelog v1.12/1.18.1/1.21) [факт].
- Runtime-генерация официальна: `dungeon.Build()` + `Config.Seed` [факт].
- **Герметичность**: SGF межмодульно герметичен by construction (неиспользуемые connection-points затыкаются стенами); внутри модулей и в Grid Builder — дисциплина авторинга тем → **seal-валидация на ингесте обязательна** (flood-fill изнутри объёма; достижимость «наружу» мимо порталов = ошибка уровня). Дешёвая, ловит дыры до того, как газ «утечёт сквозь стену».
- **Vendor-риск DA [факт]**: Unity-ветка 1.22.0 от 2024-11-29 (~18.5 мес без апдейта на сегодня), Unity 6 официально не заявлен (ревью: чинибельные скрипт-баги), автор активен в Unreal-ветке. Риск принят и изолирован: генератор за швом, замена не трогает ядро.

### 4.3 Воксализация для T2

Объём → flood-fill при необходимости на resolutionClass региона: комната 160 м³ @25 см ≈ 10k ячеек — мгновенно; весь уровень ≈ 15М ячеек ≈ секунды и не нужен разом — окна воксализируются лениво при первом wake [вывод].

### 4.4 Вердикт PGG (R3)

Факты [2026-06-12]: издатель FImpossible Creations; **v1.6.6.2.12 от 2026-02-21 — обновляется, признаков заброшенности нет**; «Beta» ~5 лет (2021→2026); min Unity 2019.4; runtime-генерация подтверждена девом [сообщается]; **публичного API-референса нет, программный структурный выход (комнаты/связи) не документирован** (node-graph-first: Build Planner); поддержка эпизодическая (Discord/email).

**Вердикт: day-one wiring НЕ рекомендован.** R3 требовал «PGG evaluated for day-one wiring» — оценка проведена, ответ: за шов. Заменяемость шва доказывается день-один **двумя провайдерами**: DA + ручные сцены стенда (MTS-паттерны нужны харнессу в любом случае, done_when 5 узла). PGG — таймбокс-спайк ≤2–3 дня за швом с единственным вопросом: «заполняет ли его runtime-выход TopologyDocument». Запуск спайка — когда решишь; промоушен — только по результату.

## §5. (d) Арифметика масштаба и сети (R1) — числа, не прилагательные

Все числа этого параграфа — **[вывод]** из явных допущений ниже, скрипт-проверены; установленные якоря помечены отдельно.

### 5.1 Профиль «огромный уровень»

Подземный город, сессия-экспедиция: footprint 200×200 м, глубина 40 м (6–8 ярусов); интерьер 15% bbox = **240 000 м³**; **~1500 объёмов** (средняя 160 м³ ≈ 8×6.5×3 м), **~3800 порталов**; партия 2–4 игрока, обычно 1–2 кластера. Параметры явные — при изменении пересчёт тривиален (всё линейно/кубично от них).

### 5.2 Память (24 Б/ячейку: 4 вида × 4 Б массы + температура + флаги, с паддингом)

| Представление | Ячейки | Память |
|---|---|---|
| T1 весь уровень (1500 узлов + 3800 порталов) | — | **~325 КБ** |
| Full-fine 50 см, весь интерьер (контрфакт) | 1.92M | 44 МБ |
| Full-fine 25 см, весь интерьер (контрфакт) | 15.36M | **352 МБ** |
| T2-окна 30 комнат @50 см | 38.4k | 0.9 МБ |
| T2-окна 30 комнат @25 см | 307k | **7 МБ** |
| Худший случай: 4 разнесённых игрока, 48 комнат @25 см | 492k | 11.8 МБ |

352 МБ на min-spec 16 ГБ (16 ГБ = 41% Steam, май 2026 [факт]) рядом с Unity, рендером и игрой — недопустимо. Здесь умирает «однородное мелкое поле» и рождается лестница.

### 5.3 Тик CPU

Min-spec CPU: 6-ядерник класса Ryzen 5 5600 (6 ядер — 28%, 8 ядер — 27.5% Steam, май 2026 [факт]); DDR4-3200 dual ≈ 51 ГБ/с теоретических [факт]. Якорь пропускной способности: Burst CA-бенчи ≈ **100–250k ячеек/мс jobbed** (диффузионный стенсил, NativeArray; 1М ячеек ≈ 4 мс) [сообщается, 2019-CPU — для 2026 консервативно].

| Нагрузка | Тик |
|---|---|
| T1: ~46k flops порталного транспорта (3800 порталов × 4 вида) | **<0.5 мс** |
| T2 30 комнат @25 см (307k ячеек) | **1.2–3.1 мс** |
| T2 worst 48 комнат @25 см (492k) | 2.0–4.9 мс |
| T2 30 комнат @50 см (38k) | 0.15–0.4 мс |
| Full-fine 25 см контрфакт (15.36M) | 61–154 мс — невозможен |

Газ-тик 10 Гц (прецеденты: SS14 — 15 Гц [факт], Stationeers — 2 Гц целевых [сообщается]) → даже worst ≈ 2–5 мс на 100 мс окна. **Бюджетная строка хоста: газ ≤ 4 мс/тик @10 Гц на 6-ядернике — выполняется с запасом ≥×2** (кроме «всё включая worst-кластеризацию одновременно» — тогда впритык, лечится бюджетом окон §5.5).

### 5.4 Сеть (uplink хоста — узкое место listen-server)

**Полы аплинка** [сообщается, 2025–2026]: UK FTTC в среднем ~9 Мбит/с ↑ (Ofcom), DE 21–26 (Opensignal 11/2025), US медиана ~57 (Ookla 04/2026; кейбл-планы 10–35 [факт, FCC]), PL 60–95, RU ~141 (09/2024). **Проектный пол: 9 Мбит/с ≈ 1.1 МБ/с на всё** (3 клиента: газ + ghosts + прочее).

**Wire-форма чанка** (8³ = 512 ячеек): заголовок ~12 Б (layer, chunkId, baseRev→rev, quantScale) + payload по виду газа: 512 × 8-бит квантованных = 512 Б сырых; RLE на гладком поле 4–10× [сообщается, Noita EW] → 50–130 Б/вид; типично 1–2 вида в чанке → **~75–280 Б/чанк; план: 300 Б средний, 600 Б худший**. Якорь-прецедент: SS14 шлёт клиенту 8×8-чанки оверлея (~450 Б сырых, ~7 Б/тайл), **3 Гц**, дельта по LastUpdate, PVS ~25 м, клиент никогда не авторитетен и молей не видит [факт, код].

**Interest на клиента**: T2-чанки в радиусе ~25–30 м + T1-сводки дальних комнат (8 Б/комната, только по изменению).

| Сценарий | На клиента | ×3 клиента | Против пола 9 Мбит/с |
|---|---|---|---|
| Стационар: ~30 грязных чанков/тик × 300 Б × 10 Гц | **90 КБ/с = 0.72 Мбит/с** | 2.16 Мбит/с | 24% — ок |
| Тот же эпизод @50 см (~4 чанка) | ~12 КБ/с | 0.3 Мбит/с | копейки |
| Breach-волна: ~100 чанков × 600 Б × 10 Гц | 600 КБ/с = 4.8 Мбит/с | **14.4 Мбит/с** | **ПРОБИВАЕТ** |
| T1-синк дальних (1500 комнат, меняются десятки/тик) | ≤ единицы КБ/с | — | — |

Стационар сидит в нормах шипнутых игр: Minecraft 50–100 КБ/с на игрока, Photon Fusion ~112 КБ/с, CS2 дефолт 786 КБ/с [сообщается/факт]. Прорыв — нет. Поэтому **в контракт стрима** (не «оптимизация потом»):

1. **Кламп полосы** на клиента (например 150 КБ/с) + **приоритетная очередь чанков** по hotness (близость × величина дельты) — прямой потомок старого Gas.Interest [факт-импорт §8.3];
2. **Лестница частот** газ-стрима 10→5→3 Гц под давлением (SS14 живёт на 3 Гц [факт]);
3. Укрупнение квантования дальних чанков;
4. Аварийный режим: дальним — только T1-сводки (комнатные проценты), T2 догоняет после пика.

Волна прорыва при клампе амортизируется за 1–2 с — для co-op horror это терпимо и даже художественно (фронт докатывается) [вывод].

**Транспортные факты NfE/UTP** [факт, доки, 2026-06-12]: NfE 6.6 (built-in с 6000.5; standalone 1.12 для 2022.3+); client-hosted listen-server — first-class (`ClientServerBootstrap`, внутрипроцессный IPC); ghost-снапшоты — unreliable, кламп ~MTU 1400 Б, partial по importance, квантование, relevancy-set; **RPC ≤ 8192 Б reliable — не для стрима; DynamicBuffer-ghosts ×32 истории на чанк на соединение — не для стрима**; UTP: фрагментация дефолт 4 КБ (до ~20 МБ unreliable), reliable-окно 32/64 в полёте (≈0.9 МБ/с @50 мс RTT). → **Стрим строим unreliable-sequenced с собственными baseline/ack на чанк, отдельным UTP-драйвером/каналом рядом с NfE.** Официального сэмпла такого канала нет [факт] — это часть рисковейшего допущения №1.

**Профиль тестов сети** [факт, Unity Network Simulator]: «Home Broadband» = 32 мс delay / 12 мс jitter / 2% loss — стандартный профиль стенда для сетевых прогонов.

### 5.5 Дормантность

Регион спит ⇔ нет игроков в interest-радиусе ∧ порталные потоки < ε ∧ нет активных источников/событий, N тиков подряд. Сон: T2→T1 + хинт распределения; пробуждение: сид из T1+хинта (§2.2). Анти-каскад (прорыв будит цепочку комнат): **бюджет одновременных T2-окон** (например ≤64 комнат) с вытеснением по hotness — потомок Gas.Interest [факт-импорт]; тот же приём у SS14 — monstermos-капы 200/2000 тайлов на событие [факт].

## §6. (e) Целостные кандидаты, рекомендация, контракты, риски

### ARCH-1 «Иерархическое поле на слоёной решётке» — РЕКОМЕНДОВАН

Состав: T1-позвоночник + T2-окна (§2) + узкий грид со слоями/фидами/фазами (§3.5) + TopologyDocument-ингест с seal-валидацией (§4) + NfE ghosts для сущностей + кастомный unreliable-sequenced chunk-stream с клампами в контракте (§5.4). Клиенты по газу — render-only (SS14-модель); по сущностям — NfE prediction.

**Почему №1:** единственный кандидат, проходящий R1 числами (§5.2–5.3); R2 — параметр региона by design; R3 — шов доказан двумя провайдерами день-один; R4 — ядро без Unity/transport-зависимостей (guardrail [факт-импорт §8.2]) = переносимый актив. **Содержит подходы C и A как конфигурации** (T2 выключен / T2 минимальными окнами) — не предрешает твой выбор в c-shape-003.

**Цена/слабости:** две репрезентации + реконсиляция (риск №2); фазовая дисциплина требует культуры (смягчение: фазовый порядок ассертится в харнессе, нарушение = красный тест).

### ARCH-2 «Однородное chunked-поле с дормантностью» — №2

Одно мелкое поле везде, чанки спят (SS14-модель, поднятая в 3D). **За:** одна репрезентация, нет хэндоффа; сеть та же. **Против числами:** @25 см — 352 МБ, мимо min-spec; @50 см — 44 МБ терпимо, но wake-каскады без T1-потолка дают full-fine хвосты тика (48–154 мс); дальняя консистентность требует комнатных агрегатов → T1 отрастает сам, но костылём, а не контрактом. **Когда выигрывает:** уровни ×5–10 меньше R1-профиля, или если T2-хэндофф окажется неукротимым. Wire-формат общий с ARCH-1 → миграция в обе стороны дёшева (страховка).

### ARCH-3 «Сервисная сетка без общей ткани» — №3

См. §3.4. Жизнеспособна на 2–3 механиках и малых уровнях; на R1+R4 проигрывает дублями инфраструктуры и O(n²) контрактов. Её дисциплина импортирована поправкой 2 вердикта (b).

### Черновики контрактов (для ARCH-1; имена иллюстративны)

1. `ITopologySource.Generate(seed) → TopologyDocument` (§4.1) — провайдеры: DA-адаптер, ручные сцены стенда, (спайк) PGG.
2. `TopologyAuthority`: `Submit(TopologyCommand) → validate → Commit(revision++)` → feed `TopologyChangeSet`. Единственный писатель топологии; breach = активация surface командой.
3. `IFieldFabric`: `RegisterLayer(schema, resolutionClass, writerSystem) → LayerHandle`; тик: `BeginTick → фазы → Publish()` → change-feeds {dirty chunks, dirty T1 nodes, revision}.
4. `IGasReadModel` (геймплей/рендер/AI): `SampleAt(worldPos)`, `RoomSummary(volumeId)`, `ExposureQuery(actor)` — потребитель не знает ни размера ячейки, ни ступени лестницы.
5. `IGasCommands` (механики → газ): `InjectSource(record)`, `Scrub(record)`, `SetPortalFlowModifier(record)` — командные записи, применяются в фазе газа (форма — modifier-records nucleus-спеки [факт-импорт §8.6]).
6. **Net stream contract**: per-client `InterestSet` (room-based) → пакеты {topologyRevision guard; T1 summaries; chunk deltas (baseRev→rev)}; правило порядка: change-set ревизии доставляется ПЕРЕД полевыми дельтами этой ревизии (update-order [факт-импорт §8.6]); клампы/лестница частот/аварийный режим §5.4 — часть контракта.
7. **Determinism policy**: хост-авторитет снимает межмашинный детерминизм; внутрихостовый replay для стенда = фиксированный порядок фаз + сидированные RNG; float-vs-fixed-point — открытое инженерное решение, архитектуру не блокирует [вывод].

### Рисковейшие допущения (ранжированы — вход для шага 6 шейпа)

1. **Сетевая консистентность chunked-delta при смене топологии на реальном аплинке** — собственный UTP-канал без официальных сэмплов, ревизионный барьер, клампы. Кандидат s-shape-001 ПОДТВЕРЖДЁН, уточнение: тест обязан включать breach-волну и джиттер-профиль 32/12/2%.
2. **T1↔T2 хэндофф**: точное сохранение массы и отсутствие осцилляций на границах окон при wake/sleep циклах (сингл-прецедент реконсиляции есть [факт-импорт]; оконный и сетевой — нет).
3. **Читаемость рендера из 50 см / T1** («каша против потоков») — решает render proof slice (критерий 6 узла); от вердикта зависит ширина T2-окон и судьба конфигурации C.
4. **Герметичность DA-ингеста** (дыры авторинга → газ течёт сквозь стены) — seal-валидация ловит, но цена правок тем — авторская.
5. **PGG**: стоимость интеграции неизвестна (нет API-доков) — спайк за швом.
6. **Каскад пробуждений** на breach-цепях — арифметика в норме, поведение не доказано (бюджет окон + вытеснение).
