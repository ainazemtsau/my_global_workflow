## §1. Требования — эхо-фиксация (R1–R4 из s-shape-002; R5–R11 — твой голос 2026-06-12)

- **R1** — огромные процедурные уровни; доказательство масштаба обязательно.
- **R2** — размер газовой ячейки = конфиг (рабочие точки 25/50 см).
- **R3** — Dungeon Architect день-один через заменяемый адаптер; PGG оценить.
- **R4** — сим-ядро = актив, переживающий концепт; продуктовые контракты.
- **R5** — мета-газы = архетипы поведения (огненные, ядовитые, электрические…), внутри — параметрические газы; особые газы: не подчиняются симуляции (как враги, скриптовые/агентные) или подчиняются с особыми эффектами; живут в тех же контрактах поля/сети/рендера.
- **R6** — симуляция везде и всегда; дальше — грубее; никогда не заморожена.
- **R7** — тяжёлые вниз, лёгкие к потолку — представлено и в грубом дальнем уровне; критично в высоких пространствах.
- **R8** — ангары/длинные коридоры/шахты: огромный объём ≠ один перемешанный узел; грубый уровень делит большие объёмы.
- **R9** — разрешение открыто вниз: 25 см может быть избыточно или недостаточно; деление мельче возможно; «поклеточно вокруг игрока» — вариант.
- **R10** — правдоподобное восстановление: грубое состояние хранит пространственные хинты (сторона заполнения, откуда утечка); рядом с игроками — максимальная детальность; телепорт-кейсы могут деградировать.
- **R11** — реакции везде и всегда: смешивание реагирует (взрыв/заморозка/превращение) на любом удалении; последствия — геймплейные события независимо от игроков.

Конкретный список метатипов и геймплей газов — узел g-d3a8, не этот бриф.

## §2. Как проверялось (кратко, для доверия результату)

V1-бриф (7 веб-агентов, 8 архивных доков) → **6 независимых опровергателей** по каждому вердикту v1 под R5–R11 → **5 исследователей дыр** (стратификация, большие объёмы, реакции, восстановление, спец-газы) → **3 судьи** с разными линзами по главному вопросу «чем быть грубой правде» (A секторно-полосный граф / B макро-воксели / C комнатный патч) → **критик полноты** → слитый дизайн → **4 враждебных ред-тимера** по слитому дизайну (численная физика; контракты против твоего A1-аудита; реализм аппетита; независимый пересчёт чисел) → исправления → этот бриф. Итого 26 агентов в двух воркфлоу + перекрёстные пересчёты. Все вердикты опровергателей: `holds_with_amendments`; ред-тим: 2×`needs_rework` (физика — два фатальных дефекта, вылечены; аппетит — день-один был ×2, вырезан), 1×`sound_with_fixes` (контракты), арифметика — 5 кластеров чисел исправлено.

## §3. Итоговая архитектура ARCH-1v2 — «секторно-полосный граф + клипованные окна»

### 3.1 Грубый уровень (T1s) — всегда, на всём уровне

**Секторизация (compile-time, в ядре-компиляторе, не в адаптере):** объём из TopologyDocument делится по архетипу и порогам (родословная — CFAST Table 2 [факт, G2]): L/W>3–5 → коридорная цепочка (сегменты 1–2 ширины); W/H<0.2 → шахтный столб; горизонтальный пролёт >8–12 м → решётка 4–8 м, клипованная к геометрии объёма; высота >4–6 м → z-слои; объём >~1000–1200 м³ → делить (кап Stationeers — прецедент [факт v1]). Обычная комната = 1 сектор (вырожденный случай — это конфигурация C). Клип-«сливеры» <~20–25% номинального объёма вливаются в соседей (стандартное cut-cell лечение). Sector id = volumeId × индекс решётки — детерминирован, стабилен навсегда.

**Состояние секторной колонны:** 2–3 вертикальные полосы — **общая геометрия для всех газов** (z-слои колонны), на полосу: массы по видам (K — конфиг) + энергия/температура. Класс плотности — только ключ сортировки, **физика — по суммарной плотности смеси полосы** (молекулярные веса + температура): горячий CO₂ корректно всплывает. Interface-высоты — **производные** (пересчитываются из масс полос при каждом коммите; полос−1 на колонну), используются для суб-полосной точности чтения/сидирования/рендера, никогда не самостоятельная правда. Хинты R10: знаковые аккумуляторы притока по порталам — текущие интеграторы с экспоненциальным затуханием (~десятки секунд; хинт = свежесть) + доли заполнения полос. [Лечения RT1 MAJOR-3/4, MINOR-3.]

**Оседание/плавучесть:** экспоненциальная релаксация к аналитически вычислимому локальному стратифицированному равновесию: Δm = (m_eq − m)·(1 − e^(−dt/τ)) — класс локально-неявных положительно-сохраняющих интеграторов (Patankar): безусловно стабильно, монотонно, работает одинаково при dt=0.1 с и dt=5 с; равновесие колонны (плотнее — ниже) — точное, обмен 2–3 полос — тривиальная система. Чекерборд/перелив холодного режима убит схемой, не тюнингом. [RT1 MAJOR-1.] Тест стенда: «10×10 ангарная решётка, полузаполненная тяжёлым газом, 1000 тиков @0.2 Гц: нет смены знака потока >2 тиков подряд, total variation убывает».

**Замыкание по давлению (НОВОЕ — фатальный дефект v2-черновика):** на сектор — алгебраическое давление (идеальный газ: суммарная масса+энергия против ёмкости); потоки по порталам — орифисное уравнение от Δp на высоте портала (sillZ + гидростатический профиль полос; CONTAM/AIRNET-родословная [факт, G1/V3]). Решается квазистационарно каждый базовый тик (red-black Gauss-Seidel по ~3.3k узлов — в бюджете, отдельная строка). **Решение потока всегда на базовых 10 Гц, исключено из любых лестниц частот** (эллиптическую сеть нельзя решать по регионам с разными частотами без рождения/уничтожения массы). Герметичный сектор с источником копит «pressurized»-скаляр → его едят venting/breach-логика. Тесты: «sealed room + source» (отток появляется с открытием портала), «multirate two-room conservation» (дрейф массы <1e-9/час). [RT1 FATAL-1.]

**Поток по порталу считается один раз на грань** (детерминированный владелец грани), применяется антисимметрично обоим концам — никаких пер-сторонних оценок. **Транспортный коммит:** все списания с (полоса, вид) за тик — порталы, оседание, entrainment — агрегируются и пропорционально масштабируются при нехватке (та же политика, что у реакций); реакции читают доступность после транспорта (фазовый порядок это даёт — зафиксировано инвариантом). [RT1 MAJOR-6, MINOR-2.]

**Частоты:** день-один — весь грубый уровень на 10 Гц (по исправленным числам это 0.2–2.5 мс ST — дёшево); лестница частот по hotness (10/5/1 Гц) — механизм масштабирования для ставки-2, с **правилом пола**: сектор с активной реакцией ≥1 Гц (внутренние подшаги ≤4–8), эмиттер прорыва → 10 Гц; капы реакций — в единицах «в секунду», не «за тик». R6 «никогда не заморожено» выполняется буквально. [RT1 MAJOR-2.]

### 3.2 Реакции и план событий (R11)

Один декларативный реестр (донор семантики: tg/SS14 gas_reactions [факт, G3]): требования в **интенсивных** величинах (доли концентрации + температурное окно — переносимы между ячейкой 0.015 м³ и полосой 50 м³), гистерезис порогов (паттерн ONI +3K/−1.5K [факт]), капы скорости в секундах. Вычисляется **ровно на одном уровне на сектор за тик**: T2, если сектор покрыт окном (T1-оценка маскируется), иначе T1s по полосам — двойной счёт запрещён правилом эксклюзивности. [RT2 M3.] Продукты — стехиометрические дельта-батчи через **атомарный мульти-слойный коммит**: арбитр (политика газ-домена, не фабрики) собирает ВСЕ батчи, сортирует по стабильному ключу, применяет с clamp+пропорциональным масштабированием СОВМЕСТНО на (полоса, вид) — детерминированно. Батчи несут **энтальпию + радиационную долю χ_r≈0.3**. Гистерезисные защёлки — sidecar-состояние газ-домена, входят в снапшот; правило переноса при open/close окна (полоса→ячейки сеется, ячейки→полоса сворачиваются по правилу). Реакция, меняющая класс вида, просто редактирует смесь полосы — рассинхрон невозможен (interface производный).

**План событий (контракт #8):** типизированные события-последствия (взрыв/заморозка/превращение: место, магнитуда, виды) собираются в фазах реакций, публикуются в ревизионированный, надёжный, **независимый от близости игроков** хостовый фид (механизм фида — generic в фабрике, схема — газ-домен); исключён из hotness-вытеснения и лестниц. До клиентов последствия едут через NfE ghosts/RPC, никогда через газ-стрим. События **самодостаточны** (потребитель не обязан иметь полевое состояние — нормативно: события системно обгоняют lossy-чанки) и несут ревизию публикации; late-join = baseline на ревизии R + события >R. Потребитель последствий (разрушение/геймплей) планируется между фазой consequences тика N и topology commit тика N+1; задержка прорыва после взрыва: **сим ≤1 тик (100 мс), воспринимаемая игроком 100–300 мс** (плюс стрим/интерполяция — честно в контракте). [RT2 M6/F6/F10, RT4.]

### 3.3 Спец-газы (R5) — контракт драйвера вида (#9)

Per-species `transportMode`: **FIELD_TRANSPORTED** (дефолтный солвер ИЛИ кастомный драйвер) | **AGENT_SUBSTRATE** (существо/спрэдер: отдельный пайплайн, общая топология + общий план событий; рендер может отличаться). Драйвер: единственный писатель своих слоёв (партиция писателей по (слой×фаза)), эмитит **скалярные** source/sink-записи в фазе драйверов (применяет один драйвер-аппликатор в конце фазы, тем же тиком, до реакций), читает мир через `IDriverReadView` (фазовая копия после T1s-транспорта — отдельно от публичного `IGasReadModel`). Прецедент дважды [факт, G5]: SS13 pyroclastic anomaly и SS14 GasProducerAnomaly — и скрипт, и одержимость игроком, поверх того же атмоса. Маска участия в дефолтном транспорте — на вид. **Направленная скорость (velocity-intent) — НЕ день-один**: беспрецедентно поверх сетевого поля, спайк-гейт под риском №1. Возможности драйвера декларируются при регистрации (виды, скоупы чтения/записи, видимость по уровням, класс детерминизма, бюджет); состояние драйвера — сериализуемый блоб (снапшот). Агентный субстрат взаимодействует с реестром только через драйверный путь. [RT2 M2/F2/F9, G5.]

### 3.4 Фазы тика и таблица писателей (детерминизм)

`P1 commands → P2 topology commit → P3 T1s (pressure/flow solve + транспорт + relaxation-оседание) → P4 drivers → P5 reactions(T1s) → P6 T2 solve+reconcile+reactions(T2) → P7 consequences → P8 publish`. Писатель на (слой×фаза) — ровно один, таблица публикуется, фабрика держит write-leases с ассертами (нарушение = красный тест): P1 командный аппликатор; P2 TopologyAuthority; P3 T1s-солвер; P4 драйвер-аппликатор; P5 арбитр реакций (атомарный коммит); P6 T2-солвер + реконсилер — **единственный** ап-пропагатор полосных дельт T1s; P7–P8 read-only. Правило отложенного коммита: команды, рождённые в тике N (реакция→брешь, геймплей), коммитятся в P2 тика N+1; каскады — шаг за тик (преднамеренный троттлинг, monstermos-родословная). Внешняя ревизия одна = published tick; внутритиковые коммиты — подшаги. [RT2 M2/F2, V2-A8.]

### 3.5 Тонкий уровень (T2/T3) — окна вокруг игроков и событий

Окно = **клипованный бокс** вокруг игрока/события (r≈8–12 м), не целый объём (целый ангар @25 см = 768k ячеек — запрещено арифметикой); **градиент разрешения внутри окна обязателен** (ядро 25 см ~8×8×6 м + оболочка 50 см): окно игрока = 47–68k ячеек; 4 игрока врозь = 190–270k [вывод RT4]. `resolutionClass` в документе = верхний кап объёма; фактическое разрешение — рантайм. **Бюджет окон в ячейках**, день-один: **75–100k суммарно** (газ ≤4 мс) или 100–160k (если хосту выделено 6–8 мс) — до ре-бенча якоря (§4); wire-ключ чанка включает ступень разрешения. T3 «hero-пузырь» 12.5 см (≤64–100k ячеек, r 3–6 м) — статья parked: контракт совместим (чанк фиксирован в ячейках), строится по измеренной нужде.

**Инвариант реконсиляции (лечение фатального дефекта №2):** масса полосы == сумма ячеек окна внутри полосы + **явный remainder-bucket** (первоклассное поле состояния на полосу). Движение окна переносит массу cell↔remainder консервативно (точно до бита); потоки через границу окна (нефизическая поверхность внутри сектора) двигают fine↔remainder, не создают. Тест: «window drag» — 10 минут ходьбы по ангару, дрейф массы <1e-9. [RT1 FATAL-2.]

**Анти-осцилляция (контракт, не тюнинг):** двойные пороги (sleep строго ниже wake) + dwell-счётчики + буферное кольцо в 1 сектор (AMR-практика + excited-groups родословная). **Сидирование:** по занятости полос (тяжёлое содержимое — снизу, лёгкое — сверху, от производных interface-высот) с уклоном к порталам притока (хинты); burn-in ≤1 скрытый тик/кадр (67–133 мс скрытой латентности — иначе спайк 6–45 мс в кадре [вывод RT4]); реакции и события в burn-in подавлены; burn-in кончается, когда инвариант сумма==полоса+remainder сошёлся.

### 3.6 Сеть — базовый режим (не «на случай аварии»)

Стрим — собственный unreliable-sequenced UTP-канал рядом с NfE (транспортные лимиты — v1 §5.4 [факт]). **База:** deadband на квантованной дельте + min-интервал пересылки чанка + hotness-очередь + лестница частот 10→5→3 Гц + кламп на клиента ~150 КБ/с + **агрегатный кламп хоста ~250–300 КБ/с на весь газ** (пропущенное v1 число: 3×150 КБ/с = 3.6 Мбит/с — связывало бы аплинк раньше всех остальных лимитов [вывод RT4]). Дальний уровень: полосные сводки (десятки байт/сектор) ~1 Гц **со своим deadband'ом и лестницей** (пол «все грязные» = 79 КБ/с — без deadband цифра ≤10 КБ/с не самоисполняется [вывод RT4]). Прорыв в ангаре — штатный staged-путь: фронт секторным туманом из сводок (≈мгновенно, ≤3 с свежесть), T2-чанки догоняют по hotness — честные цифры догона: **1–10 с** на окно игрока (256 чанков) в зависимости от измеренного размера чанка; полный ангар не догоняется никогда (и не должен). **Честный размер грязного чанка: 0.6–6 КБ** (300–600 Б из v1 верны только при ≤7% грязных ячеек на отправку) — измерение в первом же сетевом спайке, до него стрим не бюджетируется. Квантованные значения никогда не пишут правду хоста (one-way assert). Fallback Barotrauma-гибрида (клиентская локальная диффузия + пороговые коррекции [факт v1]) — контингенция с tripwire: измеренный стационарный dirty-rate > половины клампа.

### 3.7 Procgen-ингест (TopologyDocument v0.2) и DA/PGG

Поля v0.2: volumes {id, archetype (+hangar|openSpace), bounds AABB + **occupancy** {basisCellSize≈50 см, bitset|RLE} — **опционально день-один**, обязательно со ставки-2 для больших/некоробочных (его потребитель — секторизация Stage-1; держать обязательным день-один — мёртвый груз [RT3]), freeVolume, floorZ/ceilingZ}; portals {center, normal, width, height, **sillZ**} (высота нижней кромки — несущая семантика: дверь у пола вентилирует тяжёлый газ первым); surfaces {centroid, normal, extent, **zRange**, breachable}; anchors; subdivisionPolicy (пороги §3.1). **Секторизация — геометрическая и только геометрическая** (секторы, z-слои, sillZ): полосы плотности — газ-домен, по sector id (топология не знает газовых конструктов [RT2 M7b]). Прорыв добавляет **вертикальные порталы** (геометрия, не «полосы»); id breach-портала = (surfaceId, порядковый номер) — детерминирован, в снапшоте и change-set'ах. Объёмы неизменяемы; слияния = высокопоточные порталы + правило связывания interface-высот через них (релаксация к общему уровню) — без рантайм-ресекторизации. Seal-валидация: GraphClosure (hard) / ColliderAudit (report-only → калиброванный гейт по golden-сценам со вшитыми декоративными щелями) / PortalOcclusion (hard).

**DA:** день-один — **наивный** адаптер (GridDungeonModel cells→volumes/portals, sillZ из геометрии дверей, Build()-smoke на ~100–300 объёмов) — агентный параллельный трек ~1 неделя; occupancy-воксализация, маркер-харвест вентов/шахт, гейт 1000 объёмов на min-spec — ставка-2. Vendor-риск (Unity-ветка стейл 18.5 мес, Unity 6 не заявлен [факт v1]) — теперь кандидат в критический путь графика, не «дешёвый шов»: проверка на первом же ингесте. Vertical-данные из API DA **не извлекаются** (Cell.Bounds — 2D X-Z [факт V5]) — потолки/высоты идут из authoring-конвенции тем + occupancy. **PGG:** вердикт v1 стоит (за швом; ре-проверка 2026-06-12: v1.6.6.2.12 от 2026-02-21, API-референса нет [факт V5]); планка спайка поднята: «заполняет ли выход PGG **амендованный** документ (граф+occupancy+вертикаль)»; tripwire: публикация API / выход из беты → переоценка.

### 3.8 Термика (минимум, но честный)

Температура/энергия — поле состояния день-один (реакционным окнам нужно что читать); **обязательный lumped wall-node на сектор** (T_wall + две проводимости газ↔стена, стена→среда; точная экспоненциальная формула, стабильна при dt=5 с) — конфиг может занулить проводимости, но поле в схеме Stage-0, иначе R4-контракт ломается при доделке (герметичный сектор с экзотермией иначе греется вечно — пороги защёлкиваются [RT1 MAJOR-5]). **Plume entrainment** — поле реестра реакций (огонь в полосе i качает окружающую массу полосы i в i+1 пропорционально скорости реакции; линейный коэффициент; без него верхняя полоса заполняется в 5–15× медленнее) — активация в ставке-2. Динамический теплообмен между секторами — ставка-2 (день-один хватает статического/скриптового поля температуры [RT3]). Radiation-at-distance — за именованной точкой расширения реестра.

### 3.9 Шесть компонент (граница против god-object — аудит по A1)

A1 запрещает god-object [факт, архив]; слитый дизайн начал в него сползать (политика в фабрике) — раскладка после аудита [RT2 M1/F1]:

1. **TopologyCore** — ингест, seal-checks, геометрическая секторизация, стабильные ID, command→validate→commit→revision, change-sets. Никакой газовой семантики.
2. **FieldFabric kernel (механизм, не политика)** — chunked-хранилище слоёв, реестр слоёв, ревизионные change-feeds, фазовый планировщик с именованными слотами, generic транзакционный мульти-слойный коммит, generic ревизионный надёжный фид, write-leases.
3. **GasDomain** — T1s/T2/T3 солверы, реконсиляция+remainder, реестр реакций + арбитр конфликтов, реестр видов + драйверы, схема и публикация событий-последствий, термика, полосная схема (классы плотности), wake/sleep-правила.
4. **Interest/LOD-сервис** — hotness, частотные ярусы, размещение/бюджет окон (ест фиды фабрики + позиции игроков). Политика не живёт ни в фабрике, ни в сети.
5. **NetAdapter** — deadband/квантование, per-client interest, лестница отправки, клампы (per-client + агрегатный), полосные сводки, события→NfE ghosts/RPC, revision-order guard, join-baseline.
6. **UnityAdapters** — DA-ингест, рендер-потребители, NfE-мост.

**Контракты (полный набор, 13):** №1 ITopologySource · №2 TopologyAuthority (+ правило id breach-порталов) · №3 IFieldFabric (kernel) · №4 IGasReadModel (published-ревизия; разрешение точки = самый тонкий открытый уровень T3>T2>T1s; **одна нормативная формула** суб-полосной интерполяции (массы полос + interface → концентрация на высоте z) — её же использует сидирование, чтобы ИИ вдали и игрок в окне получали один ответ «летально ли здесь»; ExposureQuery = интеграл по объёму актёра той же формулой) + IDriverReadView (фазовая копия) · №5 IGasCommands (+валидация, семантика дефералов) · №6 NetStream (чанки с ключом разрешения, сводки, лестницы, клампы, порядок событий-vs-дельт, join) · №7 DeterminismPolicy · №8 EventPlane · №9 SpeciesDriverRegistry (+стабильные целочисленные id видов в конфиге — не порядок объявления) · №10 ReactionRegistry (пороги/гистерезис/эксклюзивность/арбитр) · №11 Interest&Hotness · №12 **SnapshotState** (версионированный: массы полос, remainder'ы, хинты, sparse-карты видов, защёлки гистерезиса, dwell-счётчики, отложенные команды, блобы драйверов, RNG-стримы, ярусы частот; T2-окна — derived, исключены) · №13 **ReplayJournal** (все команды + топология + **per-tick решения interest/размещения окон как записанные входы** — иначе детерминированный реплей невозможен в принципе, ибо правда зависит от того, где открывались окна [RT2 M4]). Acceptance: record→replay бит-в-бит по T1s на golden-сцене. Переносимость R4 определена: Unity-package-portable; Burst/Collections изолированы за job-слоем.

## §6. Альтернативы — почему именно это (вердикты трёх судей)

- **B «однородные макро-воксели» (1–4 м): 3.5–4 балла.** Фатально: в генератор-агностичной геометрии DA/Snap стены 10–20 см и плиты ~15 см лежат ВНУТРИ ячеек — пер-граневая проницаемость не кодирует внутриячеечную перегородку: газ сочится сквозь стены и **тонет сквозь перекрытия** (убийца R7 и доверия игрока — хоррор про чтение газа против видимой архитектуры); честное лечение (суб-ячеечная принадлежность комнатам) — ЭТО и есть конформная секторная декомпозиция, т.е. построить A после оплаченного крюка. Stationeers спасает только то, что их мир буквально построен из 2-м фреймов (свойство конструирования мира, не симуляции) [факт V6]. На 2 м обычная комната 3 м = 1–1.5 вертикальной ячейки — стратификации нет там, где игроки живут. Flip-условие: лок генерации на решётко-выровненный билдер + измеренный 1-м тик ≤2 мс.
- **C «комнатный граф + патч стратификации»: 5.5–6.** Честен в обычной комнате, но: длинный коридор остаётся одним узлом — **наступающий фронт газа** (главный читаемый хоррор-бит) не существует на грубом уровне; «виртуальные комнаты по исключению» фрагментируют идентичность объёма по швам (грязные контракты против R4); унимодальный хинт кладёт газ не с той стороны мультипортальной комнаты (ровно твой R10-пример); и его дешевизна — иллюзия: без грубого деления близкая фидельность в ангаре оплачивается полным T2-окном (768k ячеек). Накопительная траектория патчей сходится к A по кускам, с долгом. Flip: уровни ×5–10 меньше / R8-R11 ослаблены / спайк A-ингеста провалился за таймбокс.
- **A (ARCH-1v2): 8.5–8.7 у всех трёх судей** — единственный вариант, чья кривая стоимости повторяет структуру уровня (3 полосные ячейки на обычную комнату, решётка только в десятках больших объёмов), удовлетворяющий R6–R11 структурно в бюджете. Дополнительная страховка: C — вырожденная конфигурация A (1 сектор × скаляры/2 полосы), wire совместим с B-полем — миграции дёшевы в обе стороны.

**Buy-vs-build (обновление v1):** негатив стоит — купить нечего [факт, 5 углов + новый: building-science свип]. Новая строка рисков: стратифицированный мульти-видовой зонный солвер на игровом тике — **новый синтез без прямого донора**.

## §7. Рисковейшие допущения (пере-ранжированы; каждое — с именованным тестом)

1. **Сетевая консистентность кастомного chunked-delta под breach-нагрузкой на реальном аплинке** (свой UTP-канал без официальных сэмплов; честный чанк 0.6–6 КБ неизвестен). Тест: спайк недель 1–2.5 — примитивное поле, хэш-сверка хост+2 клиента, breach-волна, профиль 32/12/2, оба клампа; **первое измерение — размер грязного чанка**.
2. **Численная устойчивость полосного солвера на игровом тике** (стиффовость зонных уравнений; нулевой прецедент). Смягчено схемой (exponential relaxation + квазистационарное давление — безусловная стабильность по построению), но тесты обязательны: sealed-room+source; multirate-conservation; cold-rung checkerboard; interface-crossing; hot-CO₂ inversion (горячий тяжёлый газ детализуется К ПОТОЛКУ); Steckler-style комната (интерфейс монотонно опускается, sealed-fire остывает после выгорания).
3. **Неизмеренный якорь пропускной способности** (×2–4 множитель K; якорь уже jobbed). Первая инженерная задача: микро-бенч K-видового SoA-ядра на min-spec; до него бюджеты T2 — консервативные (75–100k).
4. **T1s↔T2 хэндофф** (remainder-bucket лечит инвариант; динамическое открытие/закрытие окон — наша новизна, статическая связка зон↔CFD в литературе есть [G4]). Тесты: window-drag 1e-9; эксклюзивность реакций; перенос защёлок гистерезиса.
5. **Согласованность исходов реакций между уровнями** («LOD-flip взрыв» при подходе игрока). Лечение: эксклюзивность + консервативные границы + гистерезис; достижимость в принципе не доказана — fallback-контракт заявлен: реакции авторитетны на грубом уровне, тонкий их рендерит.
6. **DA-адаптер как критический путь** (стейл-вендор; vertical из API не извлекается — конвенции тем; Build()-масштаб не доказан). День-один — наивный скоуп; gate 1000 объёмов — ставка-2.
7. **Читаемость полосного состояния в рендере** (клип из полос — гипотеза slice; perception ceiling — вопрос §8e).
8. **PGG** — за швом, планка спайка поднята (амендованный документ).

## §8. Твои решения (батч для c-shape-003; каждое с рекомендацией — G7)

a. **R6 и частоты:** считается ли «никогда не заморожено» выполненным при холодном ярусе 1 Гц (день-один весь грубый уровень на 10 Гц; лестница — масштабирование ставки-2)? — *Рекомендация: да; правило пола (реакции ≥1 Гц, прорыв 10 Гц) делает лестницу безопасной.*
b. **Латентность последствий вдали:** сим ≤100 мс, воспринимаемая 100–300 мс — приемлемо? — *Рекомендация: да; жёстче — дороже на порядок.*
c. **R7-представление:** 2–3 полосы + производная interface-высота = «стратификация представлена» (N полос — конфиг эскалации)? — *Рекомендация: да; полносеточная вертикаль вдали не нужна и не влезает.*
d. **Спец-газы день-один:** скалярные source/sink-драйверы (прецедентны), направленная скорость — спайк-гейт? — *Рекомендация: да (velocity поверх сетевого поля беспрецедентна — связана с риском №1).*
e. **Perception ceiling:** видна ли дальняя пространственная структура в хоррор-освещении вообще? Влияет на ширину дальней детализации. — *Рекомендация: проверить в render slice (недели 2–4) — бесплатно по ходу клипа.*
