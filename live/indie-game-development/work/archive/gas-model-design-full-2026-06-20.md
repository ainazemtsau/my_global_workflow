## A. Резюме на один экран

1. **Недовольство было про МОДЕЛЬ, не про картинку.** Тебе не хватало того, что газ УМЕЕТ и ГДЕ он точно находится — а не того, как он выглядит. Поэтому дешёвая «перекраска рендера» проблему бы не закрыла.
2. **Решение — дорога A+, не B.** A+ = оставить графовую основу (то, что уже построено и доказано по сети) и **нарастить на неё слои** (точный фронт, локальное зерно, урон от реального газа, реакции, разрушаемость). B = переписать всё в непрерывное поле — отвергнуто: оно заново открывает самую дорогую решённую проблему (детерминированная репликация по сети для коопа).
3. **Это УКРЕПЛЯЕТ уже ратифицированную архитектуру, а не сносит её.** Многое из конструкции либо уже построено, либо уже в плане Волны 3 (привязка — §K, каждый пункт сверен с NOW.md; это слой Claude, не из ресёрча).
4. **Дилбрейкеров нет**, но **перед ставкой обязательны нагрузочные пробы** (§G) — каскад взрывов, 8 «горячих» игроков, целочисленный Дейкстра на проломе, сохранение массы. Это дёшево и обязательно.
5. **«Убрать ёмкость+перелив» переосмыслено** (§J): это не баг, а видимое лицо настоящего давления; убирать без закона-замены нельзя. Настоящая боль («потерял ГДЕ газ») лечится фронтом+зерном.

## B. Словарь (простыми словами)

| Термин | Что значит на простом языке |
|---|---|
| **Узел / комната-кусок** | Одна ячейка хранения газа. Обычно = комната; сложная комната заранее порезана на несколько выпуклых «кусков». |
| **Ребро / проём** | Связь между двумя узлами, через которую газ перетекает: дверь, окно, вент, дыра, шахта. |
| **sill (высота порога)** | Высота, с которой проём начинает пропускать газ. Низкая дверь пропускает тяжёлый газ (он внизу), высокое окно — лёгкий (он вверху). Совпадает с ратифицированной моделью репо (детали — §K). |
| **Слой высоты / band** | Внутри узла газ разложен по 2–3 уровням высоты: пол / середина / потолок. (Текущее состояние репо — §K.) |
| **Грубый слой («вода»)** | Дешёвая основа: «сколько газа в каждом узле, по слоям». Считается везде, всегда, дёшево, синхронно по сети. |
| **Детальное зерно / тир** | Мелкий точный расчёт (клетки ~25 см–1 м), который поднимается ТОЛЬКО рядом с игроком/событием. |
| **Пузырь интереса** | Локальная область детального зерна вокруг игрока или активного события. |
| **Фронт** | Передний край распространения газа — «докуда дошло». Тонкая подсказка поверх массы. |
| **Геодезическое расстояние** | Расстояние «по проходимому пути» (через двери, в обход стен), а не по прямой. «Сколько идти», не «по линейке». |
| **Дейкстра** | Классический алгоритм поиска кратчайших путей по графу. Тут — чтобы честно мерить «докуда дошёл газ» сквозь двери, а не сквозь стены. |
| **Lockstep** | Все считают один и тот же шаг симуляции синхронно; никто не «опережает». |
| **Детерминизм / бит-в-бит** | Из одних и тех же входов — РОВНО один и тот же результат на любой машине. Обязательно для честного коопа. |
| **Фикс-точка vs float** | Фикс-точка/целые числа считаются одинаково на любом железе; float (дробные) могут чуть-чуть разойтись → ломают бит-в-бит, если ими хранить состояние. |
| **source-of-truth** | «Единственный источник правды». Когда правда одна, не бывает «урон говорит одно, картинка другое». |
| **Eikonal / Newton-Raphson** | Математические «непрерывные» методы (фронт волны, решатель давления). Точные, но используют корни/дроби → молча ломают бит-в-бит. В исходнике помечены как «НЕ так». |
| **Конфиг vs правило** | Конфиг = числа, которые крутим ради баланса. Правило = закон корректности, который НЕ крутим (иначе ломается честность). |
| **Сон узла** | Узел перестаёт считаться, когда в нём ничего не происходит — экономия. |
| **Каскад** | Цепочка вторичных взрывов/реакций, ползущая по уровню. |

## C. Главный вердикт (развёрнуто)

**Картинка или модель?** Разбор однозначно показал: **модель**. Недовольство было не про «некрасиво», а про то, что газ ведёт себя слишком крупно — нет точного места, нет наступающего края, нельзя по-настоящему «отравить вон тот угол». Это свойства МОДЕЛИ (что она умеет и с какой точностью), и перекраской их не лечат.

**Почему A+, а не B.** Был соблазн пойти в «непрерывное поле» (B) — переписать базу целиком. Разбор от этого отговаривает по одной жёсткой причине: **сеть**. У нас кооп без сервера — **один игрок хостит**, остальные должны видеть РОВНО то же самое. Графовая основа (целые числа, перелив по рёбрам) синхронизируется бит-в-бит дёшево. Непрерывное поле произвольной детализации синхронизировать дёшево И детерминированно — это нерешённая дорогая проблема. B = заново открыть её и переписать уже доказанные Волны 1–2 ради того, что можно получить дешевле.

**A+ = граф-основа + слои.** Вместо «снести и переписать» — оставить дешёвую честную основу и **нарастить** недостающее: точный фронт, локальное детальное зерно у игрока, урон от реально стоящего газа, реакции при подтверждённой встрече, разрушаемость через рёбра. Каждый слой ложится НА существующую архитектуру, не ломая сетевую гарантию. Отсюда «+»: это не старая модель, это она же, дорощенная до твоей цели.

**Вывод:** конструкция **укрепляет** ратифицированную архитектуру репо (многозонный граф + полосы + sillZ-проёмы + реконструкция), а не отменяет её.

## D. Полная конструкция

### D.1. Мир и резка геометрии

Мир собирается из **осевых прямоугольных коробок** по сетке (25/50 см): шахты, коридоры, ангары, этажи, ямы, помосты — всё это коробки. Никаких произвольных форм на уровне симуляции.

Сложную комнату (Г-образную, ангар, комнату с двумя входами) **режут на выпуклые куски ещё при генерации уровня** — это детерминированные данные, не рантайм-вычисление. Важно: куски ОДНОЙ комнаты **полностью открыты друг другу** — резка это разрешение «здесь свободно перетекает», а не постановка стенки. Стенки (проёмы с порогами) — только между РАЗНЫМИ комнатами.

*Зачем:* выпуклые куски нужны, чтобы «докуда дошёл газ» внутри комнаты считалось честно и чтобы фронт огибал угол Г-образной комнаты, а не шёл сквозь стену. Резка на генерации (а не в рантайме) убирает целый класс багов «как поделить массу, когда геометрия меняется на лету».

### D.2. Основа = «вода»: сохраняющийся газ, текущий по графу

Это сердце модели. **Газ ведёт себя как вода в сообщающихся сосудах.**

- Каждый **узел** (комната или кусок) хранит **целое количество** газа, разложенное по 2–3 слоям высоты, **по каждому типу газа отдельно**.
- Каждый **проём** = ребро со свойствами: **высота порога (sill)**, бит «открыт/закрыт», **проводимость (ширина)**. Низкая дверь пропускает тяжёлый газ (он внизу); высокое окно — лёгкий (он вверху); вертикальный проём (шахта, дыра в полу) = тяжёлый идёт вниз и лёгкий вверх встречными потоками.
- **Правило одного тика** (раз в ~секунду или по событию): через каждое открытое ребро перелить **ограниченную долю** газа из узла, где его уровень ВЫШЕ, в узел, где НИЖЕ. Газ при этом **только перемещается** — не создаётся и не исчезает. Поэтому суммарная масса сохраняется **точно, бит-в-бит**. Перелив останавливается сам, когда уровни сравнялись («нашёл уровень»).

**Главная красота:** из этого ОДНОГО правила само собой получается всё поведение, без отдельного кода под каждый случай:

- налив комнаты;
- перелив через высокий порог только когда дойдёт до его высоты;
- «газ нашёл свой уровень» и замер;
- слив / вентиляция / утечка через пролом;
- **отступление** газа (если источник убрали — газ растекается обратно и разбавляется);
- «источник выдохся».

Никакого спец-кода «а теперь сливаем» — всё это один и тот же перелив воды.

Частные случаи той же механики:
- **Яма** = ловушка тяжёлого газа: копится до края, потом переливается дальше.
- **Помост** = сухая высокая «земля», куда тяжёлый газ не достаёт.
- **Источник** = кран: может быть конечным (выдохнется) или бесконечным — поддержаны оба.

**Сон:** узел, в котором ничего не перетекает И нет живого взаимодействия, засыпает (не тратит расчёт). Подробнее — D.5 (сон по интересу) и D.6 (что такое «живое взаимодействие» = урон/реакции).

### D.3. Плавучесть, слои, температура, вентиляция

- Каждый газ **ищет свой уровень по весу**: тяжелее воздуха → оседает на пол, легче → поднимается к потолку, нейтральный → середина. Сами **слои высоты** (пол/середина/потолок) и есть представление «вверх-вниз».
- Слой **не прибит навсегда**: на уровень влияет вес **плюс температура**. Нагрел тяжёлый газ — он всплыл; остыл — осел обратно. **Температура — отдельный тонкий слой** поверх (текущее состояние репо — §K).
- **Вентиляция (позже)** = «труба с насосом»: принудительный поток на конкретном ребре поверх естественного перелива; может пересилить плавучесть (высосать тяжёлый газ вверх). Важно: она **встаёт на тот же граф рёбер без переделки** — это просто ещё один модификатор потока на ребре.

### D.4. Фронт = где сейчас передний край

Масса (сколько газа в узле) — это правда, но она не говорит «докуда визуально дополз газ». Для этого — **фронт**:

- Считается **целочисленное геодезическое расстояние** от источника: Дейкстра по графу проёмов, с учётом стен и высоты порогов. То есть «насколько далеко по проходимому пути».
- Тогда «фронт дошёл до точки X» = **одно сравнение** (расстояние ≤ пройденного фронтом). Дёшево.
- **Сквозь стену фронт не врёт**: если между точками нет ребра, расстояние большое — газ «не дотянулся».
- Фронт — **тонкий слой ПОВЕРХ массы**: масса = авторитетная правда, фронт = вспомогательный край для картинки и для «докуда дошло».
- Внутрикомнатный 1-мерный «докуда дополз» — **не-авторитетная подсказка** (масса = правда), поэтому его неточность безопасна.

### D.5. Зерно и сон — по ИНТЕРЕСУ, не только по игроку

- **Холодный узел** (нет игрока, нет интереса) живёт ТОЛЬКО на грубой «воде» — дёшево.
- **Детальное локальное зерно** (~1 м рядом с «интересом», ~25 см рядом с игроком) поднимается **только там, где нужно**, и кормится **реальным фронтом** (не «средним по комнате», размазанным числом).
- Узел **не засыпает, пока идёт живое взаимодействие** — то есть пока газ реально что-то «обрабатывает» здесь (наносит урон, реагирует, движется фронт). Это важный приём: он **убирает как класс** проблему «во сне досчитали дозу/состояние». Поломка предмета и её каскад — это просто обычные ЖИВЫЕ события, узел в это время не спит.
- **Дисциплина (критично для производительности):** число «не спящих» узлов и точек «интереса» нужно держать **редким**. Если интересом станет пол-уровня, экономия рушится. Это явный инвариант, за которым придётся следить.

### D.6. Лут / точки интереса / урон — железный закон

Это правило корректности, не баланс:

- **Урон = от газа, который РЕАЛЬНО стоит на предмете, по времени.** Нет пересечения газа и предмета → **0 урона** → то, чего газ не коснулся, **не ломается никогда**.
- **Числа** (скорость накопления дозы, порог срабатывания, затухание) — **конфиг**, крутятся ради баланса.
- **Источник дозы — правило, не крутится:** доза идёт от **резидентного газа** (газ реально находится тут), а НЕ от «фронт лизнул и побежал дальше».
- **Точность по дистанции:** рядом с игроком — по клеткам, точно; вдали — по «куску» (~1 м), погрешность допустима (там никто не смотрит в упор).
- **Поломка — это истинное прошлое событие.** Газ мог потом отступить или развеяться — и это ПРАВДА, мы не дорисовываем «как было бы». Урон и картинка читают **один и тот же газ**, поэтому «предмет сломался, а газа рядом нет на картинке» — **невозможно**.
- **Тот же крючок** переиспользуется для скользкого пола и прочих интерактивных предметов.

### D.7. Встреча двух газов (реакции)

- **Эмерджентно:** если два газа пришли в один кусок → событие встречи возникает само, в тик, когда пришёл второй. Не нужен скрипт «вот тут будет реакция».
- **Точная координата** реакции (нужна, если рядом игрок или реакция реально влияет) → поднимаем **этот один кусок** до клеток и ищем линию контакта.
- **Защита от ложных реакций:** необратимый взрыв жмём **только после подтверждения со-расположения на достаточно мелком зерне**. То есть грубый слой в одиночку **не выдаёт** ложный взрыв «два средних числа в одной комнате → бабах».

### D.8. Игрок в поле — одна правда, два приближения

Это закрыло главную дыру, которую нашёл аудит:

- **Игрок ВСЕГДА на детальном уровне.** И урон, и **видимость** для него считаются по клеткам = по тому, что он реально видит. Значит «умер в чистом на вид воздухе» или «прошёл сквозь дым-укрытие, которого по правде не было» — **невозможны**.
- **Это НЕ два источника правды.** Это **одна правда в двух приближениях**: детальное при «отдалении» сходится к грубому (как зум — то же самое, просто крупнее), они не противоречат, в сумме одно и то же.
- **Кооп:** детальное поле считает **хост** как одно общее поле; клетка детальна, если рядом **любой** игрок. Поэтому двое в одной точке видят **одно и то же**, и видимость безопасна **как правда**, а не как догадка клиента.

### D.9. Взрывы / цепь / огонь

- **Взрыв** (первая боевая механика) = мгновенный «бум» в точке встречи или поджога.
- **Цепь (каскад):** вторичные взрывы происходят **следующим тиком**, а не рекурсивно в том же. Это ограничено и детерминировано, и читается как **ползущий фронт** детонации. *Зачем именно так:* если разрешить рекурсию в одном тике, детонация целого крыла произойдёт за один кадр → фриз/рассинхрон. «Следующим тиком» = и красиво, и безопасно.
- **Поджог** (бегущий огонь по газовому следу) — **позже**, отдельная механика, но на том же «фронте». Сейчас не нужен.

### D.10. Разрушаемость — сложная, но БЕЗ переделки модели

Вся разрушаемость выражается через **рёбра графа**, поэтому не требует переписывания симуляции:

- Дырки любого размера, проломы, обвал-открывающий-проход, выбить стену целиком, рост дырки = **добавить / расширить / щёлкнуть ребро**. Выбил стену полностью = очень широкий проём → два узла ведут себя практически как одна комната.
- **Новая ПРЕГРАДА** (завал ПОДЕЛИЛ комнату надвое) = линии возможного разлома **нарезаны заранее при генерации**; в рантайме просто щёлкаем внутренние связи между кусками (было «открыто» → стало «закрыто»). Генератор и так знает, что разрушаемо.
- **Инвариант:** **никакой переразбивки геометрии в рантайме.** Это убирает целый класс багов «как поделить массу, когда комната делится на лету».

### D.11. Детерминизм и сеть — DO / DON'T

- **Lockstep, один хост — авторитетный писатель**, сервера нет. Клиенты **реконструируют** состояние из присланного, **не пересчитывают** его сами.
- **Всё авторитетное/реплицируемое — целые числа / фикс-точка, бит-в-бит.** Float допустим только как ВНУТРЕННИЙ расчёт, детерминированно сведённый к целому **до** хеша и до отправки по проводу.
- **ДЕЛАЙ ТАК:** целочисленная графовая математика — целый Дейкстра (фронт) + целый перелив (поток).
- **НЕ ТАК:** учебный плавающий решатель давления (Newton-Raphson), непрерывный Eikonal с корнями — они **молча ломают бит-в-бит**. ⚠️ Явное предупреждение из разбора: если инженер возьмёт готовый «солвер вентиляции из библиотеки», детерминизм умрёт на первом же тике.
- Геометрия и резка — **данные с генерации**. Сетевой трафик маленький: события + целые числа по узлам.

### D.12. Мультиплеер 4–8

- Хост держит **грубую «воду» по всему уровню** + маленький детальный **пузырь** у каждого **КЛАСТЕРА** игроков (стоящие рядом схлопываются в один пузырь).
- **8 разбежавшихся игроков = лучший случает** (8 дешёвых холодных пузырей, каждый сам по себе).
- **Реальный драйвер цены — не разбежка, а пузыри на АКТИВНЫХ событиях:** 8 игроков у 8 разных живых фронтов одновременно. Под это нужно:
  - **потолок** суммарных детальных клеток-за-тик;
  - честный **круг-по-очереди** (round-robin), чтобы при упоре в потолок правда обновлялась справедливо;
  - **политика деградации аплинка**: события — приоритет и надёжная доставка; косметика — коалесинг (склейка) и обновление по интересу.

## E. Принцип «конфиг vs правило»

- **В конфиг (крутим ради баланса):** все числа — дозы, пороги, затухание, число слоёв на комнату, проводимости проёмов.
- **Зафиксировано (законы корректности, НЕ крутим):**
  1. доза идёт от **резидентного** газа (не от «фронт лизнул»);
  2. **никакого float** в авторитетном состоянии;
  3. реакция жмётся **только** при подтверждённом со-расположении на достаточном зерне.

## F. Чем подтверждено

- **Аудит №1 (архитектура):** конструкция держится, дилбрейкеров нет; совпала с уже ратифицированной моделью репо (многозонный граф + полосы + sillZ-проёмы + реконструкция).
- **Мат-ресёрч:** «пространство как данные» = **сохраняющийся граф + геодезический фронт** — лучший фит под наши ограничения. Ключевая поправка ресёрча к первичной идее: фронт должен быть **не односторонним растущим**, а **конечным сохраняющимся количеством** (модель «воды»). Детерминизм держать **целым графом**, не float-солвером.
- **Аудит №2 (геймплей):** все 6 направлений держатся, дилбрейкеров нет; модель «воды» названа «короной» решения.

## G. ОБЯЗАТЕЛЬНЫЕ нагрузочные пробы ДО ставки

Дёшево, но обязательно — это то, что превращает «звучит хорошо» в «доказано», прежде чем вкладывать месяцы:

1. **Каскад:** один поджог в длинное газонаполненное крыло (~60 кусков) — тик ограничен по работе, глубина каскада **идентична на 2 машинах** (проверка правила «следующим тиком» + бюджета событий).
2. **8 горячих пузырей:** 8 игроков у разных активных фронтов — кадр держится в потолок, круг-по-очереди сохраняет правду.
3. **Фронт в сложной комнате на экране:** огибает угол, а не идёт «прямой стеной», без излома на швах резки.
4. **Целочисленный инкрементальный Дейкстра на проломе:** ограничен и детерминирован (полудневный проб).
5. **Сохранение массы на тике пролома + «взрыв как перенос»:** RED-тест — любая операция, меняющая суммарную массу, считается **багом, а не балансом**.
6. **Высокий зал на экране ДО выбора числа слоёв:** подобрать число слоёв глазами (слои = данные на комнату).
7. **Нет «попа»/мерцания** при перерождении грубое↔детальное (рендер-кроссфейд).

## H. Открытые куски — ещё НЕ разобрано (для планирования)

Это честно не закрытые места; их нельзя считать решёнными:

- **Операционка:** сохранение/перезагрузка состояния; **ПОТЕРЯ ХОСТА** (миграция авторитета — иначе падение хоста = смерть сессии на 4–8 человек). Серьёзный пробел.
- **Движущаяся геометрия:** лифты, бласт-двери — то, что закрывается прямо в потоке и режет газовый столб (как сохранить массу в момент движения).
- **Грифинг:** детерминированная симуляция в руках игроков = набор примитивов навредить напарнику (загазовать, перекрыть вентиляцию, заманить в яму и поджечь). Пока не осмыслено.
- **Экономия грубого слоя при МНОГИХ мелких источниках / повсеместной утечке** — чтобы правило «спящие = ноль» не рухнуло на always-on слое.
- **Believability / читаемость как РЕНДЕР-задача:** слои → туман без «полок» (видимых ступенек), читаемость «какой газ / где опасно / где фронт». Симуляция отдаёт данные; это отдельный художественный проход.

## I. Решения владельца на потом (вкус / крафт)

Намеренно НЕ решаем сейчас:

- **Подпись каждого газа** (цвет / движение / край) — отдельная большая художественная работа, не сейчас.
- **Число слоёв:** 3 по умолчанию, 4–5 в высоких витринных залах (подобрать глазами).
- **Тайминг огня:** мгновенно vs ползёт — решать, когда появится механика поджога.
- **Насколько тянуть «правый угол» реконструкции** (как точно детальное должно совпасть с «что было бы»).

## J. Развилка `d-gasmodel-redesign-001` — куда склонилось (решение за владельцем)

- **Рекомендация разбора: дорога A+** (граф-основа + наращивание слоёв), **не B** (непрерывное поле).
- **Идея «убрать ёмкость + перелив» переосмыслена.** Это не костыль — это **видимое лицо настоящего давления** (которое в модели уже есть) плюс намеренное свойство «угроза не стравливается сама собой». Полностью убрать ёмкость **без закона-замены нельзя** — иначе газ становится безграничным/осциллирующим и производительность рушится. А **реальная** боль (потеря ощущения «ГДЕ именно газ») лечится **фронтом + зерном**, а не удалением ёмкости.
- **Идея «свободное число газов» бьёт по доказанной сети.** Нужна дисциплина: **ограниченный список активных газов на узел + жёсткий потолок на проводе**. «Как богато ОЩУЩАЕТСЯ» (сколько газов в игре всего) ≠ «что реально летит по сети в один момент».
- **Проб C** (1–2 недели, проверить репликацию непрерывного поля) нужен **только если** мы всё ещё хотим B. Разбор показывает, что **не хотим** → проб C, скорее всего, не нужен. (Если на ревью останется сомнение — C дешевле, чем слепая ставка на B.)
