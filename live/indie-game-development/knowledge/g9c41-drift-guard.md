# g-9c41 — DRIFT-GUARD: 5 ловушек пересказа (owner-locked 2026-06-22)

**Кто читает / когда:** КАЖДАЯ сессия g-9c41, ПЕРВОЙ строкой PLAN, вместе с
[[g9c41-architecture-locked-slices]]. Это пять фактов, которые чаще всего ломаются, когда чат
пересказывает архитектуру «попроще». **Если твоё объяснение владельцу противоречит любому из
пяти — ты дрейфанул: остановись и перечитай канон, не пересказывай по памяти.** Канон =
locked-slices + `work/dev-plan-graph-2026-06-22.md` (+ его DECISION INDEX). Всё в `work/archive/` —
история, НЕ авторитет.

1. **«8 пузырей»? НЕТ.** Каждый пир считает грубое (комнатное) для ВСЕЙ карты + ОДИН детальный
   пузырь только у СВОЕГО игрока (~1 на пир, не N). Чужой/дальний пузырь — **пропускается**. Этот
   пропуск И ЕСТЬ экономия ~8×. (detail-authority-cost-model §1.)

2. **Детерминизм «на 2 машинах»? НИКОГДА не гейт.** Целочисленный счёт = кросс-CPU данность
   (банки, игры считают одинаково на разных процессорах годами) — это **не доказываем**.
   Единственная реальная опасность — **float** в авторитетном пути → ловит build-time **zero-float
   scan на ОДНОЙ машине** (этот скан И ЕСТЬ кросс-CPU гарантия, по конструкции). loopback-хеш
   (2 процесса на ОДНОЙ машине) = опциональный линт на порядок итерации/RNG, **не «две машины»**.
   **Реальные 2 ПК — никогда не гейт; у владельца одна машина, этого достаточно.** То же и для
   **re-flux-пробы** (всплывает в S4): её binding-гейт = loopback одна-машина + zero-float гард, а
   НЕ два железа (owner-decided 2026-06-22, `d-reflux-gate-001`; отменяет формулировку «2 машины» в
   ранних доках и в grid-vs-graph §5/§8).

3. **«Грид ИЛИ граф»? Нет — ОДНА модель, один закон сохранения.** Вблизи: грид + поток через
   **открытые ГРАНИ** (площадь = число открытых граней, реальная высота, «не сквозь стену» — всё
   даром из геометрии). Вдали: тот же закон, грубее (граф = rollup над тем же разбиением). **КОМНАТА
   = МЕТКА / region_id на ВСЕХ тирах** (транспортная «труба» сбрасывается только вблизи; разбиение
   держим везде). НЕ «при переходе на грид комнаты выбрасываем». (grid-vs-graph §1.)

4. **Деталь пишет в грубое / синхронится по сети? Нет — в ОДНУ сторону.** Грубое (комнатное) =
   ЕДИНСТВЕННАЯ общая правда, считается одинаково у всех, единственное в checksum. Деталь =
   локальное уточнение у своего игрока, **вне checksum**, на выходе из комнаты **ВЫБРАСЫВАЕТСЯ** (не
   сворачивается в авторитет). Жёсткие общие последствия (урон / реакции / поток между комнатами) =
   грубые **СОБЫТИЯ**, считаются из грубого состояния у всех. (detail-authority §2–§3.)

5. **Float-гард — несущий, это и есть защита детерминизма.** Авторитетный путь = ТОЛЬКО целые /
   фикс-поинт. Build-time zero-float scan валит сборку, если float просочился в авторитет. Owner:
   «нужен гард, который не пропустит float — это очень плохо». (dev-plan-graph §5.)

**ОТКРЫТО (не закрыто — решается per-mechanic в PLAN слайса, где встанет, НЕ заранее):** глубина
геймплея комнатно-гранулярна (Вариант A, дефолт — деталь = «для глаз», даже может быть
недетерминированной) или под-комнатно-формовая (Вариант B — дорогие ведро-3 механики, форсят «эта
комната считает деталь у всех»)? Классификация ведро-1/2/3 — в detail-authority §4 + dev-plan-graph §3.
