## The research-and-decide principle (CANDIDATE — must hit an owner G7 gate; now a CORE/root principle)
The system DECIDES every expert variable from evidence + the owner profile (recording rationale + an
evidence tag) and ASKS ONLY irreducible personal facts — minimal, skippable. Membership test:
researchable-from-evidence-plus-profile ⇒ DECIDE; otherwise ⇒ ASK. "meals/day" ⇒ DECIDE (the v1 bug).
Carve-out by NAMED CONSTRAINT (shift work, medical fasting), not by owner wish. → Mechanized in the core
as the `profile` anchors-vs-facts split.

## Disputed-term glossary (16 terms — candidate locked readings + rejected readings)
1. **preference / personal-fact** = irreducible personal fact not derivable from evidence+profile
   (dislikes, allergies, meds, hard schedule, budget, real cooking-time, equipment). Only this class is
   askable. [rejected: any owner wish about an expert variable counts (= v1 bug); preference = overridable default.]
2. **setup intake / gate** = thin, typed, NON-BLOCKING intake restricted to personal-facts; skip ≠ block
   (system fills with decided defaults). [rejected: flat questionnaire collecting everything incl. expert
   variables (v1 architecture); blocking precondition.]
3. **eating pattern** (meals/day, snacks, timing, per-meal split) = SYSTEM-DECIDED composite, decomposed
   into independently-routed sub-variables; carve-out only by named constraint. [rejected: meals/day as a
   preference asked at setup (the v1 question); blanket "always ask the pattern".]
4. **leads / "leader with brakes"** = default DECIDE-AND-INFORM; surfaces for confirmation only an explicit,
   enumerable brakes-list (safety items, below calorie floor, raising training/diet intensity, sharp course
   change). [rejected: propose-and-wait (v1 asker); silent autopilot (forbidden by guarded posture).]
5. **source-of-truth** = single basis all outputs are DERIVED from; typed store separating decided-variables
   (rationale+evidence) from generated; GitHub canonical, owner never edits files (change the upstream
   variable, not the output); apps may hold/mirror but authority stays with the decided basis; only
   summary/decision upward. [rejected: passive folder; fully federated truth.]
6. **outputs** = CLOSED typed set, each generated on demand from current decided-variables: week menu,
   swaps, shopping list, recipes, lazy food, prep plan, post-feedback corrections. [rejected: static edited
   documents; "the list is just examples".]
7. **fallback / "lazy food"** = pre-designed, system-selected low-effort meals that still hit protein/calorie;
   COUNT toward the day's accounting, NOT a slip. [rejected: cheat/lapse — mis-scoring corrupts the recalc signal.]
8. **nutrition review / checkpoint** = TWO mechanisms, both inside the Health AI System (not the strategic
   Direction OS review): (1) owner-PULL complaint transducer (taste/hunger/difficulty/price/time/slips/
   boredom/social) → MUST mutate the artifacts; (2) system-PUSH checkpoint (date/signal) → MUST emit a
   decision (hold/recalc/diet-break/change dishes/refresh evidence/rebuild). [rejected: passive summary
   (autopilot pre-mortem); = the Direction OS review (leaks daily data up).]
9. **recurring research-processes / extension points** = a structural SEAM (typed contract to add process
   specs without rebuilding the core), instantiated by owner/chat/checkpoint-triggered jobs, each bounded
   with a scope+stop/cost rule; NOT an always-on scheduler. The four done_when examples are illustrative.
   [rejected: self-firing cron (needs forbidden runtime); "4 examples = fixed feature set".]
10. **deficit** = SYSTEM-DECIDED but OUTCOME-GATED managed band (not a fixed number); throttled/paused when
    protected outcomes degrade (sleep/energy/strength/libido/mood); never below a non-overridable calorie
    floor (a brakes item). [rejected: fixed % held open-loop; defined solely by the floor.]
11. **crash diet** = an acceptance BOUNDARY never crossed; UNION of axes (too aggressive OR psychologically
    non-sustainable OR damages protected outcomes); failing ANY axis = crash; judged up-front and after-the-fact.
    [rejected: any single-axis definition — they can disagree.]
12. **tracking precision / heaviness** = DECOMPOSED: analytic precision + what-is-logged ⇒ DECIDE (minimum
    sufficient; capture low-friction photo/voice/text); owner logging capacity/cadence ⇒ ASK. Three
    heaviness axes (metric count, entry friction, precision). [rejected: all-DECIDE (refuses a real capacity
    fact); all-ASK ("how precisely do you want to track?" = v1-class expert question dressed as preference).]
13. **повторяемость / repeatability** = three layers at once: food templates / behavioral weekly cadence /
    maintenance-grade durability (~12 mo no regain); in tension with boredom → balance with variety.
    [rejected: any single layer alone.]
14. **плохая неделя / bad-week (nutrition)** = a defined nutrition minimum protocol (on lazy/fallback food,
    protein/calorie-adequate) with an entry trigger and a relaxed-but-non-negotiable floor; the activity floor
    ("2 силовые + 2 лёгкие") belongs to the training sibling; a unified whole-system mode is cross-node (deferred).
    [rejected: importing the activity protocol; unilaterally inventing a unified mode across parked siblings.]
15. **прокрастинация** = a guard against THREE forms: build-instead-of-execute (2-week tripwire) / endless
    research / decision-latency (research-and-decide stalling into over-confirmation). Each its own guard.
16. **встроено в body outcome** = a HARD closed loop: body metrics (weight trend, strength, sleep/energy) are
    INPUTS to nutrition decisions + a recalc control law → tracking + checkpoints are MANDATORY in-node, not
    deferrable; symmetrically fences both failures (break the body via extreme deficit / system-for-its-own-sake).
    [rejected: soft "aligned at setup" with no live coupling.]
