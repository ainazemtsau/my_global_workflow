# Life-reset — Vision Capture v1 (строгий само-улучшающийся личный менеджер)

Date: 2026-06-20
Session: s-life-reset-remap-001 (play: map → routed to frame)
Status: durable capture — input для re-frame корня. Не charter и не tree;
владелец подтвердил («Go»). Любая следующая сессия/чат читает это, чтобы
ничего не потерять.

## 0. Зачем этот файл

Сессия remap, со-творя с владельцем, обнаружила, что текущий корень charter
слишком узок. Старый каркас («недельный operating graph / Weekly Contract как
ядро») раз за разом тунелил в «постройте весь runtime» и был признан невалидным.
Реальная система — другая. Этот файл фиксирует её дословно.

## 1. Главный сдвиг каркаса

life-reset — **не недельный трекер** и **не набор отдельных программ-островов**.
Это **строгий, само-улучшающийся личный операционный МЕНЕДЖЕР** (недельный +
дневной ритм) как ХРЕБЕТ, в который встраивается ВСЁ.

Два полюса, оба неверны:
- старый провал — «только weekly»: контракт недели как отдельный процессор →
  слишком узко, потащило в постройку runtime;
- ошибочный перегиб (Claude, отвергнут) — «программа как король», менеджер
  задвинут → теряется объединение, всё выглядит как острова.

Правда посередине: **один координирующий менеджер держит весь ритм жизни**, и в
него втыкаются направления + программы + бэклог. Неделя — это ПУЛЬС менеджера, а
не отдельный артефакт-контракт.

## 2. Объединяющая модель (hub)

```
        НАПРАВЛЕНИЯ          ПРОГРАММЫ            БЭКЛОГ / ИНТЕЙК
     (health · game,      (что пробуешь:        (сюда стекается всё:
      читаются, инфа       ритуал · книга ·      идеи · проблемы ·
      может быть неполной)  практика)            желания)
            \                   |                     /
             \                  v                    /
        ┌───────────────────────────────────────────────────┐
        │   ЛИЧНЫЙ МЕНЕДЖЕР — недельный + дневной ритм        │
        │   координирует направления + программы + бэклог     │
        │   в ОДИН ритм жизни; день/неделю ведёт ПО ВСЕМУ     │
        │   лучшие практики: координация · мотивация ·        │
        │   решение проблем · дневник · review / мутация      │
        └───────────────────────────────────────────────────┘
                              |
                              v
        → планы достигаются · ритм жизни держится · ничего не теряется
        ───────────────────────────────────────────────────────────────
        DURABLE-СОСТОЯНИЕ (markdown / репозиторий) = портативность
```

Программа — НЕ отдельный остров: встраивается в тот же ритм, что и направления.
Старые 6 узлов карты целиком живут внутри менеджера (ритм = недельный+дневной;
интеграция = направления втыкаются; intake/knowledge/memory = бэклог+дневник+
durable-память; modules = программы; review/portable = review-петля + состояние).

## 3. Операционная петля состояний

```
1 ИНТЕЙК-ПРОЦЕСС (1-й раз / по нужде): недостающие вопросы · research под тебя · сохраняет
        v
2 СБОРКА НЕДЕЛИ: из направлений + программ + бэклога → строгий недельный граф/контракт
        v
3 ДЕНЬ ИЗ НЕДЕЛИ: тайм-слоты · приоритеты · адаптация
        v
[ВОРОТА] был review вчера? нет → требует заполнить; дальше ТОЛЬКО после явного отказа
        v
4 ПРОЖИТЬ ДЕНЬ → REVIEW ДНЯ (сохраняется; день = один чат)
        v   (└─ след. день: возврат к 3)
5 REVIEW НЕДЕЛИ (агрегирует все дневные review)
        v
[РАЗВИЛКА] неделя ок? → да: новая неделя (возврат к 2)
                       → не так: САМО-УЛУЧШЕНИЕ — новый research → переписать процесс → к 2

[СТРОГИЙ ФИЛЬТР ЖЕЛАНИЙ] перехватывает любую «хотелку» в любой момент (см. §4)
[ФУНДАМЕНТ] durable-состояние · writer-карточка или прямая запись · портативность · одна задача — один чат
```

## 4. Состав недели (планируется ВМЕСТЕ, всё через фильтр)

Направления менеджер читает, но инфа неполная → владелец дополняет.
Порядок = приоритет:
1. **МИНИМУМ обязательного** — что ДОЛЖНО быть сделано; фиксируется первым.
2. **РУТИННЫЕ FLOORS** (уборка и т.п.) — идут ВСЕГДА, не обсуждаются.
3. **ХОТЕЛКИ** — только после обязательного, каждая через строгий фильтр.
4. **EXPLICIT CUTS** — что осознанно не делаем.

Баланс «минимум ↔ рутина ↔ хотелки» — открытый design-вопрос (→ research).

## 5. Требования (R1–R31) — дословный захват

### Личность менеджера (ядро)
- R1 — строгий, прямой, нацелен ТОЛЬКО на эффективность; не подстраивается под желания вслепую.
- R2 — считает, чем сейчас объективно лучше заняться, держит баланс, но не прогибается.
- R3 — к любым «хотелкам» жёстко и критично; новое желание по умолчанию → в бэклог с явными причинами, либо отклонить совсем с причинами.
- R4 — дискуссия разрешена, но бремя доказательства на владельце, планка высокая; непробиваем к слабым доводам (скидка, «просто хочется»); сто сообщений не продавливают.
- R5 — вмешивается в принятый план посреди недели ТОЛЬКО если (а) реально влияет на текущие цели/направления, или (б) есть настоящий жёсткий due date, только что узнанный; иначе ждёт до конца недели.
- R6 — отдых/игра: не морализирует, принимает, но сразу ищет синергию (превратить в материал/пользу); реальный отдых разрешён осознанно, не как слив.
- R7 — по ощущению личность (наставник/менеджер/«моё отражение»); под капотом — распределённый процесс с циклами и блоками.
- R30 — агрессивный/прямой/критикующий режим по умолчанию в ЛЮБОЙ программе/процессе, особенно где «потихоньку ничего не делается/не предоставляется»; скептицизм строго по делу.
- R31 — смещение: «лучше перекритиковать, чем недокритиковать» — при сомнении в сторону большей критики.

### Операционный цикл и состояния
- R8 — неделя собирается из направлений + программ + бэклога; направления читаются, но инфа может быть неполной; неделя планируется ВМЕСТЕ (владелец дополняет), не чистый авто-pull; у недели строгая структура (граф/раскладка — форму проектируем).
- R9 — из недели выводятся цели/задачи дня; день адаптируется под реальность.
- R10 — review дня и недели сохраняются; дневные агрегируются в недельный.
- R11 — ворота: начинаешь день без review вчера → требует заполнить; дальше только после явного отказа.
- R12 — неделя = ограниченная процедура с чётким концом; в чате планирования недели день не начинаем; день = отдельный чат.
- R13 — неделя пошла не так → запускается цикл нового research'а, который улучшает сам процесс.
- R27 — неделя сначала фиксирует минимум обязательного, потом хотелки; порядок «сначала обязательное → потом хотелки».
- R28 — рутинные floors идут ВСЕГДА, не обсуждаются, независимо от «помогает или нет».
- R29 — баланс «минимум ↔ рутина ↔ хотелки» — design-вопрос на research.

### Само-определяющийся / само-улучшающийся процесс
- R14 — не generic; как наш workflow, всё определено: каждый процесс/задача формальны, воспроизводимы, строги (возможно, строже Direction OS).
- R15 — процесс сам себя определяет: 1-й запуск собирает данные, задаёт недостающие вопросы, сохраняет, умеет повторяться.
- R16 — делает research под стиль работы владельца, раскладывает день по тайм-слотам, применяет лучшие практики.
- R17 — гибко перестраивает собственные инструкции; в критическом случае правила переписываются, но обычно неделя идёт по принятому процессу.

### Состояние / память / портативность / writer
- R18 — память и состояние durable.
- R19 — начал в одном чате, прервался в любой момент, состояние сохранено, продолжил в другом.
- R20 — команда «сохраняй»: web (ChatGPT/Claude) → карточка в Writer → Codex/Claude Code сохраняют.
- R21 — в Codex/Claude Code напрямую (agents.md/claude.md) пишут сами, Writer вручную не нужен; либо для консистентности — карточка из отдельного чата, чтобы не загрязнять контекст.
- R22 — «одна задача — один чат», где применимо.

### Цель v1 + расширение
- R23 — первая версия: подстраивает процессы, выводит продуктивность на максимум, помогает достигать задач правильно — распределять, приоритизировать, не забывать.
- R24 — потом расширяем: новые ритуалы/активности; система и не даёт забыть, и не даёт перекрыть текущее важное.

### Мета
- R25 — имплементацию надо ресёрчить, не в лоб.
- R26 — должно работать через чат, не только в CLI.

## 6. Мои критические флаги (guardrails — несущие ограничения для frame/research)

1. **Само-переписывание правил + непробиваемость для владельца = риск.** Изменение
   собственных правил процесса должно идти через явные ВОРОТА/review (как gates в
   OS), и менеджер НЕ может тихо отключить путь override владельца через убеждение.
2. **«Математическая точность» = эвристика + evidence**, не буквальная оптимизация.
   Сила — в требовании доказательства и невёдении на слабые доводы, не в формуле.
   Не продавать ложную точность.
3. **Первая версия — крошечная**: реально прогоняет один недельный цикл по тому,
   что есть (charter pre_mortem #11: «построили систему, не прожили неделю»).
   Мотивация/дневник/синергия/само-research наслаиваются по мере доказательства.
   Фильтр строгий, НО обязан разрешать настоящее восстановление (иначе — петля
   выгорания владельца).

## 7. Открытые вопросы для research-прохода

- Как реально строят строгих self-improving личных AI-менеджеров/accountability-систем (verified-кейсы, не блог-хайп).
- Какая структура у «недели-графа»: что узлы, что рёбра, как из неё выводится день.
- Как сделать само-переписывающийся процесс БЕЗОПАСНЫМ (gate/review на изменение правил).
- Как закодировать «непрогибающуюся» политику приоритизации (что считается достаточным доводом; как взвешивать минимум/рутину/хотелки).
- Баланс минимум ↔ рутина ↔ хотелки ↔ отдых (R29).
