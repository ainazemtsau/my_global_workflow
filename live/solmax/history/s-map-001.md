## Карточки (полный rationale — detail-источник для узлов дерева)

### 1. g-kernel (W0)
- **goal:** Крошечное ядро RLK существует и доказано расширяемо: 3-й плагин = 1 register() + 1 handler, 0 строк диффа ядра.
- **done_when:** 5 частей (registry; append-only JSONL-ledger + schema_version + upcaster; типизир. конверт CALL/RESULT + boundary-валидатор; effect-tier gate 0/1/2→owner; швы Engine+Store); тест 0-дифф зелёный в CI; replay-тест; gate-тест (tier-2 без аппрува отклонён).
- **why:** механизм SC2 (0-дифф расширяемость) + предохранитель против worst-failure (вечный черновик, pre-mortem #1); тест 0-дифф делает дисциплину механически фальсифицируемой.
- **edge:** charter edge #2 (10 лет enterprise → порты/контракты/event-log сами) + edge #1 (готовый концепт EXOCORTEX + живой markdown-кернел OS). Инкумбенты (Letta/MemGPT, OpenClaw) не держат ядро намеренно крошечным — их ценность в растущих фичах.
- **risk:** ядро раздувается (тест 0-дифф падает = worst-failure); схема ledger такая, что upcaster не спасает и миграция ломает replay; gate обходим (privacy, #6).

### 2. g-engine (W1)
- **goal:** Движок на подписках надёжен: claude -p (primary, форс JSON-Schema) + codex (verifier) за швом Engine, с бюджетом и failover при credit-cliff.
- **done_when:** вызовы через шов; учёт бюджета/кредита; failover-тест без потери CALL; исчерпание = видимое событие; подмена адаптера (на API) = 0 диффа.
- **why:** без мозга ничего не живёт; снимает pre-mortem #5; шов оставляет переход на API поздней заменой адаптера.
- **edge:** charter edge #3 (уже гоняет Codex/Claude Code на подписках) + constraint «движок = подписки, не API». Инкумбент не завяжется на Agent-SDK-кредит личной подписки с кросс-вендорным failover.
- **risk:** ToS-трение always-on; деградация на fallback (codex ≠ тот же мозг); кредит кончается раньше → решение об API раньше; краевые случаи форса JSON-Schema.

### 3. g-channel (W1)
- **goal:** Первый лёгкий канал (Telegram/голос) живёт и фиксирует контракт канала; web-app вынесен в g-surface.
- **done_when:** канал = плагин kind:channel (вход→конверт→ответ); контракт заморожен (2-й канал=0 диффа); через ledger+gate; реальный dogfood-smoke.
- **why:** ранний dogfood (outside-view: пользоваться рано) → старт SC1; открывает ось «канал».
- **edge:** charter edge #5 (один пользователь → без multi-tenant/auth/onboarding) + edge #3. OpenClaw — референс оси канал/body.
- **risk:** обёртка-чат (#4), пока не приедет g-cognition (W2) → раннее использование мелкое; неверный первый канал; privacy внешней поверхности (#6).

### 4. g-surface (W2, добавлен владельцем)
- **goal:** Веб-приложение-дом: живут ассистент и инструменты, владелец работает вместе с ним (общая доска, канбан). Новая панель = register(), ~0 диффа.
- **done_when:** запущенное веб-приложение (общение + ≥1 общий инструмент вживую); kind:surface (2-я панель=0 диффа); co-work-smoke (оба меняют один артефакт); через ledger+gate; реальный dogfood.
- **why:** «тело»/дом, где живёшь → SC1+SC3; открывает ось «поверхность» (5-й вид; доп. доказательство SC2 — виды это открытый список, 0 диффа).
- **edge:** charter edge #5 (один пользователь — без multi-tenant) + edge #2 (enterprise full-stack) + edge #3 (агенты строят UI инкрементами). Инкумбент не выпустит дёшево однопользовательскую глубоко-персональную co-work-поверхность.
- **risk:** взрыв скоупа → кроличья нора (#2) и оттяжка реального использования (#1) → смягчение: после лёгкого канала, первая версия минимальна (1 инструмент); богатая поверхность искушает раздуть ядро → обязана остаться плагином kind:surface.

### 5. g-cognition (W2) — несёт неочевидный путь
- **goal:** Когнитивный цикл (compile_context + многошаговый цикл) как плагин #1 даёт глубокую персональную многошаговую помощь явно лучше обычного чата (напр. «план дня» после ресёрча; вечерний брифинг).
- **done_when:** плагин kind:process (контекст→ресёрч→рассуждение→самопроверка→результат); head-to-head вердикт «лучше» на реальной задаче (записан); многошаговость/персонализация видны в ledger; 2-й процесс = 0 диффа.
- **why:** прямо SC3 («мега-мозг»), ответ на pre-mortem #4 (обёртка чата); превращает канал+память+поверхность в реальную глубокую помощь → SC1.
- **edge:** charter edge #1 (концепт EXOCORTEX + markdown-кернел OS уже кодируют «собери контекст → гейтнутый многошаговый цикл» — портируемо) + edge #4 (OS-харнесс раскладывает на проверяемые шаги). Инкумбент-чат оптимизирует быстрый одноход в масштабе.
- **неочевидный путь (редкий пример — продвинутые/кентавр-шахматы):** очевидный путь (RAG поверх заметок + умный промпт) обычно проигрывает (#4). Неочевидный: команды человек+движок обыгрывали и чистых людей, и топ-движки — но только когда интерфейс давал человеку рулить поиском движка в ключевых точках. Перенос: преимущество цикла — структурированное руление; первый глубокий процесс (план дня) строить вокруг нескольких чекпоинтов владельца, а его правки кормят g-learning (g-megabrain). «Лучше обычной сессии» даёт связка ты+цикл.
- **risk:** правдоподобно-но-мелко (планка гэймится) → head-to-head-вердикт; медленно/дорого (cost/credit) → перестаёшь пользоваться; сбор контекста тянет лишнее → теряет фокус.

### g-megabrain (parked, W3+, под g-cognition)
- **goal:** Продвинутый «мега-мозг»: генеративный UI (сам собирает виджеты, может запрашивать разработку других); обучение на реакциях (дофамин/prediction-error); размышление во сне → приносит только хорошие идеи (иначе молчит); омниканальное «тело».
- **why:** полная EXOCORTEX-планка; открываем только после реального использования базы (pre-mortem #7 «vision sprawl», G8). Раскрывается в отдельной map при активации.

### 6. g-memory (W2)
- **goal:** Память живёт: эпизодическая поверх ledger → семантическая/RAG (→ SQLite-fold по объёму).
- **done_when:** эпизодическая (верно вспоминает прошлые сессии); семантическая (recall по смыслу без совпадения слов); kind:memory (2-й вид=0 диффа); smoke: процесс лучше благодаря памяти.
- **why:** субстрат персонализации и накопления; делает g-cognition глубоким; питает g-megabrain (learning/dreamlab читают ledger/память); открывает ось «память».
- **edge:** charter edge #2 (append-only event-log сам → эпизодическая память = чтение поверх ledger, почти даром) + edge #1 (OS уже хранит one-source-two-views). Outside-view: Letta/MemGPT.
- **risk:** шум/раздувание (garbage-in); privacy (индексируется «вся жизнь», #6); качество RAG (неверный recall хуже отсутствия); преждевременная сложность (вектор/SQLite до объёма — cost/анти-черновик).

### 7. g-accrete (W2+)
- **goal:** Возможности накапливаются по осям до ≥8 живых при 0 диффе: инструменты (доска, канбан, app-коннекторы, чтение OS) + триггеры процессов (manual/schedule(cron-MCP)/event).
- **done_when:** ≥8 используемых способностей; каждая register()+handler при 0 диффе; доска+канбан+≥1 коннектор+чтение OS (жёстко read-only, протестировано, не пишет); триггеры manual/schedule/event; владелец реально пользуется несколькими.
- **why:** SC2 вширь (доказательство расширяемости = реальные способности) + бытовые инструменты → SC1; здесь же обязательная связь «чтение OS read-only».
- **edge:** charter edge #3 (оркестрация + фоновые агенты → каждая способность = маленький agent-buildable инкремент = build mode) + edge #4. Инкумбент катит фиксированный роадмап, не личный «длинный хвост».
- **risk:** самый уязвимый к pre-mortem #1 (churn ядра + ноль используемых способностей → kill_by: заморозить ядро) и #3 (слоп агентов → неревьюемый бэклог → мелкие инкременты + гейт 0-дифф + темп ревью); grab-bag — будет разбит в shape.
