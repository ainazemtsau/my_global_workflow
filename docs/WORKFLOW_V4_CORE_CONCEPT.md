# Workflow v4 — концепт ядра (итерация 2)

Дата: 2026-06-10
Статус: концепт после обсуждения аудита; не принятое состояние
Предыстория: `docs/AUDIT_2026-06-10_WORKFLOW_V3_AND_V4_PROPOSAL.md`

Изменения относительно итерации 1 (по итогам обсуждения с владельцем):

1. Привязка чата к плейбуку — **обязательная**, а не опциональная.
2. Внутри направления восстановлена иерархия планирования (Direction → Map → фокус/задачи), но с правилом «планирование волной».
3. Закрытие чата **обязано** выдавать handoff на следующий чат — цикл замыкается.
4. Workflow-governance перестаёт быть направлением.
5. Executor-handoff получает два режима: mechanical и engineering.

---

## 1. Принципы (без изменений + уточнения)

1. **Бюджет контекста — закон.** Ядро (lifecycle + handoff + session_close + реестр плейбуков) — один файл ≤ 1500 слов. Каждый плейбук ≤ 600 слов.
2. **Машина пишет — человек решает.** Все мутации репозитория делает агент с прямым доступом (Codex / Claude Code). Человек принимает решения.
3. **Правило существует, только если его можно проверить.** Остаются только проверяемые инварианты (список в §6). Непроверяемые запреты («не срезать», «не сжимать стадии») не пишутся.
4. **Одна сущность — один вопрос.**
5. **Планирование волной.** Дальние вехи — грубо (2–5 строк), текущая веха — в задачах. Детализация вехи до того, как она стала фокусом, — запрещена.
6. **Ноль мета-направлений.** Обслуживание воркфлоу — задачи агент-сессий на этом репо, не направление.

## 2. Структура направления

```
directions_v4/<id>/
  DIRECTION.md   # цель, критерии успеха, границы, констрейнты — меняется редко
  MAP.md         # треки + вехи с примерными планами — редактируется на review_milestone
  STATE.md       # текущий фокус (одна веха) + задачи + указатель next_chat
  LOG.md         # append-only журнал сессий (5-10 строк на сессию)
  knowledge/     # принятые решения/знания, по файлу на тему, с valid_as_of
  work/          # рабочие артефакты направления
```

### MAP.md (карта направления)

```yaml
tracks:
  - id: TR-1
    name: "..."
    why: "1-2 строки"
milestones:
  - id: M-1
    track: TR-1
    goal: "что должно стать правдой"
    exit_criteria: "проверяемый критерий закрытия вехи"
    sketch: "примерный план, 2-5 строк, НЕ задачи"
    status: future | focus | done | dropped
```

Ровно одна веха может иметь `status: focus` в один момент времени (на направление).

### STATE.md (живое состояние)

```yaml
focus_milestone: M-1
tasks:
  - id: T-12
    goal: "одно предложение"
    done_when: "проверяемый критерий"
    status: open | active | blocked | done
    notes: "1-2 строки"
next_chat:            # заполняется при закрытии каждого чата
  playbook: work_task
  handoff: { ... }    # готовый к вставке handoff
open_questions: []    # вопросы к человеку, если есть
```

## 3. Цикл направления

```
define_direction ──► plan_focus ──► work_task ⟲ ──► review_milestone ──► plan_focus ...
                                        │
                                        ├─► child_research (рекурсивно)
                                        └─► executor handoff (Codex / Claude Code)
```

- **define_direction** — один чат: DIRECTION.md + первичная MAP.md (треки, 2-5 вех грубо). Не восемь чатов.
- **plan_focus** — берёт следующую веху из MAP, ставит ей `focus`, дробит в 3–7 задач с `done_when` → STATE.md.
- **work_task** — один чат на одну задачу. Закрывается session_close; next_chat обычно следующая задача; очередь пуста → review_milestone.
- **review_milestone** — сверка результата с `exit_criteria`, правка MAP.md по выученному (единственное место пересмотра карты), веха → done, next_chat → plan_focus.

Обратная связь от исполнения к плану структурно встроена в review_milestone, а не размазана.

---

## 4. Жизненный цикл чата

Каждый чат привязан ровно к одному плейбуку. Жизненный цикл универсален:

1. **Открытие.** Вход — handoff (вставлен человеком или прочитан из STATE.md `next_chat`). Модель подтверждает в ≤5 строках: плейбук, цель, `done_when`, план. Без команд-ритуалов; человек отвечает по существу или просто «да».
2. **Работа.** По шагам плейбука. Нужен побочный процесс → handoff `role: child` (рекурсивно) или `role: executor`. Родитель фиксирует блокировку задачи в STATE.md и продолжает остальное или закрывается.
3. **Закрытие.** Модель выдаёт `session_close` (включая заполненный `next_chat`). Агент применяет `state_update` к STATE.md/LOG.md и коммитит. Чат считается закрытым после коммита.

Проверка завершения = сравнение результата с `done_when` задачи / `exit_criteria` вехи / `completion` плейбука. Никаких отдельных аудитов.

### Handoff (единственный тип пакета)

```yaml
handoff:
  direction: <id>
  task: T-12                    # или milestone для plan/review
  playbook: work_task           # обязателен для role: continue|child
  role: continue | child | executor
  mode: thinking | mechanical | engineering   # для executor
  goal: "что сделать"
  context: "минимум или ссылки на файлы"
  done_when: "критерий приёмки"
  return: "что вернуть и в каком виде"
  boundaries: "что не трогать (опционально)"
```

### Session close (единственная форма закрытия)

```yaml
session_close:
  task: T-12
  result: "что реально сделано (не план, не handoff)"
  artifacts: [пути/ссылки]
  state_update: "изменения для STATE.md: статусы, next_chat"
  log_entry: "строка для LOG.md"
  knowledge_candidate: null | "тема + суть"
  open_questions: []
```

## 5. Плейбуки (реестр на старт — 7)

Плейбук = цель, входы, шаги (руководство), `completion`, допустимые next-плейбуки. ≤ 600 слов каждый.

| Плейбук | Назначение | Обычный next |
|---|---|---|
| `define_direction` | Создать направление за 1 чат | plan_focus |
| `plan_focus` | Веха → задачи в STATE.md | work_task |
| `work_task` | Рабочая сессия по одной задаче | work_task / review_milestone |
| `review_milestone` | Закрыть веху, обновить MAP | plan_focus |
| `child_research` | Ограниченный побочный вопрос, возврат родителю | возврат в родителя |
| `console` | Read-only: статус/вопросы, фиксированный формат вывода | — (без закрытия) |
| `recovery` | STATE.md противоречит реальности → восстановление по LOG/артефактам | plan_focus |

Новые плейбуки добавляются только когда один и тот же тип работы повторился ≥3 раз и не лёг в существующие.

## 6. Инварианты (все — проверяемые)

1. Чат без плейбука не существует (кроме `console`-вопросов).
2. Задача без `done_when` не попадает в STATE.md.
3. Одна веха `focus` на направление; задачи существуют только у фокусной вехи.
4. Чат не закрыт, пока агент не закоммитил `state_update`.
5. `session_close.result` — материальный результат, не handoff и не план.
6. MAP.md правится только в `define_direction` и `review_milestone`.
7. Ядро + реестр ≤ 1500 слов; плейбук ≤ 600 слов. Правка, нарушающая бюджет, отклоняется.

## 7. Роли инструментов

| Роль | Инструмент | Права |
|---|---|---|
| Мышление, планирование, формирование | ChatGPT Project (по проекту на направление) | Читает DIRECTION/MAP/STATE; не пишет в репо |
| Исполнение и запись | Codex / Claude Code | Прямой доступ к репо; применяет state_update; делает engineering-задачи |
| Арбитр | Человек | Решения; может вклиниться в любом месте, по умолчанию подтверждает |

### Executor handoff: два режима

- **`mode: mechanical`** — точное применение изменений состояния (STATE.md, LOG.md, knowledge/). Без суждений исполнителя.
- **`mode: engineering`** — бизнес-задача в кодовый репозиторий направления (пример: репозиторий игры). Конверт несёт цель, `done_when` (критерии приёмки), контекст, границы. Исполнитель сам владеет декомпозицией, дизайном, реализацией; возвращает результат + доказательства (PR/коммиты/тесты). Родительский чат НЕ микроменеджит пути/SHA/ветки.

## 8. Обслуживание воркфлоу (вместо направления workflow-governance)

- Воркфлоу — инфраструктура, у неё нет вех и цели «достигать». Целевое состояние — ноль работы над ней.
- Правка ядра/плейбуков = задача для агент-сессии (Claude Code/Codex) на этом репо.
- Инвариант: правка не увеличивает ядро сверх бюджета (§6.7).
- Журнал: `WORKFLOW_CHANGELOG.md` (строка на правку: дата, что, почему).
- Триггер правки: одно и то же трение повторилось ≥2 раз в реальных сессиях. Не «предусмотреть заранее».

## 9. Открытые вопросы к пилоту

1. Чтение STATE.md ChatGPT-проектом: GitHub-коннектор vs вставка руками (STATE.md мал по построению — вставка приемлема как старт).
2. Достаточно ли `console` для всех «обычных вопросов» или нужен формат «консоли направления» (дашборд-проекция).
3. Порог, когда engineering-репо направления заводит собственный мини-STATE (для игры — вероятно да, на стороне репо игры: CLAUDE.md/AGENTS.md + бэклог).
