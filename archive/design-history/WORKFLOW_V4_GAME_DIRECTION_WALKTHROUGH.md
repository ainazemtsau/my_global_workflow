# Workflow v4 — валидационный прогон: направление «Инди-игра»

Дата: 2026-06-10
Статус: иллюстративный сценарий для валидации концепта (`docs/WORKFLOW_V4_CORE_CONCEPT.md`)

Цель документа: показать «на пальцах», как ведут себя чаты в направлении
«создать инди-игру и сделать её коммерчески успешной», и что воркфлоу
структурно не даёт схлопнуться в одну разработку (провал v1).

Реплики сокращены; форматы соответствуют ядру v4. Владелец — технический
человек, слабые компетенции: маркетинг, бизнес, соцсети.

---

## Чат 1 — `define_direction`

**Человек:**
> Хочу сделать инди-игру и довести её до коммерческого успеха. Я технарь,
> в маркетинге и бизнесе не понимаю ничего.

**Модель** (3–5 вопросов): жанр/референсы? часов в неделю? «успех» в цифрах?
готов ли показывать незаконченное публично? бюджет на сторонние услуги?

**Человек** отвечает одним сообщением.

**Модель** формирует и показывает:

DIRECTION.md (фрагмент):
```yaml
goal: "Выпустить инди-игру и сделать её коммерчески успешной"
success_criteria:
  - "игра выпущена в Steam"
  - "≥ 5000 продаж или окупаемость затрат за 12 мес после релиза"
boundaries: "соло-разработка + ИИ; деньги на аутсорс — только после явного решения"
```

MAP.md (фрагмент):
```yaml
tracks:
  - id: design     # геймдизайн
  - id: dev        # разработка
  - id: marketing  # маркетинг/комьюнити  ← слабая компетенция владельца
  - id: business   # бизнес/финансирование ← слабая компетенция владельца

milestones:
  - id: M-1
    goal: "Концепция проверена на живых людях и на рынке"
    exit_criteria: "10 человек сыграли в прототип, ≥6 хотят ещё;
                    разбор 5 похожих инди-игр: что дало им продажи"
    track_impact: {design: "концепт, плейтесты", dev: "прототип core loop",
                   marketing: "разбор рынка", business: "not_needed: рано, в M-3"}
    sketch: "концепт-документ → прототип → плейтесты → разбор рынка"
    status: future

  - id: M-2
    goal: "Vertical slice + публичное лицо игры"
    exit_criteria: "демо играбельно 20 минут; Steam-страница опубликована;
                    первый ролик; 500 wishlist'ов"
    track_impact: {design: "...", dev: "...", marketing: "страница, ролик,
                   первые стримеры", business: "not_needed"}
    status: future

  - id: M-3
    goal: "Принято решение о финансировании производства"
    exit_criteria: "выбрано Kickstarter | издатель | самиздат — на основе
                    данных M-2 (динамика wishlist'ов, отклик)"
    track_impact: {business: "сравнение путей, pitch-материалы",
                   marketing: "данные для решения", dev: "not_needed", design: "not_needed"}
    status: future

  - id: M-4 (production)  # грубо, 2 строки
  - id: M-5 (launch)      # грубо, 2 строки
```

Заметить: **exit_criteria — во внешней реальности** («10 человек сыграли»,
«500 wishlist'ов»), не «код написан». Уже M-1 содержит маркетинговую работу —
силос разрушен с первой вехи.

**session_close** чата 1:
```yaml
result: "DIRECTION.md и MAP.md направления indie-game созданы"
state_update: "focus_milestone: none; next_chat: plan_focus M-1"
log_entry: "2026-06-10 define_direction: карта из 5 вех, 4 трека"
```
Агент (Codex/Claude Code) коммитит файлы. Человеку выдан готовый handoff —
он просто открывает следующий чат и вставляет его.

## Чат 2 — `plan_focus` (веха M-1)

Модель читает MAP/STATE, раскладывает M-1 и показывает **track coverage check**:

```
design:    ДА → T-1 "концепт-документ на 1 страницу" 
               T-4 "протокол плейтеста + проведение с 10 людьми"
dev:       ДА → T-2 "прототип core loop" (executor: engineering, репо игры)
marketing: ДА → T-3 "разбор 5 похожих инди: каналы, что дало wishlist'ы/продажи"
business:  НЕТ — причина: данных для решений ещё нет, вернёмся в M-3. Согласен?
```

**Человек:** «да». STATE.md:
```yaml
focus_milestone: M-1
tasks:
  - {id: T-1, goal: "концепт-док 1 стр", done_when: "док в work/, владелец принял", status: open}
  - {id: T-2, goal: "прототип core loop", done_when: "собирается, 1 сессия играбельна, README", status: open}
  - {id: T-3, goal: "разбор 5 похожих инди", done_when: "work/market-scan.md: каналы и выводы", status: open}
  - {id: T-4, goal: "10 плейтестов", done_when: "≥10 ответов по протоколу", status: open, notes: "после T-2"}
next_chat: {playbook: work_task, task: T-1}
```

---

## Чат 3 — `work_task` T-1 (геймдизайн)

Обычная рабочая сессия: модель-собеседник помогает сформулировать концепт.
Результат — `work/concept.md`. session_close → next_chat: work_task T-2.

---

## Чат 4 — `work_task` T-2 (разработка) — executor call

Чат уточняет с человеком 2–3 решения уровня бизнеса (движок уже выбран в
концепте) и формирует **ход ядра executor call**:

```yaml
handoff:
  direction: indie-game
  task: T-2
  role: executor
  mode: engineering
  goal: "Прототип core loop по work/concept.md"
  context: "репо: ainazemtsau/<game-repo>; концепт: directions_v4/indie-game/work/concept.md"
  done_when: "проект собирается; одна игровая сессия играбельна; README с запуском"
  return: "ссылка на PR/коммит + краткое описание решений + что не вошло"
  boundaries: "только репо игры; без платных ассетов"
```

Claude Code/Codex в репо игры **сам** решает архитектуру и реализацию,
возвращает PR. Родительский чат не управляет путями/ветками/SHA — приёмка
только по `done_when`. session_close: T-2 → done (или blocked, если возврат
не прошёл приёмку).

---

## Чат 5 — `work_task` T-3 (маркетинг) — чат как компетенция

Здесь видно, зачем не-разработческие задачи живут в воркфлоу: ChatGPT и есть
маркетолог владельца. Чат разбирает 5 игр, выдаёт:

- `work/market-scan.md`: «в этом жанре wishlist'ы приносят демо + мелкие
  стримеры; платная реклама не окупалась; Steam Next Fest — ключевое окно»;
- `knowledge_candidate`: «маркетинговый канал жанра: демо → стримеры → Next Fest»;
- внутри чата был **child call**: `child_research` «собери даты ближайших
  Steam Next Fest и условия участия» — вернулся result'ом в родителя.

---

## Чат 6 — `work_task` T-4 (плейтесты)

Протокол плейтеста; задача частично у человека (дать игру 10 людям);
чат закрывается с `open_questions: ["результаты плейтестов?"]`, T-4 → blocked.
Через несколько дней человек открывает next_chat и вбрасывает ответы;
T-4 → done. Очередь пуста → next_chat: review_milestone M-1.

---

## Чат 7 — `review_milestone` M-1 — здесь рождаются кросс-трековые решения

1. **Сверка exit_criteria:** 11 плейтестов, 7 «хочу ещё» → веха закрыта.
2. **Обязательные пер-трековые выводы** («что узнали, что меняет другие треки?»):

```
design:    фича «крафт» не зашла 9/11 → выкинуть из M-2 (правка sketch M-2)
dev:       прототип на движке X собран быстро → оценка M-2 реалистична
marketing: из market-scan: ролик и демо к Next Fest (окно через 4 мес) →
           ПРЕДЛОЖЕНИЕ: сдвинуть ролик в начало M-2 и привязать M-2 к дате Next Fest
business:  из плейтестов есть цитаты игроков → ПРЕДЛОЖЕНИЕ: новая веха-кандидат
           M-2b "pitch-ready" (deck для издателей), потому что путь «издатель»
           требует материалов раньше, чем мы планировали
```

3. Модель предлагает варианты, человек отвечает одной строкой
   («ролик сдвигаем, M-2b добавляем после M-2»). MAP.md правится, коммитится.
4. next_chat: plan_focus M-2.

**Это и есть требуемый механизм:** решение «снять ролик / готовить pitch
издателю» не зависит от инициативы владельца — его обязана предложить
структура review_milestone, владелец только выбирает.

---

## Дальше по циклу (сжато)

- `plan_focus M-2`: track coverage check даёт задачи всех четырёх линз:
  демо (dev), контент-план и Steam-страница (marketing — чат-маркетолог пишет
  тексты страницы), ролик (design+marketing; сценарий — чат, монтаж — задача
  человека или аутсорс через business-задачу), «20 стримеров жанра» (marketing).
- `review_milestone M-2`: смотрим динамику wishlist'ов → данные для M-3.
- `plan_focus M-3` (business-веха): чат-аналитик сравнивает Kickstarter vs
  издатель vs самиздат на данных M-2, готовит материалы; человек принимает
  решение. Разработка в этой вехе — `not_needed`, и это нормально: воркфлоу
  не «помогает только с разработкой», вехи бывают целиком бизнесовыми.

---

## Что валидирует этот прогон

| Требование владельца | Где обеспечено |
|---|---|
| Не схлопнуться в разработку | exit_criteria во внешней реальности; обязательный track coverage check в plan_focus; обязательные пер-трековые выводы в review_milestone |
| Комплексные решения (ролик, Kickstarter, издатель) | review_milestone обязан предлагать варианты вех из разных треков |
| Чат выдаёт следующий чат, цикл замыкается | session_close.next_chat в каждом чате |
| Минимум участия человека | человек отвечает на вопросы и выбирает из предложенного; вся запись — агентом |
| Child-чаты, рекурсивно | ход ядра child call (пример в чате 5) |
| Код игры — отдельно и по-взрослому | executor call mode: engineering (чат 4): бизнес-задача + приёмка, реализацией владеет агент в репо игры |
| Слабые компетенции владельца | чаты plan/work в треках marketing/business сами являются носителем компетенции |

Реалистичность для ChatGPT: каждый чат держит в голове только ядро (≤1500 слов),
один плейбук (≤600 слов) и маленькие MAP/STATE. Это в ~20 раз меньше контекстной
нагрузки, чем требовал v3 на один чат.
