# Аудит Workflow v3 и предложение Workflow v4

Дата: 2026-06-10
Статус: отчёт аудита, не принятое состояние

---

## 1. Вердикт

**Идея воркфлоу рабочая. Реализация — нет, и она не чинится точечными правками. Нужно пересобрать заново на других принципах, сохранив ядро идеи.**

Конкретно:

- **Не дописывать v3.** 12 из 23 процедур — заглушки. Дописывать их — значит наращивать систему, которая уже структурно перегружена.
- **Не чинить v3 правками.** Дефекты не локальные (опечатки, неудачные поля), а системные: циклические ссылки авторитета, неопределённые поля-гейты, правила, которые LLM физически не может проверить, и нагрузка в 60–80 полей на один чат.
- **Пересобрать v4 с нуля**, перенеся только смысловое ядро: направления, жизненный цикл чата, дочерние чаты, состояние в git. Объём ядра v4 — порядка 5–7 файлов вместо 124.

## 2. Главный вывод аудита: проблема не в деталях, а в модели

Через все четыре итерации (vNext-R → v2 → v3 → текущая) повторяется один и тот же цикл:

1. В чатах возникает трение (модель «срезает углы», теряет контекст, путает состояние).
2. Ответ на трение — **добавить больше управления**: больше протоколов, полей, гейтов, аудитов, запретов.
3. Больше управления → больше текста, который модель должна удержать → больше двусмысленностей и противоречий → **больше трения**.
4. Система переписывается заново. GOTO 1.

Это видно в цифрах:

| Метрика | Значение |
|---|---|
| Слов во всех .md репозитория | ~230 000 |
| Слов в workflow_v3 | ~67 500 |
| Сущностей в модели данных | 61 |
| Шаблонов | 40 (≈506 полей суммарно) |
| Полей, которые один чат обязан отслеживать/эмитить | 60–80 |
| Полей на полный цикл направления (3 узла работы) | ~600–750 |
| Процедур-заглушек | 12 из 23 (52%) |
| Мета-коммитов (rename/align/sweep/migrate) за июнь | ~40% всех коммитов |
| Реальный результат под v2 | планы созданы, исполнение «downstream_not_started» |
| Чатов до состояния «можно работать» в новом направлении (v3) | ~8–9 чатов, ~44 ручных шага |

Ключевой факт из истории: **направление health-and-beauty за 3 дня под v2 произвело 20 receipts и 145 claims чистого планирования — и ноль исполнения.** Воркфлоу, созданный для продуктивности, потребляет продуктивность. Workflow-governance стал самым активным «направлением» — система в основном обслуживает саму себя.

## 3. Корневые причины (5 штук)

### 3.1. Система борется с природой LLM вместо того, чтобы её использовать

v3 пытается превратить ChatGPT в детерминированный исполнитель процессов: юридическая проза («must not invent ad hoc compression»), 8 состояний жизненного цикла, обязательные сигналы, 30-пунктовый аудит на FINISH. LLM плохо исполняет законы, написанные прозой, и хорошо работает по компактному контексту + здравому суждению. Каждое «нельзя срезать» не предотвращает срезание — оно лишь съедает контекст и порождает церемонию: статус-вопрос на 2 минуты требует START → RUN → CHECK → FINISH → CLOSED с 4–5 раундами подтверждений (`СТАРТ`, `ДАЛЬШЕ`, `ФИНИШ`).

### 3.2. Правила, которые исполнитель не может проверить

ChatGPT обязан верифицировать commit SHA, следить за непересечением `allowed_paths`/`forbidden_paths`, считать символы payload, проверять свежесть GitHub-чтений — при этом у него нет надёжного доступа к GitHub и нет инструментов для этого. Каждое такое правило = гарантированная точка отказа «в каждом чате». Это именно те «мелкие, но серьёзные» проблемы, которые ты наблюдаешь.

### 3.3. Человек — транспортная шина между ИИ

Все записи в репозиторий идут через цепочку: ChatGPT формирует пакет (21 поле) → ты копируешь в Codex → Codex пишет → ты копируешь возврат обратно → ChatGPT верифицирует. Плюс ручное обновление Project Instructions UI, ручной подсчёт символов, ручное ведение 5+ артефактов на закрытие чата. Один 10-минутный шаг превращается в 30+ минут переключений. Это архитектурная проблема, её нельзя починить текстом протоколов.

### 3.4. Размытие ответственности и дубли сущностей

«Что делать дальше» представлено 6 сущностями (CURRENT_NEXT_MOVE, NEXT_MOVE_PACKET, NEXT_CHAT_PROMPT, continuation в FINISH_PACKET и RESULT_PACKET, copy_paste_packet в TRANSFER_PACKET). «Что сделано» — 5 сущностями. «Принято ли» — 4. Ни одна сущность не владеет вопросом целиком. Авторитет закольцован: lifecycle говорит «completion определяет процедура», registry говорит «я не владею completion-семантикой», процедуры говорят «нас выбирает lifecycle». Поля-гейты (`repair_needed`, `required_dependency_work_detected`, `open_dependencies`) используются в блокирующих условиях, но нигде не определено, кто и когда их заполняет.

### 3.5. Несколько «источников истины» одновременно

`workflow/**` (актуальный OS), `workflow_v3/**` (skeleton), `directions/**` (legacy evidence), `directions_v3/**` (пустой, будущий) + WORKFLOW_SOURCE_OF_TRUTH.md, который сам объявляет, что он не семантический авторитет. Модель, попадающая в чат, должна сначала решить нерешаемую задачу «что здесь вообще истина».

## 4. Проблемы, с которыми ты ещё НЕ столкнулся (но столкнёшься)

Ты тестировал перевод процедур, но не создание направления. Вот что ждёт впереди, если продолжать v3:

1. **Создание направления — ~8–9 чатов и ~2.5 часа до «work-ready»**, даже если ты с первого сообщения знаешь, чего хочешь (bootstrap → definition → spine → acceptance → map → acceptance → front → acceptance → contract). Любая ошибка на шаге — повтор.
2. **Хрупкий binding.** Следующий чат «нестабилен», пока ты вручную не обновишь Project Instructions UI капсулой биндинга. Забыл один раз — чат не может определить, какому направлению он принадлежит.
3. **Codex-пакеты будут падать**: несовпадение base SHA из-за движения main, пересечение path-границ, неполные поля. Без авто-ретрая каждый отказ = ручная регенерация пакета.
4. **Переполнение бюджета инструкций.** Payload уже 6440/6500 целевых символов. Капсула биндинга (~300 симв.) и любые уточнения выбивают за лимит.
5. **Параллельные направления** усилят всё перечисленное: ручная защита sibling-путей, ручная маршрутизация пакетов между проектами.
6. **Деградация памяти**: MEMORY_INDEX/ARTIFACT/CANDIDATE — три сущности без процесса устаревания; через месяцы это будет свалка, которую модель будет читать как истину.

## 5. Что в идее правильно (это сохраняем)

1. **Направления как автономные единицы** со своей целью, состоянием и знанием.
2. **Git как долговременное состояние** — переживает чаты, версионируется, проверяется.
3. **Один чат = одна работа** с явным завершением.
4. **Дочерние чаты** для побочной работы с явным пакетом передачи (рекурсивно).
5. **Явный handoff между чатами** вместо «памяти» модели.
6. **Различие черновик / принятое состояние** (acceptance) — но как один лёгкий механизм, а не размазанный по 8 шаблонам.

Иерархия «Направление → текущий фокус → задачи» тоже здравая. Spine/Map/Front/Graph/Node/GoalEvidenceGraph — пять уровней там, где хватает двух-трёх.

## 6. Предложение: Workflow v4

### 6.1. Принципы (важнее структуры)

1. **Бюджет контекста — закон.** Всё, что обязан соблюдать чат, помещается в ОДИН файл ≤ 1500 слов. Если правило не влезло — правило упрощается, а не файл расширяется.
2. **Машина пишет — человек решает.** Никаких ручных копипастов состояния. Всю запись в репозиторий делает агент с прямым доступом к репо (Codex / Claude Code). Человек только принимает решения и говорит «да/нет».
3. **Правило существует, только если его можно проверить.** Никаких «must not invent shortcuts». Вместо запретов — устойчивость: любое нарушение дёшево обнаруживается и чинится следующей сессией (recovery вместо prevention).
4. **Одна сущность — один вопрос.** «Что дальше?» живёт ровно в одном месте. «Что сделано?» — в одном. «Принято ли?» — в одном.
5. **Структура — это подсказка, а не клетка.** Playbook'и (бывшие процедуры) — это руководства, которые модель читает по необходимости, а не закон, за нарушение которого блокируется закрытие.
6. **Ноль мета-направлений.** Никакого workflow-governance как направления. Обслуживание воркфлоу — обычная задача в обычном репо.

### 6.2. Модель данных: 4 сущности, ~35 полей суммарно

**Направление** — папка `directions_v4/<id>/`:

```
directions_v4/<id>/
  DIRECTION.md      # цель, критерии успеха, границы, констрейнты (заполняется один раз, правится редко)
  STATE.md          # ЕДИНСТВЕННЫЙ живой файл: текущий фокус, активные задачи, следующий шаг, блокеры
  LOG.md            # append-only журнал закрытых сессий (по 5-10 строк на сессию)
  knowledge/        # принятые решения и знания направления (по файлу на тему)
  work/             # рабочие артефакты (планы, рецепты, код-ссылки и т.п.)
```

**Задача (Task)** — блок в STATE.md, не отдельный файл:

```yaml
- id: T-12
  goal: "одно предложение"
  done_when: "проверяемый критерий"
  status: open | active | blocked | done
  notes: "1-2 строки контекста"
```

**Handoff** — ЕДИНСТВЕННЫЙ тип пакета для всего (новый чат, дочерний чат, передача в Codex). 7 полей:

```yaml
handoff:
  direction: <id>
  task: T-12
  role: continue | child | executor      # продолжение, дочерняя работа, исполнение в Codex/агенте
  goal: "что сделать"
  context: "минимально достаточный контекст (или ссылки на файлы)"
  return: "что вернуть и в каком виде"
  boundaries: "что нельзя трогать (опционально)"
```

**Session close** — ЕДИНСТВЕННАЯ форма закрытия чата. 6 полей:

```yaml
session_close:
  task: T-12
  result: "что реально сделано (не план, не handoff)"
  artifacts: [пути/ссылки]
  state_update: "diff для STATE.md: статусы задач, следующий шаг"
  log_entry: "строка для LOG.md"
  open_questions: []   # пусто или вопросы к человеку
```

Это покрывает все 10 зон дублирования v3: FINISH/RESULT/TRANSFER/NEXT_MOVE/NEXT_CHAT/CLOSURE-пакеты схлопываются в `handoff` + `session_close`. CURRENT_STATUS/CONSOLE/BINDING не нужны — состояние читается из STATE.md.

### 6.3. Жизненный цикл чата: 3 шага вместо 8 состояний

1. **Открытие.** Чат получает handoff (или человек просто пишет задачу). Модель читает STATE.md направления, подтверждает в 3 строках: задача, критерий готовности, план. Без команды «СТАРТ» — человек просто отвечает или поправляет.
2. **Работа.** Обычная работа. Если нужна побочная работа — модель выдаёт `handoff` с `role: child` или `role: executor`. Никаких состояний ожидания: родительский чат просто фиксирует в STATE.md «T-13 blocked: ждёт возврата T-13a» и может продолжать остальное или закрыться.
3. **Закрытие.** Модель выдаёт `session_close`. Агент-исполнитель (Codex/Claude Code) применяет `state_update` к STATE.md и LOG.md и коммитит. Человек только подтверждает.

Проверка «не наврала ли модель о завершении» — это `done_when` в задаче: один проверяемый критерий вместо 30-пунктового аудита.

### 6.4. Распределение ролей по инструментам (ключевое изменение)

| Роль | Инструмент | Права |
|---|---|---|
| **Мышление и решения** (планирование, формирование направления, разбор, дочерние «исследовательские» чаты) | ChatGPT Project (по проекту на направление) | Читает STATE.md/DIRECTION.md (вставлены или через connector); НЕ отвечает за запись |
| **Исполнение и запись** (любая мутация репо: код, обновление STATE.md/LOG.md, файлы знаний) | Codex / Claude Code с прямым доступом к репо | Пишет и коммитит сам; человек не копирует пакеты руками |
| **Арбитр** | Человек | Принимает/отклоняет, задаёт направление |

Это устраняет причину 3.3 целиком: исчезают DEPENDENCY_CALL/RETURN с 21 полем, верификация SHA силами ChatGPT, ручной подсчёт символов, ручное обновление binding. Project Instructions для ChatGPT-проекта сводятся к ~1 экрану: «ты — мыслительный партнёр направления X, вот его STATE.md, на закрытии выдай session_close».

Если ChatGPT-проект имеет доступ к GitHub-коннектору — STATE.md читается сам; если нет — начало чата это одна вставка STATE.md (он маленький по построению).

### 6.5. Playbooks вместо 23 процедур

Пять коротких руководств (каждое ≤ 600 слов), читаются по необходимости, не являются законом:

1. `NEW_DIRECTION.md` — создать направление за ОДИН чат: цель, критерии, первые 3 задачи. Не 8 чатов.
2. `WORK_SESSION.md` — обычная рабочая сессия.
3. `PLANNING_SESSION.md` — пересмотр STATE.md: фокус, добор задач, чистка.
4. `CHILD_TASK.md` — когда и как порождать дочерний чат / executor-handoff.
5. `RECOVERY.md` — что делать, если STATE.md противоречит реальности (восстановление по LOG.md и артефактам).

### 6.6. Жизненный цикл знаний

- Закрытие сессии может предложить `knowledge_candidate` (1 поле в session_close).
- Принятые — отдельные файлы в `knowledge/` с заголовком `valid_as_of: <дата>`.
- Раз в N недель planning-сессия помечает устаревшее. Никаких MEMORY_INDEX.

## 7. План перехода

1. **Заморозить, не удалять.** `workflow_v3/**`, `workflow/**`, `directions/**` остаются как архив. Один абзац в README: «активная система — workflow_v4, остальное — история».
2. **Написать ядро v4**: 1 файл ядра (lifecycle + handoff + session_close, ≤1500 слов), 5 playbooks, 1 шаблон DIRECTION.md/STATE.md. Это один день работы, а не месяц.
3. **Пилот на health-and-beauty.** Это идеальный тест: планы (питание, тренировки) уже существуют и приняты — нужно именно исполнение, на котором v2 остановился. Перенести в `directions_v4/health-and-beauty/` только смысл: цель в DIRECTION.md, планы в work/, 3 стартовые задачи в STATE.md. Receipts/claims не переносить — это история, не состояние.
4. **Критерий успеха пилота:** 10 рабочих сессий подряд без правки правил воркфлоу. Если в сессии возникло трение — это баг ядра v4, чинится минимальной правкой (и правка не должна увеличивать ядро сверх бюджета).
5. **Потом** — solo-max-productive (EXOCORTEX) и новые направления.

## 8. Сводка дефектов v3 (для истории)

Полные отчёты четырёх аудитов зафиксировали, в частности:

- Коллизия терминов «owner procedure» vs «main procedure» (ROUTING_AND_DEPENDENCY_PROTOCOL.md:2 vs CHAT_LIFECYCLE_PROTOCOL.md:34).
- `CODEX_RETURN_PACKET` используется в CHAT_LIFECYCLE_PROTOCOL.md:162 при явном запрете прежних меток пакетов в ROUTING_AND_DEPENDENCY_PROTOCOL.md:30.
- Поля-гейты `repair_needed`, `required_dependency_work_detected`, `open_dependencies` нигде не имеют схемы/владельца, но блокируют CHECK/FINISH.
- Циклический авторитет: lifecycle ↔ registry ↔ procedure canon по вопросу владения completion.
- Состояния `RUN_REPAIR_REQUIRED` / «return to RUN repair» не определены поведенчески.
- Требования к ChatGPT верифицировать git-состояние (SHA, path-границы), к которому у него нет доступа.
- «Project Instructions» и «refresh classifications» используются в STORAGE_UPDATE_PROTOCOL.md:41 без определения.
- 6 сущностей для «что дальше», 5 для «что сделано», 4 для «принято ли», 4 закрывающих сводки с пересекающимися полями.
- ~44 ручных шага и ~8–9 чатов до работоспособного направления; ~5 ручных артефактов на закрытие каждого чата.
