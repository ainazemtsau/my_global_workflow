# Workflow v5 — Кодовый слой

Версия: draft для обсуждения. Дата: 2026-06-10.
Дополняет `WORKFLOW_V5_CONCEPT.md` (раздел 7). Та же философия: структура вместо дисциплины, контекст вместо запретов, контроль на границах — свобода внутри.

---

## 1. Место кодового слоя в системе

Направления порождают **Проекты** — кодовые продукты в отдельных репозиториях: трекер калорий для health-and-beauty, игра для indie-game-dev, инструменты самого воркфлоу.

Граница миров жёсткая и односторонняя:

```text
НАПРАВЛЕНИЕ (репо воркфлоу)            ПРОЕКТ (свой репозиторий)
решает ЧТО и ЗАЧЕМ                     знает КАК
знает о своих проектах        ─────►   о направлении не знает ничего
шлёт TASK-карточку                     самодостаточен для кодовой работы
получает RESULT               ◄─────   возвращает RESULT + demo
```

Направление знает о проекте (реестр в CHARTER → Параметры, решения в knowledge/). Проект о направлении не знает: всё, что код-ассистенту нужно для работы, лежит **внутри репозитория проекта**. Поэтому кодовая сессия не требует контекст-пакета из воркфлоу — её пакет это сам репозиторий + TASK-карточка. Это и делает слой совместимым с любым ассистентом (Codex, Claude Code, следующий) и готовым к автономности.

Интерфейс между мирами — те же два формата, что и везде в v5: карточка на входе, RESULT на выходе.

## 2. Анатомия проекта

Каждый проект создаётся бутстрапом (раздел 4) и несёт стандартный каркас:

```text
<project-repo>/
  AGENTS.md            конституция для код-ассистентов (стандарт Linux Foundation:
                       Codex читает нативно, Claude Code — как fallback; CLAUDE.md
                       не дублируется, при необходимости — однострочная ссылка)
  ARCHITECTURE.md      карта модулей: список, назначение, разрешённые зависимости
  docs/
    DECISIONS.md       журнал архитектурных решений (строка-абзац на решение)
    policies/          правила проекта (тесты, стиль, технология-специфика)
  specs/
    changes/<id>/      спека текущего изменения: proposal + design + tasks
    archive/           принятые/выполненные спеки
  journal/
    LESSONS.md         сырьё для ретроспектив: сбои валидации, кривые команды,
                       повторяющиеся проблемы (append-only)
  scripts/
    validate.sh        ЕДИНАЯ точка валидации (раздел 6) — её зовут все ассистенты
    init.sh            поднять окружение/дев-сервер одним действием
  .claude/agents/      сабагенты-ревьюеры для Claude Code (тонкий слой, опционально)
  <модули по правилам техпрофиля>
```

Принцип: **всё важное — файлы в репо, исполняемое — скрипты**. Ассистент-специфика (.claude/, конфиги MCP) — тонкий слой поверх, который можно потерять без потери системы.

`AGENTS.md` проекта ≤150 строк: что это, как запустить (`init.sh`), карта модулей (ссылка на ARCHITECTURE.md), как валидировать (`validate.sh`), правила работы (раздел 5), формат RESULT и demo-инструкции, политика спек. Не энциклопедия — указатель на структуру.

## 3. Модульность

Цель: ассистент работает сфокусированно в малом контексте, код переиспользуется, база не разрастается. Механизмы — в порядке силы:

1. **Карта модулей (ARCHITECTURE.md).** Список модулей, у каждого: назначение одной строкой, public API (что можно импортировать снаружи), от чего зависит. Правило зависимостей по слоям (например `app → feature → domain → core`; вглубь можно, вбок и вверх нельзя).
2. **Механическое исполнение границ — главный механизм.** Настраивается бутстрапом под стек: eslint-plugin-boundaries / dependency-cruiser (JS/TS), import-linter (Python), **asmdef-сборки (Unity — границы модулей исполняет сам движок)**, Nx/turbo graph (монорепо). Нарушил границу — валидация падает мгновенно, без LLM-ревью. Это «структура вместо дисциплины» в чистом виде.
3. **Правило работы в карточке.** Кодовая задача называет целевой модуль(и). Ассистент читает целиком только их; из остальных — MODULE-заголовок/public API. Перед созданием нового кода — обязательный поиск существующего (переиспользование) внутри разрешённых зависимостей.
4. **Новый модуль — это решение, не побочный эффект.** Создание модуля идёт отдельной строкой в плане и попадает в ARCHITECTURE.md и DECISIONS.md.

## 4. Бутстрап проекта

Создание проекта — обычная работа направления (тип build, плейбук `project-bootstrap`), которой предшествуют research/design-работы (выбор технологии, скоупа MVP — как обычно в фокусе).

Вход — карточка бутстрапа: имя, цель проекта, технология(и), целевой git-хостинг, особые требования. Исполняет код-ассистент:

1. Берёт **техпрофиль** из ядра: `kernel/tech/<stack>.md` (unity.md, python-service.md, web-app.md, …; ≤100 строк каждый). Профиль определяет: структуру каталогов и модульную схему, инструмент исполнения границ, тулчейн валидации (build/lint/test/coverage), стартовые policies (включая известные грабли технологии — например для Unity: «тестовые сцены только в `Assets/Scenes/Sandbox/`, имя `SB_<тема>`, удаляются на ретро»), рекомендуемые навыки/MCP и когда их включать.
2. Создаёт репозиторий и весь каркас раздела 2, настраивает border-линтеры, пишет validate.sh и init.sh, генерирует AGENTS.md из шаблона ядра + профиля.
3. Прогоняет validate.sh на пустом каркасе (зелёный с нуля), коммитит.
4. RESULT в направление: ссылка на репо, что настроено, demo-инструкция «как убедиться, что каркас живой».

Нет профиля под нужный стек — сначала маленькая research-работа «составить техпрофиль X» (профиль пополняет ядро и служит всем будущим проектам).

## 5. Цикл кодовой задачи

Главный сценарий: ты приносишь карточку → подтверждаешь план → уходишь → возвращаешься к готовому RESULT. Между планом и RESULT участие человека не предусмотрено.

```text
TASK-карточка (из работы направления; формат JOB + поля: проект, модули, спека-id)
  │
  ▼
ПЛАН (single gate для человека)
  ассистент исследует репо → пишет спеку изменения в specs/changes/<id>/:
  что меняем и зачем / дизайн / задачи по шагам / влияние на модули /
  план валидации / что покажет demo
  → показывает тебе. Ты: «поехали» (или правишь). Дальше — без тебя.
  │
  ▼
РЕАЛИЗАЦИЯ (автономно)
  по задачам спеки, в границах модулей; коммиты мелкие, по задаче;
  сюрпризы и обходы — строкой в journal/LESSONS.md
  │
  ▼
ВАЛИДАЦИЯ (автономно, раздел 6)
  validate.sh + ревью-сабагенты
  │  fail → доработка → снова валидация
  │  СТОП-условия выхода из цикла:
  │    та же ошибка 3-й раз ИЛИ 5-я итерация валидации ИЛИ
  │    требуется решение вне спеки (границы, скоуп, продукт)
  │  → СТОП-отчёт человеку: что пытался, что падает, гипотезы, варианты
  ▼
RESULT (формат v5) + обязательные поля кодовой задачи:
  demo: как посмотреть результат (что запустить / какая сцена / что тыкнуть)
  ручные_шаги: пошагово, если без ручной настройки не работает
                (настройки Unity-инспектора, ключи, миграции), и как проверить
  валидация: что прогнано, что НЕ покрыто
  → ты смотришь demo → «принято» → RESULT уходит в направление (apply)
```

Пороги «3 одинаковых / 5 итераций» — параметры AGENTS.md проекта, по умолчанию такие. СТОП — это не провал, а штатная эскалация: система знает, когда позвать тебя.

## 6. Валидация: три уровня

Философия: машинное — машине, смысловое — свежему взгляду, ручное — по инструкции. Никогда не «сам себя проверил и сам себе поверил».

**Уровень 0 — механика (validate.sh).** Build, типы, линт, тесты, покрытие нового кода, **границы модулей**, формат. Один скрипт, одинаково запускается любым ассистентом, CI и тобой. Настраивается бутстрапом из техпрофиля.

**Уровень 1 — свежий взгляд (ревью-сабагенты).** Отдельные контексты с критической установкой; исполнитель не оценивает сам себя:

- *code-reviewer*: diff против ARCHITECTURE.md и policies — границы, дубли, хардкоды, мёртвый код, соответствие спеке;
- *test-reviewer*: **качество тестов, а не факт их наличия** — твой опыт «оба ассистента пишут мусорные тесты» закрывается явным чеклистом: тест падает, если сломать реализацию? проверяет поведение, а не структуру? нет тавтологий и заглушек ради покрытия? кейсы граней есть?
- *профильные ревьюеры из техпрофиля*: для Unity — сцены/префабы/ресурсы по policy, аллокации в Update; для сервиса — миграции, обратная совместимость API.

В Claude Code это нативные сабагенты (.claude/agents/, создаются бутстрапом). В Codex — те же чеклисты ревью исполняются отдельной сессией по инструкции из AGENTS.md. Содержание ревью живёт в репо (docs/policies/review/*.md), ассистент-обвязка — тонкая.

**Уровень 2 — поведение.** E2e по плану валидации из спеки (если стек позволяет — авто: playwright, Unity TestRunner PlayMode). Что машиной не проверить — попадает в `ручные_шаги` RESULT с точными шагами и ожидаемым результатом. Непроверенное называется непроверенным — это поле обязательное.

## 7. Самоисправляющийся цикл (ретроспектива)

1. **Сбор — бесплатный.** Всё, что скрипело по ходу сессий, ассистент пишет строкой в journal/LESSONS.md: упавшая валидация и причина, не сработавшая команда, грабли технологии, расплодившийся мусор (те самые тестовые сцены).
2. **Ретро — отдельная работа** (плейбук `retro`, запускается по триггеру: ~10 кодовых сессий, или СТОП-эскалация, или заметная боль). Читает LESSONS + DECISIONS: какие проблемы повторялись ≥2 раз? Что чинится структурно — правкой policy, профиля, validate.sh, AGENTS.md? Выход — RESULT с конкретными правками файлов проекта (а урок уровня технологии — апстримом в kernel/tech/<stack>.md, чтобы следующий проект родился умнее).
3. **Правило то же, что в ядре:** правило добавляется после двух реальных повторов, при добавлении что-то удаляется. Policies не разрастаются в кодекс.

## 8. Инструменты: вердикты по итогам ресёрча

| Инструмент | Вердикт | Почему |
| --- | --- | --- |
| **AGENTS.md** | ядро слоя | открытый стандарт (Linux Foundation), Codex нативно, Claude Code как fallback, 60k+ репо. Один файл-истина вместо дублей под каждого ассистента |
| **Спеки изменений (паттерн OpenSpec)** | ядро слоя | лёгкий spec-driven цикл «предложение → апрув → реализация → архив» точно совпадает с нашим гейтом плана; context-light, под brownfield. Берём паттерн (папка specs/changes/), сам инструмент OpenSpec — опционально, можно включить профилем |
| **Serena MCP** | опция крупных кодовых баз | символьная навигация экономит токены на больших репо (бенчмарки это подтверждают), но: тяжёлая, известны утечки памяти, замедляет мелкие задачи, добавляет стартовый контекст. При нашей модульности потребность падает — модуль и так мал. Включается профилем, когда кодовая база переросла ~30–50k строк |
| **Taskmaster AI** | не используем | его работу в v5 делает декомпозиция Направление→Фокус→Работа→дети + задачи в спеке изменения + нативное планирование ассистентов. Отдельный менеджер задач = второй источник состояния, что нарушает «одна истина». Его главный эффект («агент работает над одной чёткой задачей») у нас даёт сама карточка |
| **Basic Memory MCP** | не используем | память проекта = docs/ + DECISIONS + LESSONS в репо; память направления = knowledge/ в воркфлоу. Ещё одно хранилище = расхождение источников |

Общий принцип выбора: инструмент допускается, если его состояние живёт в git-репо (или его потеря ничего не ломает). MCP-обвязки — расходный слой.

## 9. Сквозной пример: трекер калорий для health-and-beauty

В фокусе направления ставка «свой трекер калорий вместо MacroFactor». Работы: design «скоуп MVP и стек» (решение: python-service + web-app, в knowledge/) → build `project-bootstrap` → код-ассистент создаёт репо `calorie-tracker` по двум профилям: модули `core/` (модели, расчёты), `api/`, `web/`; import-linter на границы; validate.sh (ruff+mypy+pytest+покрытие+границы); AGENTS.md; пустой каркас зелёный. RESULT в направление с demo «как запустить и увидеть hello-экран». Дальше TASK-карточка «дневник еды: модель + API + экран» → ассистент пишет спеку (план, влияние на модули core+api+web, план валидации, что покажет demo) → ты говоришь «поехали» и уходишь → реализация → валидация: validate.sh зелёный со 2-й итерации, test-reviewer бракует 3 тавтологичных теста — переписаны, code-reviewer находит хардкод нормы калорий — вынесен в core → RESULT: demo «открой /diary, добавь яблоко 100 г, увидишь 52 ккал», ручных шагов нет, не покрыто «офлайн-режим». Ты тыкаешь demo, «принято» → строка в LOG направления, ссылка на коммиты. После десятка таких сессий — retro: в LESSONS трижды повторилась проблема с фикстурами pytest → правка policy тестов + строка в kernel/tech/python-service.md.

## 10. Что меняется в основном концепте

- Раздел 7 концепта указывает на этот документ.
- Плейбуков становится 9: + `project-bootstrap`, `code-task` (ведение кодовой сессии: план-гейт, автономия, СТОП-пороги), `retro`. Все ≤100 строк.
- В ядре появляется `kernel/tech/` — техпрофили (вне бюджета ядра: это растущая библиотека знаний, как knowledge, а не правила).
- Сущности кодового слоя: Проект, Модуль, Спека изменения, Техпрофиль. Они живут на стороне проектов и не увеличивают семь сущностей ядра воркфлоу.
