# Workflow v5 — Концепт

Версия: draft для обсуждения. Дата: 2026-06-10.
Статус: не реализовано; этот документ — предмет согласования перед сборкой.

---

## 0. Что это и какую проблему решает

Workflow v5 — система для достижения больших целей в автономных направлениях жизни (разработка игры, здоровье, продуктивность…) силами одного человека и ИИ-ассистентов. Направление может длиться годами и десятилетиями и требовать работы во всех аспектах сразу (для игры: код, геймдизайн, арт, маркетинг, бизнес).

Система решает пять задач, которые не решает «просто чат с ИИ»:

1. **Память.** Направление живёт годами, чат — часы. Нужен внешний носитель состояния, который не протухает и не распухает.
2. **Направленность.** Каждая единица работы должна приближать к цели направления, а не «быть интересной». Система должна сопротивляться расширению scope — в том числе твоему собственному.
3. **Полнота.** Структура должна сама заставлять работать над всеми аспектами направления в нужные моменты, а не только над любимыми.
4. **Адаптивность.** Детальный долгосрочный план бессмысленен. Нужен цикл: выбрали лучшее текущее движение → прошли отрезок → осмыслили → перестроили карту → выбрали следующее.
5. **Надёжность.** Каждый чат должен работать стабильно, без уточнений и допиливания на ходу.

## 1. Почему предыдущие версии не работали

Ты прошёл обе крайности, и у каждой свой механизм отказа.

**Простые версии** (протоколы чатов в проекте «питание», ранние схемы): чат получал инструкцию «как работать», но (а) не было обязательного закрытия — чаты жили долго, контекст протухал; (б) не было внешней памяти со строгим форматом — результаты растворялись в переписке; (в) не было механизма выбора работы — работа шла по настроению, scope полз; (г) не было стандартного финала — следующий чат начинал с нуля. Простота была недостаточной *структурой*: системе не хватало несущих элементов.

**Сложная версия (v3)**: те же сбои LLM лечились правилами. Каждый инцидент → новый инвариант → ~37 000 строк спецификации → модель физически не может это исполнять → больше сбоев. Сложность была избыточной *дисциплиной*: система требовала от модели того, что должна давать структура.

Вывод, на котором стоит v5: **надёжность достигается структурой и механикой, а не количеством правил.** Каждый элемент v5 — несущий: он существует потому, что без него повторится конкретный известный сбой.

| Известный сбой | Несущий элемент v5 |
| --- | --- |
| чат не знает контекста / контекст протух | контекст-пакет: собирается машиной из репо в момент старта |
| файлы из GitHub приходят молча обрезанными | EOF-маркер в каждом файле + малые бюджеты файлов |
| результат чата теряется | единая схема RESULT; работа «не существует», пока RESULT не принят и не закоммичен |
| модель объявляет победу раньше времени | definition of done в карточке работы и в ставке; review-работы |
| модель делает слишком много за раз | один чат = одна работа; работа обязана влезать в один чат |
| scope ползёт (твоя слабость) | ставки меняются только через ревью фокуса; правило «идея → строка в банк идей» |
| работа идёт только над любимыми аспектами | скан аспектов на каждом ревью фокуса |
| состояние репо портится / пишут все | единственная пишущая рука — Codex/Claude Code, три стандартные команды |
| план устаревает | карта обновляется на каждом ревью фокуса из полученного опыта |
| система пожирает себя (governance) | ядро ≤ 1200 строк; правило двух инцидентов; нулевой прирост объёма |

## 2. Принципы

1. **Структура вместо дисциплины.** Всё, что должно работать всегда, делает машина (сборка контекста, коммит, компакция) или формат (схема RESULT), а не «модель должна помнить, что нельзя…».
2. **Подавать контекст, а не запреты.** ~80% того, что видит чат, — материал для работы; правил — одна страница на всю систему.
3. **Контроль на границах, свобода внутри.** Проверяется вход чата (карточка + пакет) и выход (RESULT). Внутри чата модель работает как умеет — без пошаговых гейтов.
4. **Три горизонта.** Направление (годы) / Фокус (недели) / Работа (часы). У каждого свой носитель и свой момент пересмотра. Средний слой пересматривается не по календарю, а по событию — на ревью; но менять его МИМО ревью нельзя. Это и есть защита от хаоса и scope-creep: смена курса — осознанный ритуал, а не импульс.
5. **Аппетит вместо дедлайна** (адаптация Shape Up под соло). У ставки фиксируется аппетит — сколько времени/усилий ты готов на неё потратить. Аппетит съеден, а ставка не закрыта → она не продлевается молча, а возвращается на ревью: дорезать, упростить или убить.
6. **Один писатель.** Все изменения состояния проходят через Codex/Claude Code стандартными командами. ChatGPT/Claude-чаты — только думают и производят тексты.
7. **Готовность к автономии.** Все интерфейсы — файлы и карточки, ничего не живёт «в понимании чата». Сегодня человек переносит карточки между чатами; завтра это делает раннер по API — без редизайна.

## 3. Обзор системы

```text
ГОРИЗОНТ          НОСИТЕЛЬ                  КОГДА ПЕРЕСМАТРИВАЕТСЯ
──────────────────────────────────────────────────────────────────
Направление       CHARTER.md  (конституция) почти никогда
  «куда и зачем»  MAP.md      (карта пути)  на ревью фокуса
──────────────────────────────────────────────────────────────────
Фокус             FOCUS.md    (ставки)      на ревью фокуса —
  «что сейчас»                              по событию, не календарю
──────────────────────────────────────────────────────────────────
Работа            JOB-карточка → чат →      каждый чат
  «что делаем     RESULT → commit
   в этом чате»
──────────────────────────────────────────────────────────────────
Память            LOG.md (журнал) + knowledge/ (выжимки)
                  + work/ (артефакты) + archive/ (компакция)
```

Поток одной единицы работы:

```text
FOCUS.md содержит работу W
  → Codex: «pack W» → CONTEXT_PACK (один файл)
  → новый чат: вставляешь пакет
  → чат: блок «Коротко» (цель, критерий готовности, план) → ты: «поехали»
  → чат работает; останавливается только на развилках-решениях
  → чат выдаёт RESULT (единая схема)
  → ты: «принято» (или правки)
  → Codex: «apply RESULT» → коммит артефактов, отметка в FOCUS, строка в LOG,
    обновление knowledge, новые JOB-карточки детей — в work/
```

Два подтверждения человека на чат: старт («поехали») и приёмка («принято»). Всё остальное — содержательные развилки, когда модели реально нужно твоё решение.

## 4. Сущности (семь, других нет)

### 4.1. Направление (Direction)

Папка `directions/<id>/`. Автономна: не ссылается на другие направления, переносима целиком.

**CHARTER.md — конституция** (~50–80 строк, меняется редко и только осознанно):

```text
# Направление: <имя>
## Цель (North Star)
Одна формулировка успеха. Пример: «Выпустить коммерчески успешную игру
в сеттинге X с механиками Y: ≥N продаж / ≥M отзывов со средней ≥4».
## Критерии успеха
Измеримые. По ним ревью судит прогресс.
## Аспекты
Полный список областей, которые направление обязано покрывать.
Для игры: продукт/код, геймдизайн, арт/звук, маркетинг/комьюнити,
бизнес/деньги, издание/платформы.
## Ограничения
Бюджет времени в неделю, деньги, энергия, технологии, принципы.
## Не-цели
Явный список того, чем направление НЕ занимается. Главный якорь анти-скоупа.
## Параметры
Продуктовые репозитории; предохранитель от тишины (дней без ревью); recurring-ревью, если направлению нужен регулярный ритм.
```

**MAP.md — карта пути** (~60–120 строк, обновляется ТОЛЬКО на ревью фокуса):

```text
# Карта
## Текущее понимание пути
Крупные мазки: этапы/вехи, как сегодня видится дорога к цели. Не план-график.
## Гипотезы и неопределённости
Что мы предполагаем, но не знаем. Каждая гипотеза — кандидат на проверку работой.
## Риски
Что может убить направление; что делаем с этим.
## Банк ставок (backlog)
Кандидаты на будущие фокусы, по строке: «<что> — зачем — аспект — примерный размер».
Сюда падают все идеи. Отсюда ревью берёт ставки. Больше идеи никуда не попадают.
## Статус аспектов
По каждому аспекту CHARTER: в порядке / голодает с <даты> / осознанно заморожен до <условие>.
```

Карта — это твой «не супер детальный план»: направление держит курс крупными мазками, а опыт каждого фокуса её перестраивает. Запись в карту — только через ревью: это защищает её от импульсных правок.

### 4.2. Фокус и Ставка (Focus, Bet)

Жёстких календарных циклов в системе нет — для соло-работы они фальшивы: доступное время плавает, направления движутся с разной скоростью. Вместо этого у направления есть **Фокус** — текущий набор из 1–3 ставок, действующий от ревью до ревью. Сколько длится фокус — определяется событиями, не календарём.

**FOCUS.md** (~40–80 строк) — единственный «горячий» файл направления:

```text
# Фокус <номер>  (открыт: <дата>, ревью №<n-1> см. LOG)
## Ставки (1–3)
- BET-1: <формулировка> — definition of done: <проверяемо> —
  аспект: <…> — аппетит: <примерно сколько готов потратить>
## Работы
- [x] JOB-001 <название> → принято <дата>, см. LOG
- [ ] JOB-002 <название> (ставка BET-1)
- [ ] JOB-003 <название> (ставка BET-1, после JOB-002)
## Повторяющиеся
- еженедельно: <карточка из recurring/>
## Банк идей фокуса
Строки-вбросы по ходу работы. На ревью разносятся в MAP или умирают.
```

Правила фокуса (это почти все правила системы):

- **Ставки меняются только через ревью.** Посреди фокуса ставки не добавляются и не подменяются. Появилась идея — строка в банк идей, работа продолжается (это «if-then» из науки о целях: заранее определённая реакция на импульс). Захотел сменить курс прямо сейчас — это можно: зовёшь ревью досрочно. Смена курса легальна в любой момент, но только как осознанный ритуал с обновлением карты, а не молчаливый дрейф.
- **Ревью наступает по событию.** Триггеры: (а) все ставки закрыты или умерли; (б) ставка съела свой аппетит и не закрыта; (в) работа фокуса показала, что ставки потеряли смысл; (г) ты хочешь сменить курс; (д) предохранитель от тишины — в направлении давно ничего не происходило (параметр CHARTER, по умолчанию 30 дней; направления с регулярным ритмом, как здоровье, могут вместо него держать recurring-ревью раз в неделю).
- **Аппетит вместо дедлайна.** У ставки нет срока — есть аппетит: сколько усилий она стоит. Аппетит съеден → ставка не продлевается молча, а выносится на ревью: дорезать, упростить, убить.
- **Размеры.** Ставка маленькая — несколько работ. Работа маленькая — один чат. Не влезает — дели до уровня, который влезает.

### 4.3. Работа (Job) и Карточка

Одна работа = один чат = один результат. Карточка — единственный способ поставить работу (себе, ребёнку, Codex'у, будущему автономному агенту):

```text
JOB-<id>:
  направление: <id>
  тип: research | design | build | review | review-focus | bootstrap
  цель: <одно предложение>
  контекст: <что важно знать сверх пакета; ссылки на knowledge/work>
  definition_of_done: <по чему понять, что готово; проверяемо>
  формат_результата: <что должно лежать в RESULT: решение/файл/выжимка>
  границы: <чего НЕ делать; время/размер>
  родитель: <JOB-id или ставка>
```

Карточка одинакова на любом уровне вложенности — поэтому дочерние и до-дочерние работы не требуют отдельных правил.

### 4.4. Результат (RESULT)

Единственный формат завершения любого чата:

```text
RESULT JOB-<id>:
  итог: <2–5 строк простым языком: что получено>
  артефакты: <тексты здесь / файлы для коммита с путями>
  проверка: <чем подтверждено, что definition_of_done выполнен>
  в_журнал: <1–3 строки для LOG.md>
  в_знания: <обновления knowledge/: файл + содержимое | нет>
  в_фокус: <отметить JOB выполненным; прочие правки FOCUS | нет>
  дети: <0..N готовых JOB-карточек>
  решения_от_тебя: <вопросы, требующие твоего решения | нет>
  риски_и_недоделки: <что не сделано/не проверено; честно>
```

Свойства: (а) чат без RESULT — не существовал: ничего не принято, перезапуск безопасен (идемпотентность); (б) поля «проверка» и «риски» обязательны — это весь «контроль качества процессом»; (в) «дети» делают ветвление работ естественным; (г) «решения_от_тебя» — будущие push-уведомления автономного режима.

### 4.5. Плейбук (Playbook)

Один файл ≤100 строк на тип работы: как делать хорошо. Чеклист и образец, не машина состояний. Шесть базовых (плюс три кодового слоя — project-bootstrap, code-task, retro — см. WORKFLOW_V5_CODE_LAYER.md):

- **research** — вопрос → источники → выжимка в knowledge с уровнем уверенности. Запрещено: решать и строить.
- **design** — варианты → критерии из CHARTER → решение + обоснование в knowledge. Запрещено: молча расширять задачу.
- **build** — произвести артефакт: код (карточкой в Codex, в продуктовый репозиторий), текст/статью, спеку, маркетинг-материал. Включает проверку по definition_of_done.
- **review** — свежий контекст проверяет чужой RESULT или состояние направления против критериев. Дёшево и эффективно: свежий чат видит то, что замылилось.
- **review-focus** — закрыть фокус и выбрать новый (см. 5.3). Самый важный плейбук системы.
- **bootstrap** — завести направление: интервью с тобой → CHARTER + MAP + первый FOCUS.

### 4.6. Знание (knowledge/)

Выжимки, переживающие смену фокусов: решения с обоснованием, результаты исследований, спецификации, выученные уроки. Файл на тему, ≤200 строк, сверху `summary:` в 3 строки. `knowledge/INDEX.md` — строка на файл (для сборки пакетов). Пишется только из принятых RESULT (через apply). Это «семантическая память»; LOG — «эпизодическая»; плейбуки — «процедурная»; чат — «рабочая». Соответствует тому, как строят память долгоживущих агентов (CoALA, MemGPT).

### 4.7. Команды Codex (три)

Codex/Claude Code — единственная пишущая рука и сборщик контекста. Интерфейс — три стандартные команды (описаны в `kernel/AGENTS.md`):

- **`pack <JOB-id>`** — собрать CONTEXT_PACK.md: карточка + плейбук типа + CHARTER + MAP + FOCUS + релевантные файлы knowledge (по INDEX и полю «контекст» карточки) + хвост LOG (последние ~20 строк). Бюджет пакета ≤ ~1200 строк; при превышении — резать по приоритету и явно перечислить вырезанное. Пакет коммитится в `work/<JOB-id>/PACK.md`.
- **`apply`** (на вход — RESULT) — закоммитить: артефакты в work/ или продуктовый репо, отметка в FOCUS.md, строки в LOG.md, обновления knowledge/, карточки детей в work/. Если RESULT неполон по схеме — вернуть, что не хватает, ничего не писать.
- **`compact <direction>`** — обслуживание памяти по бюджетам: LOG >300 строк → старые записи в archive/ (остаются сводки ревью); knowledge-файл >200 строк → сжать с сохранением решений; пересобрать INDEX. Запускается, когда ревью это отметит.

## 5. Жизненные циклы

### 5.1. Жизнь направления

```text
bootstrap-чат (интервью) → CHARTER + MAP + FOCUS-1
→ [работы фокуса → ревью фокуса → новый фокус]*  — годами, основная петля
→ редкие события: пересмотр CHARTER (осознанный, отдельной работой);
  заморозка направления (запись в LOG, файлы остаются);
  закрытие (итоговая запись).
```

Направлений может быть несколько; они не знают друг о друге. Общего «портфеля» в v5 нет сознательно: твоё внимание распределяешь ты (или будущий планировщик-раннер), а не ещё один слой файлов. Единственная общесистемная вещь — ядро (kernel/).

### 5.2. Жизнь фокуса

Старт: ревью записало FOCUS.md с 1–3 ставками и стартовым списком работ. Середина: берёшь следующую работу из FOCUS → pack → чат → RESULT → apply. Появляющиеся идеи — строками в банк. Конец — по событию (триггеры в 4.2): ревью фокуса.

### 5.3. Ревью фокуса — сердце системы

Тип работы review-focus, один чат, пакет собирается как обычно. Плейбук ведёт по шагам:

1. **Факты.** Что сделано против ставок (по FOCUS и LOG). Что не сделано и почему. Сколько реально потрачено против аппетитов.
2. **Опыт → Карта.** Что нового узнали (из RESULT'ов фокуса)? Какие гипотезы в MAP подтвердились/умерли? Обновить «Текущее понимание пути», риски.
3. **Скан аспектов.** По каждому аспекту CHARTER: когда последний раз была работа (по LOG)? Статус: в порядке / голодает / осознанно заморожен. Голодающий аспект обязан получить либо ставку в новом фокусе, либо явную запись «заморожен, потому что …». Молча пропустить нельзя — это и есть механизм «форсить работать над всем».
4. **Разбор банка идей.** Каждая строка: в банк ставок MAP / умерла. Банк не обязан расти — умершие идеи это хорошо.
5. **Ставки нового фокуса.** Из банка ставок MAP по критериям: максимальный вклад в North Star, закрытие самой опасной неопределённости, разблокировка других работ, голодающие аспекты. 1–3 ставки, каждой — definition of done и аппетит. Нарезать стартовые работы.
6. **Гигиена.** Нужен ли compact. Повторяющиеся работы актуальны?

Выход — RESULT с новым FOCUS.md и правками MAP.md. Ты принимаешь — apply. Ревью же — точка «кастомизации» системы под опыт: менять MAP, набор повторяющихся работ, даже плейбуки (через отдельную работу) — здесь, осознанно, а не на лету в рабочем чате.

### 5.4. Жизнь чата (любого)

```text
1. Открываешь чат проекта направления, вставляешь CONTEXT_PACK (или даёшь
   ссылку на work/<job>/PACK.md — см. транспорт).
2. Чат отвечает блоком «Коротко»: цель своими словами, definition of done,
   план в 3–7 пунктов, чего не хватает (вопросы сразу, не в середине).
   ≤15 строк. Ты: «поехали» или правишь.
3. Работа. Чат останавливается только если: нужно твоё решение (развилка),
   нужен ребёнок (выдаёт карточку), упёрся (говорит во что).
   Никаких «ДАЛЬШЕ» после каждого шага.
4. Финал — RESULT по схеме. Ты читаешь «итог», «проверку», «риски».
   «Принято» → копируешь RESULT в Codex: apply. Или правки → доработка тут же.
5. Чат закрыт. Продолжение темы = новая работа новым чатом.
```

Дочерний чат — то же самое: карточка ребёнка (из RESULT родителя) → pack → чат → RESULT. Куда возвращается результат ребёнка, написано в его карточке (поле «родитель»): либо родительская работа ещё открыта и ты вносишь RESULT ребёнка туда как ввод, либо результат сразу применяется apply, а родитель был уже закрыт. Рекурсия любой глубины без специальных правил.

## 6. Транспорт: точные правила каналов

Факт 1: ChatGPT умеет ходить в GitHub, но файлы могут приходить **молча обрезанными** — модель не сообщает об обрезке. Факт 2: Project Files — кэш, протухает. Поэтому:

| Канал | Статус | Правило |
| --- | --- | --- |
| CONTEXT_PACK вставкой в первый месседж | **канон для старта чата** | пакет собран машиной → свежий и полный по построению |
| CONTEXT_PACK по GitHub-коннектору (один файл `work/<job>/PACK.md`) | допустимый эквивалент | файл один, с EOF-маркером: обрезку видно |
| GitHub-коннектор ad-hoc («дочитай файл X») | fallback по ходу чата | можно читать только файлы репо воркфлоу; проверка маркера конца обязательна: нет маркера → файл обрезан → сказать и не опираться на хвост |
| Project Files / память ChatGPT | контекст-подсказка | никогда не источник состояния |
| Запись в репозитории из ChatGPT/Claude-чата | **запрещено всегда** | вся запись — через Codex-команды |
| Инструкция ChatGPT-проекта | бутлоадер ≤2500 знаков | роль + 7 правил + формат «Коротко»/RESULT; одна на направление, генерируется из шаблона ядра |

EOF-маркер: каждый файл репо заканчивается строкой `<!-- EOF <имя файла> -->`. Это единственная «бюрократия» на файлах, и она механическая — её ставит Codex при коммите. (Идея унаследована из v3 — она там была здоровой.)

Бюджеты файлов (контролирует compact, а не дисциплина): CHARTER ≤80 строк, MAP ≤120, FOCUS ≤80, LOG ≤300 (дальше архив), knowledge-файл ≤200, плейбук ≤100, карточка ≤20, пакет ≤1200.

## 7. Код и проекты → кодовый слой

Направления порождают **Проекты** — кодовые продукты в отдельных репозиториях (трекер калорий, игра…). Граница односторонняя: направление знает о своих проектах, проект о направлении не знает — он самодостаточен для кодовой работы (вся нужная структура, правила и валидация лежат в его собственном репо, поэтому кодовая сессия не требует контекст-пакета из воркфлоу).

Кодовый слой описан отдельным документом — `WORKFLOW_V5_CODE_LAYER.md`: анатомия проекта (AGENTS.md-стандарт, карта модулей, спеки изменений, журнал уроков, validate.sh), бутстрап по техпрофилю (`kernel/tech/`), механическая модульность (границы исполняют линтеры/asmdef, не дисциплина), цикл кодовой задачи «план → твоё *поехали* → автономная работа → многоуровневая валидация (механика + свежие ревьюеры кода и качества тестов) → RESULT с demo-инструкцией», СТОП-пороги эскалации (3 одинаковые ошибки / 5 итераций), ретроспективы по journal/LESSONS. Слой кросс-ассистентный: работает с Codex и Claude Code, всё важное — файлы и скрипты в репо проекта.

## 8. Повторяющиеся работы

`directions/<id>/recurring/<имя>.md` — обычная JOB-карточка плюс поле `расписание: еженедельно по пятницам` и счётчик последнего запуска. Сегодня: FOCUS.md ссылается на повторяющиеся работы, ревью (и ты в начале недели) видит, что пора — открываешь чат с pack как обычно. Завтра: раннер по расписанию сам инстанцирует карточку и запускает агента (см. 10). Пример: «еженедельная статья по направлению» — карточка research+build, каждый запуск порождает новый work/JOB-id.

## 9. Качество

Три механизма вместо эвалов и acceptance-чатов:

1. **Схема RESULT**: без полей «проверка» (как подтверждён definition_of_done) и «риски» apply вернёт результат. Недоказанная работа не принимается машиной, а не твоей бдительностью.
2. **Review-работы**: для важных результатов (решение о ядре геймплея, релизный план) — карточка review в свежий чат: проверка против CHARTER/критериев. Свежий контекст — лучший ревьюер; это дешёвая замена maker/checker-разделению v3.
3. **Ревью фокуса**: систематическая сверка фактов с картой на каждой смене курса (см. 5.3).

## 10. Путь к автономности

Дизайн-инвариант: человек сегодня выполняет роль «раннера» — переносит карточки и пакеты между чатами и Codex'ом. Все интерфейсы уже файловые и машиночитаемые, поэтому автономизация — замена раннера, не редизайн:

- **Этап 0 (сейчас).** ChatGPT/Claude-чаты + Codex/Claude Code вручную. Ты — транспорт и приёмка.
- **Этап 1.** Скрипт/агент-раннер: `pack` и `apply` выполняются одной кнопкой/командой; повторяющиеся работы запускаются по расписанию; тебе остаются «поехали», «принято» и поле «решения_от_тебя».
- **Этап 2.** Раннер на API (Agent SDK/Codex): берёт следующую работу из FOCUS, исполняет чат сам, применяет RESULT с авто-приёмкой для низкорисковых типов (research, черновики), пушит тебе только «решения_от_тебя», review-результаты и ревью фокусов. Человек смещается туда, где он незаменим: цели, вкус, решения.

Поэтому в карточке есть definition_of_done и «формат_результата» (машине нужно знать, когда стоп), а в RESULT — «решения_от_тебя» (будущий канал уведомлений).

## 11. Структура репозитория

```text
my_global_workflow/            (чистая ветка; v3/v4 — в архивном теге)
  kernel/
    KERNEL.md                  правила системы — 1 страница, ≤40 строк
    LIFECYCLE.md               жизнь чата + схемы JOB/RESULT/PACK (~120 строк)
    playbooks/                 6 файлов ≤100 строк
    project_instructions/     шаблон инструкции проекта (≤2500 знаков) + готовые per-direction
    AGENTS.md                  инструкция Codex: pack / apply / compact, EOF-маркеры, бюджеты
    CHANGELOG.md               строка на изменение ядра (весь «governance»)
  directions/
    <id>/
      CHARTER.md  MAP.md  FOCUS.md  LOG.md
      knowledge/  (INDEX.md + файлы-выжимки)
      work/<JOB-id>/  (PACK.md, артефакты, карточки детей)
      recurring/
      archive/
```

Объём ядра целиком: ~10 файлов, ~900–1200 строк. Модель способна удержать его весь; ты способен прочитать его за вечер. Это сознательный предел.

**Бюджеты-предохранители** (зашиты в KERNEL.md): сущностей ≤7; ядро ≤1200 строк; новое правило — только после двух повторившихся реальных инцидентов и с удалением эквивалентного объёма (нулевой прирост); никаких правил о правилах; изменение ядра — обычная работа build с записью в CHANGELOG.

## 12. Сквозной пример: направление «игра», первые два фокуса

**Bootstrap.** Чат-интервью: цель («выпустить коммерчески успешную инди-игру в сеттинге S с механиками M: 10 000 продаж за первый год»), аспекты (код, геймдизайн, арт/звук, маркетинг, бизнес), ограничения (15 ч/нед, бюджет $X), не-цели («не делаем мультиплеер», «не учим 3D — только 2D»), предохранитель: 30 дней без ревью → пора. RESULT: CHARTER, MAP (гипотезы: «механика M интересна 30 минут», «аудитория — фанаты жанра Z»; банк ставок: прототип M, анализ конкурентов, devlog), FOCUS-1 со ставками: BET-1 «бумажный прототип M + 3 отзыва» (геймдизайн, аппетит ~3 вечера), BET-2 «карта конкурентов: 10 игр, выводы» (маркетинг/бизнес, аппетит ~2 вечера). Ты: «принято» → Codex apply.

**Фокус 1.** `pack JOB-001` (research конкурентов) → чат → RESULT: выжимка в knowledge/competitors.md, 3 строки в LOG, ребёнок: JOB-004 «позиционирование: чем отличаемся» (design). Apply. Дальше JOB-002 (design ядра механики): чат предлагает 3 варианта, останавливается на развилке — ты выбираешь B; RESULT: knowledge/core-mechanic.md + ребёнок JOB-005 «бумажный прототип» (build). По ходу тебе пришла идея «а сделаем кооп!» — в чате говоришь «в банк»; строка падает в банк идей фокуса, работа продолжается (не-цель «не мультиплеер» потом убьёт её на ревью — система отработала твой scope-импульс).

**Ревью фокуса 1** (триггер: обе ставки дошли до края — BET-2 закрыта, BET-1 съела аппетит). Факты: прототип сделан, но не протестирован. Опыт → карта: гипотеза «аудитория Z» подтвердилась частично, риск «механика скучна соло» добавлен. Скан аспектов: арт/звук и бизнес голодают с bootstrap — арт получает ставку в новом фокусе, бизнес заморожен явно («до прототипа считать нечего»). Банк: «кооп» умирает (противоречит не-целям) — записано почему. FOCUS-2: BET-1 «3 теста прототипа с записью выводов» (остаток аппетита + ещё вечер), BET-2 «арт-стиль: 3 референс-борда» + recurring «devlog раз в неделю». Apply.

Вот «элегантно вплетённый» цикл из твоего описания: поставили цель → выбрали оптимальное движение → прошли → осмыслили → перестроили карту → выбрали следующее. Заметь: никто не ждал «конца спринта» — ревью пришло, когда ставки дошли до края; и структура сама вытащила арт и убила кооп.

## 13. На чём основано

- **Anthropic, Effective harnesses for long-running agents**: агент через много контекстных окон надёжен, когда есть (а) файл прогресса + git-история для входа в курс, (б) список целей со статусами против «преждевременной победы», (в) инкрементальная работа «одна фича за раз», (г) чистое состояние в конце сессии. У нас это LOG+FOCUS, definition of done, «один чат — одна работа», RESULT+commit. https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents
- **Anthropic, Effective context engineering for AI agents**: контекст — ограниченный ресурс; сбои агентов — чаще сбои контекста; меньше правил, точнее примеры. Отсюда пакеты, бюджеты, ядро ≤1200 строк. https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents
- **Исследования памяти агентов** (MemGPT, CoALA, обзоры 2025–26): иерархия рабочая/эпизодическая/семантическая/процедурная память; компакция с риском «memory blindness» → у нас компакция сохраняет сводки циклов и решения, а сырьё уходит в archive, не удаляется. https://arxiv.org/pdf/2603.07670
- **Shape Up (Basecamp)**: appetite, betting table, fixed time / variable scope, no extensions by default — основа ставок и анти-скоупа. Адаптация под соло: календарные циклы заменены событийными ревью (аппетит на ставке выполняет роль тайм-бокса). https://basecamp.com/shapeup
- **Наука о целях** (Locke & Latham; Gollwitzer, implementation intentions): сочетание долгих и коротких целей эффективнее; if-then-планы кратно повышают исполнение → North Star + ставки фокуса; правило «идея → строка в банк». https://en.wikipedia.org/wiki/Implementation_intention
- **Факт о GitHub-коннекторе**: обрезка файлов происходит молча → EOF-маркеры + малые файлы + пакеты, собранные машиной.

## 14. Зафиксированные решения

1. **Ритм ревью**: событийный, без жёстких календарных циклов (решение Антона 2026-06-10). Триггеры — в 4.2; предохранитель от тишины — параметр CHARTER (по умолчанию 30 дней); направления с регулярным ритмом могут держать recurring-ревью.
2. **Доставка пакета в чат**: вставка руками — канон; GitHub-коннектор на единственный файл PACK.md с проверкой EOF-маркера — допустимый fallback.
3. **Репозиторий**: пересборка в этом же репо — архивный тег для v3/v4, чистка main до структуры v5.
