# Validation ‚Äî the gate pipeline

The strictest practical validation for autonomously-built code. Gates run cheap-to-expensive; a change is done only when all gates are green with attached evidence. Thresholds live in the repo's `validation.config`; rubric in its `REVIEW.md`. Per-repo customization changes thresholds and rubric content ‚Äî never the gate order or the independence rules. Threshold and rubric changes follow the repo's `docs/FRICTION.md` discipline: an explicit owner request, or ‚â•2 entries on the same point.

**Enabled ‚âÝ written.** A gate counts as ENABLED in a repo only when the repo's deliver-time check FAILS if that gate's evidence is missing or below threshold. A gate that lives only as prose ‚Äî here, or in the repo's AGENTS.md / REVIEW.md ‚Äî with nothing the deliver check trips on, is NOT enabled and never makes a leg deliverable. PROJECT_SETUP wires every load-bearing gate below as an executable-or-recorded deliver dependency, each with a seeded-miss test ‚Äî with one named exception, stated here rather than claimed enabled: **G3** (scripted end-to-end execution) is still validator-run, not yet a recorded deliver dependency (FRICTION-watch). The **review axis** ‚Äî the independent line review (**G4**) and the finding refutation (**G5** here / KERNEL-G5 fresh-session) ‚Äî is wired as of contract v10: writer-side now (`os/adapters/coding-agent.md` G10, reaching every repo before its re-sync) and a product deliver dependency once the repo ¬ßRe-syncs (¬ßReview-evidence below). **The same applies to a DELIVERABLE, not just a gate: every artifact the PLAN promises owns a default-FAIL ledger row whose closure needs an existence proof the deliver check trips on if absent ‚Äî a runnable check, or for an engine-side / non-headless artifact a RECORDED existence proof (a committed scene/prefab/asset path the check `Test-Path`s, or a recorded MCP/batch-run artifact). An owner-acceptance / eyeball axis may DEFER THE LOOK at an artifact that already passed that existence check; it may never absorb the BUILD. Existence proves the artifact is PRESENT, never that it is the NAMED approach: an engine row whose substrate/mechanism (its recorded `approach:` token, CONTOUR ¬ßBoundary) differs from the one the CALL named as its de-risk target needs a recorded owner-ack to stand, the same as a cut ‚Äî a real-but-substituted artifact, AND a silent stretch/normalization crutch the named approach did not require, is the existence gate's blind spot, closed here, not waved through green. For any leg that substituted an approach or claimed the named one infeasible, the fresh-session refutation (KERNEL G5) must try to refute THAT claim against the named code, not only value==source. Coverage is keyed to the plan's PROMISES ‚Äî each `done_when` bullet gets a disposition, so none can be silently omitted ‚Äî and cutting a promised deliverable needs a recorded owner decision, never a self-authored drop (CONTOUR ¬ßc). A deliverable with no failing row, or one discharged only by an "owner may wait without failing the leg" clause, is the gate hole's mirror: invisible to "all gates green," exactly as a prose-only gate is.** This is the one class the deliver gate currently exempts ‚Äî close it the same way. The two skipped most often, because they are cost- or judgment-heavy, are **G2 mutation** (a suite that runs but never records a mutation score has not enabled G2) and the **PLAN spec-silence audit** (CONTOUR ¬ßb ‚Äî the recorded unconstrained-regimes list, which now WALKS the cross-kind escape-class registry `os/engineering/ESCAPE-CLASSES.md` row-by-row ‚Äî an unmarked class is the same hole as a missing audit, and a catcher that does not exercise the regime its property must hold under is green where the bug does not live); both are deliver dependencies, not optional prose. When this file hardens a gate, an already-set-up repo does NOT inherit it automatically (setup runs once) ‚Äî it re-syncs via its `docs/FRICTION.md` (PROJECT_SETUP ¬ßRe-sync); until then that repo's deliver check is stale and cannot claim the hardened gate is enabled.

**Review-evidence is a deliver dependency (the review axis of *Enabled ‚âÝ written*, contract v10).** The independent line review (**G4**) and the fresh-session refutation (**KERNEL-G5**) were, until v10, prose the deliver check never tripped on ‚Äî so a leg could reach DELIVERED before any independent eye was recorded, and every review finding was by construction "after done." For a G0-frozen change (a non-archived `openspec/changes/<id>/` with a frozen spec ‚Äî the SAME mechanical trigger as the strong-check, N/A by absence for a light change), deliver now depends on a recorded `docs/reviews/review-<id>.md` whose filename `<id>` MATCHES the triggering change folder (a review naming another id, or absent, FAILs ‚Äî freshness catches a stale tree, not a mismatched change), and which records: (i) **`reviewed-commit:`** a git commit that is an ANCESTOR of the delivered HEAD, with every commit in `reviewed-commit..HEAD` that touches source accounted for by a `fixed <commit>` disposition ‚Äî an un-reviewed source change since the review FAILs, while an honest review‚Üífix‚Üídeliver does NOT, because the fix-commits ARE the listed dispositions (freshness is ancestry+accounting, deliberately NOT tree-equality, which would fail every review that fixed anything); (ii) ‚â•1 recorded pass by a model family OTHER than the builder; (iii) a findings block (a table: `id | invariant/class | disposition | marker`) where every row's `disposition` ‚àà {`fixed <commit>` | `refuted` | `owner-ack out-of-scope <token>`} ‚Äî a row missing the disposition column or valued `open` FAILs; (iv) every `refuted` row carrying a **refutation-verification marker** naming the level ‚Äî an in-session finding-verification pre-pass (G5), a binding fresh-session refutation (KERNEL-G5), or (v13) a `register:<row-id>` citation resolving to a `docs/reviews/REFUTED.md` row that carries the original marker (¬ßRefuted-register below); a `refuted` with no marker FAILs; (v) **`rounds:`** ‚â§3, or an `escalation:<id>` token that RESOLVES to a real escalation id (same resolution machinery as v8's `owner-ack`) ‚Äî a delivered frozen-change leg with neither FAILs; a structure check, not a guarantee the non-convergence was actually escalated. Same surface as the report/coverage gates ‚Äî the check asserts the artifact EXISTS, is tree-FRESH, and is STRUCTURALLY complete, NOT that the review was thorough, the independence real, or a `refuted` verdict correct (that stays the validator's, the owner's, and a binding fresh-session KERNEL-G5's job ‚Äî all sampling, all later; a recorded `reviewed-commit` proves the review TARGETED this code, never that a reviewer read it ‚Äî gates (ii)/(iv) are self-authored strings a fabricated-but-formatted review passes). The metric this buys is "no leg is DELIVERED before a review-evidence round is RECORDED tree-fresh, fully dispositioned, and rounds-bounded-or-escalated", never "review finds nothing" ‚Äî a fresh review typically finds something (measured: 93.4% of findings unique to one pass). ENABLED as a product deliver check only after the repo's ¬ßRe-sync wires the `reviewed-commit` ancestry/accounting check in; until then the writer-side bounce (`os/adapters/coding-agent.md` G10) carries the review-evidence line-presence on every carry-back.

**Oracle-soundness is a deliver dependency (the negative-control axis of *Enabled ‚âÝ written*, contract v11).** G2 mutation and G4's review-time "every negative control shown REACHABLE by a real run" both miss an EMPTY oracle ‚Äî an acceptance test with no power to fail: mutation kills mutants of code that EXISTS, so it is blind to an ABSENT property (no code ‚Üí no mutant) and to a wrong-but-consistent or too-weak assertion (one measuring the wrong quantity, or a non-strict comparison a constant satisfies), while the REACHABLE-control check lived only as review prose the deliver check never tripped on. So the "test green but empty" cluster ‚Äî telemetry measuring the wrong quantity, an unread config path, a fork-split satisfiable by pure diffusion for rounds, a `>=` a constant passes ‚Äî went green through every gate. As of v11 every acceptance property owns a COMMITTED **negative control**: a deliberately-wrong implementation, authored by the independent test-author alongside the acceptance test (the builder cannot edit it ‚Äî same protection as the tests, G0), paired with a `<Prop>_NegativeControl` test (category `NegativeControl`) that runs the property's OWN assertion against the wrong implementation and is GREEN exactly when the property REJECTS it (RED = the property passed its wrong impl = an empty oracle). `-Deliver` runs the negative-control category (e.g. `dotnet test --filter TestCategory=NegativeControl`) as a recorded pass and requires, per acceptance-property token, a control DISCOVERED (a property with no `<Prop>_NegativeControl` = oracle unverified = FAIL) and PASSED (a control the property lets through = empty oracle = FAIL) ‚Äî discovered+passed per token. This is the mechanical form of G4's REACHABLE-negative-control prose, moved from judgment-at-review to a deliver dependency; it catches exactly what G2 cannot (absent property, wrong/weak assertion) and is orthogonal to mutation (which still catches the weak test of code that DOES exist). Unlike the named-approach (v8) and review-evidence (v10) axes it has NO writer-side keystone ‚Äî the writer runs no tests ‚Äî so it is INERT until the repo ¬ßRe-syncs (same inertness as v9's registry-walk enforcement); until then it is prose here, enabled nowhere. It attacks an empty/weak/mis-targeted oracle, NOT an authored-wrong one: a test-author who writes a negative control the property catches only trivially still passes presence (the same authored-error ceiling the whole deliver layer accepts ‚Äî only the independent review, a fresh-family second derivation for a re-deriving oracle, and the owner eye verify that, all sampling, all later).

**Property-layer is a deliver dependency (the structural-seam axis of *Enabled ‚âÝ written*, contract v12).** G2's property-testing clause above was, until v12, prose with nothing behind it: zero property-testing packages in any product repo, so the cluster it exists to catch ‚Äî interface/atomicity/ordering seams, a different failure mode than v11's absent-or-weak assertion ‚Äî reached DELIVERED at ~11 P1 instances across four legs (a lost tick, `da695e8`/`8ba4491`; state left ragged when an in-flight operation is interrupted, `22d548b`/`4396ded`/`27f4b26`, surviving 8 review rounds on one leg before it closed; a seal/breach race over a tick, `147f701`; a hash keyed to insertion order, `e4d0089`/`c3b85bd`). Root cause: the only party who SEES a new throw-path or seam is the builder, whose incentive is to satisfy tests the independent test-author wrote from the frozen spec BEFORE the code existed (G0); the reviewer reads AFTER DELIVERED ‚Äî nothing sits between test-authoring and post-DELIVERED review to attack the actual diff (CONTOUR's post-build property audit closes the timing gap; this closes the enforcement gap). For a change whose PLAN-stage change-class (CONTOUR's cycle, the same taxonomy that routes it architectural) is `core algorithm`, `-Deliver` now requires each `headless`-class deliverable-coverage row (PROJECT_SETUP ¬ßStrong-check clause (c)) to carry a `prop:<TestClass>.<TestMethod>` evidence token ‚Äî the same mechanical form `exists:` already takes for `engine` rows; `engine` rows are untouched and stay on `exists:` ‚Äî cross-checked against an owner-signed **property table** recorded in the frozen spec (one row per required class: permutation-metamorphic, fault-injection-all-exit-paths, multi-actor-one-tick, boundary-biased; each row names the `[Category("Property")]` test that discharges it). A `core algorithm` change with a `headless` row carrying no `prop:` token, or naming a test absent from the property table, fails deliver. SCOPE (honest, like every prior axis): the check is INERT until ¬ßRe-sync wires the `prop:` token into `coverage-check.ps1` (confirmed absent in the one product repo checked ‚Äî not yet a lagging sync, a genuine gap); PILOT it on the next `core algorithm` leg only ‚Äî this does not retrofit already-delivered core files; presence‚âÝtruth carries over unchanged (a generator that never actually reaches the seam ‚Äî e.g. a "multi-actor" generator that happens to always serialize its actors ‚Äî passes the token check the same way a weak negative control passes presence; only the post-build audit's intent, a fresh-session refutation, and the owner eye catch that).

**Refuted-register is a deliver dependency (the disagreement-persistence axis of the review loop, contract v13).** A `refuted` finding died with its review file: `review-<id>.md` is per-change, so the rationale that killed a finding survived only inside that leg's file ‚Äî the next review of the same seam re-derived and re-litigated it from zero, and the workflow's own logs took to hand-writing "REJECTED, do not re-propose" lists (the FRICTION 2026-07-02 parked entry; the v12 log line) because a rejected finding had no durable home (witnessed: a rejected buoyancy P1 whose seam rationale lived nowhere the next reviewer reads). Persistent disagreement is load-bearing state, not noise ‚Äî an unresolved conflict must survive the round and bias the next one, not evaporate (the dual-variable mechanism of distributed optimization, imported 2026-07-04). As of v13 every `refuted` disposition ALSO appends one row to the repo's cumulative register `docs/reviews/REFUTED.md` ‚Äî `row-id | seam/invariant | claim | why-refuted | marker | review-<id> | date` ‚Äî and the review file carries a `refuted-register:` line naming the rows it appended (`none` is legal only alongside zero `refuted` rows; a review with a `refuted` row and no register line FAILs, same structure surface as v10). A later review that re-derives a registered finding closes it by CITATION: disposition `refuted`, marker `register:<row-id>` ‚Äî a legal refutation-verification marker because the cited row carries the original G5/KERNEL-G5 marker ‚Äî so an already-settled disagreement costs one lookup, not a fresh argument. The citation is legal ONLY when the re-raise adds no evidence the registered refutation did not consider; a re-raise WITH new evidence (a new witness commit, a new regime, a failing run) may NOT be citation-closed ‚Äî it escalates to the owner as a named seam decision, and the register row records the re-raise, so a seam that keeps drawing independent fire accumulates visible weight instead of being re-dropped round after round. The PLAN spec-silence audit (CONTOUR ¬ßb) walks the register the same way it walks ESCAPE-CLASSES.md: a register row naming a module/seam the change touches is marked `still-holds | re-opened <why>` ‚Äî an unmarked touching row is the same deliver FAIL as an unmarked escape class. SCOPE (honest): the writer-side keystone is ENABLED NOW (`os/adapters/coding-agent.md` G10 ‚Äî an engineering RESULT whose review-evidence reports a `refuted` disposition with no `refuted-register:` line bounces, mechanical line-presence); the product deliver checks (register row resolves per `refuted` row; the ¬ßb walk marking) are INERT until ¬ßRe-sync (same inertness as v9/v11/v12). Presence‚âÝtruth carries over unchanged: the register proves a refutation was RECORDED, never that it was right ‚Äî a wrong refutation now propagates cheaper, by citation, which is exactly why the new-evidence hatch above is load-bearing and why refutation CONTENT stays verified only by the binding fresh-session KERNEL-G5 and the owner.

**Fix-class-closure is a deliver dependency (the fix half of the review loop ‚Äî the fix axis of *Enabled ‚âÝ written*, contract v14).** The REVIEW half of the loop became structure in v10 ‚Äî each finding row names its `invariant/class`, and G4 enumerates the sibling sites ‚Äî but the FIX half never did: "classify a finding by the INVARIANT it violates; fixing the instance without the class is non-convergence by construction" lived as prose in CONTOUR, the repo run contract, and REVIEW.md, and under retry pressure a fix session satisfies the visible repro instead ‚Äî the rule sat in the fixer's own AGENTS.md and was skipped anyway (witnessed 2026-07-04, GasCoopGame dev_2 / c-visual-004: a review found a source-scan oracle satisfied by commented-out tokens; the fix stripped `//` line-comments only; the next round found `/* */` block comments, a commented-out include, and a raw-text check ‚Äî three more sites of the SAME false-green class; same pathology as the June per-species‚Üíper-transfer‚Üíper-tick 3√ó recurrence, FRICTION 92/94/96). As of v14 a `fixed <commit>` disposition in `docs/reviews/review-<id>.md` counts as closed ONLY with a recorded **sweep line** for that finding id ‚Äî `sweep: class=<the row's invariant/class token>; sites: <every sibling site the class predicts, each closed <commit> | n/a because <reason>>; controls: <for a finding that is itself a gate/scan/oracle false-green, a seeded-miss or negative-control token per class VARIANT (e.g. line-comment, block-comment, commented include, raw-text check); else n/a>` ‚Äî authored by the fixer BEFORE the row disposes `fixed`; a `fixed` row with no sweep line is the same structural FAIL as a row with no disposition (extends v10 (iii)). And the non-convergence rule gains its mechanical hook: a LATER round's finding row whose `invariant/class` token equals an earlier `fixed` row's class RE-OPENS that row ‚Äî the fix did not close the class; a string-compare inside one file, not anyone's memory ‚Äî and the pair counts as ONE recurring class toward the `rounds:`/escalation bound, exactly the recognition G4 warns per-instance fixes destroy. SCOPE (honest): the writer-side keystone is ENABLED NOW (`os/adapters/coding-agent.md` G10 ‚Äî an engineering RESULT that reports review findings fixed while carrying no per-finding class+sweep record bounces, mechanical line-presence; it keys on the RESULT's own content, not on the CALL declaring a frozen change, so it reaches a leg whose review ran OUTSIDE the recorded review machinery ‚Äî exactly where the witness lived); the product deliver checks (sweep-line presence per `fixed` row; the reopen tripwire) are INERT until ¬ßRe-sync (as v9/v11/v12/v13). Presence‚âÝtruth carries over unchanged: a hollow sweep ("searched, nothing found") passes presence ‚Äî but its sibling surfacing next round now trips the reopen instead of reading as a fresh finding; a fixer who RENAMES the class dodges the string-compare (G4's classify-recurrence-as-one-class prose and the owner stay the truth layer); and none of this makes the fix CORRECT ‚Äî only recorded, swept, and convergence-countable.

## Gates

**G0 ‚Äî Contract freeze (before build).** Acceptance criteria per feature, machine-readable, negotiated with the validator before code exists. The builder cannot edit them. No criteria ‚Üí no build. The failing acceptance tests realizing those criteria are written by an independent test-author from the spec ‚Äî not the builder ‚Äî red at freeze, and the builder cannot edit them (same protection as the criteria); the builder makes them pass. An independently-authored, spec-derived oracle is the only thing that catches a misreading the builder's own tests would share (mutation testing cannot ‚Äî a wrong-but-consistent test still kills mutants).

**G1 ‚Äî Mechanical.** Format, strict lint, typecheck, dependency-boundary check, dead-code / unused-export check, duplicate-helper detection, rules against hardcoded values/secrets/magic literals. One command; runs constantly (hooks), not just at the end.

**G2 ‚Äî Tests.** Full suite green and deterministic. Coverage % is NOT a gate ‚Äî agent-written tests look good and lie (measured ~57% mutation kill rate). The gate is **diff-scoped mutation testing** ‚â• the configured kill-rate floor, plus property-based tests for core logic; for any feature with >1 concurrent actor the generators must cover multi-actor / conflicting-input-in-one-tick regimes, not a single-actor happy path. Property tests for core-class logic run on a real property-testing framework (CsCheck; FsCheck 3.x fallback only where the repo already carries F#) under `[Category("Property")]` ‚Äî not hand-rolled loops passed off as coverage ‚Äî and cover at minimum: **permutation-metamorphic** properties (an outcome that must be invariant to insertion/iteration order ‚Äî a hash keyed to insertion order is a bug, not a feature) and **boundary-biased** generators (inputs weighted at/adjacent to type and domain limits, not uniform random, which under-samples exactly the seam a review later finds by hand). Core invariants (conservation, atomicity, non-negativity, determinism, domain-safety) are property-tested on ALL paths INCLUDING the failure/exception paths a guard introduces ‚Äî a tripped guard must leave them intact (the tick stays atomic and conserving), not only the success path; a guard tested to fire but never tested to be atomic is half-tested, and the untested half is where the next review finding lives. The kill-rate floor is a REQUIRED key in `validation.config`, and the run records the achieved score as an artifact the deliver check reads: a repo whose gate runs the suite but records no mutation score, or whose config omits the floor, has not enabled G2 (see *Enabled ‚âÝ written*). Mutation cannot catch a wrong-but-consistent test (a misread spec the builder's own tests share) ‚Äî that is G0's independent test-author and the spec-silence audit; G2 catches the weak/lying test.

**G3 ‚Äî Executed end-to-end.** A scripted scenario per feature (run the app / drive the UI headlessly / hit the API and assert on state), executed by the validator. This covers what line review is structurally blind to: missing behavior, state, sequencing.

**G4 ‚Äî Independent line-level review.** Fresh-context, read-only reviewer that did not author the code; at least one pass by a different model family (self-preference bias is measured and grows with capability). Driven by REVIEW.md: binary checklist items, `file:line` evidence required for every claim, intent-vs-diff matching against the approved spec, absence checks phrased as required-presence assertions ("every new route has an authz check: yes/no"), value-bearing/measured fields asserted EQUAL to their source rather than merely present ("on-wire bytes == bytes actually sent: yes/no") and tests pinning a claim's MEANING not its form/schema, every negative control shown REACHABLE by a real run rather than a synthetic fixture (a control no real execution or mutant can trip is dead). Each finding names the INVARIANT or class it violates (not just its surface site); where that class can recur, the reviewer enumerates the sibling sites ‚Äî not only the first instance ‚Äî so the fix can close the class, not the rung. A per-instance finding invites the per-instance fix that surfaces the next instance next round (the loop), and prevents the non-convergence rule from recognizing siblings (per-actor ‚Üí per-transfer ‚Üí per-tick) as one recurring class. For critical modules: two heterogeneous reviewers, union of findings (reviewer blind spots barely overlap).

**G5 ‚Äî Finding verification.** Each candidate finding is confirmed against actual code behavior before it counts as a failure ‚Äî the false-positive filter that keeps the loop from chasing ghosts.

**G6 ‚Äî Owner spot-check (the final report).** Severity-tallied summary, diff stats, ledger state, cost ‚Äî plus **manual acceptance instructions generated from the same scripts G3 ran**. Mandatory owner decisions regardless of gates: new dependencies, contract changes, tier-2 actions, escalations.

## Retry & escalation policy (uniform across gates)

- A failed gate returns its findings verbatim into the next attempt ‚Äî retries without the findings are wasted.
- ‚â§2 retries in-context, then 1 fresh-context retry with a rewritten prompt. Hard cap 3 per gate.
- The same finding class recurring twice = non-convergence ‚Üí stop immediately, don't burn the rest of the budget.
- Escalation to the owner is a designed terminal state, not a failure: budget exhausted, non-convergence, or an out-of-plan decision. The escalation carries the question, the findings, and 2‚Äì3 options with a recommendation.

## Test hygiene (mechanical, wired into CI at setup)

1. Tests only under `tests/<module>/` mirroring `src/` (engines: one registered test-scene root). Any test-like file elsewhere fails CI.
2. Orphaned test files (source gone) fail CI.
3. `_scratch/` contents never reach a commit (self-ignoring gitignore + CI assert).
4. No skipped/disabled tests on main (grep-gate).
5. No sleep-waits / real network / real clock in unit tests; changed-file suite runs twice ‚Äî differing results = flaky = fail.
6. Every test asserts something (assertion-free tests fail lint).
7. Dead-code and duplicate-utility gates (unused exports fail; similarity checker over helper dirs).
8. Test-to-source ratio budget per module ‚Äî a tripwire that flags sprawl for review instead of letting it accumulate.

Rule 1+3 together are the direct answer to "test scenes multiplied across the game project": scenes have exactly one legal home with a runner, throwaways have exactly one legal home that cannot be committed.
