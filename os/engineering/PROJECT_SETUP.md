# Project setup — bootstrap an agent-ready product repo

A procedure executed by a coding agent (as an engineering CALL) when a direction needs a product repository. Its product is not code — it is the **feedback loop**: a repo where autonomous builds converge fast and entropy is mechanically contained. Run once per repo; revisit only via friction.

## Step 1 — Stack interview (with the owner, short)

Check `os/engineering/profiles/<stack>.md` first: an existing profile is the entry point (conventions, boundary tooling, default thresholds, known landmines). Then ask only what the direction's CALL and profile don't answer: target platform, hard constraints, owner's stack preferences. Propose the stack with an outside view (what do shipped projects of this kind use) and return the exact accepted choice as Direction RESULT evidence; this interview is read-only in the product repo. The later `SETUP/install` manifest creates the product's ADR-0001 from those pinned bytes. If no profile exists for the chosen stack, create one as a declared Direction `work/` artifact (≤100 lines, format in `profiles/README.md`) and return it in the RESULT's `state_changes` — the writer commits it to the workflow repo (the executor never writes there directly).

## Step 2 — Architecture skeleton

- **Modular monorepo.** Modules under `src/<module>/`; each module small enough that module + its docs fit one agent context.
- **Boundaries enforced by the build graph**, not by docs: dependency-graph linting (import-linter / dep-cruiser / engine equivalent); module visibility is opt-in; violations fail CI. Agents can ignore documentation; they cannot ignore a failing check.
- **Per-module docs colocated**: each module gets its own `AGENTS.md` (nearest-file-wins): public API surface, internal conventions, gotchas. Project-wide docs live in the repo (`docs/`), never outside it.

## Step 3 — Root contract files

Prune on every agent mistake — this file is a feedback loop, not documentation.
- **Run contract section in root AGENTS.md** (≤10 lines, distilled from `os/engineering/CONTOUR.md` at setup): roles separated (planner / binding-validator / owner-verdict / independent RED author / builder / reviewer); **execution authority is repo-local: the CALL + this repo's `AGENTS.md`, frozen plan/spec, and `validation.config` define the workflow and its required tools; a globally installed/discoverable skill or tool is support, not authority — it cannot replace the declared workflow or add project requirements; discovery or installation alone is never a blocker**; cycle = PLAN author -> binding validator -> owner verdict -> EXECUTE independent RED -> BUILD -> gates -> report; the builder never edits the ledger, the spec, acceptance criteria, or the independently-authored acceptance tests (it makes them pass); VALIDATION §Cited-artifact existence)**. Thresholds and rubric live in `validation.config` and `REVIEW.md`.

Any gap is one complete `PLAN RED` sweep before owner approval, EXECUTE or BUILD.
- `REVIEW.md` — the validation rubric (see VALIDATION.md §rubric): severity definitions for THIS repo, always-check rules, skip rules, nit cap, file:line evidence bar.
- `docs/adr/` — architecture decision records; the agent proposes an ADR whenever it makes a decision a future session would otherwise re-litigate.
- `validation.config` — gate thresholds, required mutation floor, escalation, and **`synced_contract_version`**. Only successful absent/proven-clean-Git-virgin SETUP/install or lower/current-invalid RE-SYNC/install stamps live canonical-remote-main `current`. - `docs/FRICTION.md` — the repo's friction log (same format as os/FRICTION.md). Change discipline mirrors os/MAINTENANCE.md: REVIEW.md, validation.config, and the stack profile change only on an explicit owner request or ≥2 entries on the same point. Factual corrections in AGENTS.md (wrong command, wrong path) stay immediate — facts are fixed on sight, rules by friction.
- Constitution section in AGENTS.md (3–7 immutable principles, e.g. "modular always", "no second source of truth", "reuse before write") — the validator checks changes against it.

## Step 4 — Toolchain (the feedback loop itself)

**v21 install/recovery interpretation (normative over every shorthand in this file).** Every existing target, even a syntactically complete pair, evaluates trusted install history; if any trusted event exists, its disk ordinal must equal the unique descendant-most/highest recovered ordinal, else it is corrupt and routes only by recovered lower/equal/future. Any shorthand saying "matching marker" includes only a matching inner marker plus the sole exact-empty absent-SETUP sibling temp-root first-marker exception. Ordinary manifests require platform-normalized-path uniqueness across all operations. Existing-repo post commits have exactly one parent equal to pre-HEAD; absent repo-bootstrap alone is parentless.

- Strict formatter + linter + typechecker, single command, fast; wired as save/pre-commit/CI AND as agent hooks (PostToolUse format/typecheck with failures fed back).
- Test layout: `tests/<module>/` mirroring `src/` (game engines: ONE registered test-scene root with a runner — never ad-hoc scenes). Hygiene gates from VALIDATION.md §hygiene wired into CI from day one.
- **Closing-report gate** — a deliver-time check (a `-Deliver` switch on, or sibling of, the one-command gate; NOT the always-run inner-loop gate — requiring the exact configured `product-result-path` there would block every mid-build gate run) that fails when the exact configured `product-result-path` is absent at that canonical repo-relative path OR missing a required field (outcome / evidence / assumptions / cuts / cost / manual-acceptance / next). Same enforcement surface as the boundary/hygiene gates ("agents cannot ignore a failing check"); deliver/merge depends on this green check, not a prose precondition.
- **Root run-contract v21 supplement (normative over Step 3 shorthand).** Installed AGENTS names intent -> writer-finalized CALL and preserves that CALL/entry receipt/observation byte-identically. PLAN chronology remains candidate -> fresh validator -> fresh verdict -> accepted-only fresh EXEC. SETUP owns absent or proven clean initialized non-bare Git virgin; existing non-git/unborn/dirty stops. RE-SYNC owns lower/current-invalid. Author/review are read-only; finalized install alone writes exact create/overlay/delete plus journaled Git state. Matching marker or absent exact empty first-marker root resumes first. Corrupt recovery uses the unique linear descendant-most/highest trusted event. Authority-advanced terminal install returns `MANIFEST-RECONCILED`, consumes the old call without current/feature receipt, then freshly routes.
- **Transactional handoff gate (contract v21).** Session intent carries the literal locator. Writer finalization adds only schema-1 canonical resolver `https://github.com/ainazemtsau/my_global_workflow` + `refs/heads/main` + fixed authority path, freshly queries/fetches it and reads that exact object. Current/wt/local-main/tracking refs and remote-less operation are forbidden; a common-dir object is usable only after equality with the fresh remote SHA. Executor repeats independently. CALL/env/product cannot override. Writer records remote/ref@SHA evidence, authority bytes/current, product state/validity and receipts, persists finalized CALL, then emits. Audit/digest without a CALL derive the same resolver ephemerally and never persist it.
- **Strong-check enablement gate** — the same deliver-time check ALSO fails — for any change carrying a **non-archived `openspec/changes/<id>/` with a frozen spec** (the mechanical signal of a G0-frozen change; a light change has no such folder, so the gate is N/A by ABSENCE of the artifact, never by an agent declaring the change "light") — unless (a) the change's recorded **mutation score ≥ the `validation.config` floor** (proof G2 ran, not just the suite — the G2 run writes the score; the deliver check reads it, never re-runs the expensive pass), and (b) the change's frozen spec carries a **non-empty spec-silence audit** — the unconstrained regimes / actor-counts / seams / user-perceivable regimes list (CONTOUR §b), each marked intentional|unintended, AND a row-by-row walk of the cross-kind escape-class registry (`os/engineering/ESCAPE-CLASSES.md`) with every class marked addressed|n/a (an unmarked registry class fails deliver), and (c) the change's frozen spec carries a **deliverable-coverage list** (CONTOUR §c) that records the CALL done_when verbatim and gives EVERY done_when bullet a disposition — ≥1 ledger row OR an out-of-scope line — so a promised artifact cannot be silently omitted (a bullet with no disposition fails deliver); each row records the bullet's CLASS (`headless`|`engine`) and the gate keys on it, so a row whose class is `engine` disposed as built MUST carry a resolving EXISTS path under the artifact root (`Assets/**` / `docs/measurements/**`) that the check `Test-Path`s to a non-trivial file (a recorded MCP/batch-run artifact with a non-trivial measurement counts; an empty/stub file, a directory, a foreign pre-existing file, or — the leg-8 lie — a sibling headless test name in place of an artifact all FAIL), so a promised scene/prefab/composer cannot be reported done while absent (the engine-side existence dependency `VALIDATION.md` §Enabled ≠ written now requires, license-free substitute for the deferred PlayMode CI — `profiles/unity.md`); and an out-of-scope line on a promised bullet carries a recorded OWNER-ack token (plan-approval signature or escalation id) — a self-authored cut of a promise fails deliver; and (d) for the same G0-frozen change the repo holds a recorded **review-evidence** file `docs/reviews/review-<id>.md` (filename `<id>` == the non-archived change folder — absent, or naming another id, fails deliver) whose `reviewed-commit` is an ANCESTOR of the delivered HEAD with every intervening source commit accounted for by a `fixed <commit>` disposition (an un-reviewed source change since the review fails deliver; an honest review→fix→deliver passes, because the fix-commits ARE the dispositions — freshness is ancestry+accounting, never tree-equality), records ≥1 review pass by a model family other than the builder, tags every finding `scope: in-scope|class-sibling` and gives EVERY **in-scope** row a disposition (`fixed`|`refuted`|`owner-ack out-of-scope` — an in-scope row missing the disposition or valued `open` fails deliver) and EVERY **class-sibling** row a `routed <ref>` pointer (a class-sibling with no routing pointer fails deliver; a routed-not-fixed class-sibling does NOT gate — leg-gating is scoped to the diff, and a `class-sibling` row whose recorded `site` file is INSIDE the change's diff fails deliver (the mechanical anti-dodge — a git-diff name-membership test on the site, same surface as the reviewed-commit ancestry check and like it INERT until §Re-sync, the writer-G10 carrying `site`+`scope`+`routed` line-presence before then), VALIDATION §Review-evidence (iii)), carries a refutation-verification marker on every `refuted` row (in-session G5, binding fresh-session KERNEL-G5, or a `register:<row-id>` citation per clause (g) — a `refuted` with no marker fails deliver), and records `rounds:` ≤3 or a RESOLVING `escalation:<id>` (same resolution as the owner-ack token — a hollow escalation on >3 rounds fails deliver); the review-evidence dependency becomes a product deliver check only at §Re-sync (its `reviewed-commit` ancestry/accounting needs a working tree the writer lacks — the writer-side G10 bounce carries its line-presence before then); and (e) for the same G0-frozen change the `-Deliver` run includes a **negative-control pass**: for every acceptance-property token the suite holds a committed `<Prop>_NegativeControl` test (category `NegativeControl`) exercising the property's own assertion against a deliberately-wrong implementation, and the pass (`dotnet test --filter TestCategory=NegativeControl`, or the stack equivalent) requires that control DISCOVERED and PASSED — a property with no control (oracle unverified), or a control the property lets through (empty oracle: the property passed against its wrong impl), fails deliver, counted discovered+passed per token — closing what G2 mutation cannot (an absent property has no mutant; FsCheck 3.x fallback) that discharges it: a `core algorithm` change with a `headless` row carrying no `prop:` token, or one naming a test absent from the property table, fails deliver (VALIDATION §Property-layer). Like v9's registry-walk and v11's negative-control, this axis is INERT until §Re-sync wires the `prop:` token into `coverage-check.ps1` — PILOT it on the next `core algorithm` leg, not a retrofit of already-delivered core files; and (g) every `refuted` finding row in the change's `docs/reviews/review-<id>.md` resolves to a row in the repo's cumulative refuted-register `docs/reviews/REFUTED.md`: the review's `refuted-register:` line names the appended row ids (`none` is legal only alongside zero `refuted` rows), the check `Test-Path`s the register and greps each named row id, a `register:<row-id>` marker must resolve to an existing register row, and — clause (b)'s walk extended — a register row naming a module/seam the frozen spec touches is marked `still-holds|re-opened <why>`; a `refuted` row with no register row, a review with `refuted` rows and no `refuted-register:` line, a `register:` citation resolving nowhere, or an unmarked touching register row fails deliver; like v9/v11/v12 this axis is INERT until §Re-sync (the writer-side G10 bounce carries its line-presence before then — VALIDATION §Refuted-register); and (h) every `fixed <commit>` disposition row in the change's `docs/reviews/review-<id>.md` carries its **sweep line** (`sweep: class=<the row's invariant/class token>; sites: <each sibling site the class predicts, closed <commit> | n/a because <reason>>; controls: <a seeded-miss/negative-control token per class variant for a gate/scan/oracle false-green finding | n/a>` — VALIDATION §Fix-class-closure), and no later-round finding row carries an `invariant/class` token equal to an earlier `fixed` row's class without that earlier row marked re-opened: a `fixed` row with no sweep line, or an un-reopened recurring class token, fails deliver; like v9/v11/v12/v13 this axis is INERT until §Re-sync (the writer-side G10 class+sweep bounce — which keys on the RESULT's own fixed-finding content, not on a frozen declaration — carries it before then). Presence + threshold, same surface as the report gate: the gate asserts the score, the audit, the coverage list, the review-evidence file, the negative-control pass, the property-layer table/tokens, the refuted-register rows, and the fixed-row sweep lines EXIST and pass (every done_when bullet has a disposition, each `engine` row resolves a non-trivial artifact, each cut carries an owner-ack token), not that their content is true (it cannot tell a mis-classed `engine`→`headless` bullet, an under-transcribed done_when, or an honest-but-unrelated artifact — those stay the validator's / owner's / writer-G10 carry-back job) (truthfulness stays the validator's and the owner's job — same bound the report-field and converge-readiness gates accept). ## Step 5 — Harness state files

`SPEC` (per-change, from OpenSpec), feature **ledger** (machine-readable, all entries failing, not writable by the builder), `PROGRESS.md` (session handoff log), first commit.

### Canonical v21 bootstrap conformance corpus

`os/engineering/V21-CONFORMANCE.md` is the OS-owned binding-handoff authority and `v21_conformance.py` its regression witness. Absent/proven clean-Git virgin uses SETUP; lower/current-invalid uses RE-SYNC. Pre-install authority is the writer-finalized canonical remote-main resolver, never product/local/tracking bytes. Author returns a Direction bundle pinning equivalent product-native resolver without writes; fresh reviewer verifies it; fresh finalizer creates install. Only install mutates declared rows/Git journal. Disposable cases include every negative class and actual compile/discovery/run.

Bootstrap attempt identity is executable: persist 128-bit lowercase-hex nonce, canonical-LF intent SHA-256 and RFC8785 ordinary-manifest SHA-256 after platform-normalized-path then operation sort. A platform-normalized path is unique across the whole manifest regardless of operation; create+delete, overlay+delete, any cross-operation reuse and case-equivalent duplicates fail. Compute lowercase SHA-256 over the exact versioned LF frame in packets.md, excluding marker/temp/finalized CALL/receipt, then derive all paths. Create/overlay uses declared same-filesystem temp+fsync+atomic replace; recovery requires a matching inner marker, except an exact-empty absent-SETUP sibling temp-root may create it as the sole first-marker exception, and then permits any declared torn temp to be removed/regenerated from pinned postimage. Delete is exact contract/tool quarantine/removal and return verifies it; production/feature/test delete is forbidden. Finalized install journals pre/post ref/HEAD/index/tree, deterministic commit metadata/OID, temp index and declared locks. An existing-repo post commit has exactly one parent equal to pinned pre-HEAD; only absent repo-bootstrap is parentless, and wrong/multi-parent commits fail. Retry rebuilds exact Git stages, CAS-updates ref and ends clean. Every run adds separate canonical reconcile observation/hash while original CALL/receipt/entry observation remain unchanged. Old-authority terminal proof yields `MANIFEST-RECONCILED`, not a current receipt.
