# Project setup — bootstrap an agent-ready product repo

A procedure executed by a coding agent (as an engineering CALL) when a direction needs a product repository. Its product is not code — it is the **feedback loop**: a repo where autonomous builds converge fast and entropy is mechanically contained. Run once per repo; revisit only via friction.

## Step 1 — Stack interview (with the owner, short)

Check `os/engineering/profiles/<stack>.md` first: an existing profile is the entry point (conventions, boundary tooling, default thresholds, known landmines). Then ask only what the direction's CALL and profile don't answer: target platform, hard constraints, owner's stack preferences. Propose the stack with an outside view (what do shipped projects of this kind use) and return the exact accepted choice as Direction RESULT evidence; this interview is read-only in the product repo. The later `SETUP/install` manifest creates the product's ADR-0001 from those pinned bytes. If no profile exists for the chosen stack, create one as a declared Direction `work/` artifact (≤100 lines, format in `profiles/README.md`) and return it in the RESULT's `state_changes` — the writer commits it to the workflow repo (the executor never writes there directly).

## Step 2 — Architecture skeleton

- **Modular monorepo.** Modules under `src/<module>/`; each module small enough that module + its docs fit one agent context.
- **Boundaries enforced by the build graph**, not by docs: dependency-graph linting (import-linter / dep-cruiser / engine equivalent); module visibility is opt-in; violations fail CI. Agents can ignore documentation; they cannot ignore a failing check.
- **Per-module docs colocated**: each module gets its own `AGENTS.md` (nearest-file-wins): public API surface, internal conventions, gotchas. Project-wide docs live in the repo (`docs/`), never outside it.

## Step 3 — Root contract files

- `AGENTS.md` (root, ≤150 lines, command-first): what the project is, exact build/test/lint commands (single-command AND file-scoped variants), conventions, definition of done. `CLAUDE.md` = one line importing AGENTS.md + platform-specific notes. Prune on every agent mistake — this file is a feedback loop, not documentation.
- **Run contract section in root AGENTS.md** (≤10 lines, distilled from `os/engineering/CONTOUR.md` at setup): roles separated (planner / binding-validator / owner-verdict / independent RED author / builder / reviewer); v21 sequence = PLAN author candidate -> fresh binding-validator -> fresh owner verdict -> accepted-only fresh finalizer -> EXECUTE independent RED -> receipt-bound BUILD, with no planner-issued BUILD and no shared RED/builder session; **execution authority is repo-local: the CALL + this repo's `AGENTS.md`, frozen plan/spec, and `validation.config` define the workflow and its required tools; a globally installed/discoverable skill or tool is support, not authority — it cannot replace the declared workflow or add project requirements; discovery or installation alone is never a blocker**; cycle = PLAN author -> binding validator -> owner verdict -> EXECUTE independent RED -> BUILD -> gates -> report; the builder never edits the ledger, the spec, acceptance criteria, or the independently-authored acceptance tests (it makes them pass); retries ≤3 per gate, then escalate; a required tool/harness the plan depends on being UNAVAILABLE (engine Editor down, a needed MCP disconnected, headless-test license absent) is itself a mandatory STOP + owner-contact naming the tool, never a workaround/crutch, and any exception needs an explicit written owner-ack; a fix that ADDS or CHANGES behavior (a guard that can throw, a reordering) is itself a change — it re-runs the invariant-extension / spec-silence checks on its NEW failure paths and gets a test-author RED test for the touched invariant before the fix, and findings are classified by the invariant they violate so a recurring class trips the non-convergence stop, and a finding disposes `fixed` only with a recorded `sweep:` line — sibling sites closed|n/a, plus a seeded-miss/negative-control per class variant when the finding is itself a gate/scan/oracle false-green — while a later finding row re-using a prior `fixed` row's class token re-opens that row (a fix is a change, not a patch, and a patch does not close a finding); cross-module work goes through module public APIs (their AGENTS.md) — changing another module's internals is a separate feature leg; a decision whose OUTPUT the product's user directly experiences (sees/hears/feels/reads — how a mechanism looks or sounds, how output is paced or phrased) is an owner-signed acceptance property at PLAN, not a builder default (its formula/magnitude stays a builder ADR); done = gates green + evidence attached, **and the closing report the exact configured `product-result-path` (fields per Done-when) exists — producing it is part of the deliver-time gate (Step 4), so a missing/under-filled the exact configured `product-result-path` → not deliverable and a prose summary does not satisfy it; the builder routes the RESULT home (`next:` names the direction) and never authors the next task CALL; owner signals ("finish"/"summary") trigger the report, never replace it, and (contract v18) any `docs/reviews/review-*.md` / `docs/adr/*.md` path the RESULT CITES — rooted in the repo's OWN `docs/` tree — must EXIST at HEAD: a dangling citation (a report claiming a review/ADR not committed at the delivered tip) → not deliverable, for EVERY leg (this ALWAYS-RUN cited-existence floor is the light-lane counterpart of the frozen review-evidence dependency below, ORTHOGONAL to it — different key space — and a RESULT that OMITS the citation is untouched; VALIDATION §Cited-artifact existence)**. The same deliver-time gate also fails, for a G0-spec-frozen change, without a recorded G2 mutation score ≥ the `validation.config` floor, a non-empty spec-silence-audit section in the frozen spec, AND a recorded `docs/reviews/review-<id>.md` review-evidence file (id == the change folder; `reviewed-commit` ancestor-of-HEAD with every intervening source commit a `fixed <commit>` disposition; every in-scope finding dispositioned (class-siblings routed to backlog); every `refuted` marked; `rounds` ≤3 or a resolving `escalation:<id>`), AND a **negative-control pass** (a committed `<Prop>_NegativeControl` per acceptance property, GREEN only when the property rejects its deliberately-wrong impl) (Step 4 §Strong-check enablement), AND, for a `core algorithm` change, a **property-layer pass** (a `prop:<TestClass>.<TestMethod>` token per `headless` deliverable-coverage row, matched against an owner-signed property table covering permutation-metamorphic / fault-injection-all-exit-paths / multi-actor-one-tick / boundary-biased — Step 4 §Strong-check enablement clause (f)), AND a **refuted-register pass** for a review carrying any `refuted` finding (every `refuted` appends a row to the cumulative `docs/reviews/REFUTED.md`; a later review closes a re-derived duplicate by `register:<row-id>` citation; a re-raise WITH new evidence escalates to the owner, never silently re-litigated — Step 4 §Strong-check clause (g)) — running the test suite is not G2, a silence audit done only in chat does not count, a leg cannot deliver before an independent review round is recorded and dispositioned, and a suite that never proves its oracle can fail (no negative control) has not verified it. Thresholds and rubric live in `validation.config` and `REVIEW.md`.
- **Executable-test handoff in root AGENTS.md (v21).** For every G0-frozen change, the planner writes an ungated exact-base `TESTABILITY.md` binding contract with complete per-obligation `construct|act|observe|negative|source|skeleton` routes and no GREEN, probe source or candidate self-pin. A distinct fresh read-only `PLAN/binding-validator` derives the obligation union from an adapter-reviewed authority grammar independent of candidate P; that grammar rejects empty/minimal self-description and pins typed literal values, exact source/resource path+mode+blob, owner/slot lifetime and every required observer/delegation edge. Product code is never loaded into the validator controller: native compile/runtime receives an exact scratch mirror selected through the validated source registry, and the original product tree is compared with write-sensitive metadata. The validator generates scratch-only probes and records GREEN only in its separate persisted RESULT/event after actual native compile, unique discovery, filtered sentinels and an unchanged product tree. Only then may a fresh finalizer create `PLAN/verdict`; the later independent RED author reads the frozen spec plus routes/receipt, never validator probe bytes or PLAN/tasks/ADR as behavioral oracle. Any gap is one complete `PLAN RED` sweep before owner approval, EXECUTE or BUILD.
- `REVIEW.md` — the validation rubric (see VALIDATION.md §rubric): severity definitions for THIS repo, always-check rules, skip rules, nit cap, file:line evidence bar.
- `docs/adr/` — architecture decision records; the agent proposes an ADR whenever it makes a decision a future session would otherwise re-litigate.
- `validation.config` — gate thresholds: retry budget, **mutation kill-rate floor (a required key — its absence means G2 is not enabled, VALIDATION.md §Enabled ≠ written)**, escalation rules (machine-readable; the run loop reads it), and **`synced_contract_version`** — stamp it with the current `os/engineering/CONTRACT_VERSION` `current:` value at setup. This records which engineering-contract hardening level the repo is built against. Writer/finalizer/collect and executor require exact equality with independently resolved live `current`; missing, unreadable, behind, future or a stale finalized packet is a hard RE-SYNC STOP (the forcing function against silent propagation lag — `os/engineering/CONTRACT_VERSION`).
- `docs/FRICTION.md` — the repo's friction log (same format as os/FRICTION.md). Change discipline mirrors os/MAINTENANCE.md: REVIEW.md, validation.config, and the stack profile change only on an explicit owner request or ≥2 entries on the same point. Factual corrections in AGENTS.md (wrong command, wrong path) stay immediate — facts are fixed on sight, rules by friction.
- Constitution section in AGENTS.md (3–7 immutable principles, e.g. "modular always", "no second source of truth", "reuse before write") — the validator checks changes against it.

## Step 4 — Toolchain (the feedback loop itself)

- Strict formatter + linter + typechecker, single command, fast; wired as save/pre-commit/CI AND as agent hooks (PostToolUse format/typecheck with failures fed back).
- Test layout: `tests/<module>/` mirroring `src/` (game engines: ONE registered test-scene root with a runner — never ad-hoc scenes). Hygiene gates from VALIDATION.md §hygiene wired into CI from day one.
- **Closing-report gate** — a deliver-time check (a `-Deliver` switch on, or sibling of, the one-command gate; NOT the always-run inner-loop gate — requiring the exact configured `product-result-path` there would block every mid-build gate run) that fails when the exact configured `product-result-path` is absent at that canonical repo-relative path OR missing a required field (outcome / evidence / assumptions / cuts / cost / manual-acceptance / next). Same enforcement surface as the boundary/hygiene gates ("agents cannot ignore a failing check"); deliver/merge depends on this green check, not a prose precondition.
- **Root run-contract v21 supplement (normative over Step 3 shorthand).** Installed AGENTS names CALL-intent -> writer-finalized CALL; `PLAN/author` candidate event -> fresh-finalized `PLAN/binding-validator` -> validator GREEN event -> fresh-finalized verdict -> explicit accepted owner event for unchanged P/TESTABILITY/receipt -> fresh-finalized EXEC, all with strict RESULT-authorized ancestry and no self-hash/bare live edit. Revised/split re-enters a new candidate plus full binding validation; rejected creates no EXEC. PLAN entry has no future TESTABILITY authority; only the validator event carries binding GREEN. External base/acceptance, exact result routing, a closed typed binding DAG, actual validator-generated probe compile/discovery/filtered sentinels, exact-parent independent acceptance RED, fresh EXECUTE/resume and the SETUP adapter bundle/review/install contour remain required. Receipt first appearing with owner words, nickname bindings, compile booleans, planner-supplied probes, hardcoded report or self-reported bootstrap are invalid.
- **Transactional handoff gate (contract v21).** - session emits stable intent; writer alone adds current observations/receipts, runs the gate, persists finalized CALL, then emits. Modes are `install|plan-entry|plan-binding|pre-execution|red-boundary|resume|deliver|close`. Receipt preimages exclude themselves/check result; observation hash records command/exit/status. Candidate, binding-validator, verdict and execution finalizations are separate RESULT/state-change transactions; pre-execution finalization pins pre-red-head. Full semantics are in packets.md, VALIDATION.md and V21-CONFORMANCE.md.
- **Strong-check enablement gate** — the same deliver-time check ALSO fails — for any change carrying a **non-archived `openspec/changes/<id>/` with a frozen spec** (the mechanical signal of a G0-frozen change; a light change has no such folder, so the gate is N/A by ABSENCE of the artifact, never by an agent declaring the change "light") — unless (a) the change's recorded **mutation score ≥ the `validation.config` floor** (proof G2 ran, not just the suite — the G2 run writes the score; the deliver check reads it, never re-runs the expensive pass), and (b) the change's frozen spec carries a **non-empty spec-silence audit** — the unconstrained regimes / actor-counts / seams / user-perceivable regimes list (CONTOUR §b), each marked intentional|unintended, AND a row-by-row walk of the cross-kind escape-class registry (`os/engineering/ESCAPE-CLASSES.md`) with every class marked addressed|n/a (an unmarked registry class fails deliver), and (c) the change's frozen spec carries a **deliverable-coverage list** (CONTOUR §c) that records the CALL done_when verbatim and gives EVERY done_when bullet a disposition — ≥1 ledger row OR an out-of-scope line — so a promised artifact cannot be silently omitted (a bullet with no disposition fails deliver); each row records the bullet's CLASS (`headless`|`engine`) and the gate keys on it, so a row whose class is `engine` disposed as built MUST carry a resolving EXISTS path under the artifact root (`Assets/**` / `docs/measurements/**`) that the check `Test-Path`s to a non-trivial file (a recorded MCP/batch-run artifact with a non-trivial measurement counts; an empty/stub file, a directory, a foreign pre-existing file, or — the leg-8 lie — a sibling headless test name in place of an artifact all FAIL), so a promised scene/prefab/composer cannot be reported done while absent (the engine-side existence dependency `VALIDATION.md` §Enabled ≠ written now requires, license-free substitute for the deferred PlayMode CI — `profiles/unity.md`); and an out-of-scope line on a promised bullet carries a recorded OWNER-ack token (plan-approval signature or escalation id) — a self-authored cut of a promise fails deliver; and (d) for the same G0-frozen change the repo holds a recorded **review-evidence** file `docs/reviews/review-<id>.md` (filename `<id>` == the non-archived change folder — absent, or naming another id, fails deliver) whose `reviewed-commit` is an ANCESTOR of the delivered HEAD with every intervening source commit accounted for by a `fixed <commit>` disposition (an un-reviewed source change since the review fails deliver; an honest review→fix→deliver passes, because the fix-commits ARE the dispositions — freshness is ancestry+accounting, never tree-equality), records ≥1 review pass by a model family other than the builder, tags every finding `scope: in-scope|class-sibling` and gives EVERY **in-scope** row a disposition (`fixed`|`refuted`|`owner-ack out-of-scope` — an in-scope row missing the disposition or valued `open` fails deliver) and EVERY **class-sibling** row a `routed <ref>` pointer (a class-sibling with no routing pointer fails deliver; a routed-not-fixed class-sibling does NOT gate — leg-gating is scoped to the diff, and a `class-sibling` row whose recorded `site` file is INSIDE the change's diff fails deliver (the mechanical anti-dodge — a git-diff name-membership test on the site, same surface as the reviewed-commit ancestry check and like it INERT until §Re-sync, the writer-G10 carrying `site`+`scope`+`routed` line-presence before then), VALIDATION §Review-evidence (iii)), carries a refutation-verification marker on every `refuted` row (in-session G5, binding fresh-session KERNEL-G5, or a `register:<row-id>` citation per clause (g) — a `refuted` with no marker fails deliver), and records `rounds:` ≤3 or a RESOLVING `escalation:<id>` (same resolution as the owner-ack token — a hollow escalation on >3 rounds fails deliver); the review-evidence dependency becomes a product deliver check only at §Re-sync (its `reviewed-commit` ancestry/accounting needs a working tree the writer lacks — the writer-side G10 bounce carries its line-presence before then); and (e) for the same G0-frozen change the `-Deliver` run includes a **negative-control pass**: for every acceptance-property token the suite holds a committed `<Prop>_NegativeControl` test (category `NegativeControl`) exercising the property's own assertion against a deliberately-wrong implementation, and the pass (`dotnet test --filter TestCategory=NegativeControl`, or the stack equivalent) requires that control DISCOVERED and PASSED — a property with no control (oracle unverified), or a control the property lets through (empty oracle: the property passed against its wrong impl), fails deliver, counted discovered+passed per token — closing what G2 mutation cannot (an absent property has no mutant; a wrong-but-consistent or weak assertion kills only the mutants of code that exists); the control is authored by the independent test-author (builder cannot edit it), and the dependency, like v9's registry-walk enforcement, is INERT until §Re-sync (the writer runs no tests, so there is no writer-side keystone); and (f) for a change whose PLAN-stage change-class (CONTOUR's cycle, the same taxonomy that routes it architectural) is `core algorithm`, every `headless`-class deliverable-coverage row (clause (c)) carries a `prop:<TestClass>.<TestMethod>` evidence token — the same mechanical form clause (c) already requires of `engine` rows as `exists:` — matched against an owner-signed **property table** in the frozen spec naming, per required class (permutation-metamorphic, fault-injection-all-exit-paths, multi-actor-one-tick, boundary-biased), the `[Category("Property")]` test (CsCheck; FsCheck 3.x fallback) that discharges it: a `core algorithm` change with a `headless` row carrying no `prop:` token, or one naming a test absent from the property table, fails deliver (VALIDATION §Property-layer). Like v9's registry-walk and v11's negative-control, this axis is INERT until §Re-sync wires the `prop:` token into `coverage-check.ps1` — PILOT it on the next `core algorithm` leg, not a retrofit of already-delivered core files; and (g) every `refuted` finding row in the change's `docs/reviews/review-<id>.md` resolves to a row in the repo's cumulative refuted-register `docs/reviews/REFUTED.md`: the review's `refuted-register:` line names the appended row ids (`none` is legal only alongside zero `refuted` rows), the check `Test-Path`s the register and greps each named row id, a `register:<row-id>` marker must resolve to an existing register row, and — clause (b)'s walk extended — a register row naming a module/seam the frozen spec touches is marked `still-holds|re-opened <why>`; a `refuted` row with no register row, a review with `refuted` rows and no `refuted-register:` line, a `register:` citation resolving nowhere, or an unmarked touching register row fails deliver; like v9/v11/v12 this axis is INERT until §Re-sync (the writer-side G10 bounce carries its line-presence before then — VALIDATION §Refuted-register); and (h) every `fixed <commit>` disposition row in the change's `docs/reviews/review-<id>.md` carries its **sweep line** (`sweep: class=<the row's invariant/class token>; sites: <each sibling site the class predicts, closed <commit> | n/a because <reason>>; controls: <a seeded-miss/negative-control token per class variant for a gate/scan/oracle false-green finding | n/a>` — VALIDATION §Fix-class-closure), and no later-round finding row carries an `invariant/class` token equal to an earlier `fixed` row's class without that earlier row marked re-opened: a `fixed` row with no sweep line, or an un-reopened recurring class token, fails deliver; like v9/v11/v12/v13 this axis is INERT until §Re-sync (the writer-side G10 class+sweep bounce — which keys on the RESULT's own fixed-finding content, not on a frozen declaration — carries it before then). Presence + threshold, same surface as the report gate: the gate asserts the score, the audit, the coverage list, the review-evidence file, the negative-control pass, the property-layer table/tokens, the refuted-register rows, and the fixed-row sweep lines EXIST and pass (every done_when bullet has a disposition, each `engine` row resolves a non-trivial artifact, each cut carries an owner-ack token), not that their content is true (it cannot tell a mis-classed `engine`→`headless` bullet, an under-transcribed done_when, or an honest-but-unrelated artifact — those stay the validator's / owner's / writer-G10 carry-back job) (truthfulness stays the validator's and the owner's job — same bound the report-field and converge-readiness gates accept). This is what stops a strong check that exists only as prose from passing green. **General rule: every load-bearing gate in VALIDATION.md / CONTOUR.md is wired here as an executable-or-recorded deliver dependency, never left as text** — adding a gate to those files without a deliver dependency + a Done-when seeded-miss leaves it unenabled by construction.
- **Re-sync (existing repos).** Setup runs once, so when `os/engineering` hardens or adds a gate, an already-set-up repo does NOT get it automatically — its deliver check stays stale. Re-sync is a `docs/FRICTION.md` / maintenance trigger IN the product repo: re-run the affected Done-when items below and wire the newly-required gate (this is the general form of the per-gate "retrofitting is a friction trigger" note), then **bump `validation.config: synced_contract_version` to the current `os/engineering/CONTRACT_VERSION`** so the repo stops being flagged as behind. Until re-synced, the repo cannot claim the hardened gate is enabled — and `collect`/`audit`/`digest` surface the gap (the version stamp is what makes the lag visible instead of silent). The os/engineering maintenance change that hardens a gate bumps `CONTRACT_VERSION` and names the re-sync as a follow-up; it does not itself edit product repos.
- `_scratch/` — the only home for exploratory scripts/scenes; tracked dir, self-ignoring `.gitignore`; CI asserts nothing from it is committed.
- **Canonical v21 identity.** Every engineering handoff begins as intent. Only writer-finalized persisted CALL is runnable. PLAN and descendants retain external base/acceptance; PLAN candidate/receipt precedes verdict; EXECUTE gets writer run/pre-red receipt; BUILD resumes it. SETUP/RE-SYNC use external install/corpus blobs. Guessed receipts, prose, HEAD, plan tip and historical checkpoint never create authority.
- Base `validation.config` declares contract version, roots, result template, handoff command and regular trust roots. OS `V21-CONFORMANCE.md` plus executable `v21_conformance.py` supply the binding-handoff rule and OS regression witness; the product adapter must run equivalent cases with its real native compiler/discovery/runner. CALL pins the separately authored/reviewed adapter bundle and conformance event. Product installer supplies no expected verdict. Dynamic/unlisted/symlink/gitlink/reparse/outside/case-colliding/substituted load is invalid.
- Toolchain hash is SHA-256 over LF-joined `<path>\t<mode>\t<base-or-installed-blob>` rows plus final LF. PLAN/EXECUTE/BUILD use the externally persisted conformance receipt for that exact installed hash. Before every product-controlled mode, writer/executor independently derives and compares the applicable base/current rows; planner repeats this before final `plan`.
- `SETUP/interview` records inventory/profile without product writes. Only the later `SETUP/install` intent separates create rows (external source mode/blob, collision fails) from overlay rows (exact pre mode/blob and post blob or patch hash); it must cite the accepted interview/adapter events, and any changed inventory requires a fresh install intent before writes. Unspecified entries stay protected. RE-SYNC uses the same external-source model at clean HEAD==base. Both install routes close only after writer materializes/verifies OS mutations with no case/expected id and persists external conformance receipt.
- Typed TESTABILITY rows distinguish packet, frozen authority and base-qualified baseline. Every obligation has complete `construct|act|observe|negative|source|skeleton` routes and a closed typed binding DAG. A distinct validator, not the planner, generates ephemeral binding probes and actually compiles/uniquely-discovers/runs them at base without a production stub; absent new APIs use an exact reflection/dynamic route. Probe source is not persisted or reused as acceptance RED. Bare non-empty cells, direct uncompilable new-member calls or planner-authored GREEN fail PLAN.
- Base result template resolves the CALL's exact same-id `product-result-path`. Deliver hashes `D:<that path>`; archive close verifies that exact path/blob. Routing receipt can be reconstructed at D or from clean A/D=A^ after interruption and is never authority or a product self-hash.
- Independent RED receipt pins pre-red-head, author role/session, exact parent and test/oracle-registration-only delta, unique IDs/filter/runtime RED. Production-before-R, mixed/unmanifested commit or wrong parent fails. Later revisions parent exact accepted progress. Builder control/resume/deliver/close remain receipt-bound.
- **OpenSpec** lifecycle is propose -> apply -> verify -> pre-archive deliver -> archive -> close. Close relocates only packet rows, preserves frozen external authority rows, verifies baseline rows at base, and checks exact mode/type/blob. CI runs mechanical and test gates on every PR.
- Agent platform config: permission profile for autonomous runs (deny-list for hard nevers: force-push, secret paths, destructive ops; allow-list for project commands); evaluator subagent (read-only tools, instructed to report correctness gaps only) and cheap investigator subagent; verification procedures packaged as skills as they stabilize (side-effectful ones human-trigger-only); notification hooks (needs-input / finished) to the owner's push channel; stop-file and steer-file conventions.

## Step 5 — Harness state files

`SPEC` (per-change, from OpenSpec), feature **ledger** (machine-readable, all entries failing, not writable by the builder), `PROGRESS.md` (session handoff log), first commit.

### Canonical v21 bootstrap conformance corpus

`os/engineering/V21-CONFORMANCE.md` is the OS-owned binding-handoff authority and `v21_conformance.py` is its executable regression witness. After stack interview and before SETUP/RE-SYNC install, a separate Direction adapter-author session returns one adapter bundle: source plus output manifest. Writer commits its RESULT/bundle and a mechanical finalization CALL; a fresh finalizer RESULT creates a strict-descendant read-only adapter-review CALL, and that different session verifies the exact bundle and real tool identities before returning acceptance. Only then may install intent pin author/review `history/<path>@<writer-commit>` evidence plus both path/mode/blob identities. Installer/checker cannot author, review or invent expected conformance results. Product-native disposable cases must include every OS negative class and actual compile/discovery/run.

Bootstrap is acyclic: install cases receive a writer-random challenge; their rows derive a bootstrap receipt; non-install phase cases receive that opaque receipt and test missing/wrong/cross-phase values; only afterward does writer derive/persist the final receipt and run an excluded-from-preimage binding smoke. It also substitutes always-zero, case/order/root/fingerprint-switch checkers, removes the named defect from a negative, and omits one mutation. Each must fail before stamp 21. Receipt rows cover corpus, accepted adapter/specimen, salt-map hash, mutation digest, projected call, before/after trees, exit and status. A self-reported suite, same-session adapter approval or fixed-fingerprint checker is never evidence.

## Done when (mechanically checkable)

- [ ] one-command check passes locally and in CI (format+lint+type+tests)
- [ ] `SETUP/interview` is read-only, needs no adapter/manifest, and commits stack/profile/target-inventory evidence before adapter author/review/accept; any product write or install claim fails. Author RESULT/artifact commit precedes a mechanical-finalizer commit containing the fresh review CALL; review is read-only. Only `SETUP/install` carries the accepted events and manifest. Its create rows pin source mode/blob and overlay rows pin pre/post mode/blob plus patch; every unlisted byte is protected. Separate valid absent-root, empty-root and nonempty-overlay fixtures pass. Wrong create source/post mode/blob, collision, wrong overlay pre/post/patch, unrelated-line loss, undeclared overwrite, half-initialized target or missing/nonancestral interview/author/review event fails.
- [ ] RE-SYNC entry is initialized-behind, clean exact branch, external base-authority and `HEAD==base`; return verifies exact manifest path set plus create absence/source postimage, overlay pre/post/patch, and every unlisted pre-existing byte unchanged. Dirty/wrong branch/base, whole-file replacement losing unrelated lines, undeclared path, production/feature/acceptance-test touch, hidden historical diff or self-bumped stamp fails.
- [ ] adapter authority chain is strict: interview RESULT/event -> separate author RESULT plus exact event-added artifact -> fresh finalizer RESULT -> read-only review CALL/RESULT -> writer acceptance event -> exact path/mode/blob/output manifest in SETUP/install or RE-SYNC intent. Missing/non-ancestral stage or artifact, wrong artifact mode/blob, author product write, review product write, same installer/checker author, self-review or later adapter edit fails.
- [ ] SETUP/RE-SYNC bootstrap is writer-run from exact corpus bytes/mutations through that accepted adapter with nonce-salted identities. Install challenge -> bootstrap receipt -> non-install receipt cases -> persisted final receipt -> excluded binding smoke is acyclic. Valid rows exit 0, every miss exits nonzero, no mutation. Always-zero, case/order/root/fingerprint switch, defect-removed negative, omitted mutation, missing/wrong/cross-phase receipt and unpinned adapter/specimen fail; receipt rows contain salt/adapter/specimen/mutation/projected-call/tree/exit/status evidence.
- [ ] CALL materialization is two-stage: session emits stable `intent`; writer alone derives current observations, runs the gate and persists `finalized` plus a non-self-referential receipt/observation hash. Intent execution, session-supplied/guessed/tampered writer fields, changed external state or collect-time silent refresh fail; an unchanged persisted finalized CALL passes.
- [ ] An event-dependent child always uses two RESULT-authorized transactions. The first RESULT commits event plus a normal pending mechanical finalization CALL W; the fresh finalizer RESULT appends history/LOG, removes only that CALL and creates the exact finalized child W2; child work is later.
