# Project setup — bootstrap an agent-ready product repo

A procedure executed by a coding agent (as an engineering CALL) when a direction needs a product repository. Its product is not code — it is the **feedback loop**: a repo where autonomous builds converge fast and entropy is mechanically contained. Run once per repo; revisit only via friction.

## Step 1 — Stack interview (with the owner, short)

Check `os/engineering/profiles/<stack>.md` first: an existing profile is the entry point (conventions, boundary tooling, default thresholds, known landmines). Then ask only what the direction's CALL and profile don't answer: target platform, hard constraints, owner's stack preferences. Propose the stack with an outside view (what do shipped projects of this kind use) and record the choice as ADR-0001. If no profile exists for the chosen stack, create one as a byproduct of this setup (≤100 lines, format in `profiles/README.md`) and return it in the RESULT's `state_changes` — the writer commits it to the workflow repo (the executor never writes there directly).

## Step 2 — Architecture skeleton

- **Modular monorepo.** Modules under `src/<module>/`; each module small enough that module + its docs fit one agent context.
- **Boundaries enforced by the build graph**, not by docs: dependency-graph linting (import-linter / dep-cruiser / engine equivalent); module visibility is opt-in; violations fail CI. Agents can ignore documentation; they cannot ignore a failing check.
- **Per-module docs colocated**: each module gets its own `AGENTS.md` (nearest-file-wins): public API surface, internal conventions, gotchas. Project-wide docs live in the repo (`docs/`), never outside it.

## Step 3 — Root contract files

- `AGENTS.md` (root, ≤150 lines, command-first): what the project is, exact build/test/lint commands (single-command AND file-scoped variants), conventions, definition of done. `CLAUDE.md` = one line importing AGENTS.md + platform-specific notes. Prune on every agent mistake — this file is a feedback loop, not documentation.
- **Run contract section in root AGENTS.md** (≤10 lines, distilled from `os/engineering/CONTOUR.md` at setup): roles separated (planner / builder / validator / independent test-author); cycle = plan → ledger → independent red acceptance tests → build → gates → report; the builder never edits the ledger, the spec, acceptance criteria, or the independently-authored acceptance tests (it makes them pass); retries ≤3 per gate, then escalate; a required tool/harness the plan depends on being UNAVAILABLE (engine Editor down, a needed MCP disconnected, headless-test license absent) is itself a mandatory STOP + owner-contact naming the tool, never a workaround/crutch, and any exception needs an explicit written owner-ack; a fix that ADDS or CHANGES behavior (a guard that can throw, a reordering) is itself a change — it re-runs the invariant-extension / spec-silence checks on its NEW failure paths and gets a test-author RED test for the touched invariant before the fix, and findings are classified by the invariant they violate so a recurring class trips the non-convergence stop, and a finding disposes `fixed` only with a recorded `sweep:` line — sibling sites closed|n/a, plus a seeded-miss/negative-control per class variant when the finding is itself a gate/scan/oracle false-green — while a later finding row re-using a prior `fixed` row's class token re-opens that row (a fix is a change, not a patch, and a patch does not close a finding); cross-module work goes through module public APIs (their AGENTS.md) — changing another module's internals is a separate feature leg; a decision whose OUTPUT the product's user directly experiences (sees/hears/feels/reads — how a mechanism looks or sounds, how output is paced or phrased) is an owner-signed acceptance property at PLAN, not a builder default (its formula/magnitude stays a builder ADR); done = gates green + evidence attached, **and the closing report `RESULT.md` (fields per Done-when) exists — producing it is part of the deliver-time gate (Step 4), so a missing/under-filled `RESULT.md` → not deliverable and a prose summary does not satisfy it; the builder routes the RESULT home (`next:` names the direction) and never authors the next task CALL; owner signals ("finish"/"summary") trigger the report, never replace it, and (contract v18) any `docs/reviews/review-*.md` / `docs/adr/*.md` path the RESULT CITES — rooted in the repo's OWN `docs/` tree — must EXIST at HEAD: a dangling citation (a report claiming a review/ADR not committed at the delivered tip) → not deliverable, for EVERY leg (this ALWAYS-RUN cited-existence floor is the light-lane counterpart of the frozen review-evidence dependency below, ORTHOGONAL to it — different key space — and a RESULT that OMITS the citation is untouched; VALIDATION §Cited-artifact existence)**. The same deliver-time gate also fails, for a G0-spec-frozen change, without a recorded G2 mutation score ≥ the `validation.config` floor, a non-empty spec-silence-audit section in the frozen spec, AND a recorded `docs/reviews/review-<id>.md` review-evidence file (id == the change folder; `reviewed-commit` ancestor-of-HEAD with every intervening source commit a `fixed <commit>` disposition; every in-scope finding dispositioned (class-siblings routed to backlog); every `refuted` marked; `rounds` ≤3 or a resolving `escalation:<id>`), AND a **negative-control pass** (a committed `<Prop>_NegativeControl` per acceptance property, GREEN only when the property rejects its deliberately-wrong impl) (Step 4 §Strong-check enablement), AND, for a `core algorithm` change, a **property-layer pass** (a `prop:<TestClass>.<TestMethod>` token per `headless` deliverable-coverage row, matched against an owner-signed property table covering permutation-metamorphic / fault-injection-all-exit-paths / multi-actor-one-tick / boundary-biased — Step 4 §Strong-check enablement clause (f)), AND a **refuted-register pass** for a review carrying any `refuted` finding (every `refuted` appends a row to the cumulative `docs/reviews/REFUTED.md`; a later review closes a re-derived duplicate by `register:<row-id>` citation; a re-raise WITH new evidence escalates to the owner, never silently re-litigated — Step 4 §Strong-check clause (g)) — running the test suite is not G2, a silence audit done only in chat does not count, a leg cannot deliver before an independent review round is recorded and dispositioned, and a suite that never proves its oracle can fail (no negative control) has not verified it. Thresholds and rubric live in `validation.config` and `REVIEW.md`.
- `REVIEW.md` — the validation rubric (see VALIDATION.md §rubric): severity definitions for THIS repo, always-check rules, skip rules, nit cap, file:line evidence bar.
- `docs/adr/` — architecture decision records; the agent proposes an ADR whenever it makes a decision a future session would otherwise re-litigate.
- `validation.config` — gate thresholds: retry budget, **mutation kill-rate floor (a required key — its absence means G2 is not enabled, VALIDATION.md §Enabled ≠ written)**, escalation rules (machine-readable; the run loop reads it), and **`synced_contract_version`** — stamp it with the current `os/engineering/CONTRACT_VERSION` `current:` value at setup. This records which engineering-contract hardening level the repo is built against; `collect`/`audit`/`digest` compare it to the live `current` and flag the repo when it falls behind (the forcing function against silent propagation lag — `os/engineering/CONTRACT_VERSION`).
- `docs/FRICTION.md` — the repo's friction log (same format as os/FRICTION.md). Change discipline mirrors os/MAINTENANCE.md: REVIEW.md, validation.config, and the stack profile change only on an explicit owner request or ≥2 entries on the same point. Factual corrections in AGENTS.md (wrong command, wrong path) stay immediate — facts are fixed on sight, rules by friction.
- Constitution section in AGENTS.md (3–7 immutable principles, e.g. "modular always", "no second source of truth", "reuse before write") — the validator checks changes against it.

## Step 4 — Toolchain (the feedback loop itself)

- Strict formatter + linter + typechecker, single command, fast; wired as save/pre-commit/CI AND as agent hooks (PostToolUse format/typecheck with failures fed back).
- Test layout: `tests/<module>/` mirroring `src/` (game engines: ONE registered test-scene root with a runner — never ad-hoc scenes). Hygiene gates from VALIDATION.md §hygiene wired into CI from day one.
- **Closing-report gate** — a deliver-time check (a `-Deliver` switch on, or sibling of, the one-command gate; NOT the always-run inner-loop gate — requiring `RESULT.md` there would block every mid-build gate run) that fails when `RESULT.md` is absent at repo root OR missing a required field (outcome / evidence / assumptions / cuts / cost / manual-acceptance / next). Same enforcement surface as the boundary/hygiene gates ("agents cannot ignore a failing check"); deliver/merge depends on this green check, not a prose precondition.
- **Strong-check enablement gate** — the same deliver-time check ALSO fails — for any change carrying a **non-archived `openspec/changes/<id>/` with a frozen spec** (the mechanical signal of a G0-frozen change; a light change has no such folder, so the gate is N/A by ABSENCE of the artifact, never by an agent declaring the change "light") — unless (a) the change's recorded **mutation score ≥ the `validation.config` floor** (proof G2 ran, not just the suite — the G2 run writes the score; the deliver check reads it, never re-runs the expensive pass), and (b) the change's frozen spec carries a **non-empty spec-silence audit** — the unconstrained regimes / actor-counts / seams / user-perceivable regimes list (CONTOUR §b), each marked intentional|unintended, AND a row-by-row walk of the cross-kind escape-class registry (`os/engineering/ESCAPE-CLASSES.md`) with every class marked addressed|n/a (an unmarked registry class fails deliver), and (c) the change's frozen spec carries a **deliverable-coverage list** (CONTOUR §c) that records the CALL done_when verbatim and gives EVERY done_when bullet a disposition — ≥1 ledger row OR an out-of-scope line — so a promised artifact cannot be silently omitted (a bullet with no disposition fails deliver); each row records the bullet's CLASS (`headless`|`engine`) and the gate keys on it, so a row whose class is `engine` disposed as built MUST carry a resolving EXISTS path under the artifact root (`Assets/**` / `docs/measurements/**`) that the check `Test-Path`s to a non-trivial file (a recorded MCP/batch-run artifact with a non-trivial measurement counts; an empty/stub file, a directory, a foreign pre-existing file, or — the leg-8 lie — a sibling headless test name in place of an artifact all FAIL), so a promised scene/prefab/composer cannot be reported done while absent (the engine-side existence dependency `VALIDATION.md` §Enabled ≠ written now requires, license-free substitute for the deferred PlayMode CI — `profiles/unity.md`); and an out-of-scope line on a promised bullet carries a recorded OWNER-ack token (plan-approval signature or escalation id) — a self-authored cut of a promise fails deliver; and (d) for the same G0-frozen change the repo holds a recorded **review-evidence** file `docs/reviews/review-<id>.md` (filename `<id>` == the non-archived change folder — absent, or naming another id, fails deliver) whose `reviewed-commit` is an ANCESTOR of the delivered HEAD with every intervening source commit accounted for by a `fixed <commit>` disposition (an un-reviewed source change since the review fails deliver; an honest review→fix→deliver passes, because the fix-commits ARE the dispositions — freshness is ancestry+accounting, never tree-equality), records ≥1 review pass by a model family other than the builder, tags every finding `scope: in-scope|class-sibling` and gives EVERY **in-scope** row a disposition (`fixed`|`refuted`|`owner-ack out-of-scope` — an in-scope row missing the disposition or valued `open` fails deliver) and EVERY **class-sibling** row a `routed <ref>` pointer (a class-sibling with no routing pointer fails deliver; a routed-not-fixed class-sibling does NOT gate — leg-gating is scoped to the diff, and a `class-sibling` row whose recorded `site` file is INSIDE the change's diff fails deliver (the mechanical anti-dodge — a git-diff name-membership test on the site, same surface as the reviewed-commit ancestry check and like it INERT until §Re-sync, the writer-G10 carrying `site`+`scope`+`routed` line-presence before then), VALIDATION §Review-evidence (iii)), carries a refutation-verification marker on every `refuted` row (in-session G5, binding fresh-session KERNEL-G5, or a `register:<row-id>` citation per clause (g) — a `refuted` with no marker fails deliver), and records `rounds:` ≤3 or a RESOLVING `escalation:<id>` (same resolution as the owner-ack token — a hollow escalation on >3 rounds fails deliver); the review-evidence dependency becomes a product deliver check only at §Re-sync (its `reviewed-commit` ancestry/accounting needs a working tree the writer lacks — the writer-side G10 bounce carries its line-presence before then); and (e) for the same G0-frozen change the `-Deliver` run includes a **negative-control pass**: for every acceptance-property token the suite holds a committed `<Prop>_NegativeControl` test (category `NegativeControl`) exercising the property's own assertion against a deliberately-wrong implementation, and the pass (`dotnet test --filter TestCategory=NegativeControl`, or the stack equivalent) requires that control DISCOVERED and PASSED — a property with no control (oracle unverified), or a control the property lets through (empty oracle: the property passed against its wrong impl), fails deliver, counted discovered+passed per token — closing what G2 mutation cannot (an absent property has no mutant; a wrong-but-consistent or weak assertion kills only the mutants of code that exists); the control is authored by the independent test-author (builder cannot edit it), and the dependency, like v9's registry-walk enforcement, is INERT until §Re-sync (the writer runs no tests, so there is no writer-side keystone); and (f) for a change whose PLAN-stage change-class (CONTOUR's cycle, the same taxonomy that routes it architectural) is `core algorithm`, every `headless`-class deliverable-coverage row (clause (c)) carries a `prop:<TestClass>.<TestMethod>` evidence token — the same mechanical form clause (c) already requires of `engine` rows as `exists:` — matched against an owner-signed **property table** in the frozen spec naming, per required class (permutation-metamorphic, fault-injection-all-exit-paths, multi-actor-one-tick, boundary-biased), the `[Category("Property")]` test (CsCheck; FsCheck 3.x fallback) that discharges it: a `core algorithm` change with a `headless` row carrying no `prop:` token, or one naming a test absent from the property table, fails deliver (VALIDATION §Property-layer). Like v9's registry-walk and v11's negative-control, this axis is INERT until §Re-sync wires the `prop:` token into `coverage-check.ps1` — PILOT it on the next `core algorithm` leg, not a retrofit of already-delivered core files; and (g) every `refuted` finding row in the change's `docs/reviews/review-<id>.md` resolves to a row in the repo's cumulative refuted-register `docs/reviews/REFUTED.md`: the review's `refuted-register:` line names the appended row ids (`none` is legal only alongside zero `refuted` rows), the check `Test-Path`s the register and greps each named row id, a `register:<row-id>` marker must resolve to an existing register row, and — clause (b)'s walk extended — a register row naming a module/seam the frozen spec touches is marked `still-holds|re-opened <why>`; a `refuted` row with no register row, a review with `refuted` rows and no `refuted-register:` line, a `register:` citation resolving nowhere, or an unmarked touching register row fails deliver; like v9/v11/v12 this axis is INERT until §Re-sync (the writer-side G10 bounce carries its line-presence before then — VALIDATION §Refuted-register); and (h) every `fixed <commit>` disposition row in the change's `docs/reviews/review-<id>.md` carries its **sweep line** (`sweep: class=<the row's invariant/class token>; sites: <each sibling site the class predicts, closed <commit> | n/a because <reason>>; controls: <a seeded-miss/negative-control token per class variant for a gate/scan/oracle false-green finding | n/a>` — VALIDATION §Fix-class-closure), and no later-round finding row carries an `invariant/class` token equal to an earlier `fixed` row's class without that earlier row marked re-opened: a `fixed` row with no sweep line, or an un-reopened recurring class token, fails deliver; like v9/v11/v12/v13 this axis is INERT until §Re-sync (the writer-side G10 class+sweep bounce — which keys on the RESULT's own fixed-finding content, not on a frozen declaration — carries it before then). Presence + threshold, same surface as the report gate: the gate asserts the score, the audit, the coverage list, the review-evidence file, the negative-control pass, the property-layer table/tokens, the refuted-register rows, and the fixed-row sweep lines EXIST and pass (every done_when bullet has a disposition, each `engine` row resolves a non-trivial artifact, each cut carries an owner-ack token), not that their content is true (it cannot tell a mis-classed `engine`→`headless` bullet, an under-transcribed done_when, or an honest-but-unrelated artifact — those stay the validator's / owner's / writer-G10 carry-back job) (truthfulness stays the validator's and the owner's job — same bound the report-field and converge-readiness gates accept). This is what stops a strong check that exists only as prose from passing green. **General rule: every load-bearing gate in VALIDATION.md / CONTOUR.md is wired here as an executable-or-recorded deliver dependency, never left as text** — adding a gate to those files without a deliver dependency + a Done-when seeded-miss leaves it unenabled by construction.
- **Re-sync (existing repos).** Setup runs once, so when `os/engineering` hardens or adds a gate, an already-set-up repo does NOT get it automatically — its deliver check stays stale. Re-sync is a `docs/FRICTION.md` / maintenance trigger IN the product repo: re-run the affected Done-when items below and wire the newly-required gate (this is the general form of the per-gate "retrofitting is a friction trigger" note), then **bump `validation.config: synced_contract_version` to the current `os/engineering/CONTRACT_VERSION`** so the repo stops being flagged as behind. Until re-synced, the repo cannot claim the hardened gate is enabled — and `collect`/`audit`/`digest` surface the gap (the version stamp is what makes the lag visible instead of silent). The os/engineering maintenance change that hardens a gate bumps `CONTRACT_VERSION` and names the re-sync as a follow-up; it does not itself edit product repos.
- `_scratch/` — the only home for exploratory scripts/scenes; tracked dir, self-ignoring `.gitignore`; CI asserts nothing from it is committed.
- CI running mechanical + test gates on every PR.
- **OpenSpec** initialized (`openspec/`): living specs + per-change proposal/tasks/delta-specs; the change-folder lifecycle (propose → apply → verify → archive) is the contract surface with the workflow OS. Archive is part of done — stale specs are the known failure mode. An unbuilt promised artifact still failing its coverage row must NOT be retired by archiving the change (the coverage scan skips `archive/`, so archive-with-open-row is an evasion the validator/owner catches at archive time — a pre-archive tripwire is a FRICTION candidate, not yet wired).
- Agent platform config: permission profile for autonomous runs (deny-list for hard nevers: force-push, secret paths, destructive ops; allow-list for project commands); evaluator subagent (read-only tools, instructed to report correctness gaps only) and cheap investigator subagent; verification procedures packaged as skills as they stabilize (side-effectful ones human-trigger-only); notification hooks (needs-input / finished) to the owner's push channel; stop-file and steer-file conventions.

## Step 5 — Harness state files

`SPEC` (per-change, from OpenSpec), feature **ledger** (machine-readable, all entries failing, not writable by the builder), `PROGRESS.md` (session handoff log), first commit.

## Done when (mechanically checkable)

- [ ] one-command check passes locally and in CI (format+lint+type+tests)
- [ ] dependency-boundary check exists and fails on a seeded violation
- [ ] root AGENTS.md ≤150 lines with working commands; ≥1 module AGENTS.md
- [ ] run contract section present in root AGENTS.md (clauses per step 3)
- [ ] run contract names the closing-report path (`RESULT.md`) + its field list (outcome/evidence/assumptions/cuts/cost/manual-acceptance/next), forbids the builder authoring the next task CALL, names the user-perceivable-behavior class as an owner-signed acceptance property, names the strong-check enablement deliver dependency (G2 mutation score ≥ floor + non-empty spec-silence-audit section + deliverable-coverage list that records the done_when verbatim and gives every bullet a disposition — a ledger row, or an owner-acked out-of-scope line — with engine-side EXISTS paths resolving to non-trivial artifacts, + a recorded review-evidence file `docs/reviews/review-<id>.md` (reviewed-commit ancestor-of-HEAD with intervening source commits accounted, every in-scope finding dispositioned (class-siblings routed), every refuted carrying a G5/KERNEL-G5 marker, rounds ≤3-or-escalated) + a negative-control pass (a committed `<Prop>_NegativeControl` per acceptance property that its test goes RED against) + a property-layer pass for a `core algorithm` change (a `prop:` token per `headless` row matched to an owner-signed property table — permutation-metamorphic / fault-injection-all-exit-paths / multi-actor-one-tick / boundary-biased) + a refuted-register pass (every `refuted` review finding appended as a row to the cumulative `docs/reviews/REFUTED.md`; a re-derived duplicate closes by `register:<row-id>` citation; a re-raise with NEW evidence escalates to the owner), for a G0-frozen change — BUILD-exists is gated, only the owner LOOK may defer, and a cut is never silent), names the fix-is-a-change clause (a behavior-adding fix re-runs the invariant / spec-silence checks on its new failure paths, with a test-author RED test before the fix, and a `fixed` finding carries its recorded class+sweep line — a recurring class token re-opens the earlier fix), and routes the RESULT home (`next:` names the direction) — repo produces the report with no OS access
- [ ] the deliver-time report gate FAILS on a seeded miss: a repo with no `RESULT.md`, and a `RESULT.md` missing a required field, both block deliver/merge (presence + structure)
- [ ] the always-run cited-artifact-existence gate FAILS on a seeded miss (contract v18): a `RESULT`/`docs/results/<id>.md` citing an absent `docs/reviews/review-*.md` (or `docs/adr/*.md`) path rooted in the repo's own `docs/` tree blocks deliver; the same citation once the file lands, and a `RESULT` that omits the citation, both deliver (existence-only — cross-repo/prose-only/empty-stub out of scope)
- [ ] `validation.config` declares the mutation kill-rate floor (required key)
- [ ] `validation.config` stamps `synced_contract_version` = the current `os/engineering/CONTRACT_VERSION` `current:` value
- [ ] the strong-check enablement gate FAILS on a seeded miss (seed a non-archived `openspec/changes/<id>/` with a frozen spec): a missing or below-floor recorded mutation score blocks deliver, a frozen spec with no spec-silence-audit section blocks deliver, and a frozen spec whose silence audit leaves an `ESCAPE-CLASSES.md` class unmarked (neither addressed nor n/a) blocks deliver; a repo with no such change-folder delivers (gate N/A by absence)
- [ ] the review-evidence gate FAILS on a seeded miss (same non-archived `openspec/changes/<id>/` frozen-spec trigger): a missing `docs/reviews/review-<id>.md`, or one whose filename id ≠ the change folder, blocks deliver; a `reviewed-commit` that is NOT an ancestor of the delivered HEAD, or an intervening source commit not accounted for by a `fixed <commit>` disposition (an un-reviewed change slipped in), blocks deliver; an in-scope finding row missing its disposition or valued `open` blocks deliver; a class-sibling row with no `routed <ref>` pointer blocks deliver (a routed-not-fixed class-sibling passes — leg-gating is scoped to the diff); a class-sibling row whose recorded `site` file is INSIDE the diff blocks deliver (the anti-dodge — an in-diff defect cannot be routed away); a `refuted` row with no G5/KERNEL-G5 refutation marker blocks deliver; `rounds` >3 with no resolving `escalation:<id>` blocks deliver; a review file naming this id, whose reviewed-commit is an ancestor with every intervening source commit a `fixed` disposition, all in-scope findings dispositioned, all class-siblings routed with an out-of-diff `site`, every `refuted` marked, and rounds ≤3, passes; a repo with no such change-folder delivers (gate N/A by absence)
- [ ] the refuted-register gate FAILS on a seeded miss (same non-archived `openspec/changes/<id>/` frozen-spec trigger): a `review-<id>.md` containing a `refuted` row but no `refuted-register:` line blocks deliver; a `refuted-register:` row id absent from `docs/reviews/REFUTED.md` blocks deliver; a `register:<row-id>` marker resolving to no register row blocks deliver; a frozen spec whose silence audit leaves a touching REFUTED row unmarked (neither `still-holds` nor `re-opened`) blocks deliver; a review whose every `refuted` row resolves to a register row (or an explicit `refuted-register: none` with zero refuted rows) passes; a repo with no such change-folder delivers (gate N/A by absence)
- [ ] the fix-class-closure gate FAILS on a seeded miss (same non-archived `openspec/changes/<id>/` frozen-spec trigger): a `review-<id>.md` containing a `fixed <commit>` finding row with NO `sweep:` line blocks deliver; a later-round finding row whose `invariant/class` token equals an earlier `fixed` row's class, with that earlier row not marked re-opened, blocks deliver (the recurrence is ONE class, not a fresh finding); a review whose every `fixed` row carries its sweep line (sites closed|n/a; controls per variant for a gate/scan/oracle-class finding) passes; a repo with no such change-folder delivers (gate N/A by absence)
- [ ] the negative-control gate FAILS on a seeded miss (same non-archived `openspec/changes/<id>/` frozen-spec trigger): (i) an acceptance property with NO committed `<Prop>_NegativeControl` blocks deliver (oracle unverified); (ii) a property whose test does NOT catch its negative control — the deliberately-wrong impl passes the property (empty/weak oracle) — blocks deliver; validated on the historical witnesses each of which MUST be caught: pure diffusion must fail the fork-split property, a constant-middle field must fail OR1, `params(16,0)` vs `(16,16)` must differ in output bytes; a suite where every acceptance property has a discovered control its own test goes RED against passes; a repo with no such change-folder delivers (gate N/A by absence)
- [ ] the property-layer gate FAILS on a seeded miss (same non-archived `openspec/changes/<id>/` frozen-spec trigger, and the PLAN change-class recorded as `core algorithm`): (i) a `headless` deliverable-coverage row with NO `prop:<TestClass>.<TestMethod>` token blocks deliver; (ii) a `prop:` token naming a test not marked `[Category("Property")]`, or absent from the frozen spec's property table, blocks deliver; (iii) a property table missing one of the four required classes (permutation-metamorphic, fault-injection-all-exit-paths, multi-actor-one-tick, boundary-biased) blocks deliver; a `core algorithm` change whose every `headless` row carries a `prop:` token resolving to a table entry in all four classes passes; a change whose PLAN change-class is not `core algorithm` delivers (gate N/A by absence)
- [ ] the deliverable-coverage check FAILS on a seeded miss: (i) a frozen spec with no deliverable-coverage section blocks deliver; (ii) a recorded done_when bullet with NO disposition blocks deliver; (iii) an `engine`-class row whose EXISTS path does not resolve — or resolves to an empty/stub file, a directory, or a file outside the artifact root — blocks deliver (a promised scene/prefab/composer cannot be reported done while absent); (iv) an `engine`-class row disposed as built with only a headless TEST NAME and no EXISTS path blocks deliver (the leg-8 lie: an engine deliverable cannot be discharged by a sibling test); (v) an out-of-scope line on a promised bullet with NO owner-ack token blocks deliver; a fully-disposed list with each `engine` row resolving a non-trivial artifact and each cut owner-acked passes
- [ ] REVIEW.md, validation.config, docs/adr/ADR-0001, docs/FRICTION.md, openspec/ exist
- [ ] stack profile created or updated, returned in the RESULT's state_changes
- [ ] `_scratch/` exists and a seeded file in it cannot be committed
- [ ] test-hygiene gates fail on seeded violations (test outside tests/, skipped test, assertion-free test)
- [ ] notification hook fires a test push
- [ ] evaluator subagent answers a smoke question read-only

The setup CALL's RESULT reports this checklist with evidence per item.
