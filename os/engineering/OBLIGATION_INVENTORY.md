# Engineering leg-obligation inventory

This file is the load-bearing census for the engineering contract. It counts what one applicable
product leg must do, not how many words describe it. The historical `N=71` in the 2026-07-14 audit
was a hypothesis; this is the fresh contract-v24 census made before the contract-v25 valve landed,
re-verified by v26, with X41 replaced in v28 and the listed carrier/version duties replaced in v29.

## Counting unit

One **leg obligation** is one normative action, stop, artifact, or independently removable
pass/fail predicate imposed by `os/engineering/**` on one product leg. Apply these tests manually:

1. **Removal test.** If one clause can be removed while a neighbouring clause still has a useful,
   enforceable meaning, they are two obligations and need two ids. A long sentence, table cell, or
   shared checker cannot hide them in one row.
2. **Schema boundary.** Fields and thresholds are attributes, not extra obligations, only when they
   belong to one named gate or evidence object and share its trigger, actor, phase, evidence path,
   and retirement. A clause with another actor, phase, artifact, gate, seeded-miss route, or
   retirement is a separate obligation even when compressed into the same line.
3. **Occurrence rule.** Count an obligation type once per leg, not once per generated acceptance row
   or review finding. The fixed escape-class registry is the exception: every row is a separate
   question and therefore a separate obligation.
4. **Deduplication rule.** Repeated wording and a cross-reference to the same predicate count once.
   Historical/superseded routes, rationale, examples, `SHOULD`, and one-time PROJECT_SETUP work do
   not count. Conditional duties count in `N` only when one legally completable leg can make them all
   applicable together.
5. **Profile rule.** Compute each profile separately. `N` is the largest current applicable total,
   never the sum of mutually exclusive stack profiles.

All authority pointers below are `path:line` in `ainazemtsau/my_global_workflow`. They are manually
read, not generated by a semantic parser or scanner.

## Verified ceiling

The maximizing current case is a frozen, compiled Unity product-code leg that is architectural,
touches a core algorithm and more than one concurrent actor, contains both headless and engine-side
deliverables, and exercises fixed, refuted, owner-ack and escalation branches. Its count is:

|---|---:|
| Gate predicates (`G`) | 49 |
| Cross-cutting axes (`X`) | 97 |
| Escape-class rows (`R`) | 16 |
| Required spec payloads (`S`) | 19 |
| Required artifacts/evidence objects (`E`) | 31 |
| Unity profile predicates (`U`) | 27 |
| **Maximum `N`** | **239** |

`N = 239` remains the contract-v29 ceiling. B4 removed no obligation id: compacting historical
narrative is a reading reduction, not a semantic deletion. The post-change maximum may not exceed
it. An addition that
would otherwise raise `N` is legal only when the same commit names and actually removes enough
inventory ids from every authoritative surface to keep `N <= 239`. Renaming, compressing, merging
rows, moving text to a product repo, or replacing two predicates with one sentence is not removal.
A one-for-one replacement names both ids and leaves `N` unchanged. A later batch may lower this
ceiling only after the corresponding obligations are proved removed.

## Change protocol for this inventory

- Every engineering-contract change re-reads its affected authority, updates every added, removed, or
  replaced obligation here in the same commit, and recomputes the affected profile totals. An unlisted
  duty still counts and makes the change invalid. A replacement's `CONTRACT_VERSION` entry names
  `added: <ids>` and `removed: <ids>`; the diff must prove removal from every authority.
- The AI reads the authoritative files and edits this inventory manually. Do not create or run a
  parser, regex scanner, schema, or conformance program to decide semantic obligation count.
- Any claimed witness for adding, keeping, replacing, or retiring an obligation must resolve as a
  concrete `path:line` or Git SHA in the repository explicitly named with it. Prose, a chat claim,
  an unresolved filename, or an approximate line count is not a witness.
- A temporary obligation declares `retire_on: <specific leg/CALL>` and that leg's `done_when`
  includes removal from every authority plus this inventory. That leg cannot close while retirement
  remains future cleanup. A `watching` line or a later voluntary maintenance session is not an owner.
- A defect in this budget/inventory mechanism discovered while applying it may be repaired in the
  same maintenance session; that repair is part of the same B3 problem, not an unrelated second job.
