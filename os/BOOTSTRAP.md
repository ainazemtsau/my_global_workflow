# BOOTSTRAP — запуск направления, по шагам

Точная инструкция первого запуска. Вариант A — ChatGPT/Claude Project, вариант B — всё в одной агентной сессии. Проблемы с самим воркфлоу — не здесь: `os/MAINTENANCE.md`.

Главное правило входа: **ты никогда не пишешь структурные CALL-пакеты руками.** Пакеты — машинный формат: их выдают сессии и writer, ты их только копируешь, если хочешь точный вход. Человеческий вход — обычные слова или «продолжаем».

## Вариант A: ChatGPT / Claude Project

**Шаг 0.** `os/**` лежит в `main`.

**Шаг 1. Проект.** Новый Project → имя направления.

**Шаг 2. Инструкции.** `os/adapters/SESSION_PAYLOAD.md` → скопируй блок в Project Instructions → замени `<direction-id>` (встречается несколько раз).

**Шаг 3. Доступ к репо.** Подключи GitHub-коннектор к `ainazemtsau/my_global_workflow`. Нет коннектора → загрузи как Project Files: `os/KERNEL.md`, `os/schema/packets.md`, `os/schema/direction-files.md`, все файлы `os/plays/`. Из `live/**` — никогда ничего.

**Шаг 4. Первые чаты — фундамент строится с тобой, не за тебя.** Просто напиши (или надиктуй) своими словами:

> Хочу создать направление: <твоя амбиция парой предложений>.

Дальше два чата, в каждом ты утверждаешь каждый артефакт (G9 — без твоего «да» в состояние ничего не попадает):
1. **frame** — вопросы батчем ≤7 → хартия (миссия, критерии успеха, пре-мортем) → твоё утверждение → корень дерева. Целей-узлов frame НЕ создаёт.
2. **map** — глобальный план. Frame сам выдаёт сначала research-сессию `map_evidence` (актуальный baseline с цифрами + как сопоставимые амбиции реально дошли до критериев, всё с датированными источниками; запускается в deep-research режиме — сессия выдаст готовый адаптированный промпт; пропуск — только твой явный отказ), её результат приходит в map. Дальше: твои кандидаты одной строкой → скелет всей карты одним экраном (строка на ветку, с пометками: твоё / закрывает линзу / неочевидный путь / из evidence; твои строки без автоприоритета) → твоя реакция на состав и порядок → узлы карточками ПО ОДНОМУ (цель, когда готово, почему ведёт к успеху, что убьёт) → по каждой твой вердикт принять/править/отклонить → итоговое дерево → рекомендация первого бета.

Контроль: каждая реплика начинается с `📍 <direction>/...`, в конце чата — блок `RESULT` с отметками `owner_approved`.

**Шаг 5. Применение (writer).** Открой **свежую** сессию Claude Code/Codex на репо воркфлоу → вставь RESULT → `Apply per os/adapters/coding-agent.md, role: writer` → она применит, закоммитит, вернёт локальное продолжение или свежий фронтир → **закрой её**. Writer всегда свежий и одноразовый — постоянных сессий в системе нет.

**Шаг 6. Цикл.** Новый чат в проекте. С коннектором **«продолжаем»** открывает единственный готовый CALL/вопрос; если их несколько, показывает короткий выбор по трекам и рекомендует один, ничего не записывая в state; если готовых нет, показывает уже запущенное, ожидания и блокировки. **«Что можно делать»** показывает все готовые CALL по трекам, а «продолжаем канон» матчится по человеческой метке. Id помнить не нужно. Без коннектора — вставь CALL из шага 5 или попроси writer: `collect next for <direction-id>` / `collect next for <direction-id>/<track>`: один actionable даст готовый блок, несколько — выбор. Дальше всегда: чат → RESULT → свежий writer → новый чат.

**Сбой / переход между платформами посреди работы.** Ничего не теряется, кроме недописанного разговора: `NOW.tracks` группирует работу, а `open_calls` хранит готовые, уже запущенные, ожидающие и заблокированные продолжения. Открой чат на любой платформе и назови трек обычным словом; без уточнения единственная готовая работа откроется, а несколько дадут выбор. Потерянный запуск назови явно: время само не возвращает его в очередь. Живую длинную сессию можно закрыть checkpoint-RESULT'ом — его локальный `RESULT.next` продолжит тот же трек где угодно.

**Управление треками.** Пиши обычными словами: «добавь трек звук», «поставь визуал на паузу», «закрой канон». Сессия сама выберет work/map/review и сохранит остальные треки; если owner-approved WIP-лимит заполнен, сначала предложит, что поставить на паузу/закрыть.

**Регулярно:** раз в неделю — pulse (свежая агентная сессия: `play: pulse, все направления`). Разъехалось состояние → новый чат: `почини состояние direction <id>` (repair).

## Вариант B: всё в агентной сессии

Открой свежую сессию Claude Code на этом репо и напиши то же самое человеческое сообщение с припиской «работай по os/KERNEL.md; ты сам себе writer — применяй state_changes и коммить». Один чат = одна работа действует и здесь: после RESULT сессию закрой, продолжение — в новой. Для state-тяжёлых plays (shape, review, pulse, repair) этот вариант дешевле по твоим действиям.

## Параллельные направления и треки

Несколько направлений одновременно — на каждое направление свой git worktree/проект; писатели разных направлений работают параллельно. Внутри одного направления (например, одна игра) `NOW.tracks` группирует CALL основной разработки, канона, визуала, маркетинга и любых будущих линий. Это диспетчеризация, не дополнительные беты: только `primary` владеет `NOW.bet/tasks`, а `parallel`-треки состоят из bounded CALL. RESULT'ы одного направления применяются по одному и семантически сливаются по стабильным id CALL/треков. Правила: `os/adapters/worktrees.md`.

Можно запустить сразу несколько показанных независимых CALL. После запуска скажи управляющей сессии обычными словами, например: «запустил Grid и Demo» — она пометит их `running` и больше не предложит повторно. Внешний ChatGPT-чат сам по себе невидим для Git, поэтому нужен этот один короткий сигнал; «чат Demo потерян/отменён» явно возвращает его в готовую очередь.

## Чего НЕ делать

- Не держать «живущие» сессии — ни рабочие, ни writer. Любая сессия одноразовая.
- Не создавать направления «про запас» (gate G8).
- Не редактировать `live/**` руками — только через RESULT/writer.
- Проблема или хотелка к самому воркфлоу → свежая Claude Code-сессия + `os/MAINTENANCE.md` (один запрос = одна сессия).
