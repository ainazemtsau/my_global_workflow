# Adapter: coding / working agents (Claude Code, Codex, Gemini CLI, ...)

Two distinct roles run on agentic tools. Any vendor works; switching vendors changes nothing outside this file.

## Role 1: writer (kind: mechanical)

Applies a RESULT's `state_changes` to `live/**` and commits.

Contract:
- Input: one complete RESULT packet. A bare `state_changes` section is incomplete because replay identity, G9/G10, owner-verdict/G5 and `next` validation require the full packet; bounce it before writes.
- Apply the RESULT's declared state-change intent atomically against fresh current state; append the log line once to LOG.md; save the full RESULT once to `history/<date>-<session-id>.md`; commit with a descriptive message (`<direction> <play> <node/task>: <log line>`).
- Maintain the `END_OF_FILE: <path>` trailer on every state file it writes (truncation guard, schema/direction-files.md).
- **Fresh-state semantic rebase is mandatory.** Re-read every target immediately before apply. Blob/SHA/commit ids, expected old text and exact anchors in `state_changes` are the base for a three-way rebase, not freshness locks. A mismatch by itself is information that concurrent state landed, never a reason to bounce. Derive only the RESULT's explicit delta, locate targets by stable path/id/key, apply that delta to the current files, and preserve every concurrent change outside those targets. `Preserve unchanged` means preserve the current value after rebase, not restore the packet's base. Compatible edits to a named target are merged; moved anchors are relocated by their stable identity.
- **Bounded integration judgment only.** The writer may resolve stale anchors and compatible parallel updates; it may not invent or alter outcomes, evidence, owner verdicts, gates, tasks, CALL meaning, or unrelated state. Compare base-to-RESULT and base-to-current at semantic-field level: shared path/id or whole-object inequality alone is not a collision.
- **Bounce threshold.** Bounce only when the RESULT or its `next` CALL is invalid/incomplete under the checks below, the intended delta is itself ambiguous, or a fresh-state rebase leaves mutually exclusive meanings for the same semantic field (for example, the returning call was consumed by a different RESULT, or a named target was deleted/retargeted so the declared outcome is no longer meaningful). Report that semantic conflict. Never bounce merely because commits, blobs, anchors, `updated:`, unrelated tasks/open calls, or global `NOW.next` changed meanwhile.
- **Whole-RESULT replay is idempotent.** Before normal apply, look up the committed history path for the incoming session-id. A byte-identical saved full RESULT is definitive proof that the whole atomic apply already completed, regardless of later state that superseded its postconditions: make no mutation and return the existing commit/outcome plus current valid routing (the incoming `next` if still pending, otherwise current `NOW.next`). Never restore old postconditions during replay. Different RESULT content under the same session-id is a logical collision. A history/LOG mismatch is state inconsistency and routes to repair; an interrupted uncommitted apply is discarded by the mandatory fresh-worktree reset and then applied normally.
- **Deterministic merge operations:**
  - `preserve unchanged` or an unnamed field: keep the fresh current value.
  - add/upsert a list or map entity by stable id: absent means add; existing means merge field-by-field. Preserve current-only changes, apply RESULT-only changes, and treat identical changes as a no-op; a field changed on both sides follows the scalar rule below.
  - remove by stable id: remove the fresh current entity. If absent, same-RESULT replay is a no-op; a different recorded consumer is a logical collision.
  - explicitly set a scalar: use true three-way comparison. If RESULT equals base, keep current; if current equals base, apply RESULT; if current equals RESULT, no-op; if both changed differently, bounce as a semantic collision. Two explicit exceptions are last-applier-wins: `updated:` always names the applying RESULT, and explicitly set `NOW.next` takes the applying RESULT's valid self-contained CALL/pointer. Unnamed or declared-unchanged `NOW.next` keeps current. Final merged state must still pass binding evidence, owner-verdict and hard-gate checks.
  - append LOG/history: key by session-id/history path and write once; identical existing content is a no-op, different content under that identity is a collision.
  - create a complete work/CALL file: absent means create, byte-identical means no-op, different existing content is a collision unless the RESULT explicitly declared a patch to that file.
- Reject CHARTER.md/TREE.md changes whose RESULT carries no `owner_approved` mark (gate G9) — reply with the missing approval instead of applying.
- Validate before applying (gate G10) — bounce with the specific miss, never apply partially:
  - RESULT fields complete per os/schema/packets.md, `play_check` included (one line per play step);
  - `(owner)`-marked steps in play_check carry the owner's actual words, not a bare "done";
  - **owner-verdict CALL guard (session RESULTs)** — when the returning CALL's `goal`/`done_when` asks for an owner-readable
    verdict (`accepted/revised/rejected/split`, approve/reject, choose, "можно записывать", or similar), a RESULT that
    clears the open_call or opens a downstream CALL must carry the owner's actual verdict words in `evidence` or
    `play_check`. A CALL packet, a recommendation, or the agent's phrase "accepted/revised" is not a verdict. Without
    owner words, bounce or checkpoint while leaving the same open_call pending.
  - the `next` CALL's goal is an outcome with no method/procedure paraphrase (CALL hygiene, schema/packets.md).
  - **CALL materialization (ENABLED NOW)** - engineering session returns `packet-stage: intent` with stable author fields only. During apply, writer may add only schema-enumerated observations: current inventory/HEAD/toolchain, plan/run/pre-RED/progress receipts and gate observation. Receipt preimage is intent bytes plus resolved observed identities/clean status, excluding itself and check exit/output; observation hash separately covers command/exit/before-after status. Writer persists the finalized self-contained CALL before payload. Guessed writer fields bounce. Collect never runs an intent or stale finalization; it renders STOP for a fresh writer apply.
  - **engineering transaction identity (v21, ENABLED NOW)** - every engineering CALL has normalized phase and absolute `worktree`; `repo` is logical only. PLAN/EXECUTE/BUILD/RE-SYNC also name exact branch/base and external `base-authority: sha:<sha>|ref:<ref>@<observed-sha>`; base equals that SHA and a ref resolves exactly. PLAN/EXECUTE/BUILD retain byte-identical stable parent `acceptance-source` and exact done_when `acceptance-hash`. Writer resolves both from Direction state before apply/collect; it never advances authority to HEAD or accepts a child-rewritten base/acceptance. SETUP alone is branch/base-free.
  - **SETUP preservation/bootstrap (ENABLED NOW)** - intent manifest is disjoint create vs overlay. Create rows pin external source mode/blob and fail on collision. Overlay rows pin exact pre mode/blob plus exact post blob or exact patch hash; writer, not executor, chooses them in a fresh amended intent after interview. Every unspecified pre-existing byte is protected. Finalization adds target/git/inventory receipt. Return verifies create/overlay postimages, protected inventory, no undeclared path and clean installed HEAD, then materializes the OS-owned V21-CONFORMANCE specimen/mutations with externally pinned corpus/specimen/adapter blobs. Writer applies/verifies exact deltas in randomized roots, passes no case/expected id, observes exits/status, and persists conformance receipt. Always-zero, case-switch, defect-absent, unpinned adapter/specimen, create collision, wrong overlay preimage/postimage or unrelated-line loss bounces.
  - **RE-SYNC boundary (ENABLED NOW)** - CALL requires initialized-behind, exact authority/branch, clean `HEAD == base`, and exact ordinal contract/toolchain/seeded-fixture/doc install manifest; no feature run or production/acceptance-test path. Writer preflights before payload. On return it requires clean installed HEAD, exact `base..HEAD` path equality to that manifest, no source/feature/test change, new stamp 21, independently derived installed trust-root hash and the same OS-side negative conformance receipt before accepting. RE-SYNC cannot redefine a feature base; the next feature starts with a fresh Direction-pinned PLAN authority.
  - **PLAN entry/final (ENABLED NOW)** - intent carries external id/base/acceptance and installed toolchain/conformance receipts; writer finalizes clean entry and runs plan-entry. Before any approval prompt, planner commits exact candidate, re-verifies trust roots/delta/acceptance/typed recipes and red realizations, runs pinned plan, then persists plan-candidate-head and a non-self-referential receipt over candidate/base/acceptance/toolchain/conformance/TESTABILITY plus clean before/after status. Owner verdict evidence must be chronologically later and cite that receipt. Edit, skipped plan, stale receipt or approval-before-receipt forces rerun. PLAN RESULT returns stable EXEC intent; writer verifies receipt/candidate/owner chronology before finalizing next CALL.
  - **fresh EXECUTE and authorized BUILD resume (ENABLED NOW)** - EXEC session supplies stable intent without run/pre-execution receipt. Writer verifies plan receipt/candidate and packet-only external-base delta, runs pre-execution, then adds/persists deterministic run-id, pre-red-head, receipt and observation hash before payload. First independent RED checkpoint is direct child of pre-red-head or an exact declared test-author-only chain; its receipt pins role/session, parent and exact test/oracle-registration-only delta. No source/frozen/toolchain/result/unmanifested change may enter R. BUILD resume requires same finalized run/plan/base/pins, protected RED revision and accepted progress; writer finalizes current resume observation only. Foreign diff, mixed source+test R, wrong parent/run/base, rewritten RED, unaccepted progress or pre-v21 checkpoint bounces.
  - **typed packet, complete recipes, red realization and oracle (NORMATIVE)** - every obligation passes all six construct|act|observe|negative|source|skeleton grammar records plus red-realization; no bare non-empty shortcut. Fully qualified signature/visibility/owner/args/fixtures/returns/observe/inject/delegate tokens resolve. Validator renders the skeleton against exact base and proves it compiles and uniquely discovers without production stub. A new absent symbol uses exact reflection/dynamic test-local signature route and fails runtime by criterion; an uncompilable direct call is PLAN RED. Test-author R pins regular rows, unique IDs, registration/filter/status and exact allowed delta; later revisions parent exact accepted progress and use the same path rule. Deliver/close require protected history and discovery/pass.
  - **result/archive/recovery (ENABLED NOW)** - base config declares result template; CALL pins canonical in-worktree `product-result-path`. At D deliver computes `D:<that path>` blob; product result never self-hashes/commits. Close verifies typed packet relocation, frozen external rows, base-only baseline rows, toolchain/oracle/result identities and one next archive commit A. The routing receipt may be reconstructed after interruption: rerun deliver at D, or at clean A derive `D=A^` and D:path/blob before structural close. Missing stdout is not loss of authority.
  - **v21 PLAN split (NORMATIVE over PLAN shorthand above)** - `PLAN/author` may return only a terminal PLAN-GATED checkpoint, never owner words. Writer reruns the pinned plan, recomputes `PLAN_OBSERVATION_HASH` and receipt, writes them through the checkpoint's declared Direction `history/` state change, commits, then creates the separate finalized `PLAN/verdict` open_call carrying `history/<path>@<writer-commit>`. That fresh session obtains owner words. Before EXEC finalization, writer requires the receipt-event commit to be an ancestor of the verdict CALL/RESULT and all candidate/receipt identities equal. A receipt/timestamp first appearing in the verdict RESULT, missing/nonzero/tampered observation, verdict in author stage or event mismatch bounces.
  - **v21 conformance bootstrap (NORMATIVE)** - install mode receives writer-random `CONFORMANCE_BOOTSTRAP_CHALLENGE`, accepted `ADAPTER_EVENT` and blob, not `VALIDATION_CONFORMANCE_RECEIPT`. Writer runs install cases, derives bootstrap receipt, then runs non-install cases with CALL receipt plus independently derived `CONFORMANCE_AUTHORITY_RECEIPT`, including missing/wrong/cross-phase mutations. It derives/persists final receipt/event and runs an excluded final binding smoke. Only final receipt/event enters real PLAN/EXECUTE/BUILD and stays byte-identical through close. This two-pass order is the only checker-environment exception.
  - **v21 RE-SYNC return (NORMATIVE over path-equality shorthand above)** - writer verifies every create row was absent and now equals source mode/blob, every overlay row matched pre mode/blob and now equals post mode/blob/patch hash, every unlisted pre-existing byte remains identical, and no undeclared path exists. Exact path-set equality without these content/postimage/protected-byte checks bounces.
  - Uninitialized SETUP and contract-only RE-SYNC are the only routes to install v21. PLAN requires entry+final passes; EXECUTE requires fresh pass; BUILD requires a v21 resume receipt. None substitutes for another phase.
  - **checker environment (NORMATIVE)** - set exact applicable `DIRECTION_PHASE|WORKTREE|BRANCH|BASE|BASE_AUTHORITY|ACCEPTANCE_SOURCE|ACCEPTANCE_HASH|CHANGE_ID|TESTABILITY|TESTABILITY_MODE|TESTABILITY_BLOB|INPUT_MANIFEST_HASH|VALIDATION_TOOLCHAIN_HASH|VALIDATION_CONFORMANCE_RECEIPT|PRODUCT_RESULT_PATH|PLAN_CANDIDATE_HEAD|PLAN_RECEIPT|RUN_ID|PRE_RED_HEAD|PREEXECUTION_RECEIPT|RED_AUTHOR_SESSION|RED_PARENT_HEAD|RED_CHECKPOINT|ORACLE_MANIFEST_HASH|PROGRESS_COMMIT|PROGRESS_LEDGER_BLOB|PREARCHIVE_COMMIT|RESULT_BLOB|ARCHIVE_TARGET|SETUP_INVENTORY_HASH|INSTALL_MANIFEST|OS_CONFORMANCE_BLOB|SPECIMEN_HASH|ADAPTER_BLOB` values from CALL/external receipts. A mode ignores only fields declared N/A for that phase; it never reads authority from prose or current HEAD.
  - **builder-return is not close (engineering RESULTs)** — a product-repo RESULT, owner playtest summary, merge/push request, or prose like "formally closed on dev/dev2" is evidence input, not authority to clear `NOW.open_calls` or mark done. The writer may clear the returning engineering CALL only when the Direction-OS RESULT/checkpoint carries the required close evidence: deliverable reconciliation plus the binding fresh-session G5/review (or stricter gate) named by the CALL/state. Product gates, screenshots, owner-eye, merge, and push may support that close; they do not replace it. A CALL phrase such as "no heavy multi-agent refutation" reduces fan-out only and is not a G5 waiver. If the close evidence is absent, bounce or write a checkpoint that leaves the `open_call` pending; never improvise a "merged therefore closed" state.
  - NOW hygiene (mechanical only): if state_changes write `NOW.md`, reject when `open_calls` keeps the returning CALL id or adds text explicitly marked `returned|done|superseded|cancelled`; when `decisions` adds text explicitly marked answered/accepted/declined; or when `next` is neither a CALL packet nor a one-line pointer to a self-contained CALL artifact under `work/`. Do not judge whether a compaction is semantically good — pulse/repair/review own that.
  - **deliverable reconciliation (engineering RESULTs only)** — diff the RESULT against the CALL's OWN `done_when` (the writer holds it in `open_calls` — the independent copy the product-repo deliver gate never sees, so this is where an UNDER-transcribed or merged promise is caught even if the product coverage list was sanitized to omit it). Every `done_when` bullet of the CALL must carry a disposition in the RESULT: a named evidence line, or an out-of-scope line with an owner-ack token (a plan-approval signature or escalation id). A bullet naming a built artifact (a scene/prefab/composer/adapter) needs an EXISTENCE evidence line citing a path or recorded run-artifact — not a test name (so this stays mechanical line-presence, never a judgment call on "related vs unrelated test"). A missing bullet, a build-bullet discharged only by "owner may look later" or by a headless test in place of an artifact, or a cut with no owner-ack = conflict → bounce with the specific bullet, not applied. Two FURTHER bounce cases, same mechanical line-presence check (never a judgment call): (a) **approval-claim** — any RESULT text asserting the owner approved a deviation, substitution, or cut (e.g. a bare `(owner-approved)`) that does NOT resolve to a real `owner-ack:<signature|escalation-id>` token elsewhere in the RESULT is the fabricated-approval class (same anti-fabrication principle as the `(owner)` play_check words check above) → bounce; (b) **approach-substitution** — when the CALL's `done_when` names the ONE approach/substrate the leg exists to prove (CONTOUR §Boundary) and the RESULT closes that bullet with a built artifact whose recorded `approach:` token differs from the named one, that is a self-authored substitution needing the same owner-ack token to stand → otherwise bounce. Return-path mirror of the dispatch-side contract-drift guard below: read the leg's promises, compare, surface loudly, never a silent pass.
  - **review-evidence (engineering RESULTs only, ENABLED NOW — reaches every repo before its §Re-sync)** — when the returning CALL's `done_when` declares a frozen-spec / strong-check change or names an `openspec/changes/<id>` change-id (the signal the writer already holds in `open_calls` — never re-derived from the product repo), the RESULT must carry a **review-evidence line**: a reference to `docs/reviews/review-<id>.md` with its finding dispositions and refutation markers present. Bounce when: no review-evidence reference is present; an **in-scope** finding is left `open` (or an in-scope row missing its disposition), or a **class-sibling** row missing its `routed <ref>` pointer or its `site` column (a routed-not-fixed class-sibling does NOT bounce the leg; the diff-membership anti-dodge — a class-sibling's `site` must be OUTSIDE the diff — is product-side, INERT until §Re-sync as with the reviewed-commit ancestry, so the writer carries only `site`+`scope`+`routed` line-presence, VALIDATION §Review-evidence (iii)); a `refuted`/false-positive disposition carries no refutation-verification marker (in-session G5, binding fresh-session KERNEL-G5, or a `register:<row-id>` citation — VALIDATION §Refuted-register); or the review-evidence reports ≥1 `refuted` disposition while carrying no `refuted-register:` line (the v13 register append — `none` is legal only alongside zero refuted rows); or the RESULT reports done while naming `rounds` >3 whose `escalation:<id>` does NOT resolve to a real escalation id elsewhere in the RESULT/history (same resolution as the `owner-ack` token above — a hollow escalation is the fabricated-approval class). A CALL whose `done_when` declares a light change (no frozen/openspec change) carries an explicit `review: n/a — light change` line instead; its ABSENCE on a frozen-declared change is itself the bounce. Mechanical line-presence against the CALL's own declaration — never a judgment on review quality or true independence (same bound as the deliverable-reconciliation and approval-claim checks above); the product `-Deliver` check adds the `reviewed-commit` ancestry/accounting freshness once the repo §Re-syncs, this is the writer-side teeth until then.
  - **fix-class-closure (engineering RESULTs only, ENABLED NOW — reaches every repo and every leg, frozen-declared or NOT)** — when the RESULT reports review findings fixed (any `fixed <commit>` disposition, or a review-repairs/"findings addressed" line), it must carry, PER fixed finding, its **class+sweep record**: the `invariant/class` token plus a `sweep:` line (sibling sites each closed|n/a — an out-of-diff sibling closes `routed <ref>`, a class-sibling per VALIDATION §Review-evidence, never an in-leg fix; for a gate/scan/oracle false-green finding, a seeded-miss/negative-control token per class variant) — inline in the RESULT or via the `docs/reviews/review-<id>.md` rows it references (VALIDATION §Fix-class-closure, contract v14). A fixed-finding report with no class+sweep record = bounce. This deliberately keys on the RESULT's own content, NOT on the CALL declaring a frozen change — the witnessed recurrence (dev_2 c-visual-004: `//` fixed, `/* */` + commented-include + raw-text found next round) ran on a leg with no openspec folder, where the review-evidence bullet above never fires. Mechanical line-presence; the writer never judges sweep quality or completeness (the reopen tripwire and the next review round own that).
  - task-play lifecycle: for `play: work|guide`, reject `TREE.md` edits, active_bet removal/done/retargeting, or activation of another node. If the RESULT marks the last open/active task of the current active bet done, `next` must be a `review` CALL for that same node. Review closes bets and harvests TREE; repair may reconstruct contradictory state under its own play.
  - converge Definition-of-Ready (shape activation): reject a `play: shape` RESULT that makes a node `active` (node->active, NOW gains the bet) unless that node is converge-ready — either the direction's `history/` holds a `converge-verify` RESULT for it (a clean set handed to shape), or the RESULT's `play_check` carries an explicit `triage: <type> — converge OFF — because <…>` line. Neither present → bounce with the miss (routes back to converge). This is the "shape's Definition-of-Ready" `os/plays/converge.md` names; it binds however the node reached shape (map/review/owner). Mechanical: file-presence + line-presence, no judgment — the owner-visible `because` reason is where a wrong "trivial" triage is caught, not here.

- **Owner-panel regeneration (direction-defined).** If the direction declares an owner panel — render-only view file(s) whose location and regeneration rules live in that direction's `knowledge/` (e.g. a lanes/venues map naming `work/board/dashboard.html`) — then any apply that changes `open_calls`, `tasks`, `decisions`, `next`, or other state the panel's live sections mirror ALSO regenerates those live sections per that knowledge entry, in the same commit. No declared panel → skip silently. A declared panel whose rules are unreadable this apply → apply the RESULT anyway and say the panel was NOT regenerated in the reply — never a silent skip.
- `collect` command: on "collect next for <direction>" (or a CALL id), emit ONE paste-ready block for chat platforms: the SESSION_PAYLOAD block, the play file, then NOW.md, then the CALL last (rules first, context next, CALL last). Always include the authoritative NOW.md as written; never compact or filter it in collect. If NOW hygiene drift is visible (closed-like `open_calls`, answered `decisions`, non-CALL/non-pointer `next`, or archive-scale hot state), prepend a loud warning and suggest a repair CALL; the payload still carries the real NOW.md. For SETUP, an absent target or incomplete run-contract/stamp pair is expected bootstrap state, not drift-unknown; a complete initialized pair bounces SETUP. Missing/unreachable worktree or unreadable stamp surfaces `WARNING contract: drift-unknown (<worktree> stamp unreadable)`, never a silent pass. Non-engineering CALLs skip this guard.
- `audit` command: on "audit <direction>" — read-only consistency sweep of `live/<id>/`, one boolean line per check: every state file ends with its END_OF_FILE trailer; nothing present beyond the six state types plus `work/` and local `plays/`; files conform to `os/schema/direction-files.md` (allowed statuses, `why` on every non-root tree node, NOW hygiene: pending-only `open_calls`/`decisions`, CALL-or-pointer `next`; hot-state hygiene across NOW.md/CHARTER.md/TREE.md/LOG.md — any of the four past the schema's soft line ceiling, any field not in that file's template (an inlined owner-quote block, a running narrative like a stray `current_truth`), or a LOG.md line longer than its one-line/≤2-sentence rule, is hygiene drift); cross-references resolve (NOW.md bet/task/open_calls targets exist in TREE.md/NOW.md, LOG.md lines point to existing history/ files); gates G1–G4 hold in the state as written. For a direction with a product `repo:`, also check contract drift: the repo's `validation.config: synced_contract_version` vs `os/engineering/CONTRACT_VERSION` `current:` — behind → name the gap (the missing log lines). The audit fixes nothing: state drift, NOW hygiene drift, or hot-state hygiene drift → a ready repair CALL; contract drift → a drafted §Re-sync trigger (PROJECT_SETUP §Re-sync); a rule defect → a draft MAINTENANCE REQUEST; clean → one line saying so. Flow health (stale calls, kill dates, cadences) stays pulse's job.
- `digest` command: on "digest" (optionally "digest <direction>" / "digest since <date>") — read-only morning report, default scope all `live/*` directions, default window since yesterday 00:00. Per direction with activity: its LOG.md lines from the window, current bet and task statuses from NOW.md, pending decisions count, the `next` CALL's goal in one line; a direction without activity gets a single line; for a direction whose product `repo:` is behind on `synced_contract_version` vs `os/engineering/CONTRACT_VERSION` `current:`, one `contract: vN < vM BEHIND — §Re-sync owed` line (current repos stay silent); os/** commits in the window close the report, one screen total. Render only: no proposals, no new ideas, no state changes — judgment stays with pulse (weekly) and outside_review (on demand). Built for a scheduled headless run (`claude -p "digest"`) — autonomy stage 2.

**The writer is ephemeral — never a standing chat.** Every RESULT is self-contained, so the writer needs no memory; a long-lived writer session is accumulated noise and cost (one chat = one job applies to the writer too). Manual loop: open a fresh agent session → paste the RESULT → it applies, commits, hands back the next CALL → close it. Headless one-shot invocations (`claude -p` / `codex exec`) do the same without a chat at all; an orchestrator replaces the hop entirely at autonomy stage 2+.

**Concurrency.** Parallel directions: one git worktree per direction, writers of different directions run in parallel; within one direction RESULTs apply one at a time. A queued RESULT may therefore have stale bases and is semantically rebased when its turn arrives. Sync/push protocol and rules: `os/adapters/worktrees.md`.

For `collect`, only a persisted `packet-stage: finalized` CALL is runnable. An intent, guessed writer field, stale inventory/HEAD/toolchain/plan/run receipt, changed candidate, or missing owner-after-plan chronology renders `STOP - writer finalization required`; collect does not invent or silently refresh it. Then run the phase guard: SETUP create/overlay receipt and external corpus blobs; RE-SYNC clean HEAD==base/exact manifest; PLAN entry identity; fresh EXECUTE exact plan receipt/candidate plus packet-only pre-execution; BUILD valid v21 run/pre-RED/oracle/progress resume. Any mismatch, nonconformant checker, nonzero exit or mutation emits no product payload.

## Role 2: executor (kind: engineering)

For `PLAN/verdict` collect, the finalized CALL must cite a real `history/<path>@<writer-commit>` receipt event already present in the current Direction history; that commit must precede the open_call commit and bind the exact candidate/observation/receipt. Collect never accepts a receipt or timestamp supplied only by the verdict session.

Receives a business task CALL whose `repo` is a logical product identity and whose absolute `worktree` is the sole filesystem/execution target. The full engineering process - plan gate, autonomous build, validation gates, retry/escalation, final report - is defined in `os/engineering/CONTOUR.md`; repo bootstrap in `os/engineering/PROJECT_SETUP.md`. The contract summary:

Contract:
- Execute only `packet-stage: finalized`. Recompute its non-self-referential receipt preimage and observed current identities before the first write; intent/placeholder/guessed/tampered/stale finalization stops. SETUP rechecks create collisions and overlay preimages. PLAN must produce plan receipt before owner prompt. Fresh EXECUTE starts at finalized pre-red-head. BUILD resumes only exact recorded run/oracle/progress.
- The CALL gives goal / context / boundaries / done_when / return / budget. The agent owns everything else: design, implementation, refactoring, branch/PR mechanics.
- Repo-internal conventions (build, test, style, architecture rules) live in that repo's AGENTS.md / CLAUDE.md — command-first, written once per repo. The OS never duplicates them.
- **Execution authority is repo-local.** The CALL + product-repo `AGENTS.md`, frozen plan/spec, and `validation.config` define the workflow and its required tools. A globally installed/discoverable skill or tool is support, not authority: it cannot replace the workflow or add project requirements. The required-tool STOP below applies only after one of those repo authorities explicitly requires the tool; discovery or installation alone is never a blocker.
- The run contract installed in the repo's root AGENTS.md at setup governs the run (roles, gates, retry/escalation). No complete run-contract plus validation-stamp pair means the target is uninitialized: only `phase: SETUP` may proceed, interactively per `os/engineering/PROJECT_SETUP.md`, preserving existing files and implementing no feature. SETUP against an initialized product, or PLAN/EXECUTE/BUILD/RE-SYNC against an uninitialized one, stops and reports the exact identity/scope mismatch.
- **At every boundary**, derive trust/typed rows independently before product code. PLAN renders/builds/discovers each no-implementation skeleton against external base; an absent new symbol must use exact base-compilable reflection/dynamic route without production stub. Before approval persist candidate/plan receipt, then obtain owner verdict. Fresh EXECUTE preserves pre-red-head. Test-author R has exact parent and test/oracle-only delta, unique discovery/filter and expected runtime RED; any production/mixed/unmanifested commit stops before builder. Later revisions follow the same exact-parent rule. Resume/deliver/close verify history and all receipts.
- Exit evidence carries finalized CALL preimage/observation hashes; PLAN candidate/receipt and later owner-verdict citation; run/pre-red/pre-execution identity; RED author/session/parent/delta/oracle revisions; accepted progress; exact result/archive evidence. SETUP/RE-SYNC carries external corpus/specimen/adapter blobs and writer-observed mutation rows. Product result never self-hashes. Writer recomputes all before accepting handback.
- "Done" is the CALL's done_when verified by a runnable check, not the agent's assertion. A task that can't carry a runnable check should say how it is to be verified instead — at CALL time, not after.
- Blockers surface early: if the task looks infeasible or 2x over budget, stop and report — a fast "blocked, because" is a good result; days of silent struggle are not. A required tool/harness being UNAVAILABLE (engine Editor not running, a needed MCP disconnected, headless-test license absent) is a blocker: STOP and tell the owner which tool to launch — never a workaround or crutch around it; any exception stands only on the owner's explicit written ack (CONTOUR ESCALATE).

- For a frozen/OpenSpec execution, `TESTABILITY.md` is part of the frozen packet; neither the RED author nor builder may edit it.

## Sizing rule

One executor CALL ≤ a focused half-day of human-equivalent work. Bigger means the task needed splitting at shape time — send it back as such.

## Running thinking sessions here too

Any OS play can run in an agentic CLI session instead of a chat app: the agent reads `os/KERNEL.md`, the play file, and `live/<dir>/NOW.md` itself, and can act as its own writer for state changes (one relay hop instead of two, or zero with an orchestrator). Recommended default for shape / review / pulse / repair, which are state-heavy; chat apps stay attractive for frame and long work dialogs.
