# Adapter: coding / working agents (Claude Code, Codex, Gemini CLI, ...)

Two distinct roles run on agentic tools. Any vendor works; switching vendors changes nothing outside this file.

## Role 1: writer (kind: mechanical)

Applies a RESULT's `state_changes` to `live/**` and commits.

Contract:
- Input: one complete RESULT packet. A bare `state_changes` section is incomplete because replay identity, G9/G10, owner-verdict/G5 and `next` validation require the full packet; bounce it before writes.
- Apply the RESULT's declared state-change intent atomically against fresh current state; append the log line once to LOG.md (track-mode prefixes track id after session-id); save the full RESULT once to `history/<date>-<session-id>.md`; commit with a descriptive message (`<direction>[/<track>] <play> <node/task>: <log line>`).
- Maintain the `END_OF_FILE: <path>` trailer on every state file it writes (truncation guard, schema/direction-files.md).
- **Fresh-state semantic rebase is mandatory.** Re-read every target immediately before apply. Blob/SHA/commit ids, expected old text and exact anchors in `state_changes` are the base for a three-way rebase, not freshness locks. A mismatch by itself is information that concurrent state landed, never a reason to bounce. Derive only the RESULT's explicit delta, locate targets by stable path/id/key, apply that delta to the current files, and preserve every concurrent change outside those targets. `Preserve unchanged` means preserve the current value after rebase, not restore the packet's base. Compatible edits to a named target are merged; moved anchors are relocated by their stable identity.
- **Bounded integration judgment only.** The writer may resolve stale anchors and compatible parallel updates; it may not invent or alter outcomes, evidence, owner verdicts, gates, tasks, CALL meaning, or unrelated state. Compare base-to-RESULT and base-to-current at semantic-field level: shared path/id or whole-object inequality alone is not a collision.
- **Bounce threshold.** Bounce only when the RESULT or its `next` CALL is invalid/incomplete under the checks below, the intended delta is itself ambiguous, or a fresh-state rebase leaves mutually exclusive meanings for the same semantic field (for example, the returning call was consumed by a different RESULT, or a named target was deleted/retargeted so the declared outcome is no longer meaningful). Report that semantic conflict. Never bounce merely because commits, blobs, anchors, `updated:`, unrelated tasks/calls/tracks, or the default `NOW.next` changed meanwhile.
- **Whole-RESULT replay is idempotent.** Before normal apply, look up the committed history path for the incoming session-id. A byte-identical saved full RESULT is definitive proof that the whole atomic apply already completed, regardless of later state that superseded its postconditions: make no mutation and return the existing commit/outcome plus its still-pending calls and current `NOW.next`. Never restore old postconditions during replay. Different RESULT content under the same session-id is a logical collision. A history/LOG mismatch is state inconsistency and routes to repair; an interrupted uncommitted apply is discarded by the mandatory fresh-worktree reset and then applied normally.
- **Deterministic merge operations:**
  - `preserve unchanged` or an unnamed field: keep the fresh current value.
  - add/upsert a list or map entity by stable id: absent means add; existing means merge field-by-field. Preserve current-only changes, apply RESULT-only changes, and treat identical changes as a no-op; a field changed on both sides follows the scalar rule below.
  - remove by stable id: remove the fresh current entity. If absent, same-RESULT replay is a no-op; a different recorded consumer is a logical collision.
  - explicitly set a scalar: use true three-way comparison. If RESULT equals base, keep current; if current equals base, apply RESULT; if current equals RESULT, no-op; if both changed differently, bounce as a semantic collision. `updated:` alone is always last-applier-wins. Legacy mode keeps the prior exception: an explicitly set `NOW.next` takes the applying RESULT's valid CALL/pointer. In track-mode, clear/upsert `open_calls` by stable call id: different ids merge even inside one track, while competing meanings for the same call follow the scalar rule. `NOW.next` is `call: <id>` or `awaiting_decision`; preserve it unless `state_changes` explicitly selects a valid default. A RESULT returning the selected default must name its valid successor/focus. Final merged state must still pass binding evidence, owner-verdict and hard-gate checks.
  - append LOG/history: key by session-id/history path and write once; identical existing content is a no-op, different content under that identity is a collision.
  - create a complete work/CALL file: absent means create, byte-identical means no-op, different existing content is a collision unless the RESULT explicitly declared a patch to that file.
- Reject CHARTER.md/TREE.md changes whose RESULT carries no `owner_approved` mark (gate G9) — reply with the missing approval instead of applying.
- Validate before applying (gate G10) — bounce with the specific miss, never apply partially:
  - RESULT fields complete per os/schema/packets.md, `play_check` included (one line per play step);
  - when current NOW uses track-mode or these state_changes introduce it, the RESULT and any newly issued CALL name an existing track or one atomically created here (a pre-migration return may inherit the one exact track on its `open_calls` row); an undeclared or ambiguous track bounces;
  - track-mode NOW hygiene overrides the legacy `next` shape below: a positive owner-approved `track_wip_limit` caps tracks with non-paused roots or pending decisions (waiting/blocked count; children do not); track ids/normalized current labels are unique and each `for` resolves to approved scope/done_when; exactly one track is `primary` and, while a bet exists, serves `bet.node`; every open call/decision has a unique id and existing track; each track has at most one parentless root call, every child resolves an existing same-track parent, appears in its `waiting_on`, and has acyclic ancestry to that root; every current track has a root or decision; calls carry `ready|waiting|blocked|paused` plus a self-contained artifact, `waiting` has non-empty `waiting_on`, and blocked has `unblock_when`; `next` is an honest pending decision, otherwise a ready call when one exists or existing non-ready focus when none does; new call/track ids were never used as those identities in current/history, and a new/retired/WIP-limit/primary-handoff change has the owner's cited words; a return clears only its registered call while preserving unrelated calls/tracks;
  - `(owner)`-marked steps in play_check carry the owner's actual words, not a bare "done";
  - track-mode status fields are mechanical: `waiting` has `waiting_on`, `blocked` has `unblock_when`, and `paused` has an owner-verdict `paused_by` pointer;
  - a tracked child return removes only its id from direct-parent `waiting_on`, appends its history receipt once, preserves sibling ids/receipts, and changes that parent to `ready` iff no waits remain; reverse return order must produce the same call graph;
  - **owner-verdict CALL guard (session RESULTs)** — when the returning CALL's `goal`/`done_when` asks for an owner-readable
    verdict (`accepted/revised/rejected/split`, approve/reject, choose, "можно записывать", or similar), a RESULT that
    closes the pending work or opens a downstream CALL must carry the owner's actual verdict words in `evidence` or
    `play_check`. A CALL packet, a recommendation, or the agent's phrase "accepted/revised" is not a verdict. Without
    owner words, bounce or checkpoint by replacing the returning id with a new continuation CALL for the same pending work.
  - **compiled carrier and RED handoff guard (contract v23; engineering PLAN/PLAN-AMEND/SURFACE-FREEZE/RED-FREEZE)** - for a compiled product-code leg, a PLAN or PLAN-AMEND RESULT may open only SURFACE-FREEZE, never RED/EXEC/BUILD; the retired v22 prose PLAN-AMEND carrier is not launch authority. SURFACE-FREEZE may open independent RED-FREEZE only when evidence records: exact freeze commit and parent; its Git manifest (not a prose folder list); the owner-approved decision-page path and word count <=400 with every named fixture fully defined or absent; skeleton production-source additions <=400 from `git diff --numstat <parent>..<freeze>` (excess = STOP and split, never a raised cap); real public signatures with field-preserving data/value construction and no acceptance meaning moved into `///`; repo-native build and hygiene GREEN; and all mandatory platform sidecars tracked in the same commit (Unity: each new `Assets/**/*.cs` has tracked `.cs.meta`; unavailable real importer = tool-unavailable STOP). RED-FREEZE changes tests/test-support only, pins the carrier, and may open only binding fresh RED refutation; its repo-native test command must compile, discover the intended tests and fail on behavior, never compile-RED. The fresh review may open BUILD only when it attempts to refute the exact carrier, fixture meaning, actual immutable RED commit/diff and runner evidence; the plan's complete `behavioral-red | evidence-only` inventory remains comparison evidence. BUILD pins both reviewed commits, forbids edits/re-authoring of frozen RED and the frozen public surface, and returns to SURFACE-FREEZE if the public carrier changes. A prose `N/N`, v22 recipe table, test count or named-blocker-only GREEN never satisfies this guard. For a non-compiled repo, require its declared native executable carrier or an explicit owner-approved `n/a`; never copy the C# mechanism. The writer checks recorded evidence shape, not semantic truth, and never substitutes a parser/regex/conformance tool.
  - the `next` CALL's goal is an outcome with no method/procedure paraphrase (CALL hygiene, schema/packets.md).
  - **builder-return is not close (engineering RESULTs)** — a product-repo RESULT, owner playtest summary, merge/push request, or prose like "formally closed on dev/dev2" is evidence input, not authority to clear `NOW.open_calls` or mark done. The writer may close the pending engineering work only when the Direction-OS RESULT/checkpoint carries the required close evidence: deliverable reconciliation plus the binding fresh-session G5/review (or stricter gate) named by the CALL/state. Product gates, screenshots, owner-eye, merge, and push may support that close; they do not replace it. A CALL phrase such as "no heavy multi-agent refutation" reduces fan-out only and is not a G5 waiver. If close evidence is absent, bounce or checkpoint by replacing the returning id with a continuation CALL for the same work; never improvise a "merged therefore closed" state.
  - NOW hygiene (mechanical only): if state_changes write `NOW.md`, reject when `open_calls` keeps the returning CALL id or adds text explicitly marked `returned|done|superseded|cancelled`, or when `decisions` adds answered/accepted/declined text. Legacy `next` must be a CALL/pointer or honest `awaiting_decision`; track-mode `next` follows the stricter selector check above. Do not judge whether a compaction is semantically good — pulse/repair/review own that.
  - **deliverable reconciliation (engineering RESULTs only)** — diff the RESULT against the CALL's OWN `done_when` (the writer holds it in `open_calls` — the independent copy the product-repo deliver gate never sees, so this is where an UNDER-transcribed or merged promise is caught even if the product coverage list was sanitized to omit it). Every `done_when` bullet of the CALL must carry a disposition in the RESULT: a named evidence line, or an out-of-scope line with an owner-ack token (a plan-approval signature or escalation id). A bullet naming a built artifact (a scene/prefab/composer/adapter) needs an EXISTENCE evidence line citing a path or recorded run-artifact — not a test name (so this stays mechanical line-presence, never a judgment call on "related vs unrelated test"). A missing bullet, a build-bullet discharged only by "owner may look later" or by a headless test in place of an artifact, or a cut with no owner-ack = conflict → bounce with the specific bullet, not applied. Two FURTHER bounce cases, same mechanical line-presence check (never a judgment call): (a) **approval-claim** — any RESULT text asserting the owner approved a deviation, substitution, or cut (e.g. a bare `(owner-approved)`) that does NOT resolve to a real `owner-ack:<signature|escalation-id>` token elsewhere in the RESULT is the fabricated-approval class (same anti-fabrication principle as the `(owner)` play_check words check above) → bounce; (b) **approach-substitution** — when the CALL's `done_when` names the ONE approach/substrate the leg exists to prove (CONTOUR §Boundary) and the RESULT closes that bullet with a built artifact whose recorded `approach:` token differs from the named one, that is a self-authored substitution needing the same owner-ack token to stand → otherwise bounce. Return-path mirror of the dispatch-side contract-drift guard below: read the leg's promises, compare, surface loudly, never a silent pass.
  - **review-evidence (engineering RESULTs only, ENABLED NOW — reaches every repo before its §Re-sync)** — when the returning CALL's `done_when` declares a frozen-spec / strong-check change or names an `openspec/changes/<id>` change-id (the signal the writer already holds in `open_calls` — never re-derived from the product repo), the RESULT must carry a **review-evidence line**: a reference to `docs/reviews/review-<id>.md` with its finding dispositions and refutation markers present. Bounce when: no review-evidence reference is present; an **in-scope** finding is left `open` (or an in-scope row missing its disposition), or a **class-sibling** row missing its `routed <ref>` pointer or its `site` column (a routed-not-fixed class-sibling does NOT bounce the leg; the diff-membership anti-dodge — a class-sibling's `site` must be OUTSIDE the diff — is product-side, INERT until §Re-sync as with the reviewed-commit ancestry, so the writer carries only `site`+`scope`+`routed` line-presence, VALIDATION §Review-evidence (iii)); a `refuted`/false-positive disposition carries no refutation-verification marker (in-session G5, binding fresh-session KERNEL-G5, or a `register:<row-id>` citation — VALIDATION §Refuted-register); or the review-evidence reports ≥1 `refuted` disposition while carrying no `refuted-register:` line (the v13 register append — `none` is legal only alongside zero refuted rows); or the RESULT reports done while naming `rounds` >3 whose `escalation:<id>` does NOT resolve to a real escalation id elsewhere in the RESULT/history (same resolution as the `owner-ack` token above — a hollow escalation is the fabricated-approval class). A CALL whose `done_when` declares a light change (no frozen/openspec change) carries an explicit `review: n/a — light change` line instead; its ABSENCE on a frozen-declared change is itself the bounce. Mechanical line-presence against the CALL's own declaration — never a judgment on review quality or true independence (same bound as the deliverable-reconciliation and approval-claim checks above); the product `-Deliver` check adds the `reviewed-commit` ancestry/accounting freshness once the repo §Re-syncs, this is the writer-side teeth until then.
  - **fix-class-closure (engineering RESULTs only, ENABLED NOW — reaches every repo and every leg, frozen-declared or NOT)** — when the RESULT reports review findings fixed (any `fixed <commit>` disposition, or a review-repairs/"findings addressed" line), it must carry, PER fixed finding, its **class+sweep record**: the `invariant/class` token plus a `sweep:` line (sibling sites each closed|n/a — an out-of-diff sibling closes `routed <ref>`, a class-sibling per VALIDATION §Review-evidence, never an in-leg fix; for a gate/scan/oracle false-green finding, a seeded-miss/negative-control token per class variant) — inline in the RESULT or via the `docs/reviews/review-<id>.md` rows it references (VALIDATION §Fix-class-closure, contract v14). A fixed-finding report with no class+sweep record = bounce. This deliberately keys on the RESULT's own content, NOT on the CALL declaring a frozen change — the witnessed recurrence (dev_2 c-visual-004: `//` fixed, `/* */` + commented-include + raw-text found next round) ran on a leg with no openspec folder, where the review-evidence bullet above never fires. Mechanical line-presence; the writer never judges sweep quality or completeness (the reopen tripwire and the next review round own that).
  - task-play lifecycle: for `play: work|guide`, reject `TREE.md` edits, active_bet removal/done/retargeting, or activation of another node. If the RESULT marks the last open/active task of the current active bet done, `next` must be a `review` CALL for that same node. Review closes bets and harvests TREE; repair may reconstruct contradictory state under its own play.
  - converge Definition-of-Ready (shape activation): reject a `play: shape` RESULT that makes a node `active` (node->active, NOW gains the bet) unless that node is converge-ready — either the direction's `history/` holds a `converge-verify` RESULT for it (a clean set handed to shape), or the RESULT's `play_check` carries an explicit `triage: <type> — converge OFF — because <…>` line. Neither present → bounce with the miss (routes back to converge). This is the "shape's Definition-of-Ready" `os/plays/converge.md` names; it binds however the node reached shape (map/review/owner). Mechanical: file-presence + line-presence, no judgment — the owner-visible `because` reason is where a wrong "trivial" triage is caught, not here.
  A bounced RESULT routes back to its session (or repair) — same path as an irreconcilable logical collision after rebase.
- Also emits the `next` CALL back to the owner/orchestrator if one is present.
- Record issued CALLs in `NOW.md → open_calls`; in track-mode require track/status/call and preserve every unrelated call id, including calls in the same track. Clear the returning entry when its RESULT arrives. If that id remains, default dangles, or default is non-ready while another call is ready, bounce as a mechanical contradiction.
- **Owner-panel regeneration (direction-defined).** If the direction declares an owner panel — render-only view file(s) whose location and regeneration rules live in that direction's `knowledge/` (e.g. a lanes/venues map naming `work/board/dashboard.html`) — then any apply that changes `tracks`, `open_calls`, `tasks`, `decisions`, `next`, or other mirrored state ALSO regenerates those live sections per that knowledge entry, in the same commit. The panel is a render over NOW+git: regenerating it never alters state meaning and needs no judgment; format and cleanliness rules (plain-language explanation per ID, closed items with their outcome, journal window, no stale sections) come from the knowledge entry, not from here. No declared panel → skip silently. A declared panel whose rules are unreadable this apply → apply the RESULT anyway and say the panel was NOT regenerated in the reply — never a silent skip.
- `collect` command: on "collect next for <direction>" (or a CALL id), emit ONE paste-ready block for chat platforms: the SESSION_PAYLOAD block, the play file, then NOW.md, then the CALL last (rules first, context next, CALL last). Always include the authoritative NOW.md as written; never compact or filter it in collect. If NOW hygiene drift is visible (closed-like `open_calls`, answered `decisions`, non-CALL/non-pointer `next`, or archive-scale hot state), prepend a loud warning and suggest a repair CALL; the payload still carries the real NOW.md. This replaces the owner assembling the sources by hand on platforms without a reliable connector. **Contract-drift guard (engineering CALLs):** when the CALL targets a product `repo:`, read that repo's `validation.config: synced_contract_version` and compare to `os/engineering/CONTRACT_VERSION` `current:`; if the repo is behind, PREPEND a loud `⚠️ RE-SYNC OWED — <repo> is on contract vN, current vM; this leg should §Re-sync first (missing: <the version rows N+1..M>)` banner. This is the forcing function with teeth — drift is surfaced at the exact moment work is dispatched into the repo. The CALL's `repo:` field IS the working-copy path to read; a CALL with no `repo:` (not an engineering leg) → skip silently, but a `repo:` whose working copy or `synced_contract_version` stamp is unreachable → surface `⚠️ contract: drift-unknown (<repo> stamp unreadable)`, never a silent pass (an unread stamp is the one no-op that would defeat the guard).
- `audit` command: on "audit <direction>" — read-only consistency sweep of `live/<id>/`, one boolean line per check: every state file ends with its END_OF_FILE trailer; nothing present beyond the six state types plus `work/` and local `plays/`; files conform to `os/schema/direction-files.md` (allowed statuses, `why` on every non-root tree node, NOW hygiene: pending-only `open_calls`/`decisions`, CALL-or-pointer `next`; hot-state hygiene across NOW.md/CHARTER.md/TREE.md/LOG.md — any of the four past the schema's soft line ceiling, any field not in that file's template (an inlined owner-quote block, a running narrative like a stray `current_truth`), or a LOG.md line longer than its one-line/≤2-sentence rule, is hygiene drift); cross-references resolve (NOW.md bet/task/open_calls targets exist in TREE.md/NOW.md, LOG.md lines point to existing history/ files); gates G1–G4 hold in the state as written. For a direction with a product `repo:`, also check contract drift: the repo's `validation.config: synced_contract_version` vs `os/engineering/CONTRACT_VERSION` `current:` — behind → name the gap (the missing version rows). The audit fixes nothing: state drift, NOW hygiene drift, or hot-state hygiene drift → a ready repair CALL; contract drift → a drafted §Re-sync trigger (PROJECT_SETUP §Re-sync); a rule defect → a draft MAINTENANCE REQUEST; clean → one line saying so. Flow health (stale calls, kill dates, cadences) stays pulse's job.
- `digest` command: on "digest" (optionally "digest <direction>" / "digest since <date>") — read-only morning report, default scope all `live/*` directions, default window since yesterday 00:00. Per direction with activity: its LOG.md lines from the window, current bet and task statuses from NOW.md, pending decisions count, the `next` CALL's goal in one line; a direction without activity gets a single line; for a direction whose product `repo:` is behind on `synced_contract_version` vs `os/engineering/CONTRACT_VERSION` `current:`, one `contract: vN < vM BEHIND — §Re-sync owed` line (current repos stay silent); os/** commits in the window close the report, one screen total. Render only: no proposals, no new ideas, no state changes — judgment stays with pulse (weekly) and outside_review (on demand). Built for a scheduled headless run (`claude -p "digest"`) — autonomy stage 2.

**The writer is ephemeral — never a standing chat.** Every RESULT is self-contained, so the writer needs no memory; a long-lived writer session is accumulated noise and cost (one chat = one job applies to the writer too). Manual loop: open a fresh agent session → paste the RESULT → it applies, commits, hands back the next CALL → close it. Headless one-shot invocations (`claude -p` / `codex exec`) do the same without a chat at all; an orchestrator replaces the hop entirely at autonomy stage 2+.

**Concurrency.** Parallel directions use one workflow worktree each; within one direction RESULTs apply one at a time and merge independent calls/tracks by stable id. A queued RESULT may have stale bases and is semantically rebased when its turn arrives. Sync/push rules: `os/adapters/worktrees.md`.

### Track-mode command overrides

These clauses replace only the legacy single-`next` assumptions in the command bullets above:

- `collect next for <direction>/<track>` emits that track's unique ready call; if several are ready it returns a compact choice/recommendation instead of guessing. `<direction>` alone resolves `NOW.next`, and a CALL id still resolves directly. A valid `call: <id>` default is not a hygiene warning; waiting/blocked/paused calls are reported, never launched as ready.
- In track-mode, `audit` replaces the legacy next check with: positive owner-approved WIP limit and occupancy within it; one primary track serving `bet.node` while a bet exists; every `parallel` TREE node in a current workstream is indexed; unique track/call/decision ids and normalized current labels; resolvable track scope/done_when; at most one root call per track; an acyclic same-track child graph with each child in parent `waiting_on` and resolvable receipts; required call/decision track/status/artifact and valid wait/block/pause conditions; every track has a root or decision; and default is an honest pending decision, else a ready call when available or existing non-ready focus. Missing track-mode in a direction with multiple current workstreams is state drift and yields a repair CALL.
- `digest` renders the default plus one compact `track: ready/waiting/blocked/paused counts — ready call goals` line per current track, instead of pretending the default is the only continuation. It remains render-only and one screen.

## Role 2: executor (kind: engineering)

Receives a business task CALL targeting a product repo (`repo:` field). The full engineering process — plan gate, autonomous build, validation gates, retry/escalation, final report — is defined in `os/engineering/CONTOUR.md`; repo bootstrap in `os/engineering/PROJECT_SETUP.md`. The contract summary:

Contract:
- The CALL gives goal / context / boundaries / done_when / return / budget. The agent owns everything else: design, implementation, refactoring, branch/PR mechanics.
- Repo-internal conventions (build, test, style, architecture rules) live in that repo's AGENTS.md / CLAUDE.md — command-first, written once per repo. The OS never duplicates them.
- **Execution authority is repo-local.** The CALL + product-repo `AGENTS.md`, frozen plan/spec, the v23 frozen
  carrier where applicable, and `validation.config` define the workflow and its required tools. A globally
  installed/discoverable skill or tool is support, not authority: it cannot replace the workflow or add project
  requirements. The required-tool STOP below applies only after one of those repo authorities explicitly requires the
  tool; discovery or installation alone is never a blocker.
- The run contract installed in the repo's root AGENTS.md at setup governs the run (roles, gates, retry/escalation). No run contract → the repo is uninitialized: stop and report — the first CALL for this repo is setup per `os/engineering/PROJECT_SETUP.md` (interactive: stack interview with the owner).
- Exit report (= the RESULT's evidence): descriptive commits or PR link, diff summary, output of the checks run (tests/build), explicit list of assumptions made, anything cut for budget.
- Every executor stage ends with the current CALL's `return` handback HOME, even when a downstream stage is only eligible or blocked. It never authors or asks the owner to find/create a successor CALL; a Direction session consumes the handback and the writer hands back its issued continuation.
- "Done" is the CALL's done_when verified by a runnable check, not the agent's assertion. A task that can't carry a runnable check should say how it is to be verified instead — at CALL time, not after.
- Blockers surface early: if the task looks infeasible or 2x over budget, stop and report — a fast "blocked, because" is a good result; days of silent struggle are not. A required tool/harness being UNAVAILABLE (engine Editor not running, a needed MCP disconnected, headless-test license absent) is a blocker: STOP and tell the owner which tool to launch — never a workaround or crutch around it; any exception stands only on the owner's explicit written ack (CONTOUR ESCALATE).

## Sizing rule

One executor CALL ≤ a focused half-day of human-equivalent work. Bigger means the task needed splitting at shape time — send it back as such.

## Running thinking sessions here too

Any OS play can run in an agentic CLI session instead of a chat app: the agent reads `os/KERNEL.md`, the play file, and `live/<dir>/NOW.md` itself, and can act as its own writer for state changes (one relay hop instead of two, or zero with an orchestrator). Recommended default for shape / review / pulse / repair, which are state-heavy; chat apps stay attractive for frame and long work dialogs.
