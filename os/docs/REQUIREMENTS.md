# Workflow OS — требования

Статус: извлечено из всего корпуса (формулировки владельца, аудит 2026-06-10, уроки поколений vNext-R / v2 / v3); основание для дизайна с нуля.
Прошлые реализации — только источник информации, не источник решений.

Формат: `R-N` — требование; `[слова владельца]` / `[урок поколения]` / `[аудит]` — происхождение.

---

## A. Назначение системы

- **R-1. Система ведёт направления жизни до результата.** Несколько автономных направлений; каждое — годы работы; система помогает достигать глобальных целей в каждом. [слова владельца]
- **R-2. Масштаб — «охрененно сложные» направления.** Стресс-тест: соло-разработка коммерчески успешной инди-игры (разработка + геймдизайн + маркетинг + бизнес). Если работает там — работает везде. Воркфлоу НЕ подстраивается под это направление. [слова владельца]
- **R-3. Система заменяет команду соло-человеку.** Носит компетенции, которых у владельца нет (маркетинг, бизнес); закрывает его слабости, главная из которых — разрастание скоупа («пытаюсь сделать круче, скоуп растёт, проект не доходит до конца»). [слова владельца]
- **R-4. Владелец — арбитр, не исполнитель и не транспорт.** По умолчанию подтверждает и выбирает из предложенного; может вклиниться всегда. Целевое состояние — автономность: уведомление только когда нужно решение. Сейчас — ручная пересылка, но концепт не должен зависеть от способа транспорта. [слова владельца]

## B. Чат (сессия)

- **R-5. Один чат — одна работа.** Чат = вызов функции: получил вход, сделал работу, вернул результат, закрылся. Контекст не разрастается; продолжение — новый чат. [слова владельца]
- **R-6. Закрытие чата выдаёт вход следующего чата.** Цикл замыкается без усилия владельца. [слова владельца]
- **R-7. Каждый чат привязан к ровно одной явно определённой процедуре (play).** Никакой магии: что чат делает, откуда берёт вход, что имеет право менять, как порождает дочерний чат — всё определено заранее. [слова владельца; аудит: в v1 свобода → расползание]
- **R-8. Ориентация за секунды.** Владелец ведёт десятки чатов; вернувшись в любой, он должен мгновенно понять: какое направление, какая работа, какой шаг, что от него нужно. Человекочитаемое «где мы» в каждом ответе, без сохранения. [слова владельца]
- **R-9. Дочерние чаты — рекурсивно.** Чат может породить дочерний для побочной работы; дочерние — своих дочерних. Возврат результата родителю определён. [слова владельца]

## C. Состояние и память

- **R-10. Всё состояние — в git, вне чатов.** Любому чату можно дать доступ/ссылку и продолжить работу. История переживает любые чаты и платформы. [слова владельца]
- **R-11. Слоистая память.** Горячее состояние (мало, всегда читается) → журнал (строка на чат + ссылка) → архив (полный результат каждого чата, можно вернуться) → принятые знания/решения. [слова владельца]
- **R-12. Платформо-независимость.** ChatGPT / Claude / Gemini взаимозаменяемы и для мышления, и для исполнения; привязка к платформе — только в отделяемых адаптерах. [слова владельца]
- **R-13. Десинхронизация состояния — ожидаемое событие.** Дёшево обнаружить и починить (recovery), а не «запрещено протоколом». [урок v3; аудит]

## D. Цели и планирование

- **R-14. От огромной цели до конкретной задачи — рекурсивно.** Цель делится на подцели, те — на свои, до задач; выполнение собирается обратно вверх по уровням. Глубина не фиксирована («элегантный способ формирования от огромной цели до её достижения»). [слова владельца]
- **R-15. Детализация — по мере приближения.** Дальнее — грубо, ближнее — конкретно. Запрет на детальную проработку далёкого будущего (в v2 планирование съело всё: 20 receipts, ноль исполнения). [урок v2; слова владельца «примерные планы на каждом уровне с возможностью редактировать»]
- **R-16. Возникающая работа вытекает из структуры, не теряется.** «Собрать отзывы у друзей» порождает трекаемые задачи: кого звать, как тестировать, на что смотреть; «снять ролик» может вырасти в подцель с собственной структурой. Захват — из любого чата. [слова владельца]
- **R-17. Жёсткий отбор следующего шага.** Выбор «что дальше» — по явным правилам (что прямее всего двигает цель уровнем выше; самое рискованное предположение — первым), с показанными альтернативами. [слова владельца «жёсткое грамотное планирование выбора»]
- **R-18. Встроенная сила отсечения лишнего.** Обязательный, механически проверяемый шаг урезания скоупа в каждом планировании («отсекать, пока не станет неловко»; Парето). Не совет — гейт. [слова владельца; R-3]
- **R-19. Кросс-функциональность вытекает из структуры.** Маркетинг/бизнес-работа появляется потому что критерии целей сформулированы во внешней реальности, а не потому что владелец вспомнил. Запрет на обособленные треки-силосы. [урок v1: «зависла на том, что это только разработка»]
- **R-20. Внутри направления — несколько затей параллельно или сменяя друг друга** (две игры, переключение между ними) — без хаоса, с явным ограничением одновременной работы. [слова владельца]

## E. Исполнение

- **R-21. Инженерная работа делегируется кодинг-агентам как бизнес-задача.** Цель + критерии приёмки + границы; агент сам владеет реализацией; возвращает результат с доказательствами (PR, тесты). Workflow не микроменеджит пути/SHA/ветки. Почти в каждом направлении будет большой поддерживаемый код — шов предусмотрен в ядре, детальные правила — отдельный модуль потом. [слова владельца; аудит v3: микроменеджмент = главный источник трения]
- **R-22. Запись состояния — машиной.** Чаты не пишут в git руками владельца; мутации применяет агент-писатель. [слова владельца; аудит: человек-как-шина — корневая причина трения]

## F. Сама система

- **R-23. Никакой магии, никаких непроверяемых правил.** Каждое правило либо механически проверяемо, либо его нет. Поля без владельца, циклический авторитет, юридическая проза — запрещены. [аудит v3]
- **R-24. Бюджет размера — закон.** Всё, что чат обязан соблюдать, помещается в малый объём (ядро + одна процедура + горячее состояние). Правило, не влезшее в бюджет, упрощается. [аудит: 230k слов ≠ исполнимо]
- **R-25. Модульность.** Ядро / процедуры / схемы хранения / адаптеры платформ / контракт исполнителя — независимо прорабатываемые модули со слабой связностью. [слова владельца]
- **R-26. Система — инфраструктура, не направление.** Обслуживание — обычные агент-сессии; изменение правил — только после повторившегося трения; мета-работа не должна стать аттрактором (в июне 40% коммитов были мета). [аудит; урок workflow-governance]
- **R-27. Дизайн обоснован исследованием.** Решения опираются на документированные практики работы с фронтирными моделями (GPT-5.x, Claude/Fable 5) и проверенные методики планирования, а не на привычки прошлых версий. [слова владельца]

## G. Антитребования (чего НЕ должно быть)

- **AR-1.** Команд-ритуалов (СТАРТ/ДАЛЬШЕ/ФИНИШ) и многораундовых церемоний для простых вещей. [аудит]
- **AR-2.** Дублирующих сущностей: один вопрос — одно место («что дальше», «что сделано», «принято ли»). [аудит: 6/5/4 сущности на вопрос]
- **AR-3.** Правил, требующих от модели того, чего она не может (верифицировать SHA без доступа, считать символы, гарантировать свежесть чтения). [аудит]
- **AR-4.** Нескольких одновременных «источников истины». [аудит: 4 неймспейса]
- **AR-5.** Ручного копирования состояния владельцем как части протокола. [слова владельца]

## H. Дополнение 2026-06-10 (после ревью владельцем): переносимость и инженерный контур

- **R-28. Настройка платформы — копированием из репо.** Всё, что нужно для подключения новой платформы (ChatGPT / Claude / Gemini / любая), хранится в одном месте в репозитории; владелец ничего не сочиняет. Несколько платформ параллельно для одного направления — норма (простое → дешёвая, сложное → сильная). [слова владельца]
- **R-29. Инженерный контур.** Отдельный слой определяет, как создаются и развиваются продуктовые репозитории: процедура сетапа проекта (через Claude Code/Codex), всегда модульная архитектура, у каждого модуля своя колокированная документация (AGENTS.md: внешний API + внутреннее устройство); документация продукта живёт в продуктовом репо, не в workflow. [слова владельца]
- **R-30. Автономный цикл сборки.** Бизнес-задача → планирование (вопросы владельцу допустимы) → владелец утверждает план → работа идёт полностью без владельца, прерывание только в критических случаях → финальный отчёт. Управление возвращается владельцу только когда всё сделано и проверено. [слова владельца]
- **R-31. Жесточайшая валидация.** Не только тесты: полное построчное ревью (расположение, отсутствие хардкода, best practices). Валидация — в отдельной сессии, не автором. Валидационные узлы после модулей/этапов. Провал → доработка; N провалов подряд (по умолчанию 5) → уведомление владельца. База валидации настраивается процедурой сетапа, кастомизируется от проекта к проекту. [слова владельца]
- **R-32. Переиспользование и анти-распухание.** Структура обязана подталкивать к переиспользованию существующего кода; «голый» агент даёт дорогой и/или плохой код — нужна настроенная база. [слова владельца]
- **R-33. Гигиена тестов.** Тестовые артефакты (сцены, скрипты) не плодятся бесконтрольно — у них есть владелец, место и жизненный цикл. [слова владельца: «тестовые сцены расплодились»]
- **R-34. Ручная приёмка.** Финальный отчёт включает инструкцию, как владельцу самому проверить результат. [слова владельца]
- **R-35. Выбор инструментов — по ресёрчу.** Инструменты, skills и plugins — adopt/adapt/skip по данным, не по привычке; установленное не становится обязательным автоматически. [слова владельца]

## I. Дополнение 2026-06-10 (внешнее design-stage ревью, принято владельцем)

- **R-36. Сессии, где исполнитель — владелец.** Инструмент без агентного доступа (Blender, редактор движка, железо, физический мир): сессия ведёт пошагово, по одному действию, проверяет по скриншотам, фиксирует результат с подтверждением владельца. [слова владельца через ревью]
- **R-37. Лёгкая кастомизация направления.** Направление может иметь собственные процедуры, политики и параметры, не трогая ядро; путь повышения локального в общее определён. Покрыть все стадии заранее невозможно — расширение должно быть дешёвым и слойным. [слова владельца через ревью]
- **R-38. Повторяющиеся обязательства.** Регулярная работа направления (devlog, тренировки) переживает беты и не теряется при их смерти; инстанцирование — регулярным сторожем, не вручную. [ревью, сквозной прогон]
- **R-39. Защита от молчаливой обрезки транспорта.** Чтение файла без его END_OF_FILE-маркера трактуется как обрезка; сборка канонического входа для слабых платформ механизирована (writer collect). Конкретизация R-23/AR-3. [ревью]

## J. Дополнение 2026-06-10 (UX-ревью владельца перед пилотом)

- **R-40. Человеческий вход — свободный.** Владелец начинает чат парой предложений, вопросом или словом «продолжаем» (в т.ч. диктовкой); сессия сама разрешает это в play по состоянию. Структурные пакеты — машинный формат: владелец их не сочиняет и полей не помнит. [слова владельца]
- **R-41. Никаких постоянных сессий.** Все сессии одноразовые, включая writer: «один чат = одна работа» распространяется на служебные роли. [слова владельца]
- **R-42. Восстановимость незавершённого.** Сбой чата или смена провайдера посреди работы не теряет картину: реестр запущенного (open_calls — что выдано, кто кого ждёт) живёт в состоянии; длинная работа может закрываться чекпойнтом с CALL-продолжением. История завершённого — LOG (строка+ссылка) + history/ (полный RESULT) — уже покрыта R-11. [слова владельца]

- **R-43. Параллельная работа над направлениями.** Несколько направлений одновременно (worktree на направление, отдельный проект в code assistant); внутри направления допустимы параллельные сессии и исполнители, но применение состояния одного направления сериализовано («много умов — одно перо»). [слова владельца]

## K. Дополнение 2026-06-12 (первый контакт пилота)

- **R-44. Глобальные планы — в со-творчестве.** Хартия и дерево целей создаются и меняются только с глубоким участием владельца: артефакты предъявляются по одному (узел = артефакт с обязательным `why`), каждый получает явный вердикт владельца; запись без отметки одобрения отклоняется писателем. Планирование сущности — отдельный чат (frame ≠ map). [слова владельца: «дерево сформировалось без меня»]
- **R-45. Хранить полно — загружать минимально.** Полные обоснования и история лежат в state (history/), в контекст сессии попадает компактная форма (одна строка why), детали достаются по ссылке по запросу. [слова владельца]

## Трассировка

Каждый компонент дизайна обязан ссылаться на требования, которые он закрывает; требование без компонента = белое пятно; компонент без требования = кандидат на удаление.
