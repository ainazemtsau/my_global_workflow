# Play: map

Purpose: build or deeply revise the direction's goal tree WITH the owner — the global planning ritual. One node at a time; nothing enters the tree without explicit owner approval (gate G9).

Reads: CHARTER.md, TREE.md, NOW.md; evidence sources the owner names (e.g., archive material).
Writes: TREE.md, NOW.md (next), LOG.md.

Trigger: right after frame; or the owner says "перепланируем карту"; or a review harvest demands restructuring bigger than small updates.

## Steps

1. **Recite** — restate the mission and success criteria in plain words, and the current tree state ("пусто" for a new direction).
2. **Candidate cards** — propose 3–6 top-level outcomes. Each is presented as a separate card:
   - goal (an outcome in the world, not an activity)
   - done_when (verifiable)
   - **why — exactly how this node leads to the root's success criteria**
   - risk — what would kill or invalidate it
   Draw on frame captures and owner-named evidence sources.
3. **Per-node verdict** — the owner accepts / edits / rejects each card. One card — one verdict; no silent batch acceptance. Rejected cards are recorded with the reason (parked or dropped).
4. **Order** — propose the sequence: what unblocks what, riskiest assumption first. Owner confirms.
5. **Depth check** — top level only (rolling wave, G2). Deeper splits happen later in shape, when a node becomes the focus.
6. **Lens sweep on the map** — does the tree cover every lens in the charter? For each gap: propose a node or an explicit `not_needed: reason`.
7. **Close** — show the final tree compactly; the owner approves the whole (G9). RESULT: full TREE.md (every non-root node carries its one-line why), log line, next = shape CALL on the recommended first node (2–3 options + recommendation).

## Done when

TREE.md contains an owner-approved tree where every non-root node has its one-line why; the next CALL is ready.

## Notes

- Rerunnable anytime: "перепланируем карту" is a normal owner request. It revises, never recreates — done/dropped nodes stay as one-line history.
- Detail beyond the one-line why lives automatically in this session's history file; a node may reference it (`detail: history/<file>.md`).
- If the owner's answers reveal the charter itself is wrong, stop and route to frame — map does not edit CHARTER.md.
