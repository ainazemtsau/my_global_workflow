## Phase fit

This Goal advances the active Phase by defining the kernel hypothesis before research, architecture, or implementation.

The strongest near-term framing is a persistent personal operating loop that observes work, compiles relevant context, remembers decisions/patterns, detects mismatch, and applies Reality Alignment to improve future decisions/actions.

## Working assumptions

- The full EXOCORTEX seed is directional input, not a roadmap.
- This is development of the current personal Workflow, not a separate commercial roadmap.
- The current Workflow is the first test environment.
- Future LLMs are replaceable/upgradable cognitive engines.
- The kernel is the persistent personal substrate around the models.
- Biology language is metaphorical and must not become architecture requirements.
- One distributed agent means one coherent assistant/operator across tools and surfaces, not a multi-agent build.

## Model vs kernel boundary

LLM/model layer:

- reasoning;
- language generation;
- synthesis;
- planning assistance;
- decision support;
- tool-use intelligence when routed through the workflow.

Kernel/substrate layer:

- persistent context;
- memory and decisions;
- tools and interfaces;
- source-of-truth rules;
- workflow stage contracts;
- mismatch/outcome capture;
- Reality Alignment loops;
- expansion seams for future rooms, dashboards, widgets, automations, and observation surfaces.

## Single Distributed Agent Model

The target is not many unrelated agents.

The target is one coherent helper/operator that can appear through multiple tools and surfaces while preserving one context, one memory structure, and one operating loop.

## Core loop candidate

1. Observe user work/input.
2. Compile relevant context.
3. Shape decision/action.
4. Capture outcome and mismatch.
5. Update memory, rule, handoff, or context loading behavior.
6. Reuse improved context in a later decision/action.

## Expansion seams

These are future seams only, not current implementation scope:

- tool integrations;
- observation/vision;
- working rooms;
- dashboards;
- widgets;
- automations;
- richer memory;
- stronger model adapters;
- cross-surface interaction.

## Validation requirements

The Kernel Min Proof Brief must pass these checks:

- It defines one smallest proof.
- It explains why future LLMs strengthen rather than replace the kernel.
- It clearly separates model capabilities from persistent kernel value.
- It includes one current-workflow test slice.
- It has explicit hard cuts.
- It can be reviewed without building anything.

## Stop conditions

Stop or reroute if:

- the work expands into app architecture;
- implementation is requested before E1/C1/C2 readiness;
- external evidence is required before defining the proof;
- the full EXOCORTEX seed becomes necessary but is not loaded;
- repository/source-of-truth conflict appears.
