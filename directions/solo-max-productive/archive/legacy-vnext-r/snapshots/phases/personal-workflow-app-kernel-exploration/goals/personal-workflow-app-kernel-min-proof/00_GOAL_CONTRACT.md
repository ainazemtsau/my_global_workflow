## WHAT

Create a compact Kernel Min Proof Brief for the personal Workflow/app/kernel direction.

The brief defines the smallest testable kernel hypothesis for an expandable external operating substrate around LLMs: context, memory, tools, interaction surfaces, feedback loops, working rules, and Reality Alignment mechanisms.

## WHY now

The app/EXOCORTEX idea is too broad to research, architect, or build safely until the kernel hypothesis is defined.

This Goal defines the foundation future LLMs should amplify rather than replace.

## Design philosophy

Future frontier LLMs are treated as replaceable/upgradable cognitive engines.

The kernel is the persistent personal substrate around them: context, memory, tools, surfaces, feedback loops, operating rules, and Reality Alignment mechanisms.

Stronger future models should increase the kernel's power, not make it obsolete.

## DONE

A Kernel Min Proof Brief exists and includes:

- Max Vision: one short orientation paragraph.
- Min Proof: one testable kernel hypothesis.
- Single Distributed Agent Model: one coherent assistant/operator distributed across tools and surfaces, not a swarm of unrelated agents.
- Model vs Kernel Boundary: what the LLM does versus what the persistent kernel provides.
- Core Loop: observe/input -> compile relevant context -> shape action/decision -> capture outcome/mismatch -> update memory/rules.
- Expansion Seams: future places for tools, vision/observation, working rooms, dashboards, widgets, automations, and richer interfaces.
- Hard Scope Cuts: no app build, no stack choice, no full architecture, no literal human-biology copy, no broad research sweep.
- Validation Signal: the user can confirm this is the smallest kernel worth testing, and E1 can proceed without loading the full EXOCORTEX seed by default.

## Acceptance floor

A fresh E1 chat can understand what kernel is being tested, why future LLMs should amplify rather than replace it, what is in scope, what is cut, and how the min proof will be validated.

## Validation signal

The user can say: "This is the smallest external-brain/workflow-kernel proof worth testing," and the next stage can turn it into an execution brief without inventing app architecture.

## Validation method

Review the Kernel Min Proof Brief against the Phase constraint.

It passes only if it:

- preserves future-model leverage;
- defines one observable test slice;
- prevents broad app/research/architecture drift;
- keeps biological language metaphorical rather than technical requirements.

## Smallest testable slice

Use the current ChatGPT + GitHub + Codex Workflow to test one minimal external-brain loop:

1. LLM receives relevant context and tools.
2. LLM helps shape one action or decision.
3. Workflow captures outcome/mismatch.
4. That mismatch improves a future rule, context bundle, or handoff.

No app, UI, database, dashboard, automation layer, or agent framework is required for the first proof.

## Scope in

- Kernel hypothesis.
- Max Vision versus Min Proof.
- Single Distributed Agent Model.
- Model-vs-kernel boundary.
- One current-workflow test slice.
- Reality Alignment / mismatch-capture loop.
- Expansion seams only.
- Research gaps only where they affect the min proof.

## Non-goals

- Build the app.
- Choose technical stack.
- Design full architecture.
- Design or implement an agent framework.
- Build dashboards, widgets, rooms, or automations.
- Create multi-agent system.
- Copy human biology literally.
- Turn this into commercial roadmap.
- Load full EXOCORTEX seed unless exact wording is needed.

## Scope cuts

- "AI companion" breadth is cut to one coherent distributed assistant/operator model.
- "External brain" breadth is cut to one testable context/tool/mismatch loop.
- "Dopamine system" language is cut to a metaphor for feedback learning.
- "Vision/tools/surfaces" are cut to future expansion seams.
- Research is cut to later evidence gaps only.

## Risk triggers

- Scope expands into full app/product roadmap.
- Kernel proof depends on unvalidated external evidence.
- Model-vs-kernel boundary becomes unclear.
- Metaphors become technical requirements.
- Repository/runtime source-of-truth conflicts appear.
