## 5. Lessons learned

- Foundation selection should close once the foundation decisions, specification, execution profile, and readiness gates are accepted; setup and implementation belong to later lifecycle work.
- Old project material is useful as evidence/reference, not as an automatic source of production code or requirements.
- Codex-driven product work needs explicit project/tool bindings, bounded slices, module/dependency rules, and validation surfaces before X0/X1 execution.
- Validation scenes/harnesses are not optional decoration; they are the way implementation results become visible and reviewable.

## 6. Constraints discovered

- Codex product/project execution remains blocked until concrete project/tool bindings, validators, scope, permissions, and execution route are verified.
- Unity bootstrap and product repository creation are not authorized by this Phase close.
- Game Documentation promotion requires a later explicit documentation route.
- H1_G4 durable technical nucleus remains premature until a later P0/G0/G1/E1 route scopes it.

## 7. Carryovers

- P0 must show why the next Phase is not a repeat of this foundation-selection Phase.
- The next Phase should be selected from the initialized Direction Map / H1 active horizon and must preserve the no-product-execution boundary until E1/X0/X1 readiness is proven.
- Candidate next-phase themes are only candidates, not selected by P9:

## 8. Do not repeat

- Do not reopen `Core Co-op Technical Foundation Selection` merely to start implementation.

- Do not run X0/X1 without a later registry-valid execution route.

- Do not make old project code the starting point.

- Do not create a broad research/planning loop when the next needed delta is an executable setup or implementation-readiness campaign.
- Do not create a phase that is only a renamed version of H1_G3 readiness.
