Verdict: `adopt_with_amendments`.

## 9. Technical Invariants To Adopt

Verdict: `adopt_with_amendments`.

Gas invariants:

- Gas has authoritative simulation truth, not visual-only state.
- Gas should update through fixed-step or explicit deterministic boundary.
- Gas state tracks typed mass/concentration across spatial containers.
- Source/sink behavior, propagation, accumulation, clearing, and topology-sensitive behavior are foundation requirements.
- Heavy/light/neutral vertical density behavior is useful as a validation band, not a final taxonomy.
- Reaction/transform hooks are core potential, while final reaction graph is deferred.
- Runtime dependencies, invalid topology, invalid mass, duplicate or missing IDs, and missing anchors should fail fast.

Grid/topology invariants:

- Grid/topology is the shared spatial substrate.
- Stable region/room IDs, connection/portal IDs, door/opening IDs, anchors, observation points, and topology revision IDs are required where gas/debug/future replication depend on identity.
- Connectivity/passability must be explicit.
- Effective topology must distinguish base layout from passage state, vent availability, and controlled mutation overlays.
- Dynamic topology mutation should follow command/validate/deterministic commit/version/change-set publication or equivalent.
- Spatial change propagation should prefer events, batches, dirty regions, versioning, snapshots, or justified hybrid models over naive per-cell spam.
- Grid must not become a god object for Gas, AI, interaction, multiplayer, Unity presentation, or gameplay state.

Interaction invariants:

- Gas reads effective topology after topology revision advances.
- Door/open/closed state changes must be applied before the gas tick that should read them.
- Grid owns room/door/topology authority; Gas consumes explicit read models/events/change sets.
- Cross-system interaction records should expose source, target owner, target ID, interaction type, payload summary, tick/revision, authority/scope, and debug visibility.

Validation/performance invariants:

- No-player validation is a hard first boundary.
- Debug surfaces should expose region inventory, connection inventory, passage state, gas mass/concentration, effective topology snapshot, topology revision, gas tick, update order, source/sink effects, and pass/fail explanations.
- Foundational hot paths require performance/scale evidence before execution-readiness acceptance; old performance evidence is reference-only.

Multiplayer boundary invariants:

- Foundational systems should be multiplayer-ready authoritative simulation cores without building full multiplayer first.
- Stable IDs, command/intent boundaries, authoritative tick or explicit step boundaries, deterministic/server-authoritative consistency, snapshot/delta/change-set output, validation, and transport-free domain core are required.
- No final networking stack is adopted now.
