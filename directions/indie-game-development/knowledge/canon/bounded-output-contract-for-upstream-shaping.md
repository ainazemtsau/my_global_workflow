# Bounded output contract for upstream shaping

Date:
2026-04-25

Source:
S7 Review — Выбрать одну primary product bet через evaluation frame и explicit experience model

Status:
Knowledge, not Canon

## Observation

For upstream shaping / narrowing Goals, a strict output contract prevents the work from expanding into downstream proof, design, tech, metrics or documentation cleanup.

In this Goal, the output contract was narrow:

- one honest comparison artifact;
- one selected primary product bet;
- one explicit experience model;
- one downstream proof question;
- explicit boundary that playable proof and full design were out of scope.

The Goal stayed inside that boundary. It did not create playable proof, prototype backlog, validation plan, metrics plan, tech-stream, concrete reaction recipes or full game design.

## Working rule

For shaping / narrowing Goals, define:

1. a small number of named output artifacts;
2. explicit downstream artifacts that must not be produced yet;
3. replan triggers for scope drift;
4. acceptance criteria that prove the narrowing happened.

## Why this is Knowledge, not Canon

This is a useful operating pattern, but it is already mostly covered by the workflow structure: Execution Pack, scope in/out, no-gos, rabbit holes, milestone map and replan triggers. It should become Canon only if future Goals repeatedly drift despite those mechanisms.

## Exception

Does not apply in this form when the Goal itself is a build/proof Goal. In that case proof design, prototype backlog or validation plan may be scope in.
